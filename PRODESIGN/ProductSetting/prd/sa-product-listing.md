# SA 运营后台 — 产品上架交互规范

> 文档版本：v2.0 | 最后更新：2026-02-23
> 关联文档：`product-attributes.md` / `productfee.md` / `productionopening.md`

---

## 概述

SA（EX 运营后台）代替服务商（SP）完成产品上架，原因：
- 本期体系内只有 BB、IPL 两个 SP，均有自己的运营后台
- 服务商自助上架能力本期不开放，由 EX 运营/产品代运营

**三个核心模块：**

| 模块 | 说明 |
|------|------|
| 服务商管理 | SP 基本信息维护 |
| 服务商产品管理 | 已上架的正式产品列表，可编辑/启停 |
| 产品上架记录 | 上架申请的草稿/审核流程 |

**核心流程：**
```
SA 发起上架申请（草稿）
  → 填写产品属性 → 提交审核
  → 审核通过 → 产品正式上架（进入"服务商产品管理"列表）
  → 审核不通过 → 退回草稿可修改后重新提交
```

## 产品目录与分类

### 产品三层分类

| 分类 | 产品 | 产品编码 | 核心定义 |
|------|------|----------|----------|
| **账户类**（开户 + 持有 + 同名存取） | 法币账户 | FIAT_ACCOUNT | 开户、持有法币余额、同名充值、同名提现 |
| | 数币钱包 | CRYPTO_WALLET | 开户、持有数币余额、同名充币、同名提币 |
| **收付类**（非同名收付） | Payins 收款 | PAYINS | 非同名 VA 代收、非同名链上代收 |
| | Payouts 付款 | PAYOUTS | 非同名法币付款、非同名链上付币 |
| **业务类**（编排 / 增值） | OnRamp 入金 | ON_RAMP | 法币 -> 数币承兑（编排层） |
| | OffRamp 出金 | OFF_RAMP | 数币 -> 法币承兑（编排层） |
| | VCC 发卡 | VCC | VCC 发行与管理 |
| | Crypto Checkout 收单 | CRYPTO_CHECKOUT | Crypto Checkout 在线收单 |

### 产品依赖关系

```
法币账户 ──────────────────────── 独立，无依赖
数币钱包 ──────────────────────── 独立，无依赖

Payins  ── 依赖 ──> 法币账户 或 数币钱包（资金入账到哪个账户）
Payouts ── 依赖 ──> 法币账户 或 数币钱包（资金从哪个账户扣款）

OnRamp  ── 依赖 ──> 法币账户（同名充值能力，资金入口）
        ── 依赖 ──> 数币钱包（承兑后数币入账）

OffRamp ── 依赖 ──> 数币钱包（承兑前数币扣款）
        ── 依赖 ──> 法币账户（同名提现能力，资金出口）

VCC     ── 依赖 ──> 法币账户（卡资金来源）

Crypto Checkout ── 依赖 ──> 数币钱包（收单资金入账）
```

### 为何这样分类

#### 核心问题：账户的"同名存取"和"非同名收付"应该放在一起还是分开？

**结论：分开。同名存取归账户产品，非同名收付归 Payins/Payouts 产品。**

#### 理由

**1. 同名存取是账户的天然属性，非同名收付是独立业务**

同名充值/提现本质上是"往自己的账户里存钱/取钱"，类似银行开户后自带的存取款能力。而非同名收付（VA 代收、付款给第三方）是一种独立的业务能力，需要额外的合规审核、收款人管理、风控规则。

**2. 避免"部分开通"的尴尬**

如果把同名和非同名都放在 Payins/Payouts 里，OnRamp 商户开通后只需要"同名充值"，却被迫开通了整个 Payins 产品（含非同名代收）。这导致：
- 商户获得了不需要的能力（非同名代收）
- 或者 Payins 产品内部需要复杂的子能力开关

分开后，OnRamp 只依赖法币账户（自带同名充值），不需要开通 Payins，边界清晰。

**3. 各场景开通组合干净**

| 业务场景 | 需要开通的产品 | 说明 |
|----------|---------------|------|
| 只做 OnRamp（法币换数币） | 法币账户 + 数币钱包 + OnRamp | 不需要 Payins/Payouts |
| 只做 OffRamp（数币换法币） | 法币账户 + 数币钱包 + OffRamp | 不需要 Payins/Payouts |
| 做非同名代收代付 | 法币账户 + Payins + Payouts | 不需要 OnRamp/OffRamp |
| 做汇款（Remittance） | 法币账户 + Payins + Payouts + Remittance | 编排层产品 |
| 做发卡（VCC） | 法币账户 + VCC | 不需要 Payins/Payouts |
| 做 Crypto Checkout | 数币钱包 + Crypto Checkout | 不需要法币账户 |
| 未来 SVF（纯持有） | 法币账户（关闭同名充值/提现开关） | 只持有余额 |

**4. 账户作为独立产品仍有意义**

即使账户只剩"持有 + 同名存取"，它仍然值得作为独立产品，因为：
- **开户是合规行为**：为商户在某个 SP 下创建账户需要 KYB、开户审核、账户协议，这和交易是不同维度的事情
- **跨 SP 账户选择**：TP 需要选择"法币账户用 IPL 的还是 BB 的"，这个选择需要产品级载体
- **不是所有商户都需要账户**：Crypto Checkout 场景下资金可直接结算到外部银行，不需要在 EX 开法币账户
- **账户能力可输出**：传统收单 SP 自己没有账户能力，BB/IPL 可以输出账户能力给它，这需要账户作为独立产品来管理

**5. 未来扩展性**

- 未来 SVF/MPI 场景：商户只需要"持有余额"，不需要任何收付款产品 -> 只开通法币账户即可
- 未来账户能力输出：BB/IPL 的账户产品可以被其他 SP 的交易产品引用
- 如果未来需要更细粒度拆分（如同名充值独立成产品），数据模型只需从"账户的子模块"变成"独立产品"，改动可控

## 二、服务商产品管理

### 2.1 列表页

展示所有**已审核通过、正式上架**的产品。

**列表字段：**

| 字段 | 说明 |
|------|------|
| 产品名称 | 如"数币钱包"、"法币账户"、"入金（OnRamp）" |
| 产品编号 | 系统生成，如 `PROD-BB-CRYPTO-001` |
| 服务商编号 | 关联 SP ID |
| 服务商名称 | 如 Bonbil |
| 产品状态 | 启用 / 停用 |
| 上架时间 | 审核通过时间 |
| 操作 | 编辑 / 启用 / 停用 |

**操作说明：**
- **编辑**：进入产品属性编辑页，修改后需重新提交审核（生成新的上架记录，原产品保持现状直到新记录审核通过）
- **停用**：二次确认后立即停用，已开通该产品的商户不受影响（存量订单正常处理），新商户无法开通
- **启用**：将已停用产品重新启用，无需审核

## 四、上架产品交互流程

### 4.1 入口

- 服务商产品管理列表页右上角 → **「上架产品」** 按钮
- 产品上架记录列表页右上角 → **「新建上架申请」** 按钮

### 4.2 Step 1 — 基本信息

**字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 选择服务商 | 下拉单选 | ✅ | 从已有 SP 列表选择，如 Bonbil / IPL |
| 选择产品分类 | 下拉单选 | ✅ | 账户类 / 收付类 / 业务类 |
| 选择产品类型 | 下拉单选 | ✅ | 根据分类联动，见下表 |

**产品类型选项（按分类）：**

| 分类 | 可选产品 |
|------|----------|
| 账户类 | 法币账户 / 数币钱包 |
| 收付类 | Payins 收款 / Payouts 付款 |
| 业务类 | OnRamp 入金 / OffRamp 出金 / VCC 发卡 / Crypto Checkout 收单 |

**联动规则：**
- 选择服务商后，产品类型下拉过滤：仅显示该 SP 尚未上架的产品类型（已上架的不可重复上架）
- 选择产品分类后，产品类型下拉仅显示该分类下的产品
- 选择产品类型后，下方展示对应的属性填写区（Step 2）
- 选择业务类产品时，系统自动校验依赖关系（见"产品依赖关系"章节），若依赖产品未上架则阻断

**底部操作：**
- 「保存草稿」：保存当前填写内容，状态为草稿
- 「下一步」：校验必填项后进入 Step 2

### 4.4 Step 3 — 确认提交

**确认页展示：**
- 汇总展示所有已填写的属性（只读）
- 分区块展示：基本信息 / 各能力配置 / 限额
- 底部操作：「返回修改」 / 「保存草稿」 / 「提交审核」

**提交审核后：**
- 状态变为「待审核」
- 上架记录列表新增一条记录
- 审核人（SA 内部）收到待审核通知

---

### 4.5 审核流程

**审核页面展示：**
- 所有属性只读展示
- 底部操作：「审核通过」 / 「审核不通过」
- 审核不通过时，必须填写不通过原因（文本输入，必填）

**审核通过后：**
- 上架记录状态 → 审核通过
- 系统自动生成产品编号（如 `PROD-BB-ONRAMP-001`）
- 产品进入「服务商产品管理」正式列表，状态为「启用」
- 若为编辑已有产品的新版本，新版本生效，旧版本归档

**审核不通过后：**
- 上架记录状态 → 审核不通过
- 申请人可查看不通过原因
- 可点击「重新编辑」进入草稿状态修改后再次提交

## 五、字段通用交互规范

### 5.1 多选字段
- 以 Tag/Chip 形式展示已选项，点击 × 可移除
- 支持搜索过滤（选项超过 5 个时）
- 至少选 1 个时，清空最后一项时提示"至少保留一个选项"

### 5.2 金额输入
- 仅允许正数，支持两位小数
- 输入框右侧显示币种单位（如 USD / USDT）
- 失焦时自动格式化为千分位（如 1,000,000）

### 5.3 开关字段
- 默认值见各字段说明
- 开关联动展开/收起相关配置区，收起时保留已填内容（不清空）

### 5.4 保存草稿
- 任意步骤均可保存草稿
- 草稿保存时不校验必填项，允许部分填写
- 草稿保存成功后提示"草稿已保存"，可继续编辑或关闭

### 5.5 表单校验
- 提交审核时校验所有必填项
- 错误提示显示在对应字段下方，红色文字
- 页面滚动至第一个错误字段

## 六、产品依赖关系提示

上架有依赖关系的产品时，系统在 Step 1 产品类型选择后自动校验依赖，并展示提示。

### 6.1 依赖关系汇总

| 产品 | 依赖的产品 | 附加条件 |
|------|-----------|----------|
| 法币账户 | 无 | — |
| 数币钱包 | 无 | — |
| Payins | 法币账户 或 数币钱包 | 至少已上架其中一个 |
| Payouts | 法币账户 或 数币钱包 | 至少已上架其中一个 |
| OnRamp | 法币账户 + 数币钱包 | 法币账户需已启用同名充值能力 |
| OffRamp | 法币账户 + 数币钱包 | 法币账户需已启用同名提现能力 |
| VCC | 法币账户 | — |
| Crypto Checkout | 数币钱包 | — |

### 6.2 提示规则

**依赖已满足时 — 信息提示：**

| 产品 | 提示文案 |
|------|----------|
| Payins | ⚠️ Payins 收款依赖账户产品（法币账户 或 数币钱包）。当前该 SP 已上架：[已上架的账户产品列表]。 |
| Payouts | ⚠️ Payouts 付款依赖账户产品（法币账户 或 数币钱包）。当前该 SP 已上架：[已上架的账户产品列表]。 |
| OnRamp | ⚠️ OnRamp 入金依赖「法币账户」（需启用同名充值）和「数币钱包」。 |
| OffRamp | ⚠️ OffRamp 出金依赖「数币钱包」和「法币账户」（需启用同名提现）。 |
| VCC | ⚠️ VCC 发卡依赖「法币账户」产品。 |
| Crypto Checkout | ⚠️ Crypto Checkout 收单依赖「数币钱包」产品。 |

**依赖未满足时 — 错误阻断：**
> ❌ 该 SP 尚未上架「[缺失的依赖产品]」，无法上架 [当前产品] 产品。请先完成依赖产品的上架审核。

**附加条件未满足时 — 错误阻断：**
> ❌ 该 SP 的「法币账户」未启用同名充值能力，无法上架 OnRamp 产品。请先编辑法币账户产品，开启同名充值。
