## 产品定位

**VCC（Virtual Card）** 是 EX 平台的虚拟卡发卡产品：

- SP 向商户提供虚拟卡发卡能力，商户可为其用户/业务场景开立虚拟卡
- 用于线上消费、广告投放、订阅支付等场景
- **本期所有配置均在 Bin 维度**，不设项目（Programme）维度
- **本期只做预付卡（Prepaid）**，授信卡（Credit / Deferred Credit）下期再做

**配置层级（本期2层）：**

```
第1层：SP 能力定义（SP 上架时按 Bin 维度填写，不可被 TP/商户修改）
  → 卡 Bin 基本信息 / 卡账户能力（含共享卡账户数量） / 交易能力（受理渠道、消费模式、授权、3DS） / 卡限制配置 Card Limits（数量+金额+次数） / 风控能力 / 持卡人要求

第2层：TP 产品配置（TP 签约后配置，可对商户做特殊配置；本期商户级由 TP 代配）
  → Bin 选择 / 转入币种 / 授权与 3DS / 限制收紧（卡片数量+转入转出+消费限额+交易次数+受理渠道）

下期：第3层 商户自助配置
```

## 1. SP 能力定义（Bin 维度）

> SP 上架时定义卡 Bin，一个 SP 可上架多个 Bin，每个 Bin 独立配置以下所有属性。
> **本期不设项目（Programme）维度，所有配置直接挂在 Bin 上。**

### 1.1 卡 Bin 基本信息

| 字段       | 字段名（EN）         | 类型     | 说明                                  | 枚举值 / 示例                          |
| ---------- | -------------------- | -------- | ------------------------------------- | -------------------------------------- |
| 卡 Bin     | bin                  | 文本     | 6～8 位卡 Bin 号码（当前主流为 8 位） | 如：53422618                           |
| Bin 描述   | bin description      | 多语言文本 | Bin 的描述信息，支持多语言，格式要求 2-3 行 | 见下方说明                             |
| 卡组织     | card-scheme          | 下拉     | 卡组织标识                            | Visa / Mastercard                      |
| 发卡地区   | region               | 下拉     | 发卡所在国家/地区                     | 美国 / 英国 / 香港 等                  |
| 卡账户币种 | card-account ccy     | 下拉     | 卡片账户币种                          | USD / EUR / GBP 等                     |
| 卡面 Logo  | logo                 | 图片上传 | 卡组织 Logo / 卡面设计                | —                                     |
| 资金模式   | funding_model        | 下拉     | 卡片资金所有权归属（见下方说明）      | Programme Managed / Cardholder Managed |
| 卡类别     | Card Programme Type  | 下拉     | 使用主体维度：企业卡 / 个人卡         | Corporate / Consumer                   |
| 卡有效期   | Card Validity Period | 文本     | 卡片有效期限（Bin 固有属性）          | 如：3 years                            |

> **Bin 描述格式要求：**
>
> - 支持多语言配置（中文、英文为必填，其他语言可选）
> - 每种语言的描述限制为 2-3 行
> - 每行建议不超过 50 个字符（中文）或 100 个字符（英文）
> - 用于在商户端、TP 端展示该 Bin 的用途和特点
> 
> **示例：**
> ```json
> {
>   "zh-CN": "适用于广告投放场景的企业虚拟卡\n支持 Google Ads、Facebook Ads 等主流广告平台\n单笔限额 50,000 USD，适合大额广告投放",
>   "en-US": "Corporate virtual card for advertising scenarios\nSupports major platforms like Google Ads, Facebook Ads\nTransaction limit: 50,000 USD per transaction"
> }
> ```

> **卡 Bin 位数说明：** 行业已从 6 位 Bin 过渡到 8 位 Bin（IIN），当前主流为 8 位。系统同时兼容 6 位和 8 位，SP 上架时按实际 Bin 填写。

> **资金模式说明（Visa/MC 发卡标准术语）：**
>
> | 维度     | Programme Managed (PM)                                 | Cardholder Managed (CM)                          |
> | -------- | ------------------------------------------------------ | ------------------------------------------------ |
> | 资金归属 | 资金归属**Programme（项目/企业）**，企业控制余额 | 资金归属**持卡人个人**，到账后企业无权扣回 |
> | 充值方   | 企业/TP 统一充值，持卡人不能自行充值                   | 持卡人自己充值（或企业代充，到账即归持卡人）     |
> | 余额控制 | 企业可随时充值/扣回/冻结                               | 持卡人完全控制，企业无权操作                     |
> | KYC 要求 | 持卡人 KYC 要求较低（企业已 KYB）                      | 持卡人需完成个人 KYC                             |
> | 典型场景 | 广告投放卡、差旅卡、供应商付款卡                       | 个人消费卡、工资卡、自由职业者收款卡             |
>
> **一句话区分：** PM = 钱是企业的，企业说了算；CM = 钱到卡上就是持卡人的，持卡人说了算。

### 1.2 卡账户能力

> 每个 Bin 的卡账户级别属性，定义卡产品类型、共享卡能力等。

| 字段               | 字段名（EN）             | 类型 | 说明                                                     | 枚举值 / 示例                      |
| ------------------ | ------------------------ | ---- | -------------------------------------------------------- | ---------------------------------- |
| 卡产品类型         | Card Product Type        | 下拉 | 资金模式维度的卡类型（**本期仅 Prepaid**）         | Prepaid / Credit / Deferred Credit |
| 是否支持共享卡     | is shared                | 下拉 | 是否支持多卡共享一个账户                                 | Shared / Not Shared                |
| 共享卡账户最大数量 | shared account max cards | 数字 | 单个共享卡账户下可挂载的最大卡片数量（开启共享卡时必填） | 如：10                             |
| 转入币种           | recharge ccy-default     | 下拉 | 支持转入/充值的币种                                      | USD / EUR / USDT / USDC 等         |

**联动规则：**

- is shared = Shared 时，共享卡账户最大数量必填
- is shared = Not Shared 时，共享卡账户最大数量不可配置

> **本期只考虑 Prepaid 能力**，Credit / Deferred Credit 下期再做。Prepaid 模式下 `credit amount` 表示卡片最大可充值余额上限。

### 1.3 交易能力

> 定义卡片的受理渠道、消费次数、授权模式、浮动规则和 3DS 安全验证。

| 字段             | 字段名（EN）                   | 类型     | 说明                                                           | 枚举值                                                                                        |
| ---------------- | ------------------------------ | -------- | -------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| 交易受理渠道     | transaction acceptance channel | 多选     | 支持的交易渠道                                                 | ATM / POS / 线上                                                                              |
| 卡消费次数       | card usage type                | 下拉     | 卡片消费模式（能力开关，具体次数上限见 1.4.4）                 | 单次（Single-use）/ 多次（Multi-use）                                                         |
| 授权模式         | authorization_mode             | 下拉     | 交易授权方式                                                   | Real-time（实时授权）/ Standard（事后同步）                                                   |
| 授权浮动         | auth-padding enable            | 下拉     | 是否支持授权金额浮动                                           | 允许 / 不允许                                                                                 |
| 授权浮动值       | auth-padding value             | 区间输入 | 浮动比例范围（开启授权浮动时可配置）                           | 如：5%、10%、15%                                                                              |
| 3DS 开关         | 3ds_enabled                    | 开关     | 是否启用 3D Secure（EMVCo 3DS2）                               | Enabled / Disabled                                                                            |
| 3DS 默认挑战策略 | 3ds_default_challenge_policy   | 下拉     | ACS 侧默认风控策略（发卡行最终决定 Frictionless 或 Challenge） | Prefer Frictionless / Prefer Challenge / Always Challenge                                     |
| 3DS 挑战方式     | 3ds_challenge_method           | 多选     | ACS 支持的验证手段（可启用一种或多种）                         | OTP（一次性验证码）/ Static Password（静态密码）/ Out-of-Band（APP推送/邮件确认链接）         |
| 3DS 通知递送     | 3ds_notification_delivery      | 下拉     | 验证请求的递送路径                                             | Direct to Cardholder（ACS 直接发给持卡人）/ Webhook to Merchant（回调商户，商户转发给持卡人） |

> **3DS 决策流程说明（EMVCo 3DS2）：**
>
> 在 VCC 发卡场景中，EX 的 SP 即发卡处理方（Issuer Processor），SP 配置的是 **ACS（Access Control Server）侧** 的行为：
>
> ```
> 收单机构 → 发起 3DS 请求（附带 challengePreference 偏好）
>     ↓
> 卡组织 Directory Server（Visa/MC）
>     ↓
> 发卡行 ACS（SP 配置的策略在这里生效）→ 最终决策者
>     ├→ 风险评估通过 → Frictionless（无感通过，持卡人无需操作）
>     └→ 风险评估不通过 → Challenge（要求持卡人验证）
> ```
>
> - **Prefer Frictionless**：ACS 默认倾向无感通过，仅高风险交易才触发 Challenge
> - **Prefer Challenge**：ACS 默认倾向挑战验证，仅低风险交易才可能 Frictionless
> - **Always Challenge**：所有交易强制 Challenge（最严格）
> - **注意：** 收单机构也会发送偏好参数，但 **发卡行 ACS 是最终决策者**，可以覆盖收单侧的偏好
>
> **挑战方式（Challenge Method）：**
>
> - **OTP**：ACS 向持卡人手机/邮箱发送一次性验证码
> - **Static Password**：持卡人输入预设的安全密码（如 Verified by Visa 密码）
> - **Out-of-Band（OOB）**：ACS 通过 APP 推送或邮件发送确认链接，持卡人在外部渠道确认
>
> **通知递送：**
>
> - **Direct to Cardholder**：ACS 直接向持卡人发送验证请求（短信/邮件/APP）
> - **Webhook to Merchant**：平台通过 Webhook 回调通知商户，商户负责将验证请求转发给持卡人

**联动规则：**

- 卡消费次数 = 单次时，1.4.4 交易次数上限默认 = 1，不可修改
- 卡消费次数 = 多次时，1.4.4 交易次数上限可填写具体数字或留空表示不限
- 授权浮动 = 允许时，授权浮动值必填
- 3DS 开关 = Enabled 时，3DS 默认挑战策略、3DS 挑战方式、3DS 通知递送必填
- 3DS 开关 = Disabled 时，其余 3DS 字段不可配置

### 1.5 风控能力

> 按 Bin 维度配置 MCC 和地区限制规则，以及支持的主体类型与国家组合。

| 字段           | 字段名（EN）         | 类型     | 说明                                                    | 示例                                                                     |
| -------------- | -------------------- | -------- | ------------------------------------------------------- | ------------------------------------------------------------------------ |
| MCC 控制模式   | MCC CONTROL MODE     | 单选     | MCC 限制模式：正向选择（白名单）或反向选择（黑名单）    | Whitelist / Blacklist                                                    |
| MCC 白名单     | MCC WHITELIST        | 多选     | 允许消费的商户类别码（正向选择模式时使用）              | 5411, 5812, 7311                                                         |
| MCC 黑名单     | MCC BLACKLIST        | 多选     | 禁止消费的商户类别码（反向选择模式时使用）              | 7995（赌博）, 6012（金融）                                               |
| 地区控制模式   | REGION CONTROL MODE  | 单选     | 地区限制模式：正向选择（支持）或反向选择（不支持）      | Accepted / Restricted                                                    |
| 支持业务地区   | ACCEPTED REGION      | 多选     | 允许交易的国家/地区（正向选择模式时使用）               | US, GB, HK, SG                                                           |
| 不支持业务地区 | RESTRICTED REGION    | 多选     | 禁止交易的国家/地区（反向选择模式时使用）               | IR, KP, CU                                                               |
| 客户主体限制   | ENTITY RESTRICTIONS  | 组合列表 | 定义支持的主体类型（个人/企业）与所在国家的组合关系列表 | 中国大陆-个人、中国大陆-企业、美国-企业、英国-企业、香港-个人、香港-企业 |

**规则说明：**

**MCC 控制：**
- **正向选择（Whitelist）**：配置 MCC 白名单，只允许白名单内的 MCC 交易，其他全部拒绝
- **反向选择（Blacklist）**：配置 MCC 黑名单，禁止黑名单内的 MCC 交易，其他全部允许
- 两种模式互斥，必须选择其中一种
- 如果不配置任何 MCC（控制模式为空），则表示不做 MCC 限制

**地区控制：**
- **正向选择（Accepted）**：配置支持业务地区，只允许这些地区的交易，其他全部拒绝
- **反向选择（Restricted）**：配置不支持业务地区，禁止这些地区的交易，其他全部允许
- 两种模式互斥，必须选择其中一种
- 如果不配置任何地区（控制模式为空），则表示不做地区限制

**客户主体限制：**
- 以列表形式配置，每项包含"国家/地区"和"主体类型"（Individual 个人 / Enterprise 企业）的组合
- 示例：中国大陆支持个人和企业，其他国家只支持企业
- 配置格式：`[{country: "CN", entityTypes: ["Individual", "Enterprise"]}, {country: "US", entityTypes: ["Enterprise"]}, {country: "GB", entityTypes: ["Enterprise"]}]`
- 为空时表示不限制主体类型和国家

**联动规则：**
- MCC 控制模式 = Whitelist 时，MCC 白名单必填，MCC 黑名单不可配置
- MCC 控制模式 = Blacklist 时，MCC 黑名单必填，MCC 白名单不可配置
- 地区控制模式 = Accepted 时，支持业务地区必填，不支持业务地区不可配置
- 地区控制模式 = Restricted 时，不支持业务地区必填，支持业务地区不可配置

### 1.6 持卡人要求

> 定义该 Bin 是否需要持卡人信息，以及持卡人的类型和地区限制。

| 字段           | 字段名（EN）      | 类型 | 说明                                       | 枚举值                  |
| -------------- | ----------------- | ---- | ------------------------------------------ | ----------------------- |
| 是否需要持卡人 | cardholder needed | 下拉 | 开卡时是否必须绑定持卡人信息               | Yes / No                |
| 持卡人类型     | cardholder 类型   | 下拉 | 持卡人身份类型（需要持卡人时可配置）       | Individual / Enterprise |
| 持卡人地区     | holder regions    | 多选 | 允许的持卡人所在地区（需要持卡人时可配置） | US, GB, HK, SG 等       |

**联动规则：**

- cardholder needed = Yes 时，持卡人类型和持卡人地区必填
- cardholder needed = No 时，持卡人类型和持卡人地区不可配置

## 2. TP 产品配置

> TP 在 SP 能力范围内收紧，不可超出 SP 上限。
> **本期仅支持：卡 Bin 选择、资金与充值、卡限制配置。**

### 2.1 卡 Bin 选择与描述

| 字段          | 字段名（EN）      | 类型       | 说明                                       | 约束             | 默认值     |
| ------------- | ----------------- | ---------- | ------------------------------------------ | ---------------- | ---------- |

## 3. 资金与余额模型

> 本章说明两种资金模式的区别，本期仅支持 Prepaid（预付）。

### 3.1 资金模式对比

| 维度     | Programme Managed（项目管理）   | Cardholder Managed（持卡人管理） |
| -------- | ------------------------------- | -------------------------------- |
| 资金归属 | 资金归属于 Programme（商户/TP） | 资金归属于持卡人个人             |
| 充值方式 | 商户从法币账户向卡片充值        | 持卡人自行充值（或商户代充）     |
| 余额控制 | 商户统一管理所有卡片余额        | 持卡人自行管理余额               |
| 适用场景 | 企业费用管理、广告投放、采购卡  | 个人消费卡、员工福利卡           |
| 退款流向 | 退款回到商户法币账户            | 退款回到持卡人卡片余额           |
| 本期支持 | ✅                              | ✅                               |

### 3.2 本期资金流向（Prepaid 模式）

```
Programme Managed：
  商户法币账户余额 → 卡片余额（商户充值）
  卡片退款 → 商户法币账户余额

Cardholder Managed：
  持卡人充值 → 卡片余额
  卡片退款 → 卡片余额
```

### 3.3 下期扩展（Credit 模式）

```
下期规划：
  ☐ Credit（信用卡）：先消费后还款，有账单周期和还款日
  ☐ Deferred Credit（延期信用卡）：类似信用卡但无循环信用
  ☐ 授信额度管理：额度审批、额度调整、临时额度
```

## 4. 产品开通联动规则

```
商户开通 VCC 产品时：
  ├── 依赖：法币账户（FIAT_ACCOUNT）— 用于卡片充值/余额转入
  ├── 可选：数币钱包（CRYPTO_WALLET）— 支持 USDT/USDC 转入卡片时需要
  └── 自动开通：卡片管理能力（开卡/销卡/冻结/解冻）

VCC 与其他产品的资金流向：
  充值到卡：法币账户余额 → VCC 卡片余额（Programme Managed）
  数币充值：USDT/USDC → 自动换汇 → VCC 卡片余额（需 OnRamp 能力，下期）
  退款到账：VCC 卡片退款 → 法币账户余额 / 卡片余额（取决于资金模式）
```

## 6. 本期 vs 下期

### 本期实现（Phase 1）

```
✅ 所有配置在 Bin 维度，不设项目（Programme）维度
✅ 卡 Bin 基本信息：8 位 Bin 为主（兼容 6 位）、卡组织、地区、币种、资金模式、卡类别
✅ 卡类型：仅 Prepaid（预付卡）
✅ 资金模式：Programme Managed / Cardholder Managed
