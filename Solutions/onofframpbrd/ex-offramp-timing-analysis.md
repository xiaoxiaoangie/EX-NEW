## 1. 推演目的

### 背景

之前的决策是直接拍板做 OffRamp。核心出发点有两个：

```
1. 集中精力：不想什么都做，先聚焦一个产品方向
2. 市场判断：行业数据显示 OffRamp 市场更大，所以选了 OffRamp
```

这个决策逻辑本身没有问题 —— **聚焦 + 选大市场**，方向是对的。

### 为什么要重新推演

但"方向对"不等于"现在就该这样做"。实际推进过程中，发现了一些需要面对的现实：

```
· 现有客户的卡点（OnRamp VA 入账 + OffRamp POBO）一直没解决
· 海外客户 = 零（签了 1 个但未入网，能不能跑通还不确定）
· BB 正在对接新渠道（智朗、MetaCom、SGB），这些渠道本身可能就能解决客户问题
· 如果做新的 OffRamp 多渠道产品，还要面对迁移、双系统、运营复杂度等问题
```

所以这份文档的目的是：

```
不是否定"做 OffRamp"的方向
而是基于实际情况推演：到底怎么做更合适？

具体来说：
  Q1：现在全力做 OffRamp 多渠道新产品，时机对不对？
  Q2：有没有更务实的路径，既能解决现有客户问题，又能为未来做准备？
```

### 推演逻辑

```
原来的决策路径：
  行业数据 → OffRamp 市场大 → 做 OffRamp        ← 外部视角，方向判断

本文档的推演路径：
  现有客户问题是什么？                            ← 内部视角，问题驱动
  + BB 渠道对接能解决多少？                       ← 资源盘点
  + 做新产品的代价和回报是什么？                  ← 成本收益
  + 海外客户能不能跑通？什么时候？                ← 时间线
  =
  到底先做什么，后做什么                          ← 优先级判断
```

## 2. 关键事实：国内 vs 海外的结构性差异

在展开推演之前，先摆出一个影响全局判断的关键事实：

```
EX 目前有两支销售团队：国内销售 + 海外销售

国内销售：
  · 已有客户 ✅（目前所有客户都来自国内销售）
  · 客户痛点 = OnRamp 阻碍多（不是 OffRamp）
  · OffRamp 痛点 小于 10%，且主要是费率问题，不是覆盖面

海外销售：
  · 客户数 = 0（签了 1 个但未入网，还不确定能否跑通）
  · 行业趋势 = 海外 OffRamp 确实需求大
  · 但目前连能不能跑通都是未知数
```

**这意味着一个结构性矛盾：**

```
OffRamp 多渠道主要服务的是 → 海外客户（OffRamp 需求大的市场）
但我们目前的客户全部是 → 国内客户（痛点不在 OffRamp 覆盖面）
海外客户数量 → 0

→ 做了 OffRamp 多渠道，短期内谁来用？
```

这个矛盾是本文档推演的核心。以下每个维度都会围绕这个矛盾展开。

## 4. 维度一：现有客户痛点

### 需要回答的问题

```
Q1：现有客户提出的需求/投诉中，OffRamp 相关的占多少比例？
  → 不到 10%

Q2：OffRamp 的具体问题是什么？
  a. 付不出去：
     · 大额到账显示 BB，收款行不接受
     · 走某渠道要的材料多，不符合客户画像
  b. 费率问题 ← 这是卓能-唯一一个 OffRamp体量可观的 客户痛点

Q3：如果不是 OffRamp，客户主要痛点是什么？
  → 国内客户主要痛点在 OnRamp 侧（阻碍多）
  → OffRamp 客户数量很少，唯一的客户目前遇到的问题是费率

⚠️ 关键洞察：
  · 现有客户 = 全部来自国内销售
  · 国内客户的 OffRamp 痛点 = 费率，不是覆盖面
  · 多渠道解决的是覆盖面和冗余，不直接解决费率
  · 除非：多渠道引入更低费率的 SP → 间接降费（但这依赖 SP 的实际费率）
```

### 推演逻辑

```
如果 OffRamp 痛点占大头：
  → 做 OffRamp 多渠道直接解决现有客户问题 → 时机对 ✅
  → 客户留存率提升、满意度提升 → 短期就有回报

如果 OffRamp 痛点不是主要的：
  → 做了 OffRamp 多渠道，现有客户感知不到改善
  → 短期无法体现价值
  → 需要依赖新客户来消化这个能力 → 见维度二（销售方向）
```

### 初步判断

```
→ OffRamp 痛点只占不到 10%，且主要是费率不是覆盖面
→ 多渠道不直接解决现有客户最痛的问题
→ 维度一判断：⚠️ 方向对但对现有客户短期价值有限
```

## 5. 维度二：销售团队与客群结构

### 已知事实

```
EX 有两支销售团队，处于完全不同的阶段：

┌─────────────────────────────────────────────────────────┐
│ 国内销售                                                │
│ · 状态：已有客户，已产生交易                            │
│ · 客群特征：国内跨境商户                                │
│ · 客户主要需求：OnRamp（充法币买U）                     │
│ · OffRamp 痛点：不到10%，主要是费率                      │
│ · OffRamp 多渠道对他们的价值：有限                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 海外销售                                                │
│ · 状态：还没跑通，0 个客户                              │
│ · 客群特征：海外企业/商户（潜在）                       │
│ · 行业趋势：海外 OffRamp 需求确实大                     │
│ · OffRamp 多渠道对他们的价值：高（如果能签下来的话）    │
│ · 当前阻碍：未知（是产品能力？合规？品牌？还是其他？）  │
└─────────────────────────────────────────────────────────┘
```

### 需要回答的问题

```
关于国内销售：
  Q1：国内客户的 OnRamp 阻碍具体是什么？解决了能带来多少增量？
  Q2：国内客户中有没有 OffRamp 大户？如果有，多渠道对他们有帮助吗？

关于海外销售：
  Q3：海外销售团队目前零入网客户，能否跑通还是未知数 — 阻碍到底是什么？
      a. 产品能力不够（OffRamp 覆盖面不够 → 多渠道能解决）
      b. 合规/牌照问题（多渠道解决不了）
      c. 品牌/信任问题（多渠道解决不了）
      d. 销售能力/市场策略问题（多渠道解决不了）
      e. 还在早期拓展阶段，需要时间
  Q4：如果做了 OffRamp 多渠道，能帮海外销售签下第一个客户吗？
  Q5：海外第一个客户的预计签约时间线？
```

### 推演逻辑

```
情况 A：海外跑不通是因为产品能力不够
  → 做了 OffRamp 多渠道能直接解锁海外市场 → 时机对 ✅
  → 但需要验证：真的是因为产品能力吗？还是其他原因？

情况 B：海外跑不通是因为其他原因（合规/品牌/销售策略）
  → 做了 OffRamp 多渠道，海外客户还是签不下来
  → 产品做了没客户用 → 时机偏早
  → 应该先解决海外销售的真正阻碍

情况 C：海外销售还在早期，需要时间
  → 产品可以先做，等海外签下客户时能力就绪 → "提前布局"
  → 但风险：如果海外 6-12 个月还签不下来，研发投入就冻结了
  → 需要评估：这个"提前量"是否值得？

⚠️ 关键问题：
  做 OffRamp 多渠道的增量交易额 100% 依赖海外客户
  但海外客户 = 0，且签约时间不确定
  → 这是一个高度不确定的投入
```

### 初步判断

```
→ 国内客户痛点不在 OffRamp 覆盖面 → 多渠道对国内短期价值低
→ 海外客户 = 0 → 多渠道做了短期没人用
→ 海外目前连能不能跑通都不确定，零客户入网
→ 除非海外阻碍是产品能力 → 那多渠道是解锁钥匙
→ 维度二判断：⚠️→❌ 海外能否跑通未知，做了多渠道也可能没人用
```

## 11. 维度八：机会成本与迁移复杂度

### 核心论点

```
我们现在有一批客户天天在用、天天有问题 → 但我们要去做一个新产品
这个新产品的客户画像目前是未知的，或者只存在于“别人的嘴里”

这是一个经典的资源配置矛盾：
  现有客户有真实的、每天发生的痛点 → 但没人解决
  新产品服务的客户还不存在 → 但要投入研发资源
```

### 三层问题分析

```
第一层：机会成本 — 放着现有客户问题不解决

  现状：
  · 国内客户 OnRamp 阻碍多 → 影响每天的交易量
  · 国内客户 OffRamp 费率高 → 影响客户满意度和留存
  · 这些问题每天都在发生，影响的是真实的收入和客户关系

  如果研发资源去做 OffRamp 多渠道：
  · OnRamp 阻碍谁解决？→ 没人解决 → 现有客户继续受苦
  · OffRamp 费率谁优化？→ 没人优化 → 现有客户继续抱怨
  · 结果：新产品还没做完，老客户可能已经流失

  → 这是典型的“捡了花生丢了西瓜”

第二层：新产品的隐性成本 — 不只是研发

  做 OffRamp 多渠道不只是写代码，还有：
  · 研发成本：系统设计 + 开发 + 测试 + SP 对接
  · 客户沟通成本：跟客户解释新流程、新接口、新的操作方式
  · 内部运营成本：新的运营流程、新的对账流程、新的异常处理流程
  · 多系统维护成本：老系统不能丢，新系统要维护 → 同时养两套

  这些成本往往被低估，实际可能是研发成本的 2-3 倍

第三层：迁移复杂度 — 老客户怎么办？

  如果 OffRamp 多渠道做在新 EX 上，会出现一个很尴尬的局面：

  场景 A：老客户迁移到新 EX
    · 客户要重新对接（API/流程/账户）
    · 客户问：“OnRamp 的问题解决了吗？” → 没有
    · 客户问：“那我为什么要迁移？” → 回答不了
    · 迁移 = 成本高 + 风险高 + 客户体验差

  场景 B：新客户用新 EX，老客户留老系统
    · 两套系统并行运行
    · 运营团队要同时维护两套流程
    · 技术团队要同时维护两套系统
    · 产品团队要同时管理两套产品逻辑
    · 资源消耗 = 双倍

  场景 C：新客户 OnRamp 用老系统，OffRamp 用新 EX
    · 同一个客户的业务跨两个系统
    · OnRamp 走老的（体验不顺），OffRamp 走新的
    · 客户要对接两套接口、两套账户、两套运营流程
    · 客户体验极差：“你们连一个统一的平台都做不出来？”
    · 运营复杂度爆表
```

### 推演逻辑

```
核心问题：值不值得？

  投入侧：
  · 研发资源（系统 + 对接 + 测试）
  · 客户沟通成本（解释新流程、培训、对接）
  · 运营成本（新流程、新对账、新异常处理）
  · 多系统维护成本（老系统不能停）
  · 迁移成本 / 双系统并行成本

  回报侧：
  · 客户画像 = 未知（海外客户 0 个）
  · 交易增量 = 未知（依赖海外签约）
  · 回报时间 = 未知（海外能不能跑通都不确定）

  同时放弃的：
  · 解决国内 OnRamp 阻碍 → 直接提升现有客户交易量
  · 优化 OffRamp 费率 → 直接解决现有客户最大痛点
  · 这些是确定性高、见效快的投入

  → 用确定性高的投入 换 不确定性高的产出，这个账算不算得过？
```

### 初步判断

```
→ 现有客户问题是真实的、紧迫的、每天发生的
→ 新产品的回报是不确定的、依赖海外的、时间未知的
→ 迁移/双系统并行带来的复杂度会进一步拉高成本
→ 维度八判断：❌ 机会成本高，新产品投入的确定性不足以证明放弃现有客户问题的合理性
```

## 12. 核心矛盾分析

结合前面各维度的已知信息，核心矛盾已经清晰：

```
矛盾结构：

  行业趋势 ──→ 海外 OffRamp 需求确实大 ──→ 应该做 OffRamp 多渠道
                                                   │
                                                   ▼
  但是 ────→ 海外客户 = 0 ─────────────→ 做了谁来用？
                                                   │
                                                   ▼
  国内客户 → 痛点 = OnRamp + OffRamp 费率 ──→ 多渠道不直接解决
```

### 矛盾的三种可能解法

```
解法 A："先做产品，等客户来"（提前布局）
  · 先把 OffRamp 多渠道做好
  · 等海外销售跑通后，产品能力已就绪
  · 风险：海外能否跑通未知，零客户入网，研发投入可能长期冻结
  · 适用条件：研发资源充裕 + 对海外市场有信心

解法 B："先跑通客户，再做产品"（需求驱动）
  · 先让海外销售跑通第一个客户入网（哪怕用 BB 内部渠道先跑）
  · 验证海外 OffRamp 需求真实存在
  · 客户量起来后，再投入做多渠道
  · 风险：客户因为覆盖面不够签不下来 → 鸡生蛋问题
  · 适用条件：海外阻碍不是因为产品能力

解法 C："两条腿走路"（并行推进）
  · 产品侧：做最小可行版（只接 1 个 SP，验证架构）
  · 销售侧：同步推进海外第一个客户
  · 国内侧：同步解决 OnRamp 阻碍和 OffRamp 费率问题
  · 风险：资源分散
  · 适用条件：能控制多渠道的研发投入在合理范围内
```

### 决定解法前需要回答的关键问题

```
❓ Q1：海外销售目前零入网客户，能否跑通未知 — 阻碍到底是什么？
   · 如果是产品能力（覆盖面不够）→ 必须先做多渠道（解法 A 或 C）
   · 如果是其他原因 → 先解决那个原因（解法 B）

❓ Q2：国内客户的 OffRamp 费率问题，多渠道能不能顺带解决？
   · 如果新 SP 费率更低 → 多渠道间接帮国内客户降费 → 国内也有价值
   · 如果新 SP 费率跟 BB 差不多 → 国内客户无感

❓ Q3：研发资源是否允许"提前布局"？
   · 如果团队有余力 → 提前做不亏
   · 如果团队已经满载 → 应该先做对现有客户更有价值的事

❓ Q4：不做 OffRamp 多渠道的话，研发资源做什么更有价值？
   · 比如：解决国内 OnRamp 阻碍 → 直接提升现有客户交易量
   · 比如：优化现有 OffRamp 费率/流程 → 直接解决 OffRamp 最大痛点
   · 这些替代方案的 ROI 可能更高
```
