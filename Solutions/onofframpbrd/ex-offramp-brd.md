## 1. 产品概述

### 1.1 什么是 OffRamp

OffRamp 是承兑产品的核心场景之一：客户持有 USDT（稳定币），通过承兑引擎将其转换为法币（如 USD），并付款给指定收款人。

```
OffRamp 一句话定义：

  客户的 USDT → 承兑为法币 → 付款到收款人银行账户

  本质 = 数币变现 + 法币出金
```

### 1.2 为什么先做 OffRamp

**行业背景：OffRamp 是更大的市场**

| 数据点                                                                        | 来源                    |
| ----------------------------------------------------------------------------- | ----------------------- |
| 2024 年稳定币总交易量超**$32 万亿**                                     | Visa On-Chain Analytics |
| B2B 稳定币付款约**$2,260 亿**，占全球稳定币支付量 ~60%                  | McKinsey 2025           |
| B2B 稳定币付款同比增长**733%**                                          | McKinsey 2025           |
| 稳定币关联卡消费（OffRamp）2025 年达**$45 亿**，同比增长 **673%** | McKinsey 2025           |
| 跨境汇款成本仍高达**6.5%**                                              | BIS 2025 Q1             |

**OffRamp 是刚需：**

- 企业/商户收到 USDT 后需要变现为法币 → 发工资、付供应商、缴税
- 跨境支付场景：收款方（新兴市场）收到 USDT → 需要 OffRamp 到本地法币 → 这是"最后一英里"
- OffRamp 是降低跨境汇款成本的关键环节

**OffRamp 在架构上更可控：**

- 客户的 U 在链上 → 可以先进 BB → BB 选渠道执行 → 切换渠道 = 换执行方
- OnRamp 则相反：法币一旦汇到某个 SP 就定了，切换不了

### 1.3 Phase 1 现状与痛点

Phase 1 的 OffRamp 是 BB 内部闭环：

```
客户 USDT → BB 承兑 → BB 自有渠道付款 → 收款人

  · 承兑方 = BB（唯一）
  · 付款渠道 = BB 自有银行账户 / XPAY（BB 内部）
  · 不涉及外部 SP
```

| 痛点                 | 说明                                         | 严重程度 |
| -------------------- | -------------------------------------------- | -------- |
| **覆盖面不足** | BB 自有渠道只能覆盖部分地区/币种             | 🔴 高    |
| **单点故障**   | 所有付款依赖 BB 自有渠道，出问题则整体不可用 | 🔴 高    |
| **无冗余**     | 付款失败只能退款，不能自动切换重试           | 🟡 中    |
| **扩展受限**   | 新增地区/币种 = BB 自己签约银行/PSP，周期长  | 🟡 中    |

### 1.4 本期目标

引入外部 SP，建立多渠道 OffRamp 能力：

| 维度           | Phase 1     | 本期目标             |
| -------------- | ----------- | -------------------- |
| 地区/币种覆盖  | BB 自有渠道 | 外部 SP 补充         |
| 付款成功率     | 无冗余      | 失败可自动切换       |
| 单渠道故障影响 | 整体不可用  | 自动降级到其他渠道   |
| 客户感知 SP    | 不感知      | **继续不感知** |

## 2. 业务约束与设计原则

### 2.1 硬约束

| #  | 约束                            | 为什么                                                                                                                 |
| -- | ------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| ❶ | **客户不感知底层 SP**     | 防止客户绕过 EX 直接找 SP 签约。EX 前期不具备多 SP 生态吸引力，如果客户知道底层 SP 是谁，可能直接跟 SP 签约，EX 被架空 |
| ❷ | **EX 不持牌，绝不碰资金** | EX 是科技平台，没有钱包、承兑能力、银行账户。所有资金（数币和法币）必须在 BB 或 SP 体系内流转，EX 只做编排和路由       |
| ❸ | **SP 间资金不互通**       | SP 是独立机构，不会帮你把客户的钱转给竞争对手。客户资金一旦进了某个 SP，就"锁"在那里了                                 |

### 2.2 核心推导

```
约束❶ + ❸ 产生一个结构性矛盾：

  "客户不感知 SP" + "资金在某个 SP 处" = 切换 SP 时资金转不过去

展开：
  · 不暴露 SP → 客户不知道钱在哪 → 不能让客户自己转
  · SP 间独立 → EX 控制不了 SP 间资金流动
  · 要保证体验 → 必须能无缝切换 SP

→ 三个条件不能同时满足
→ 除非：客户资金不直接进 SP，而是先进一个"资金中枢"

约束❷ → EX 不能做资金中枢

→ BB 是唯一解（有数币钱包 + 承兑能力 + 持牌）
```

> 完整推理过程见 `ex-offramp-v2-reasoning.md`

## 3. 角色分工

### 3.1 三方角色

| 角色         | 定位                  | 核心职责                                                                      |
| ------------ | --------------------- | ----------------------------------------------------------------------------- |
| **EX** | 科技平台（不持牌）    | SP 分组管理、组内路由引擎、风控规则、商户运营、对账监控。**绝不碰资金** |
| **BB** | Web3 基础设施（持牌） | 资金中枢 + 承兑引擎 + 支付底座。同时 BB 自身也是一个 SP（BB Channel）         |
| **SP** | 付款能力提供方        | 接收 BB 的资金/指令，执行法币出金。挂在 BB 名下，对客户不可见                 |

### 3.2 BB + EX 双轮驱动

```
EX（科技）                          BB（基建）
┌──────────────────┐          ┌──────────────────┐
│ · 路由引擎        │          │ · 数币钱包        │
│ · 风控规则        │  ←协同→  │ · 承兑引擎        │
│ · SP 分组管理     │          │ · 法币银行账户    │
│ · 商户运营        │          │ · 渠道执行        │
│ · 对账监控        │          │ · 清结算          │
│                  │          │                  │
│ ❌ 不碰资金       │          │ ✅ 资金中枢       │
└──────────────────┘          └──────────────────┘

· EX 提供交易流量 + 科技能力 → BB 提供资金基建 → 飞轮效应
· BB 始终是核心基建角色，不会"退为普通 SP"
```

### 3.3 BB 的双重身份

BB 既是资金中枢，也是一个 SP：

```
BB = 资金中枢
    ├── 接收客户 USDT
    ├── 执行承兑
    └── 选择渠道执行付款
            │
            ├── BB Channel（BB 自己的付款能力 = BB 作为 SP）
            │     · BB 自有银行账户
            │     · JRL（BB 体系内 MSO）
            │     · BB 已有 PSP
            │
            └── EX SP（外部渠道）
                  · SP-1、SP-2、SP-N ...
```

## 4. 核心概念模型

### 4.1 渠道模型

BB 是资金中枢，名下有 N 个渠道。渠道分两类：

| 渠道类型             | 包含                       | 管理方      | EX 可见性                               |
| -------------------- | -------------------------- | ----------- | --------------------------------------- |
| **BB Channel** | BB 自有银行、JRL、已有 PSP | BB 内部管理 | EX 看到"BB Channel"整体，不关心内部细节 |
| **EX SP**      | 通过 EX 接入的外部 SP      | EX 管理     | EX 可见，可配置路由                     |

对客户和租户：**全部不可见**。客户只看到余额和交易状态。

#### 为什么系统上要区分 BB Channel 和 EX SP

| 维度 | BB Channel | EX SP |
|------|-----------|-------|
| **签约与共享** | BB 自己签约的渠道，只服务 BB 自己的业务 | EX 平台签约/对接的 SP，可分配给 EX 平台上的多个租户使用 |
| **管理权归属** | BB 自己管理（上下线、限额、费率等），EX 无权干预内部细节 | EX 管理（上下线、熔断、路由权重、分组、备付金预警线） |
| **路由粒度** | EX 路由引擎只看到"BB Channel"整体，BB 内部自己决定用哪个子渠道（银行/JRL/PSP） | EX 路由引擎直接到每个 SP，无二级路由 |
| **资金风险** | 资金在 BB 自己的银行账户（BB 名下），冻结风险低，**不需要敞口管理** | 资金在 SP 账户（SP 名下），冻结/跑路风险高，**需要单 SP 敞口上限** |
| **结算** | BB 内部对账，无外部结算 | BB 和 SP 之间需要轧差对账结算 |
| **扩展性** | BB 未来可能独立发展渠道，不通过 EX | EX 可持续接入新 SP，扩大平台能力 |

> 一句话：BB Channel 是 BB 的私有能力，EX SP 是平台的公共能力。两者的管理权、风险模型、结算流程、共享范围全都不同，系统上必须区分。

### 4.2 两种资金安排模式

所有 SP（含 BB Channel）的资金安排归为两种模式：

#### 模式一：预存

BB 提前在 SP 处存放资金，付款时从预存中扣。

| 子类型             | 说明                                                 | 谁承兑 |
| ------------------ | ---------------------------------------------------- | ------ |
| **预存法币** | BB 承兑后，指令 SP 从预存的法币备付金中付款          | BB     |
| **预存 U**   | BB 提前存 USDT 到 SP，SP 收到付款指令后自行承兑+付款 | SP     |

```
预存模式的特点：
  · BB 资金占用 🔴 高（需在多个 SP 处预存）
  · 切换渠道 ✅ 简单（换指令对象即可，备付金已在各 SP 处）
  · 风险：备付金冻结/SP 跑路 → 需设单 SP 敞口上限
```

#### 模式二：转账

BB 按笔实时转给 SP，不提前预存。

| 子类型             | 说明                                     | 前提条件                                  | 谁承兑 |
| ------------------ | ---------------------------------------- | ----------------------------------------- | ------ |
| **转 U**     | BB 按笔转 USDT 给 SP，SP 自行承兑+付款   | SP 有链上地址 + 承兑能力                  | SP     |
| **法币转账** | BB 承兑后，从 BB 银行账户实时转法币给 SP | BB 和 SP 在同一家银行，且行内转账实时到账 | BB     |

```
转账模式的特点：
  · BB 资金占用 🟢 低（按笔转出，不预存）
  · 切换渠道：
    - 法币同行转账 ✅ 简单（退回也是同行实时）
    - 转 U ⚠️ 较复杂（U 已转出，退回时间不确定）
  · 法币转账前提：同一家银行 + 行内转账实时到账
```

#### 模式对比

| 维度                   | 预存（法币/U）          | 转账（法币同行/U）                           |
| ---------------------- | ----------------------- | -------------------------------------------- |
| **资金占用**     | 🔴 高（长期锁在 SP 处） | 🟢 低（按笔转出）                            |
| **冻结风险**     | 🔴 高（备付金在 SP 处） | 🟢 低（资金在 BB 账户，用时才转）            |
| **切换渠道**     | ✅ 简单（换指令）       | 法币同行 ✅ 简单 / 转U ⚠️ 较复杂           |
| **渠道准入门槛** | 低（纯法币渠道即可）    | 转U：高（SP 需承兑能力）/ 法币：中（需同行） |
| **结算速度**     | 取决于 SP 执行速度      | 法币同行：秒级 / 转U：链上确认 + SP 执行     |

### 4.3 SP 分组与路由

因为客户不感知底层 SP（约束❶），我们通过 **SP 分组** 来对客户暴露差异化的服务选项：

```
SP 分组机制：

  ┌──────────────────────────────────────────────┐
  │                 渠道池                        │
  │                                              │
  │  ┌─ 分组 A（费率高 / 资料要求少）─────────┐   │
  │  │  BB Channel、SP-1、SP-3                │   │
  │  └────────────────────────────────────────┘   │
  │                                              │
  │  ┌─ 分组 B（费率低 / 资料要求多）─────────┐   │
  │  │  SP-2、SP-4                            │   │
  │  └────────────────────────────────────────┘   │
  │                                              │
  └──────────────────────────────────────────────┘
```

| 概念                          | 说明                                                                  |
| ----------------------------- | --------------------------------------------------------------------- |
| **分组维度**            | 按费率、资料要求等业务维度将渠道分为若干组                            |
| **客户选组**            | 客户选择一个分组（如"快速到账"vs"低费率"），不感知具体 SP             |
| **组内路由**            | EX 在客户选定的分组内，按可用性、备付金余额、渠道优先级自动选最优渠道 |
| **不同分组 = 不同收费** | 每个分组有独立的费率标准（见 §6）                                    |

### 4.4 架构图

```
                    ┌─────────────────────────┐
                    │       客户 / 租户         │
                    │    (只看到余额+分组选项)   │
                    └─────────┬───────────────┘
                              │ 发起 OffRamp + 选分组
                              ▼
                    ┌─────────────────────────┐
                    │      EX 科技平台          │
                    │  · 风控检查               │
                    │  · 按分组做组内路由        │
                    │  · 监控 + 对账            │
                    │  · ❌ 不碰资金            │
                    └─────────┬───────────────┘
                              │
                              ▼
              ┌───────────────────────────────────┐
              │          BB（资金中枢 + 承兑）      │
              │                                   │
              │   客户 U ──→ BB 钱包               │
              │   BB 承兑（或不承兑，看模式）        │
              │   BB 选择渠道执行付款               │
              └──┬──────────┬──────────┬──────────┘
                 │          │          │
       ┌────────▼──┐ ┌────▼─────┐ ┌──▼───────┐
       │ BB Channel │ │ EX SP-1  │ │ EX SP-2  │
       │ (BB自身)   │ │ (外部)   │ │ (外部)   │
       │            │ │          │ │          │
       │ 自有银行   │ │ 预存法币 │ │ 法币同行 │
       │ JRL/PSP    │ │ 或预存U  │ │ 或转U    │
       │            │ │          │ │          │
       │ BB内部管理 │ │ EX管理   │ │ EX管理   │
       └────────────┘ └──────────┘ └──────────┘
```

## 5. 关键业务流程

### 5.1 正常 OffRamp 流程

```
1. 客户发起 OffRamp
   · 输入：卖 100 USDT → 付到收款人银行账户
   · 选择 SP 分组（如"快速到账"）

2. EX 处理
   · sp风控检查 → 通过
   · 在客户选定的分组内，按路由规则选最优渠道

3. BB 执行
   · 冻结客户 BB 钱包中的 100 USDT
   · 根据渠道的资金安排模式执行：
     - 预存法币：BB 承兑 100 USDT → 96 USD → 指令 SP 从备付金付 96 USD
     - 预存 U：BB 指令 SP 从预存 U 中扣 100 USDT → SP 承兑 + 付款
     - 法币同行转账：BB 承兑 → 银行内部实时转 96 USD 给 SP → SP 付款
     - 转 U：BB 转 100 USDT 给 SP → SP 承兑 + 付款

4. 完成
   · SP 付款成功 → 扣除冻结的 USDT → 通知客户"付款成功"
```

### 5.2 异常切换流程

```
渠道付款失败时：

  客户 100 USDT 在 BB 钱包（始终在 BB 手里）
      │
      ▼
  EX 组内路由选渠道 A → BB 执行 → 失败
      │
      ▼
  EX 自动切换到组内渠道 B → BB 执行 → 失败
      │
      ▼
  EX 自动切换到组内渠道 C → BB 执行 → 成功 ✅

关键：
  · 客户的 USDT 始终在 BB → 切换渠道 = BB 换一个执行方
  · 预存模式的渠道：备付金在各渠道处，切换 = 换指令对象
  · 转账模式的渠道：BB 重新转一笔（前一笔等 SP 退回）
  · 最大重试次数可配置
```

### 5.3 客户端无感知

```
不管后端切换了几次渠道：

  · 客户看到的始终是：处理中 → 成功 / 失败
  · 不展示"尝试了渠道 A 失败，切换到渠道 B"
  · 所有渠道都失败 → 客户看到"付款失败，已退款"
  · 切换过程的耗时 = 客户的等待时间
```

## 6. 收费模型

### 6.1 对客户收费

**沿用现有承兑手续费模式**，但按 SP 分组差异化收费：

| 分组   | 特点                             | 手续费     |
| ------ | -------------------------------- | ---------- |
| 分组 A | 费率高 / 资料要求少 / 到账快     | X%（较高） |
| 分组 B | 费率低 / 资料要求多 / 到账可能慢 | Y%（较低） |

> 具体分组方案和费率标准待确认（见 §9）

### 6.2 承兑汇率

本期沿用 BB 现有汇率逻辑，不做改动。

### 6.3 SP 通道费

SP 向 BB 收取通道费（按笔或按比例），由渠道合同约定。不同 SP 的通道费不同，这也是 SP 分组和费率差异的底层原因。

### 6.4 BB ↔ EX 分润

BB 的基建服务费 / EX 的平台技术服务费 — **待定**。

## 8. 关键风险与应对
