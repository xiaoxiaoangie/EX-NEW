Bonbillhk 收单结算解决方案

一、背景与目标

Bonbillhk 通过 Worldpay 进行收单业务。Worldpay 定期将收单结算款打给 Bonbillhk，Bonbillhk 需要与各供应商进行结算。

问题具体情况
结算收款Worldpay 结算款需打入银行账户，Bonbillhk 已在智朗银行成功开户
供应商结算供应商分布在不同地区，传统 SWIFT 逐笔结算成本高、到账慢
账户体系缺乏统一的账户体系来管理商户资金、结算、提现全流程

方案总览

维度方案一VA 法币收付方案二数币结算
核心思路Bonbillhk 在 IPL 开 VA 接收 Worldpay 结算款，Payout 给供应商结算款→OnRamp→Winingbees 钱包直接提币到商户钱包
结算收款VA 直接接收 Worldpay 结算款智朗银行→行内转账→OnRamp 数币化
下发方式法币 Payout（银行转账）给供应商数币钱包直接提币到商户钱包
承兑环节无两次（OnRamp + OffRamp）
商户入网不需要需要（BB 开 MID + 钱包）
适用场景供应商少、单笔大、快速上线多地区、需账户体系、追求时效

方案一 VA 法币收付方案

核心思路

Bonbillhk 入网 EX 平台，在 IPL（法币 SP）开立 VA。该 VA 作为 Worldpay 结算收款账户，Bonbillhk 收到结算款后通过法币 Payout 向各供应商结算。

各方角色

实体角色说明
BonbillhkEX 租户在 IPL 开 VA，接收 Worldpay 结算款，发起 Payout 给供应商
IPL法币 SP为 Bonbillhk 开立 VA，执行 Payout
EX技术服务平台API 编排层，不碰钱
供应商Bonbillhk 的供应商提供银行账户信息，接收法币结算款

账户结构

IPL 侧

└── Bonbillhk（MID-B） └── VA（HKD/USD） 接收 Worldpay 结算 + Payout 给供应商

供应商侧

├── 供应商 A —— 银行账户 ├── 供应商 B —— 银行账户 └── 供应商 N ...
关键：供应商不需要入网任何平台。只需提供银行账户信息给 Bonbillhk。Bonbillhk 自己作为 EX 租户直接调用 API。

方案二 数币结算方案

核心思路

  1. Worldpay 结算款数币化 — 收到法币后通过 OnRamp 转为 USDT
  2. 数币链路结算给商户 — Winingbees 钱包直接提币到各商户钱包（一跳到位）
  3. 商户自主提现 — 商户可通过 OffRamp 兑换为法币提现到银行

角色关系

商户视角（商户只看到这一层）
收单商户刷卡 / 查看结算 / 提现
⟷
Bonbillhk清算 + 结算 + 钱包账本
▼ 通过 Winingbees 渠道执行（商户不可见）
WiningbeesEX 租户 / API
⟷
EXAPI 编排层
⟷
BB持牌承兑通道

各方身份

实体现实角色系统身份说明
收单商户Bonbillhk 签约商户Winingbees 下商户（代开）在 BB 有独立 MID + 数币钱包，不感知底层
BonbillhkWorldpay 收单方不在 BB 入网负责清算、结算触发，不拥有 MID/数币钱包
Winingbees下发渠道EX 租户 + BB 商户双重身份：调 API + 拥有数币钱包，OnRamp 后直接提币给商户
BB持牌承兑通道SP执行 OnRamp/OffRamp，管理钱包
智朗银行法币通道银行Bonbillhk 和 Winingbees 均在智朗开户

账户结构

BB 侧（持牌服务商）： │ ├── Winingbees（MID-W） │ ├── 数币钱包（USDT）— OnRamp 入账 → 直接提币给商户 │ └── 智朗银行账户 — 法币收付 │ ├── 商户 A（MID-A） │ └── 数币钱包（USDT）— 接收提币 + OffRamp │ ├── 商户 B（MID-B2）└── 数币钱包 └── 商户 N ...
注意：Bonbillhk 不在 BB 入网，没有 MID 和数币钱包。OnRamp 以 Winingbees MID 执行，数币直接从 Winingbees 钱包提币到各商户钱包。

Bonbillhk 商户钱包体系

Bonbillhk 为每个商户建立钱包账本体系。底层数币钱包由 BB 托管，Bonbillhk 维护商户可见的钱包视图。

Bonbillhk 平台（商户可见）

├── 商户 A │ ├── 待结算余额 — 法币计价 │ ├── 可用余额 — USDT 计价 │ ├── 冻结余额 — 拒付/风控 │ └── 提现中 └── Bonbillhk 自有账户 ├── 待 OnRamp 余额 └── 手续费收入

BB 侧（商户不可见）

├── Winingbees 数币钱包 │ OnRamp 入账→提币给商户 ├── 商户 A 数币钱包（USDT） ├── 商户 B 数币钱包（USDT） └── ...

五、端到端资金流程

流程全景

Phase 1: 消费者刷卡 → Worldpay 收单 → 结算款打入 Bonbillhk 智朗 Phase 2: Bonbillhk 智朗 → 行内转账 → Winingbees 智朗 → OnRamp → USDT 入 Winingbees 钱包 Phase 3: Winingbees 钱包 → 直接提币 → 各商户钱包（一跳到位） Phase 4: 商户数币钱包 → OffRamp → 法币到商户银行

5.3 数币结算：Winingbees → 商户钱包

结算是 Bonbillhk 系统内部发起的动作。先在账本中将「待结算余额」转为「结算中」，然后触发 Winingbees 提币到商户钱包。

sequenceDiagram autonumber participant BH as Bonbillhk 系统 participant WB as Winingbees participant EX as EX 平台 participant MW as 商户钱包 Note over BH: 结算周期到达，计算各商户金额 Note over BH: 账本：「待结算」→「结算中」 BH->>WB: 结算指令（商户列表 + USDT 金额） loop 逐商户 WB->>EX: 提币：Winingbees钱包→商户钱包 EX-->>WB: 完成 Note over MW: 到账 end WB-->>BH: 返回执行结果 Note over BH: 账本：「结算中」→「可用余额」↑
核心要点：
• 结算主体是 Bonbillhk，逻辑全在 Bonbillhk 系统内
• Winingbees 是执行层，从自己钱包直接提币到商户钱包（一跳到位）
• 商户只看到 Bonbillhk 平台余额变化，底层 BB 钱包不可见
• 双层一致性：Bonbillhk 账本 与 BB 实际余额 定期对账

5.4 商户提现 OffRamp

sequenceDiagram autonumber participant M as 商户 participant BH as Bonbillhk 平台 participant WB as Winingbees participant EX as EX 平台 participant BB as BB 承兑 participant BANK as 商户银行 M->>BH: 点击「提现」 BH->>WB: 转发提现请求 WB->>EX: 调 OffRamp（商户MID, 金额, 目标法币） EX-->>WB: 返回订单 BB->>BB: 从商户钱包扣 USDT → 兑换法币 BB->>BANK: 法币到商户银行 EX-->>WB: Webhook: OffRamp 完成 WB-->>BH: 通知完成 BH-->>M: 平台展示提现成功
关键：OffRamp 使用商户自己的 MID，从商户自己的数币钱包出款。全程在 Bonbillhk 平台操作，底层对商户透明。
