### 已确认的核心决策（不要再讨论）

> 1. **BB 是资金中枢**：所有客户资金（U）先进 BB，BB 统一管理，再通过渠道执行付款
> 2. **EX 不持牌、不碰资金**：EX 是科技平台，只做路由/编排/监控，资金始终在 BB 体系内
> 3. **统一渠道模型**：BB 内部渠道（BB Channel）和 EX 平台 SP（EX SP）并存，对客户完全不可见
> 4. **三种资金安排**：预存法币、按笔转U、同行转账（BB 和 SP 在同一银行有账户，银行内部实时转账）
> 5. **混合切换策略**：SP 间互通时直接转（解法A，含同行转账），不通时走 BB 中枢兜底（解法D）；一期走 D，A 二期探索
> 6. **BB+EX 双轮协同**：EX 提供科技平台（路由/风控/商户管理），BB 提供 Web3 基础设施（资金/承兑/支付底座）；BB 始终是基建角色，不会"退为普通 SP"
> 7. **v2 只做 OffRamp 多渠道**，OnRamp 保持 BB 内部闭环（原因见第 3.4 节）

## 1. Phase 1 现状与瓶颈

### 1.1 Phase 1 怎么做的

```
Phase 1 OffRamp = BB 内部闭环：

  客户数币钱包(USDT) → BB承兑引擎 → BB法币账户(USD)（或直接付到收款人）

  · 承兑方 = BB（唯一）
  · 付款渠道 = BB 代收付账户 / XPAY（BB 内部 Channel）
  · 不涉及外部 SP
```

### 1.2 为什么要做 v2

| 瓶颈             | 说明                                                   | 严重程度 |
| ---------------- | ------------------------------------------------------ | -------- |
| **覆盖面不足**   | BB 自有渠道只能覆盖部分地区/币种，客户付不出去         | 🔴 高   |
| **单点风险**     | 所有付款依赖 BB 自有渠道，渠道出问题则整体不可用       | 🔴 高   |
| **无冗余**       | 没有备选渠道，付款失败只能退款，不能自动切换重试       | 🟡 中   |
| **扩展受限**     | 新增地区/币种 = BB 自己去签约银行/PSP，周期长          | 🟡 中   |

**一句话**：Phase 1 能跑通但"不够用"，需要接外部渠道扩大能力。

### 1.3 v2 要达到什么

| 目标               | Phase 1 现状 | v2 期望             |
| ------------------ | ------------ | ------------------- |
| 地区/币种覆盖      | BB 自有渠道  | 外部渠道补充        |
| 付款成功率         | 无冗余       | 失败可自动切换      |
| 单渠道故障影响     | 整体不可用   | 自动降级            |
| 客户感知 SP        | 不感知       | **继续不感知**      |

## 2. 核心矛盾与推理链条

> 完整推理过程见 `ex-offramp-v2-reasoning.md`，以下为精华摘要。

### 2.1 核心矛盾（这是整个方案的起点）

```
我们有一个硬约束：客户不能感知底层 SP

这导致一个连锁问题：

  "客户不感知 SP" + "资金在某个 SP 处" = 切换 SP 时资金转不过去

展开说：
  · 不暴露 SP → 客户不知道钱在哪 → 不能让客户自己去转
  · SP 之间独立 → EX 控制不了 SP 间资金流动
  · 要保证体验 → 必须能无缝切换 SP

  这三个条件不能同时满足 → 除非引入一个"资金中间方"
```

### 2.2 推理链条（从约束到方案）

```
推理链条（产品团队必须理解这个逻辑）：

Step 1：客户不感知 SP + SP 间资金不通
    ↓
Step 2：客户资金不能直接进 SP → 必须进一个"中间方"
    ↓
Step 3：中间方 = BB（唯一有数币钱包+承兑能力的体系内角色）
    ↓
Step 4：BB 持有客户 U 后，怎么让 SP 执行付款？四种模式：
    · 模式A（预存法币）：BB 在 SP 处预存法币，指令 SP 付款
    · 模式B（按笔转U）：BB 按笔转 U 给 SP，SP 自己承兑+付款
    · 模式C（BB自己付）：BB 通过自有渠道直接付
    · 模式D（同行转账）：BB 和 SP 在同一银行有账户，BB 承兑后银行内部实时转给 SP
    ↓
Step 5：四种模式不是互斥的 → 统一为"BB 的渠道池"
    · BB 自己付 → 也是用渠道（自有银行/JRL/PSP）
    · 每个渠道有不同的"资金安排"：预存法币 / 按笔转U / 同行转账 / 直接转账
    ↓
Step 6：EX 做路由，选最优渠道 → 客户无感知
```

### 2.3 为什么是 BB 做资金中枢（而不是别的方案）

| 备选方案                  | 为什么不行                                           |
| ------------------------- | ---------------------------------------------------- |
| 客户的 U 直接进 SP        | SP 失败时资金在 SP 处，转不到别的 SP → 切换不了      |
| EX 自己做资金中枢         | EX 不持牌、无钱包/承兑能力；**EX 绝不碰资金**是硬约束，不论现在还是未来 |
| IPL 做资金中枢            | IPL 是法币收付SP，没有数币钱包/承兑能力              |
| 不做 SP 切换              | 客户体验差，单点故障无法恢复                         |
| **BB 做资金中枢** ✅      | BB 有数币钱包 + 承兑能力 + 体系内核心角色            |

## 3. 架构决策与理由

### 3.1 BB 和 EX 的关系（双轮驱动飞轮）

```
两个硬约束决定了 BB 和 EX 的关系：

硬约束 1：EX 不持牌，绝不碰资金
  · EX 是开放式科技平台（路由/风控/商户管理/合规工具）
  · EX 没有钱包、没有承兑能力、没有银行账户
  · 不论现在还是未来，客户资金（U 或法币）不能在 EX
  · 这意味着：EX 永远不可能自己做资金中枢

硬约束 2：BB 纯承兑故事已经不好讲了
  · BB 定位 = Web3 基础设施（不是单纯的承兑商/OTC）
  · BB 聚焦稳定币发行与支付，筑牢链上支付底座，支持大规模交易处理
  · BB 需要科技加持 → BB + EX = 科技 + 承兑 = 完整故事

→ BB + EX = 双轮驱动的增长飞轮（1 + 1 > 2）

  ┌────────────────────┐         ┌────────────────────┐
  │  EX：开放式科技平台  │ ──────→ │  BB：Web3 基础设施   │
  │                    │ 提供     │                    │
  │ · 数法融合赋能      │ 交易流量 │ · 稳定币发行与支付   │
  │ · 端到端 AI 风控    │         │ · 链上支付底座       │
  │ · 商户/合规工具     │ ←────── │ · 大规模交易处理     │
  │                    │ 提供     │                    │
  │ 飞轮：数据驱动 →    │ 基建支撑 │ 飞轮：规模经济 →     │
  │  模型迭代 → 赋能    │         │  体验优化 → 基建完善  │
  └────────────────────┘         └────────────────────┘

这决定了架构怎么分：
  · EX 负责：路由引擎、渠道管理、风控规则、商户运营、合规工具
  · BB 负责：资金持有、承兑引擎、银行账户、渠道执行、清结算
  · BB 不是 EX 的一个"渠道选项"，而是 EX 的基建支撑层
  · 外部 SP 通过 EX 接入后，也是走 BB 的资金中枢执行

关键区别（vs 之前的"BB退为普通SP"表述）：
  · ❌ 旧说法：BB 一期顶上，终态退为普通 SP
  · ✅ 新定位：BB 始终是 Web3 基建层，EX 始终是科技平台层
  · BB 可能不再是唯一的承兑方（未来可接入其他承兑方）
  · 但 BB 始终是核心基建，不是"普通 SP 之一"
  · EX 也不会"替代 BB 做资金中枢"（EX 不持牌，不碰资金）
```

### 3.2 BB 跟所有渠道的关系

```
不论 BB 是什么身份（VASP/MSB/OTC），BB 跟所有渠道的关系都是"渠道合作"：

  · BB 出钱（法币备付金 / U / 银行内部实时转账）
  · 渠道出能力（付款渠道 / 承兑+付款）
  · 签的是渠道合作协议

"BB 自己付"的背后也是渠道：
  · BB 通过自有银行转账 → 银行 = BB 的渠道
  · BB 通过 JRL 付款 → JRL = BB 的渠道
  · 跟 BB 通过外部 SP 付款没有本质区别

"同行转账"的特殊优势：
  · BB 和 SP 在同一银行有账户 → 银行内部实时转账 → 无需预存
  · 解决了"预存法币"的资金占用和冻结风险
  · 解决了"按笔转U"的链上确认延迟和SP承兑风险

→ 统一模型：BB = 资金中枢，名下有 N 个渠道，EX 做路由
```

### 3.3 EX vs BB 职责边界

| 职责             | EX 负责                    | BB 负责                      |
| ---------------- | -------------------------- | ---------------------------- |
| **定位**         | 开放式科技平台（不持牌）    | Web3 基础设施（持牌）        |
| **渠道接入**     | 外部 SP 在 EX 上注册管理   | BB 内部渠道自行管理          |
| **路由决策**     | ✅ EX 路由引擎             | ❌                           |
| **资金持有**     | ❌ **绝不碰资金**          | ✅ 客户 U 在 BB 钱包        |
| **承兑**         | ❌                         | ✅（预存法币/同行转账模式下 BB 承兑） |
| **渠道监控**     | 外部 SP 的监控             | BB 内部渠道的监控            |
| **对账结算**     | EX 与外部 SP 对账          | BB 内部对账                  |
| **风险承担**     | 平台级风控规则             | 垫资风险（备付金冻结/跑路）  |
| **合规**         | 合规工具/规则引擎          | 持牌主体，承担合规法律责任   |

### 3.4 为什么 v2 只做 OffRamp（不做 OnRamp 多渠道）

#### 原因一：OffRamp 交易体量显著大于 OnRamp

```
行业数据（2024-2025）：

  · 稳定币总交易量：2024 年超 $32 万亿（Visa On-Chain Analytics）
  · 稳定币支付相关交易量：约 $5.7 万亿（Visa 估算）
  · B2B 稳定币付款：约 $2,260 亿，占全球稳定币支付量的 ~60%（McKinsey 2025）
  · B2B 稳定币付款同比增长 733%（McKinsey 2025）
  · 稳定币关联卡消费（Off-Ramp）：2025 年达 $45 亿，同比增长 673%（McKinsey 2025）

核心趋势：
  · OffRamp（Crypto-to-Fiat）增长速度快于 OnRamp
  · 企业/商户收到稳定币后需要变现为法币（发工资、付供应商、缴税）→ OffRamp 刚需
  · 跨境支付场景：收款方（新兴市场）收到 USDT → OffRamp 到本地法币 → 这是最后一英里
  · 跨境汇款成本仍高达 6.5%（BIS 2025 Q1），OffRamp 是降低成本的关键环节

我们的客户画像也验证了这一点：
  · 客户主要需求 = 卖 U 提法币（付款给供应商/收款人）
  · OnRamp 需求相对弱（客户自己有 U，不太需要我们帮他买）
  · OffRamp 交易量 > OnRamp 交易量（具体比例待业务确认）
```

#### 原因二：OffRamp 在架构上更容易做多渠道

```
OffRamp（卖U提法币）：
  · 客户 U 在链上 → 可以先进 BB → BB 选渠道执行付款 → 可控 ✅
  · 资金始终在 BB 手里 → 切换渠道 = 换执行方 → 简单

OnRamp（充法币买U）：
  · 客户汇法币 → 必须有一个收款账户
  · 收款账户不能是 EX 的 → EX 不持牌，绝不碰资金（硬约束）
  · 收款账户是 BB 的 → BB 需要在各地区有 VA/银行账户 → 覆盖面有限
  · 收款账户是 SP 的 → 钱进了 SP → 资金在 SP 处 → 切换难
  · 法币一旦汇出 → 物理上在收款方 → 不像 U 可以链上秒转

→ OnRamp 的多渠道切换 = 资金物理移动（法币跨银行转账）→ 慢且不可控
→ OffRamp 的多渠道切换 = 换一个指令对象（U 始终在 BB）→ 快且可控
→ 而且 EX 不持牌 → 法币收款必须走 BB 或 SP → 进一步限制了 OnRamp 的灵活性
```

#### 原因三：投入产出比

```
OffRamp 多渠道：
  · 体量大 + 架构简单 + 客户痛点强 → 投入产出比高 ✅

OnRamp 多渠道：
  · 体量相对小 + 架构复杂 + 客户可自行解决（自己有U来源）→ 投入产出比低 ❌

结论：v2 集中资源做 OffRamp 多渠道，OnRamp 保持 BB 内部闭环
  · OnRamp 多渠道方案可以开始调研，但不在 v2 范围内
```

# Part 2：做什么（What）

## 4. 渠道模型

### 4.1 统一视角：一切都是"BB 的渠道"

```
BB 的所有付款能力 = BB 名下的渠道池

渠道池：
├── BB Channel（内部渠道，EX 不可见）
│   ├── BB 自有银行账户（直接转账）
│   ├── JRL（BB 体系内 MSO）
│   └── BB 已有 PSP
│
└── EX SP（外部渠道，EX 可见）
    ├── SP-1（纯法币渠道 → 预存法币模式）
    ├── SP-2（有承兑能力 → 按笔转U模式）
    ├── SP-3（与BB同行 → 同行转账模式）
    └── SP-N ...

对客户：全部不可见
对租户：全部不可见
对 BB 后台：可见 BB Channel
对 EX 后台：可见 EX SP，BB Channel 作为一个整体
```

### 4.2 渠道分类维度

| 维度           | 分类       | 说明                                                 |
| -------------- | ---------- | ---------------------------------------------------- |
| **资金安排**   | 预存法币   | BB 在该渠道处预存法币备付金，指令付款时从备付金扣     |
|                | 按笔转U    | BB 按笔转 USDT 给渠道，渠道自己承兑+付款             |
|                | 同行转账   | BB 和 SP 在同一银行有账户，BB 承兑后银行内部实时转账给 SP，SP 执行付款 |
|                | 不需预存   | BB 自有银行账户，直接转账（资金始终在 BB 名下）       |
| **BB 控制力** | 高         | BB 自有银行账户、JRL                                 |
|                | 中         | 通过 EX 对接的 SP（有合同约束）                       |
|                | 低         | 第三方 PSP（合作关系较松散）                          |
| **承兑能力**   | 有         | 渠道能收 U 自己换法币（OffRamp SP）                   |
|                | 无         | 渠道只能收发法币（纯付款渠道）                        |
| **EX 可见性** | EX 可见    | 通过 EX 对接的渠道，在 EX 上注册为 SP                 |
|                | EX 不可见  | BB 自有渠道（银行账户等），BB 内部管理                 |

### 4.3 三种资金安排模式对比

| 维度 | 预存法币 | 按笔转U | 同行转账（新增） |
| --- | --- | --- | --- |
| **资金流** | BB承兑→指令SP用备付金付 | BB转U给SP→SP承兑+付 | BB承兑→银行内部实时转给SP→SP付 |
| **谁承兑** | BB | SP | BB |
| **BB资金占用** | 🔴 高（多渠道预存） | 🟢 低（按笔转出） | 🟢 低（按笔转出，不预存） |
| **冻结风险** | 🔴 高（法币长期在SP处） | 🟢 低（U按笔转，在途小） | 🟢 低（资金在BB银行账户，实时转出即到） |
| **切换渠道** | ✅ 简单（换指令对象） | ⚠️ 复杂（U已转出需追回） | ✅ 简单（换指令对象，实时到账） |
| **渠道准入** | 低（纯法币渠道即可） | 高（需有承兑能力） | 中（SP需在同一银行有账户） |
| **汇率控制** | ✅ BB统一定价 | ❌ 各SP自行定价 | ✅ BB统一定价 |
| **结算速度** | 取决于SP执行速度 | 链上确认+SP执行 | ✅ 银行内部实时（秒级） |
| **适用渠道** | BB Channel、纯法币SP | 有OffRamp能力的SP | BB和SP在同一银行有账户 |
| **前提条件** | BB在SP处开备付金账户 | SP有承兑能力+链上地址 | BB和SP在同一家银行开户 |

#### 同行转账模式详解

```
同行转账 = 解法A的可行特例

原始推理文档中，解法A"让SP-1把钱转给SP-2"被判定为"不可控、不可行"
原因：SP间是独立机构，跨行转账慢且不可控

但有一个特例：
  · 如果 BB 和 SP 在同一家银行都有账户
  · 银行内部转账 = 实时到账（不走跨行清算）
  · BB 承兑后，直接从 BB 的银行账户 → SP 的银行账户（同行内部转账）
  · SP 收到法币后执行付款

这种模式的优势：
  ✅ 不需要预存备付金（按笔实时转，资金占用 ≈ 0）
  ✅ 没有冻结风险（资金在BB自己银行账户，转出才走）
  ✅ 实时到账（银行内部转账秒级完成）
  ✅ BB统一承兑（汇率可控）
  ✅ 切换简单（跟预存法币一样，换指令对象即可）

前提条件：
  · BB 和 SP 必须在同一家银行有账户
  · 该银行支持实时内部转账（API或指令）
  · SP 有对应的法币付款能力

适用场景：
  · BB 和某些 SP 碰巧在同一家银行开户
  · 或者 BB 主动在 SP 的开户行开设账户
  · 特别适合交易量大的核心SP（值得为它开一个同行账户）

与"预存法币"的关键区别：
  · 预存法币：BB 提前把钱存在 SP 处 → 资金被占用 → 冻结风险
  · 同行转账：BB 的钱始终在 BB 自己的银行账户 → 按笔实时转 → 无占用无冻结
```

### 4.4 为什么架构上要区分 BB Channel 和 EX SP

```
即使目前所有渠道都是 EX 帮 BB 对接的，架构上先做区分：

1. BB 未来可能独立发展
   · BB 自己对接新渠道，不通过 EX
   · 今天不区分 → 未来拆分很痛苦

2. 风险管理不同
   · BB 自有渠道：资金在 BB 名下，冻结风险低
   · 外部 SP：资金在 SP 名下，冻结/跑路风险高

3. 结算流程不同
   · BB 自有渠道：BB 内部对账
   · 外部 SP：需要跟 SP 轧差结算

4. 费率结构不同
   · BB 自有渠道：BB 内部成本
   · 外部 SP：SP 收费 + BB 加价
```

### 5.2 四种模式下的正常流程

**模式一：预存法币（BB承兑 + SP执行付款）**

```
客户发起 OffRamp：卖 100 USDT → 付到收款人银行

1. EX 接收请求 → 风控 → 冻结 BB 钱包 100U
2. EX 路由选 EX SP-1（预存法币模式）
3. BB 承兑：100 USDT → 96 USD
4. BB 指令 SP-1：用备付金付 96 USD → 收款人
5. SP-1 执行 → 回调成功 → 扣除冻结 → 通知客户
```

**模式二：按笔转U（SP自己承兑+付款）**

```
1-2. 同上，路由选 EX SP-2（有承兑能力）
3. BB 不承兑，转 100 USDT 给 SP-2 链上地址
4. SP-2 收U → 自己承兑 → 付 96 USD → 收款人
5. 同上
```

**模式三：BB Channel（BB自己付）**

```
1-2. 同上，路由选 BB Channel
3. BB 承兑：100 USDT → 96 USD
4. BB 通过自有银行直接转 96 USD → 收款人
5. 同上（内部完成，无外部 SP）
```

**模式四：同行转账（BB承兑 + 银行内部实时转给SP + SP执行付款）**

```
1-2. 同上，路由选 EX SP-3（与BB同行）
3. BB 承兑：100 USDT → 96 USD
4. BB 从自己的银行账户 → 银行内部实时转账 96 USD → SP-3 同行账户（秒级到账）
5. SP-3 收到法币 → 执行付款 96 USD → 收款人
6. 同上

与预存法币的区别：
  · 预存法币：钱提前存在SP处（资金占用）
  · 同行转账：钱始终在BB银行账户，按笔实时转（无资金占用）
```

## 6. 异常场景与切换机制

### 6.1 渠道切换逻辑

```
OffRamp 付款失败时：

  客户 100U 在 BB 钱包（始终在 BB 手里）
      │
      ▼
  EX 路由选 BB Channel → 付款失败
      │
      ▼
  EX 路由换 EX SP-1 → BB 用备付金指令 SP-1 付款 → 失败
      │
      ▼
  EX 路由换 EX SP-2 → BB 转 U 给 SP-2 → SP-2 承兑+付款 → 成功 ✅

关键：客户的 U 始终在 BB，切换渠道 = BB 换一个执行方
  · 预存法币渠道：备付金在各渠道处，切换 = 换指令对象
  · 按笔转U渠道：BB 重新转一笔 U（前一笔等 SP 退回）
```

### 6.2 异常场景矩阵

| #  | 异常场景                     | 处理方式                                                 |
| -- | --------------------------- | -------------------------------------------------------- |
| E1 | 渠道付款失败                 | EX 路由自动切换下一个可用渠道重试                        |
| E2 | 所有渠道不可用               | 告知客户暂时不可用；后台告警；运营补充备付金             |
| E3 | 备付金不足（单个渠道）       | 跳过该渠道，选下一个；告警运营补充                       |
| E4 | 备付金不足（全局）           | OffRamp 暂停；后台紧急告警                               |
| E5 | 渠道结算账户被冻结           | 自动摘除该渠道；在途订单切换；**BB 可能损失被冻备付金**  |
| E6 | SP 跑路/破产                 | 紧急下线；在途订单退款；**BB 承担垫资损失**              |
| E7 | 按笔转U后 SP 不执行          | 超时 → 标异常 → BB 用其他渠道重做 → 向 SP 追偿          |
| E8 | 承兑后汇率波动               | 锁汇有效期内完成付款；超时重新询价                       |
| E9 | 多笔并发切换消耗备付金       | 冻结串行化 + 实时余额监控                                |

### 6.3 切换策略配置

| 配置项               | 说明                                     | 默认值       |
| -------------------- | ---------------------------------------- | ------------ |
| **最大重试次数**     | 单笔 OffRamp 最多尝试多少个渠道          | 3            |
| **切换超时**         | 单个渠道执行超时后触发切换               | 渠道级可配置 |
| **备付金预警线**     | 低于此值告警运营补充                     | 渠道级可配置 |
| **备付金冻结线**     | 低于此值自动停止路由到该渠道             | 渠道级可配置 |

### 7.1 风险矩阵

```
在这个方案里：
  · "渠道" = BB 名下的付款能力（自有银行/JRL/外部SP都是渠道）
  · "SP" = 在 EX 上注册的外部服务提供商

BB Channel 不是 SP，是 BB 内部渠道。
EX SP 是 SP，也是 BB 的渠道。
产品设计时要统一用"渠道"描述付款能力，不要用"SP"。
```

### 10.2 容易混淆的两层

```
❗ EX 的渠道中心 vs BB 的渠道管理

EX 渠道中心：管 EX SP（外部SP的注册、路由、对账）
BB 渠道管理：管 BB Channel（BB自有银行/JRL，BB内部管理）

两层要分开设计：
  · EX 渠道中心只看到 "BB Channel" 这一个整体 + N 个 EX SP
  · BB 内部 BB Channel 下面有几个子渠道，EX 不需要知道
  · 路由引擎在 EX 层，选的是 "BB Channel / EX SP-1 / EX SP-2"
  · BB Channel 内部用哪个子渠道，是 BB 自己的事
```

### 10.3 切换时的资金状态

```
❗ 三种资金安排模式的切换体验完全不同

预存法币渠道切换：
  · BB 指令 SP-1 付款 → 失败 → 备付金留在 SP-1 账户（没出去）
  · BB 换指令 SP-2 付款 → 用 SP-2 的备付金 → 不涉及资金追回
  · 切换 = 换一个指令对象 ← 简单

同行转账渠道切换：
  · BB 银行内部转 96 USD 给 SP-3 → SP-3 付款失败 → 96 USD 在 SP-3 账户
  · SP-3 需退回 96 USD（同行内部退回也是实时的）
  · BB 收到退回后 → 换指令其他渠道
  · 切换 = 等退回（秒级）+ 换指令对象 ← 较简单（比按笔转U快很多）

按笔转U渠道切换：
  · BB 转 100U 给 SP-1 → SP-1 失败 → 100U 在 SP-1 手里
  · BB 要重新转 100U 给 SP-2 → BB 需要额外的 U 流动性
  · SP-1 那笔 100U 什么时候退回？不确定
  · 切换 = 追回资金 + 重新转出 ← 复杂

→ PRD 里必须区分三种模式的切换流程
→ 按笔转U模式需要额外设计"U退回追踪"机制
→ 同行转账模式需要设计"法币退回确认"机制（但因同行实时，复杂度低）
```

### 10.4 客户端无感知的实现

```
❗ 客户端不做任何渠道相关的展示

不管后端用了几个渠道、切换了几次：
  · 客户看到的始终是：订单状态（处理中/成功/失败）
  · 不展示"尝试了渠道A失败，切换到渠道B"
  · 如果所有渠道都失败 → 客户只看到"付款失败，已退款"
  · 切换过程的耗时 = 客户的等待时间 → 需要设置合理的超时
```

### 10.5 BB Channel 在 EX 上怎么建模

```
❗ 这是跟技术团队讨论时的关键设计问题

选项A：BB Channel 作为一个"虚拟SP"注册在 EX 上
  · 优点：路由引擎统一处理，不需要特殊逻辑
  · 缺点：BB Channel 不是真的 SP，硬塞进 SP 模型会别扭

选项B：BB Channel 作为独立的渠道类型
  · 优点：概念清晰，跟 EX SP 是并列关系
  · 缺点：路由引擎要处理两种不同类型

建议：选项A，BB Channel 建模为一个特殊类型的"内部SP"
  · 路由引擎统一处理
  · 用 channel_type 字段区分 "INTERNAL" vs "EXTERNAL"
  · 但这需要跟技术确认可行性
```
