## 1. 业务背景

### 1.1 EX 平台定位

EX 是一家科技公司，提供 **Payment Platform as a Service（PPaaS）** 服务。EX 自身不持有支付牌照，不直接处理资金，而是作为**科技编排层**连接多个持牌服务提供商（SP），为租户和商户提供统一的支付产品体验。

### 1.2 为什么是多 SP 架构

单一 SP 无法覆盖所有业务场景：

| 业务能力         | 需要的资质/条件                             | 单一 SP 能否覆盖 |
| ---------------- | ------------------------------------------- | ----------------- |
| 数币钱包 + 承兑  | 数币牌照、承兑引擎、链上基础设施            | BB 能做           |
| 法币 VA 收款     | 各地区银行合作、法币收款牌照                | IPL 能做          |
| 法币 POBO 出款   | 各地区银行合作、法币汇款牌照                | IPL 能做          |
| 人民币结汇       | 中国跨境支付牌照、结汇通道                  | BB/IPL 都做不了   |
| VCC 发卡         | 发卡 BIN、卡组织合作                         | 专属发卡 SP       |

**核心结论：EX 必须支持多 SP 共存，不同 SP 承担不同业务线的底层能力。**

### 1.3 当前业务背景

目前 EX 平台上有两个核心 SP：

```
┌──────────────────────────────────────────────────────────────┐
│                        EX Platform                            │
│                     （科技编排层）                              │
└────────┬─────────────────────────────────┬───────────────────┘
         │                                 │
         ▼                                 ▼
┌─────────────────────┐         ┌─────────────────────┐
│     BB (SP)          │         │     IPL (SP)         │
│   承兑服务商         │         │   法币收付服务商      │
│                     │         │                     │
│  · 数币钱包          │         │  · VA 收款           │
│  · 承兑引擎          │         │  · POBO 出款         │
│  · BB 法币账户       │         │  · IPL 法币账户      │
│  · 代收付账户/XPAY   │         │                     │
│                     │         │                     │
│  承兑 + 数币 + 法币  │         │  法币收付款           │
└─────────────────────┘         └─────────────────────┘
```

**关键约束：**

- BB 和 IPL **各自独立**，账户不互通
- 商户在 BB 有 BB 账户，在 IPL 有 IPL 账户
- **承兑业务（OnRamp/OffRamp）**固定走 BB，资金在 BB 内部闭环
- **法币收付款（Payins/Payouts）**走 IPL
- 跨 SP 资金移动通过**独立的划转产品**，商户自行发起

## 2. 核心架构：多 SP 共存模型

### 2.1 架构总览

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           EX Platform                                    │
│                                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │  产品中心     │  │  账户中心     │  │  交易中心     │  │  清算中心   │ │
│  │  Product      │  │  Account      │  │  Transaction  │  │  Settlement │ │
│  └──────────────┘  └──────────────┘  └──────────────┘  └────────────┘ │
│                                                                         │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │                        SP 抽象层                                  │  │
│  │   · SP 注册 & 管理                                               │  │
│  │   · 产品 → SP 映射                                               │  │
│  │   · SP 能力声明（支持的币种、地区、交易类型）                      │  │
│  │   · SP 审核流程编排                                               │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│     ┌───────────┐    ┌───────────┐    ┌───────────┐    ┌───────────┐  │
│     │   BB      │    │   IPL     │    │  SP-C     │    │  SP-D     │  │
│     │  (承兑)   │    │ (法币)    │    │ (结汇)    │    │ (发卡)    │  │
│     └───────────┘    └───────────┘    └───────────┘    └───────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 设计原则

| # | 原则                           | 说明                                                                                     |
| - | ------------------------------ | ---------------------------------------------------------------------------------------- |
| 1 | **每个 SP 独立账户**          | 商户在每个 SP 下拥有独立账户，余额按 SP 区分，不自动合并                                 |
| 2 | **产品绑定 SP**               | 每个产品在当前阶段绑定唯一 SP（如承兑=BB，法币收付=IPL），未来支持多 SP 路由              |
| 3 | **SP 间不自动打通**           | SP 间资金不自动流转，需要通过独立的划转产品显式操作                                       |
| 4 | **SP 对商户透明（默认）**     | 商户端默认不暴露 SP 身份，只看到产品和余额。特殊场景除外（如平级 SP / 托管钱包）          |
| 5 | **SP 审核独立**               | 每个 SP 独立审核商户资料，一个 SP 通过不代表另一个 SP 也通过                              |
| 6 | **EX 不做合规审核**           | 合规审核由各 SP 负责，EX 只做流程编排和状态同步                                          |

## 3. 当前 SP 分工

### 3.1 BB — 数币资金中心 + 承兑引擎

BB 是 EX 平台的**数币资金中枢**，承担所有与数字货币相关的业务。

| 能力             | 说明                                                     |
| ---------------- | -------------------------------------------------------- |
| **数币钱包**    | 商户 USDT/USDC 钱包，支持链上充币/提币                   |
| **承兑引擎**    | USDT/USDC ↔ USD 等法币的实时兑换                         |
| **BB 法币账户** | 承兑资金落入的法币账户，支持法币充值/提现                 |
| **代收付账户**  | BB 自有的法币收付能力                                    |
| **XPAY 渠道**   | BB 内部 Channel，提供额外的 VA/POBO 能力（对外不可见）   |

**BB 承兑闭环原则：**

```
承兑资金必须在 BB 内部流转：

  OnRamp（买入数币）：BB 法币账户(USD) → BB 承兑引擎 → BB 数币钱包(USDT)
  OffRamp（卖出数币）：BB 数币钱包(USDT) → BB 承兑引擎 → BB 法币账户(USD)
  OffRamp一步到位 ：BB 数币钱包(USDT) → BB 承兑引擎 → 直接付到收款人银行

  ⚠ 承兑过程中不涉及 IPL 账户，不发生跨 SP 资金流转
```

### 3.2 IPL — 法币收付款

IPL 是 EX 平台的**法币收付款服务商**，承担传统法币场景。

| 能力              | 说明                                           |
| ----------------- | ---------------------------------------------- |
| **VA 收款**      | Virtual Account 收款，支持多币种同名入金       |
| **POBO 出款**    | Payment on Behalf Of，支持异名法币出款         |
| **IPL 法币账户** | 法币余额账户，支持法币充值/提现                |

**IPL 与 BB 的关系：**

- IPL 和 BB 是集团内部合作关系，但在 EX 架构中是**完全独立的两个 SP**
- 商户如果同时签约 BB 和 IPL，会拥有**两套独立的法币账户**
- IPL **不参与承兑流程**，BB **不依赖 IPL 的收付能力**
- 两者各自独立审核商户

### 3.3 为什么不打通

| 原因                   | 说明                                                                                                       |
| ---------------------- | ---------------------------------------------------------------------------------------------------------- |
| **架构清晰**          | 打通意味着承兑流程中嵌入跨 SP 划转逻辑，增加复杂度，且一旦任一 SP 故障影响另一个 SP 的业务                  |
| **合规独立**          | BB 和 IPL 分属不同牌照体系，资金混合会带来合规审计问题                                                      |
| **资金安全**          | BB 承兑资金在 BB 内部闭环，不暴露给其他 SP，降低资金风险                                                    |
| **Phase 1 简化**      | 跨 SP 自动划转需要建设统一清算、对账、差错处理等基础设施，Phase 1 暂不具备                                  |

**Phase 2 演进方向：** 承兑流程中支持自动跨 SP 划转（A2/B2 场景），前提是 EX 建成统一清算引擎。

## 4. 账户模型

### 4.1 账户按 SP 区分

商户在每个 SP 下拥有独立账户。账户归属于 SP，不归属于 EX。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     商户账户视图（Phase 1）                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌──────────────────────────────┐   ┌──────────────────────────────┐       │
│  │        BB 账户体系            │   │       IPL 账户体系            │       │
│  │                              │   │                              │       │
│  │  ┌──────────────────────┐   │   │  ┌──────────────────────┐   │       │
│  │  │ BB 数币钱包           │   │   │  │ IPL 法币账户          │   │       │
│  │  │ USDT: 5,000          │   │   │  │ USD: 3,000           │   │       │
│  │  │ USDC: 2,000          │   │   │  │ EUR: 1,500           │   │       │
│  │  └──────────────────────┘   │   │  └──────────────────────┘   │       │
│  │                              │   │                              │       │
│  │  ┌──────────────────────┐   │   │  能力：                       │       │
│  │  │ BB 法币账户           │   │   │  · VA 收款                   │       │
│  │  │ USD: 7,000           │   │   │  · POBO 出款                 │       │
│  │  │ EUR: 1,000           │   │   │  · 法币充值 / 提现           │       │
│  │  └──────────────────────┘   │   │                              │       │
│  │                              │   └──────────────────────────────┘       │
│  │  能力：                       │                                         │
│  │  · 充币 / 提币               │   ┌──────────────────────────────┐       │
│  │  · 承兑 (OnRamp/OffRamp)    │   │  未来：SP-C 账户体系（鲲鹏）  │       │
│  │  · BB 法币充值 / 提现       │   │                              │       │
│  │  · BB POBO 出款             │   │  CNY 托管钱包: ¥0            │       │
│  └──────────────────────────────┘   │  ...                         │       │
│                                     └──────────────────────────────┘       │
│                                                                             │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─           │
│  划转通道：BB 法币 ↔ IPL 法币（商户自行发起，独立产品）                       │
│  未来：BB 法币 ↔ SP-C 托管钱包（商户自行发起）                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 4.2 账户生命周期

| 阶段           | 触发条件                 | 系统行为                                                           |
| -------------- | ------------------------ | ------------------------------------------------------------------ |
| **账户创建**  | 商户产品开通，SP 审核通过 | 在对应 SP 下自动创建账户（如开通承兑→创建 BB 数币钱包 + BB 法币账户） |
| **账户激活**  | 创建即激活               | 账户可接收和发起交易                                                |
| **余额归属**  | 交易入账                 | 余额归属于具体 SP 账户，不跨 SP 合并                                |
| **账户冻结**  | SP 合规要求              | 该 SP 下账户冻结，不影响其他 SP 下的账户                            |

### 4.3 为什么会有多个法币账户

商户可能同时拥有 **BB 法币账户**和 **IPL 法币账户**，看起来有两个 USD 余额，但用途不同：

| 账户           | 资金来源                         | 用途                                           |
| -------------- | -------------------------------- | ---------------------------------------------- |
| BB 法币账户    | 承兑卖出数币所得、BB 法币充值    | 承兑买入数币、BB 法币提现、BB POBO 出款        |
| IPL 法币账户   | IPL VA 收款、IPL 法币充值        | IPL POBO 出款、IPL 法币提现                    |

**关键规则：** 承兑只认 BB 法币账户。如果商户想用 IPL 法币账户的钱买数币，必须先通过划转产品将资金从 IPL 转到 BB。

## 5. 资金流转规则

### 5.1 SP 内部流转（自动）

```
BB 内部（承兑闭环）：
  BB 法币账户 ←→ BB 承兑引擎 ←→ BB 数币钱包
  · 买入数币：BB 法币 → 承兑 → BB 数币钱包
  · 卖出数币：BB 数币钱包 → 承兑 → BB 法币

IPL 内部（法币收付闭环）：
  IPL VA → IPL 法币账户 → IPL POBO 出款
  · 收款入账：VA 收款 → IPL 法币账户
  · 付款出金：IPL 法币账户 → POBO → 收款人
```

### 5.2 SP 间流转（需商户主动发起）

```
划转产品（FIAT_TRANSFER）：
  BB 法币账户(USD) ←→ IPL 法币账户(USD)

  · 商户在 MP 端主动发起
  · 同币种划转，不涉及换汇
  · 前提：商户同时拥有 BB 和 IPL 法币账户
  · 独立产品，不属于承兑或收付款范畴
```

### 5.3 资金流转矩阵

| 资金流向                          | 触发方式     | 涉及 SP  | 产品归属         |
| --------------------------------- | ------------ | -------- | ---------------- |
| 链上 → BB 数币钱包                | 自动（充币） | BB       | 数币钱包         |
| BB 数币钱包 → 链上                | 商户发起     | BB       | 数币钱包         |
| 银行 → BB 法币账户                | 同名转账     | BB       | BB 法币账户      |
| 银行 → IPL 法币账户               | 同名转账     | IPL      | IPL 法币账户     |
| BB 法币 ↔ BB 数币                 | 商户发起     | BB       | 承兑（OnRamp/OffRamp） |
| BB 数币 → 承兑 → 收款人银行       | 商户发起     | BB       | 承兑（OffRamp 一步到位） |
| BB 法币账户 → 收款人银行           | 商户发起     | BB       | BB POBO          |
| IPL 法币账户 → 收款人银行          | 商户发起     | IPL      | IPL POBO         |
| BB 法币 ↔ IPL 法币                | 商户发起     | BB + IPL | 划转（FIAT_TRANSFER） |
| BB 法币 → SP-C 托管钱包（未来）   | 商户发起     | BB + SP-C | 划转             |

## 6. SP 扩展模型：平级 SP

### 6.1 两类 SP 关系

从 `ex-offramp-v2-reasoning.md` 的推理中得出：不是所有 SP 都在 BB 下面。当 BB 的能力不足以覆盖某些业务场景时，需要引入与 BB **平级**的 SP。

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  类型一：BB 渠道型 SP（BB 能力范围内）                               │
│                                                                     │
│    客户 → BB → SP 执行                                              │
│    · 客户不感知 SP                                                   │
│    · BB 做资金中枢，SP 是 BB 的执行方                                │
│    · EX 自动路由，客户无需操作                                       │
│    · 例：BB 内部的 XPAY 渠道、未来 BB 下接入的付款渠道              │
│                                                                     │
│  类型二：平级 SP（BB 能力范围外）                                     │
│                                                                     │
│    客户 → 自己操作转账 → SP                                          │
│    · 客户感知 SP（看到独立账户 / 托管钱包）                          │
│    · SP 跟 BB 平级，不是 BB 的下级                                   │
│    · 客户自己选择何时、是否转账                                      │
│    · 例：人民币结汇 SP（鲲鹏等）、特殊地区法币出金 SP               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 为什么需要平级 SP

| 场景               | BB 能否处理 | 原因                               | 需要的 SP 类型   |
| ------------------ | ----------- | ---------------------------------- | ---------------- |
| USDT ↔ USD 承兑    | ✅          | BB 核心能力                        | —                |
| USD POBO 出款      | ✅          | BB 有代收付账户                    | —                |
| 人民币结汇         | ❌          | 数币相关业务在中国合规限制         | 平级 SP（如鲲鹏）|
| 特殊地区法币出金   | ❌          | BB 无当地牌照/银行合作             | 平级 SP          |

**核心逻辑：BB 做不了的业务，必须由独立 SP 承接。资金不能先进 BB 再出去（BB 不是这笔业务的执行方），所以商户必须主动将资金转到该 SP。**

### 6.3 平级 SP 的运作方式

以人民币结汇为例（鲲鹏 SP）：

```
前提：
  · 商户在 BB 有 USD 余额（OffRamp 卖出数币所得）
  · 鲲鹏 SP 有人民币结汇能力
  · 商户在鲲鹏下有一个托管钱包 / 账户

流程：
  1. 商户在 BB 法币账户中有 10,000 USD
  2. 商户在 MP 端看到"鲲鹏账户（人民币结汇）"
  3. 商户主动发起划转：BB 法币账户 → 鲲鹏账户（10,000 USD）
  4. 资金到达鲲鹏 → 鲲鹏执行 USD → CNY 结汇
  5. 商户在鲲鹏账户中看到 CNY 余额
  6. 商户发起提现 → CNY 到国内银行卡

关键：
  · 步骤 3 是商户自己发起的（不是系统自动路由）
  · 商户明确知道资金去了鲲鹏（SP 暴露）
  · BB 在此过程中只是"资金来源"，不是"执行方"
```

### 6.4 平级 SP vs BB 渠道型 SP 对比

| 维度                 | BB 渠道型 SP（类型一）       | 平级 SP（类型二）              |
| -------------------- | ---------------------------- | ------------------------------ |
| **客户感知**        | ❌ 不感知                    | ✅ 感知（看到独立账户）        |
| **资金路径**        | 客户 → BB → SP 执行         | 客户自己划转 → SP              |
| **谁做路由**        | EX 自动路由                  | 客户自己选                     |
| **与 BB 关系**      | BB 下级（BB 的渠道）         | BB 平级（独立 SP）             |
| **为什么需要**      | 扩大 BB 的覆盖面             | BB **做不了**的业务            |
| **承兑方**          | BB（或 SP 自己，看资金安排） | SP 自己                        |
| **EX 的角色**       | 路由引擎自动选渠道           | 提供账户入口 + 划转工具        |
| **典型场景**        | OffRamp USD/HKD/EUR          | 人民币结汇、特殊地区法币       |

### 6.5 暴露平级 SP 不会导致客户绕过 EX

为什么暴露 SP 身份是安全的：

- 平级 SP 提供的是**特殊能力**（如人民币结汇），不是通用 OffRamp
- 客户不能只用平级 SP 完成全部业务（还需要 BB 做 USDT → USD 承兑）
- BB + 平级 SP 是**互补关系**，不是替代关系
- 客户仍然需要 EX 作为入口和管理平台

## 7. Phase 规划

### 7.1 Phase 1（当前）

> **BB 承兑内部闭环 + IPL 独立法币收付款，账户按 SP 区分**

| 能力                 | 状态 | 说明                                             |
| -------------------- | ---- | ------------------------------------------------ |
| BB 承兑（OnRamp/OffRamp） | ✅ | BB 内部闭环，数币钱包 ↔ BB 法币账户               |
| BB 数币钱包          | ✅   | 链上充币/提币                                     |
| BB 法币账户          | ✅   | 法币充值/提现/POBO                                |
| IPL 法币账户         | ✅   | VA 收款 / POBO 出款 / 法币充值提现                |
| 划转（BB ↔ IPL）     | ✅   | 独立产品，商户自行发起                             |
| 每个产品绑定唯一 SP  | ✅   | 无需路由                                           |

### 7.2 Phase 2（近期）

> **引入平级 SP + 跨 SP 划转扩展**

| 能力                          | 说明                                                       |
| ----------------------------- | ---------------------------------------------------------- |
| 平级 SP 接入（如鲲鹏）       | 人民币结汇等 BB 做不了的场景，商户端暴露为独立账户          |
| 划转扩展                      | BB ↔ SP-C 划转，IPL ↔ SP-C 划转                            |
| 承兑流程中自动跨 SP 划转       | A2/B2 场景：OnRamp/OffRamp 可自动从 IPL 划转到 BB（或反向） |
| 多 SP 承兑（可选）            | 不同 SP 提供承兑服务，EX 统一汇率引擎做路由                 |

### 7.3 Phase 3（远期）

> **统一账户视图 + 智能路由**

| 能力                    | 说明                                                               |
| ----------------------- | ------------------------------------------------------------------ |
| 统一账户视图            | 商户端看到总余额（按币种），底层按 SP 拆分（商户不感知）            |
| EX 统一汇率引擎        | 比较多个 SP 汇率，智能路由到最优 SP                                  |
| 多 SP 产品路由          | 同一产品可由多个 SP 提供，EX 根据费率/成功率/额度自动选择            |
| 统一清算                | EX 统一清算引擎，跨 SP 自动轧差                                      |

### 7.4 架构演进路线

```
Phase 1（当前）                Phase 2（近期）               Phase 3（远期）
┌──────────────┐            ┌──────────────┐            ┌──────────────┐
│ BB + IPL      │            │ BB + IPL      │            │ BB + IPL      │
│ 各自独立      │   ──→     │ + 平级 SP     │   ──→     │ + N个 SP     │
│ 手动划转      │            │ 跨SP自动划转  │            │ 统一路由     │
│ 无路由        │            │ 初步路由      │            │ 智能路由     │
│ 按SP看余额    │            │ 按SP看余额    │            │ 统一余额视图 │
└──────────────┘            └──────────────┘            └──────────────┘
```

## 8. 术语定义

| 中文 | 英文 | 定义 | 粤语（广东话） |
| ---- | ---- | ---- | -------------- |
| 服务提供商 | Service Provider (SP) | 在 EX 平台上提供底层支付能力的持牌机构 | 服務提供商 |
| 租户 | Tenant (TP) | 购买 SP 能力，为商户提供服务的机构（EX 的客户） | 租戶 |
| 商户 | Merchant (MP) | 最终使用支付服务的企业客户 | 商戶 |
| 承兑 | Exchange / OnRamp / OffRamp | 数字货币与法币之间的兑换 | 承兌 |
| 数币钱包 | Crypto Wallet | 商户在 BB 开设的数字货币钱包 | 數幣錢包 |
| 法币账户 | Fiat Account | 商户在某 SP 下开设的法币余额账户 | 法幣賬戶 |
| 充币 | Crypto Deposit | 从外部链上地址将数币转入平台钱包 | 充幣 |
| 提币 | Crypto Withdrawal | 从平台钱包将数币转出到外部链上地址 | 提幣 |
| 法币充值 | Fiat Deposit | 从外部银行账户将法币转入平台法币账户（同名） | 法幣充值 |
| 法币提现 | Fiat Withdrawal | 从平台法币账户将法币提现到商户银行账户（同名） | 法幣提現 |
| 划转 | Fiat Transfer | 跨 SP 法币账户间的资金转移 | 劃轉 |
| 平级 SP | Peer SP | 与 BB 平级的独立 SP，提供 BB 做不了的特殊能力 | 平級 SP |
| 托管钱包 | Custodial Wallet | 平级 SP 为商户开设的账户，商户端可见 | 託管錢包 |
| VA | Virtual Account | 虚拟账户，用于法币收款 | 虛擬賬戶 |
| POBO | Payment on Behalf Of | 代付，支持异名法币出款 | 代付 |
