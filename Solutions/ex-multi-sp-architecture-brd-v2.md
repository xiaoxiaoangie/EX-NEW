## 1. 起点：EX 是什么

### 1.1 三个基本事实

```
事实❶：EX 是一家科技公司
  · EX 自己没有支付牌照
  · EX 不持有客户资金
  · EX 不做合规审核

事实❷：EX 提供 Payment Platform as a Service（PPaaS）
  · 连接持牌 SP，为租户/商户提供统一的支付产品
  · EX 的价值 = 编排 + 路由 + 统一体验

事实❸：EX 目前有两个核心 SP
  · BB — 有数币牌照、承兑引擎、链上基础设施、法币收付能力
  · IPL — 有法币收款牌照（VA）、法币汇款牌照（POBO）
```

### 1.2 一个直觉问题

> 如果 EX 自己不持牌、不碰钱，那商户的钱在谁那里？
> 答案：在 SP 那里。哪个 SP 提供服务，钱就在哪个 SP 的账户体系里。

这个答案是后续所有推理的起点。

## 2. 推理❶：为什么需要多个 SP

### 2.1 出发点：列出所有需要的业务能力

| 业务能力        | 需要的资质/条件                  |
| --------------- | -------------------------------- |
| 数币钱包 + 承兑 | 数币牌照、承兑引擎、链上基础设施 |
| 法币 VA 收款    | 各地区银行合作、法币收款牌照     |
| 法币 POBO 出款  | 各地区银行合作、法币汇款牌照     |
| 人民币结汇      | 中国跨境支付牌照、结汇通道       |
| VCC 发卡        | 发卡 BIN、卡组织合作             |

### 2.2 逐个检验：单个 SP 能否覆盖全部？

```
检验 BB：
  · 数币钱包 + 承兑 → ✅ BB 核心能力
  · 法币 VA 收款 → ⚠️ BB 有一定能力（代收付+XPAY），但覆盖面有限
  · 法币 POBO 出款 → ⚠️ 同上
  · 人民币结汇 → ❌ BB 做不了（数币相关业务在中国合规敏感）
  · VCC 发卡 → ❌ BB 没有发卡 BIN

  结论：BB 不能覆盖全部 ❌

检验 IPL：
  · 数币钱包 + 承兑 → ❌ IPL 没有数币牌照
  · 法币 VA 收款 → ✅
  · 法币 POBO 出款 → ✅
  · 人民币结汇 → ❌
  · VCC 发卡 → ❌

  结论：IPL 也不能覆盖全部 ❌

任何单一 SP 都有能力边界。
```

### 2.3 推导结论

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  推导❶：EX 必须支持多 SP 共存                                       │
│                                                                     │
│  · 没有任何一个 SP 能覆盖所有业务能力                               │
│  · 不同业务需要不同资质 → 需要不同 SP 承担                          │
│  · EX 作为平台，必须能接入 N 个 SP，按业务类型分配                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. 推理❷：数币余额应该放在哪里

### 3.1 出发点：谁有能力管理数币？

```
数币钱包需要什么？
  · 数币牌照（合规前提）
  · 链上基础设施（地址生成、链上监控、热/冷钱包管理）
  · 承兑引擎（USDT ↔ USD 等法币的实时兑换）
  · AML/KYT 链上反洗钱能力

谁有这些能力？
  · BB → ✅ 全部具备
  · IPL → ❌ 没有数币牌照，没有链上基础设施
  · 鲲鹏 → ❌ 纯法币机构
  · 其他 SP → ❌（目前平台上没有第二个有数币能力的 SP）
```

### 3.2 假设：如果让多个 SP 都有数币钱包？

```
假设将来有多个 SP 都有数币能力（BB、SP-X、SP-Y 都能开数币钱包）：

技术层面：
  · 数币没有 nesting 这种强监管约束（不像法币账户间转账受限）
  · 链上转账自由，技术上可以让多个 SP 都有数币钱包

那为什么不让多个 SP 都有数币钱包？
  → 需要从商业策略层面分析
```

### 3.3 商业策略约束：EX 前期的生存问题

```
约束❶：客户短期不应感知背后是哪个 SP（数币）
  · 原因：防止租户绕过 EX 直接找 SP 签约
  · EX 前期不具备多 SP 生态吸引力
  · 客户端只看到"余额 100U"，不看到"BB 余额 100U"

约束❷：EX 在数币承兑领域还不具有特别优势
  · 技术层护城河正在被 AI/Vibe Coding 抹平
  · 1-2 年后客户自己几天就能对接一个 SP
  · 数据网络效应有鸡生蛋问题（需要交易量→需要更多 SP→但只有 IPL+BB）

约束❸：如果过早暴露多个 SP
  · 客户发现 BB 和 SP-X 都能做承兑
  · 客户会问：为什么我要通过 EX？我直接找 BB 不就行了？
  · EX 的中间层价值被质疑
  → 客户流失风险

约束❹：数币没有 nesting 监管约束
  · 法币之所以要按 SP 独立账户，是因为监管限制 SP 间资金流动
  · 数币没有这个约束 → 链上转账自由
  → 但正因如此，EX 更需要主动建立"唯一数币资金中心"的架构
  → 否则客户可以随意把钱转到任何 SP，EX 的编排价值更弱
```

### 3.4 推导：BB 作为唯一数币资金中心的必然性

```
从商业策略推导：

  假设 A：多个 SP 都有数币钱包
    → 客户看到"BB 数币余额 3000U" + "SP-X 数币余额 2000U"
    → 客户意识到有多个承兑 SP 可选
    → 客户可能绕过 EX 直接找 SP-X
    → EX 失去客户 ❌

  假设 B：只有 BB 有数币钱包（唯一数币资金中心）
    → 客户只看到"数币余额 5000U"（不暴露 BB）
    → 客户不知道背后是哪个 SP
    → 客户必须通过 EX 承兑
    → EX 保持客户粘性 ✅

  同时满足：
    · BB 是唯一有数币牌照+链上能力+承兑引擎的 SP（技术前提）
    · 数币没有 nesting 监管约束（反而更需要主动建立唯一中心）
    · EX 前期需要保护客户不被 SP 抢走（商业前提）

  → BB 作为唯一数币资金中心，是技术+商业双重约束的必然结论
```

### 3.5 与法币账户的对比

```
为什么法币可以多 SP，数币只能单 SP？

  法币：
    · 监管约束（nesting）→ 必须 按 SP 独立账户
    · 客户理解"我在不同账户有不同余额"
    · 不涉及承兑汇率问题
    → 多 SP 是监管要求，不是选择

  数币：
    · 无监管约束 → 可以多 SP，也可以单 SP
    · 但承兑涉及汇率 → 多 SP 导致汇率不一致
    · 且承兑是 EX 的核心业务 → 暴露多 SP = 暴露核心供应商
    → 单 SP 是商业选择，不是监管要求

  关键区别：
    · 法币多 SP = 监管强制
    · 数币单 SP = 商业策略
```

### 3.6 推导结论

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  推导❷：数币余额只在 BB                                             │
│                                                                     │
│  · BB 是唯一有数币牌照+链上能力+承兑引擎的 SP（技术前提）           │
│  · 数币无 nesting 监管约束，但 EX 主动建立唯一资金中心（商业策略）  │
│  · 过早暴露多 SP 会导致客户绕过 EX 直接找 SP                        │
│  · 所有数币（USDT/USDC）统一由 BB 管理                              │
│  · 客户要做承兑，都基于 BB 作为唯一数币资金中心的能力                │
│                                                                     │
│  这个结论在可预见的未来不会改变。                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 4. 推理❸：法币余额能不能只放一个 SP

### 4.1 出发点：法币账户的现状

```
目前有两个 SP 都有法币账户能力：
  · BB 法币账户 — 承兑资金落地、法币充值/提现
  · IPL 法币账户 — VA 收款、POBO 出款、法币充值/提现

商户可能同时签约 BB 和 IPL → 会拥有两个法币账户
→ 看起来有两个 USD 余额

这正常吗？能不能合并成一个？
```

### 4.2 假设A：把所有法币都放在 BB

```
如果商户只用 BB 法币账户：
  · IPL VA 收到的钱 → 需要先转到 BB 法币账户
  · IPL POBO 出款 → 需要从 BB 法币账户取钱给 IPL 执行

问题①：IPL 有自己的商业计划和独立发展
  · BB 可能只是 IPL 的其中一条通道
  · IPL 还有别的通道关系需要维护
  · 如果所有法币都放 BB，IPL 的其他通道关系无法利用
  → IPL 的商业利益受损 ❌

问题②：BB 的出款渠道目前还没有 IPL 丰富
  · IPL 通过 BB 出款技术可行（IPL 和 BB 在可以 nesting 的银行开户）
  · 但 BB 的出款渠道覆盖有限
  · IPL 自己的出款渠道更丰富
  → 渠道能力受限 ❌

问题③：nesting 在某些银行可行，但不是所有银行
  · IPL 和 BB 目前在可以 nesting 的银行开户
  · 但不是所有银行都允许 nesting
  · 如果只用 BB 账户，IPL 的银行关系受限

结论：技术上部分可行，但商业和渠道能力受限 ⚠️
```

### 4.3 假设B：把所有法币都放在 IPL

```
如果商户只用 IPL 法币账户：
  · BB 承兑后法币落入 IPL 账户
  · 所有法币资金都在 IPL

核心问题：IPL 是 MSO（金钱服务经营者），合规政策限制
  · IPL 作为 MSO，不能结束数币背景的资金
  · 承兑（OnRamp/OffRamp）的资金来源/去向都是数币
  · 合规政策不太允许 MSO 账户承接数币背景资金
  → 合规障碍 ❌

额外问题：
  · BB 承兑引擎需要法币账户做即时扣款/入账
  · 如果法币在 IPL，BB 承兑时需要跨 SP 调用 → 延迟+失败风险

结论：合规不允许，不可行 ❌
```

### 4.4 假设C：EX 自己开一个统一法币账户

```
EX 自己没有牌照 → 不能持有客户资金 → 不可能有 EX 的法币账户 ❌
（事实❶已明确：EX 不持牌、不碰钱）
```

### 4.5 推导结论

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  推导❸：法币余额必须按 SP 区分（多 SP 独立账户）                    │
│                                                                     │
│  原因不是单纯的 nesting 限制，而是：                                 │
│                                                                     │
│  · 假设A（全放BB）→ IPL 商业独立性受损 + BB 出款渠道不够丰富        │
│  · 假设B（全放IPL）→ IPL 是 MSO，不能接数币背景资金（合规障碍）     │
│  · IPL 和 BB 在可以 nesting 的银行开户，技术上部分可行              │
│    但商业和合规约束决定了必须各自独立                                │
│                                                                     │
│  结论：                                                             │
│  · BB 有 BB 法币账户（承兑资金落地）                                │
│  · IPL 有 IPL 法币账户（VA 收款、POBO 出款）                        │
│  · 各自独立，不能合并                                               │
│                                                                     │
│  推论：商户可能看到两个 USD 余额（BB USD + IPL USD）                 │
│  → 这不是设计缺陷，是商业+合规约束的必然结果                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 5. 推理❹：承兑流程中要不要打通法币账户

### 5.1 出发点：一个真实场景

```
场景：商户在 IPL 有 10,000 USD（VA 收款所得），想买入 USDT

  · 买 USDT = OnRamp = 承兑业务 = 走 BB
  · 但钱在 IPL 法币账户
  · BB 承兑引擎需要扣 BB 法币账户的钱

  → 是否应该在 OnRamp 流程中自动从 IPL 划转到 BB？
```

### 5.2 如果打通（自动跨 SP 划转）

```
OnRamp 流程变成：
  IPL 法币(USD) →[自动划转]→ BB 法币(USD) → BB 承兑 → BB 数币钱包(USDT)

技术上可以打通，但核心问题是：法币-数币 SP 是否接受？

  法币 SP（IPL）视角：
    · IPL 把资金转给 BB 用于承兑
    · IPL 是否愿意？是否有合规顾虑？
    · IPL 是 MSO，合规政策对数币相关业务有限制

  数币 SP（BB）视角：
    · BB 接收来自 IPL 的资金用于承兑
    · BB 是否愿意接受非直接客户的资金？
    · BB 是否愿意承担跨 SP 的合规责任？

  关键约束：
    · 不是 EX 技术能力问题
    · 是 SP 之间的商业协议和合规政策问题
    · 需要法币 SP 和数币 SP 双方都接受这种模式

结论：可以打通，但取决于法币-数币 SP 是否接受 ⚠️
```

### 5.3 Phase 1 的选择

```
Phase 1 选择不打通的原因：

  · 法币-数币 SP 之间的协议尚未明确
  · SP 双方的合规政策需要评估
  · 自动跨 SP 划转涉及责任划分（谁对客户负责？）
  · Phase 1 优先保证核心流程稳定

  → 先用独立划转产品，让商户手动操作
  → 待 SP 协议明确后，Phase 2 再考虑自动打通
```

### 5.4 如果不打通（商户手动划转）

```
商户操作：
  1. 发起划转：IPL USD → BB USD（10,000 USD）
  2. 划转完成后，发起 OnRamp：BB USD → BB USDT

trade-off：
  · 用少量体验损失换取架构清晰
  · SP 之间责任边界清晰
  · 合规风险可控 ✅
```

### 5.5 推导结论

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  推导❹：Phase 1 不打通法币账户，跨 SP 划转是独立产品                 │
│                                                                     │
│  · 承兑只认 BB 法币账户，不涉及 IPL                                  │
│  · 划转是独立产品（FIAT_TRANSFER），商户自行发起                    │
│  · 打通取决于法币-数币 SP 是否接受，不是技术问题                    │
│  · Phase 2 待 SP 协议明确后，可支持自动跨 SP 划转                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 6. 推理❺：BB 做不了的业务怎么办

### 6.1 出发点：一个 BB 做不了的场景

```
场景：商户想把 USDT 背景的USD 变成 CNY（人民币）

  · BB 不能直接做结汇业务
  · IPL 也做不了
  · 但有些 SP（如鲲鹏）可以做人民币结汇
```

### 6.2 检验现有模型

```
现有模型：客户资金先进 BB → BB 选渠道执行

应用到人民币结汇：
  · BB 做不了 → 不能走"BB→渠道"路径 ❌
  · BB 指令下级 SP = BB 代客结汇 = BB 承担人民币合规风险 ❌

现有模型三个假设全部被打破：
  假设① 客户不感知 SP → 结汇场景必须暴露 SP ❌
  假设② 资金先进 BB → BB 做不了，资金不能走 BB ❌
  假设③ SP 是 BB 渠道 → 结汇 SP 不是 BB 的下级 ❌
```

### 6.3 推导：需要平级 SP

```
当客户需要的能力超出 BB 的边界时：
  · BB 不能做 → 不能走"BB→渠道"路径
  · 需要一个有这种能力的 SP 来做
  · 资金不应该先进 BB 再出去（BB 不是执行方）
  · 商户自己决定是否、何时把钱转给这个 SP

  → 这个 SP 跟 BB 是平级的独立服务提供方
```

### 6.4 人民币结汇的风险转移

```
核心设计原则：将涉及人民币的合规风险直接转移给 SP

  · 商户主动划转法币到结汇 SP 账户（不是系统自动路由）
  · 从划转那一刻起，资金归结汇 SP 管理
  · 结汇 SP（如鲲鹏）自行承担人民币合规风险
  · EX 和 BB 不参与结汇执行，只提供账户入口+划转工具
```

### 6.5 推导结论

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  推导❺：BB 做不了的业务，由平级 SP 承接                              │
│                                                                     │
│  · 平级 SP 跟 BB 平级，不是 BB 的下级渠道                           │
│  · 客户感知 SP（看到独立账户），自己决定是否转账                     │
│  · 涉及特殊合规风险（如人民币）直接转移给 SP                        │
│  · EX/BB 不承担 → 只提供入口和划转工具                              │
│                                                                     │
│  为什么暴露平级 SP 不会导致客户绕过 EX：                             │
│  · 平级 SP 提供特殊能力（如人民币结汇），不是通用 OffRamp            │
│  · 客户还需要 BB 做 USDT→USD 承兑 → BB+平级SP 是互补，非替代       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 7. 推理❻：EX 自己的角色是什么

### 7.1 出发点：EX 不持牌，那 EX 做什么？

```
已经推导出：
  · 数币 → BB 管
  · 法币 → 各 SP 各自管
  · 承兑 → BB 做
  · 法币收付 → IPL 做
  · 特殊业务 → 平级 SP 做

那 EX 做什么？
```

### 7.2 EX 的三层价值

```
第1层：渠道对接平台
  · EX 帮 BB/IPL 对接底层渠道
  · 通过 EX 接入的渠道都注册为 SP
  · 当前仅开放给 BB 和 IPL，后续可开放给其他租户
  · EX 的价值 = SP 池的规模效应

第2层：科技编排层
  · 统一产品定义（承兑、收付款、划转等）
  · SP 抽象层（SP 注册、能力声明、审核编排）
  · 路由引擎（当前简单映射，未来智能路由）

第3层：合规运营平台（核心价值）
  · 统一进件审核流程编排
  · 合规规则引擎
  · 风控决策中台
  · 监管报告自动化
  · SP 承担法律合规责任，EX 提供合规基础设施工具
```

### 7.3 推导结论

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  推导❻：EX 是渠道对接+科技编排+合规运营平台                         │
│                                                                     │
│  · EX 不碰钱、不持牌 → 但提供统一的产品体验和合规基础设施           │
│  · 通过 EX 接入的渠道注册为 SP，当前仅开放 BB 和 IPL                │
│  · 后续可直接开放给其他租户 → 新租户无需重新对接渠道                 │
│  · SP 审核独立 → 每个 SP 独立审核商户，EX 做流程编排                │
│  · EX 不做合规审核 → 合规责任由各 SP 承担                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 8. 汇总：推导出的架构全貌

### 8.1 六个推导结论汇总

| #  | 推导结论                                | 推理依据                              |
| -- | --------------------------------------- | ------------------------------------- |
| ❶ | **EX 必须支持多 SP 共存**         | 没有任何单一 SP 能覆盖所有业务能力    |
| ❷ | **数币余额只在 BB**               | BB 是唯一有数币牌照+承兑引擎的 SP     |
| ❸ | **法币余额按 SP 区分**            | 法币 SP 间打款涉及 nesting，必须独立  |
| ❹ | **Phase 1 不打通法币账户**        | 跨 SP 清算基础设施未建成              |
| ❺ | **BB 做不了的业务由平级 SP 承接** | BB 能力有边界，某些业务必须独立 SP 做 |
| ❻ | **EX 是渠道对接+编排+合规平台**   | EX 不持牌，价值在编排和合规工具       |

## 10. 资金流转规则（推导结果）

### 10.1 SP 内部流转（自动）

```
BB 内部（承兑闭环，推导❹）：
  买入数币：BB 法币 → 承兑 → BB 数币钱包
  卖出数币：BB 数币钱包 → 承兑 → BB 法币
  ⚠ 不涉及 IPL 账户

IPL 内部（法币收付闭环）：
  收款入账：VA 收款 → IPL 法币账户
  付款出金：IPL 法币账户 → POBO → 收款人
```

### 10.2 SP 间流转（商户主动发起，推导❹）

```
划转产品（FIAT_TRANSFER）：
  BB 法币账户(USD) ←→ IPL 法币账户(USD)
  · 商户在 MP 端主动发起
  · 同币种划转，不涉及换汇
  · 独立产品，不属于承兑或收付款范畴
```

### 10.3 资金流转矩阵

| 资金流向                      | 触发方式     | 涉及 SP | 产品归属                     |
| ----------------------------- | ------------ | ------- | ---------------------------- |
| 链上 → BB 数币钱包           | 自动（充币） | BB      | 数币钱包-充币                |
| BB 数币钱包 → 链上           | 商户发起     | BB      | 数币钱包-提币                |
| 银行 → BB 法币账户           | 同名转账     | BB      | BB 法币账户-充值             |
| 银行 → IPL 法币账户          | 同名转账     | IPL     | IPL 法币账户-充值            |
| BB 法币 ↔ BB 数币            | 商户发起     | BB      | 承兑                         |
| BB 数币 → 承兑 → 收款人银行 | 商户发起     | BB      | OffRamp 一步到位             |
| BB 法币 → 收款人银行         | 商户发起     | BB      | BB 法币账户-BB 提现（含pobo) |
| IPL 法币 → 收款人银行        | 商户发起     | IPL     | IPL 法币账户提现             |
| BB 法币 ↔ IPL 法币           | 商户发起     | BB+IPL  | 划转                         |
| BB 法币 → 平级SP账户         | 商户发起     | BB+SP-C | 划转                         |
