## 一、行业痛点：支付机构"开户难"

### 1.1 现状

全球范围内，**新申请或刚购买牌照的 MSO / PSP 面临严重的银行账户困境**：

| 痛点 | 说明 |
|------|------|
| **传统银行拒绝开户** | 银行将 MSO/PSP 归为高风险客户（money laundering risk），新牌照机构几乎无法通过银行合规审核 |
| **审核周期极长** | 即使部分银行愿意受理，审核周期通常 3~6 个月，严重影响业务启动 |
| **Nesting 合规壁垒** | 其他支付机构（PSP/EMI）也不敢给同行开 VA——监管层面担忧"Nesting"（支付机构嵌套），大部分 PSP 的合规政策直接禁止给支付机构开户 |
| **业务无法启动** | 没有银行账户 = 无法接收结算款 = 无法开展收单/汇款业务。**账户问题是支付机构的生死问题** |

### 1.2 受影响群体

- **新申请的 MSO**：刚拿到牌照，没有银行关系和交易历史
- **刚买牌的 MSO**：通过收购获得牌照，银行对新股东/新管理层存疑
- **小型 PSP / EMI**：规模不够大，银行不愿意承担合规成本
- **跨境支付公司**：涉及高风险走廊（如非洲、东南亚），银行更加谨慎

### 1.3 为什么 Moolahgo 能解决

**Moolahgo 作为 MAS 持牌 Major Payment Institution，具备独特优势：**

| 优势 | 说明 |
|------|------|
| **合规框架允许** | Moolahgo 的 MAS MPI 牌照和合规框架支持为支付机构提供 VA 服务，不构成 Nesting |
| **风控能力成熟** | 拥有成熟的 AML/CFT 风控体系，能够有效管控支付机构客户的风险 |
| **多币种 VA** | 已支持 14+ 货币（含 SGD、USD、EUR、GBP、IDR、PHP、VND 等），覆盖东南亚核心市场 |
| **ASEAN Treasury Network** | 已建成东南亚资金网络，具备跨境收付、换汇等完整能力 |
| **MAS 监管背书** | 新加坡 MAS 监管，合规信誉高，合作方信任度强 |

## 三、核心业务场景

### 3.1 场景一：MSO 作为 PF 接收收单行结算款 ⭐

**这是最核心、最刚需的场景。**

```
收单行（Acquirer）
    │
    │  ① 日终/周结算：将 MSO 的收单款结算到 MSO 的 Moolahgo VA
    │
    ▼
MSO 的 Moolahgo VA（如 USD VA）
    │
    ├── ② 资金归集：多个收单行的结算款统一汇入同一 VA
    │
    ├── ③ 换汇：USD → 本地货币（如 IDR/PHP）
    │
    └── ④ 付出：从 VA 付款给商户 / 合作方 / 自有银行账户
```

**为什么这个场景成立：**
- MSO 作为 Payment Facilitator，需要一个银行/VA 账户来接收收单行的结算款
- 传统银行不给新 MSO 开户 → Moolahgo 可以
- 收单行不关心 MSO 用哪个银行/VA 接收结算款，只要账户合规即可

### 3.2 场景二：资金归集 + 换汇 + 清算

```
MSO 运营全链路：

外部入金                    Moolahgo VA 体系                    外部出金
─────────                ─────────────────                 ─────────
收单行结算 ──→ ┐                                    ┌──→ 付款给商户
合作方付款 ──→ ├── USD VA ──→ 换汇 ──→ IDR VA ──→ ├──→ 运营费用支付
客户汇入   ──→ ┘                                    └──→ 利润汇出
```

**日常运营中，MSO 的资金流转全部在 Moolahgo 体系内完成：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| **归集** | 多个来源的资金统一进入 VA | 收单结算、合作方付款、客户汇入 |
| **查看** | 实时查询各币种余额 | 通过 EX API 或 Dashboard |
| **换汇** | 多币种之间实时兑换 | 批发 FX 汇率，优于市场零售价 |
| **内转** | 体系内 MSO 之间转账 | 合作清算、服务费结算 |
| **付出** | 付款到外部银行 | SWIFT / 本地清算 |

### 3.3 场景三：MSO 之间行内清算

```
MSO-A（汇款公司，收端）          MSO-B（汇款公司，付端）
        │                              │
        │  客户从 A 发起汇款             │
        │  A 收到法币                   │
        │                              │
        │  ① A 通过行内转账付给 B         │
        ├──────────────────────────────→│
        │  （实时到账、零手续费）          │
        │                              │
        │                    ② B 在目的国付款给收款人
```

**传统模式 vs Moolahgo 模式：**

| 维度 | 传统跨行 | Moolahgo 行内 |
|------|---------|--------------|
| 到账速度 | T+1~3 | **实时** |
| 手续费 | $15~50/笔 | **零/极低** |
| 可用时间 | 工作日 + 清算窗口 | **7×24** |
| 对账复杂度 | 高（跨行对账） | 低（系统内记录） |

### 3.4 场景四：MSO 为下游商户开子账户

```
MSO-A（收单机构）
  │
  ├── 商户 1 · USD VA（电商卖家）
  ├── 商户 2 · USD VA（SaaS 公司）
  └── 商户 3 · EUR VA（旅游平台）
  
  MSO-A 代商户收款后，结算到各商户的子 VA
  商户可从自己的 VA 发起换汇、付出
```

## 四、开户流程

### 4.1 MSO/PSP 租户开户

```
1. MSO 通过 EX 平台注册 → 获取 MID
2. 提交 KYB 材料：
   - 牌照文件（MSO License / PSP License）
   - 公司注册证明
   - 董事及 UBO 信息
   - 业务说明书（业务模式、预估交易量、主要市场）
   - AML/CFT 政策文件
3. EX 初审（材料完整性 + 基础风险筛查）
4. → 转 Moolahgo 合规终审
5. Moolahgo 审核通过 → 创建多币种 VA
6. 返回 VA 信息（账户号、路由信息等）
7. MSO 可开始使用
```

**预计周期：5~10 个工作日**（对比传统银行 3~6 个月）

### 4.2 商户子账户开户

```
1. MSO 通过 EX API 提交商户注册
2. 提交商户 KYC/KYB 材料
3. EX 初审 → Moolahgo 终审
4. 审核通过 → 在 MSO 名下创建商户子 VA
5. MSO 对商户子 VA 有查询和管理权限
```

## 五、风控与合规设计

### 5.1 为什么不构成 Nesting

> **Nesting（嵌套）** 指支付机构通过另一支付机构的账户间接接入银行体系，规避银行的 KYC/AML 审查。

**Moolahgo 方案不构成 Nesting，原因：**

| 维度 | Nesting（违规） | Moolahgo 方案（合规） |
|------|---------------|---------------------|
| **KYC 执行** | 中间机构不对最终客户做 KYC | Moolahgo + EX **双层 KYC**：EX 初审 + Moolahgo 终审 |
| **交易可见性** | 银行无法看到最终客户的交易 | Moolahgo **完全可见**所有 VA 的交易明细 |
| **监管报告** | 中间机构不配合 STR/SAR | Moolahgo 直接承担监管报告义务 |
| **账户归属** | 最终客户用中间机构的账户 | **每个 MSO 拥有独立 VA**，账户关系透明 |
| **合规关系** | 银行只认中间机构 | Moolahgo **直接认知每个 MSO**，有完整的客户档案 |

**核心逻辑**：Moolahgo 不是"借通道"，而是**直接为 MSO 提供账户服务**，每个 MSO 都经过完整的尽职调查。EX 是技术层，不是合规隔离层。

### 5.2 三层风控体系

```
第一层：EX（技术风控）
  ├── 材料完整性检查
  ├── 交易实时监控（限额、频次、异常模式）
  ├── 制裁名单筛查
  └── 可疑交易标记 → 转人工

第二层：Moolahgo（合规风控）
  ├── KYB 终审 + 风险评级
  ├── 持续尽职调查（CDD/EDD）
  ├── STR/SAR 报告
  └── 定期复审

第三层：MAS 监管
  ├── Moolahgo 向 MAS 报告
  └── 接受 MAS 定期审计
```

### 5.3 MSO 专属风控规则

| 规则 | 说明 |
|------|------|
| **牌照验证** | 开户前验证 MSO 牌照有效性、持牌范围 |
| **业务范围限制** | VA 用途必须与 MSO 牌照允许的业务范围一致 |
| **交易量监控** | 实际交易量与开户时声明的预估量对比，偏差过大触发审查 |
| **资金来源核实** | 大额入金需说明资金来源（如收单行结算、客户汇入等） |
| **定期复审** | 每 6~12 个月复审 MSO 的牌照状态、业务合规性 |

## 十、FAQ

**Q1：给 MSO 开 VA 会不会有 Nesting 合规风险？**
> 不会。Moolahgo 直接对每个 MSO 执行完整 KYB + 尽职调查，每个 MSO 拥有独立 VA，交易完全透明可追溯。详见第五章合规设计。

**Q2：MSO 的牌照过期或被吊销怎么办？**
> EX + Moolahgo 定期复审牌照状态。牌照失效时冻结账户，启动资金退出流程。

**Q3：MSO 之间的行内转账需要审批吗？**
> 可配置。默认走自动风控规则（限额、频次等），大额转账可设置人工审批。

**Q4：收单行怎么知道 MSO 用的是 Moolahgo VA？**
> MSO 向收单行提供 Moolahgo 分配的 VA 信息（银行名称 + 账户号 + 路由号）即可。收单行只需确认账户合规。

**Q5：EX 和 Moolahgo 的分工是什么？**
> Moolahgo = 账户和资金（开 VA、记账、清结算）；EX = 技术和分发（API 平台、风控引擎、租户管理、Webhook 通知）。MSO 只需对接 EX，不需直接对接 Moolahgo。
