## 三、收单结算场景 — 痛点分析

### 3.1 收单平台（PF）的结算困境

收单平台（Payment Facilitator）通过收单行（如 Worldpay）为其商户提供收单服务。收单行定期将结算款打给 PF，PF 再分发给旗下商户。跨境场景下面临：

| 痛点                   | 具体表现                            | 影响                  |
| ---------------------- | ----------------------------------- | --------------------- |
| **结算成本高**   | 给中国商户打款走 SWIFT，单笔 $15-40 | 小额结算利润被吃掉    |
| **到账时效慢**   | SWIFT T+2~T+5，叠加收单结算周期 T+3 | 商户实际 T+5~T+8 到账 |
| **换汇不透明**   | 中间行换汇加点多                    | 商户到手金额不可预期  |
| **合规文件繁琐** | 中国商户结汇需还原订单明细          | 商户操作成本高        |
| **入网效率低**   | 逐个为商户开立收款账户              | 运营成本高            |
| **缺乏账户体系** | PF 没有自己的账户系统，逐笔打款     | 无法统一管理商户资金  |

### 3.2 典型场景

```收单平台（PF）：线下
收单行（如 Worldpay）：PF 的收单行，负责清算

当前流程：
收单行 ──T+3结算──→ PF 银行账户
                          │
                          │ PF 逐笔 SWIFT 打款给商户
                          ▼
                     商户银行账户（大陆）

痛点：
├── 中国商户占比高
│   ├── SWIFT 单笔 $25+，月结算量大 → 手续费成本高昂
│   ├── 商户自行结汇 + 申报，操作复杂
│   └── T+4 才到账（T+3 结算 + T+1 SWIFT）
│
├── 无账户体系
│   ├── 无法给商户开账户，每次结算都是一次性打款
│   └── 无法冻结商户资金（拒付场景）
│
└── 效率问题
    ├── 逐笔人工打款，运营成本高
    └── 无系统化对账
```

## 四、解决方案 — EurewaX 如何帮助收单 PF

### 4.1 方案概述

EurewaX 为收单 PF 提供以下核心能力：

| # | 能力                       | 说明                                                                    |
| - | -------------------------- | ----------------------------------------------------------------------- |
| 1 | **开 VA 给收单 PF**  | PF 在 SP 开立 VA，用于接收收单行的结算款                                |
| 2 | **账户 + A2A 转账**  | PF 主账户收到结算款后，通过 A2A 转账即时结算给商户                      |
| 3 | **商户下发能力**     | 商户收到结算款后，可使用换汇/结汇/提现等能力（可能由其他租户提供）      |
| 4 | **同 SP 跨租户转账** | 不同租户在同一 SP 内可直接 A2A 转账，无需出金再入金，**即时到账** |

### 4.2 整体架构

```
┌──────────────────────────────────────────────────────────────┐
│              收单行（如 Worldpay）                             │
│              负责收单清算                                      │
└──────────────────┬───────────────────────────────────────────┘
                   │
       收单行结算款 │ 打入 PF 的 VA
                   ▼
┌──────────────────────────────────────────────────────────────┐
│         收单 PF（通过 EurewaX API 管理）                       │
│                                                              │
│  ┌─────────────────────────────────────────────────┐         │
│  │  PF VA（接收收单行结算款）                        │         │
│  │  → 收单行结算款入账 → PF 余额增加                 │         │
│  └─────────────────────────────────────────────────┘         │
│                   │                                           │
│        A2A 转账   │ （即时到账）                               │
│                   ▼                                           │
│  ┌─────────────────────────────────────────────────┐         │
│  │  商户账户（PF 下的子商户）                        │         │
│  │  商户 A ── 余额 → 换汇/结汇/提现                 │         │
│  │  商户 B ── 余额 → 换汇/结汇/提现                 │         │

全部通过 EurewaX 统一 API 操作
合规 · 风控 · 开户 · 资金托管 —— 由持牌服务商执行
```

### 4.3 各方角色

| 角色              | 身份              | 说明                                                       |
| ----------------- | ----------------- | ---------------------------------------------------------- |
| **收单行**  | 外部              | 负责收单清算，将结算款打入 PF 的 VA                          |
| **收单 PF** | EurewaX 租户      | 通过 EurewaX API 管理商户、开 VA 接收结算款、A2A 转账结算给商户 |
| **商户**    | PF 的子商户       | PF 下的签约商户，拥有独立账户                              |

### 4.4 账户关系

```
在 SP 内：
├── 收单 PF（租户）
│   ├── PF 主 VA：接收收单行结算款
│   └── PF 余额账户：A2A 转出给商户
│
├── 商户 A（PF 的子商户）
│   ├── 余额账户：接收 PF 的 A2A 结算款
│   └── 可操作：换汇 / 结汇 / 提现
│
├── 商户 B
│   └── ...
│
└── （可选）其他场景（如承兑服务商）
    ├── 租户给商户提供其他业务用到其他SP
    └── 同 SP 内 A2A → 即时到账
```

## 五、核心业务流程

### 5.1 端到端流程

```
Phase 1: 入网
PF ──API──→ EurewaX ──→ 审核通过 → PF 开 VA + 商户开账户
                                    │
                                    ▼
                        PF 将 VA 提供给收单行作为结算账户

Phase 2: 收单行结算 → PF 的 VA
消费者刷卡 → 收单行收单清算 → 结算款打入 PF 的 VA
                                    │
                                    ▼
                        PF VA 余额增加（Webhook 通知）

Phase 3: PF 结算给商户（A2A）
PF ──API──→ EurewaX ──→ A2A 转账执行
                        │
                        ▼
              商户账户余额增加（同 SP 内，即时到账）

Phase 4: 商户自助下发
商户余额 ──→ 换汇 / 结汇 / 提现到银行账户
```

### 5.2 资金入账（收单行 → PF VA）

```
收单行
    │
    │ 按结算周期（如 T+3）打款
    ▼
PF 的 VA（在 SP 开立的虚拟账户）
    │
    │ SP 入账确认
    ▼
PF 余额增加 → Webhook 通知 PF
```

> **关键**：VA 是持牌服务商开立的真实银行账户，收单行直接打款到该 VA。

### 5.3 PF 结算给商户（A2A 转账）

```
PF 系统
    │
    │ ① API 提交结算指令（可批量）
    │   [{商户ID, 金额, 币种, 参考号}, ...]
    ▼
EurewaX 平台
    │
    │ ② 执行 A2A 转账
    ▼
持牌服务商即时完成
    │
    │ ③ 同 SP 内即时完成（无需银行通道）
    │   PF 账户扣款 → 商户账户入账
    ▼
商户余额增加 → Webhook 通知
```

> **核心优势**：A2A 在同一 SP 体内完成，等同于"记账"操作，**即时到账、零银行手续费**。

### 5.4 商户自助操作

```
商户账户余额（外币，如 USD/GBP）
    │
    ├──→ 换汇：USD → 其他币种
    ├──→ 结汇：USD → CNY（自动匹配结汇额度）
    └──→ 提现/付款：到绑定的银行账户
```

### 5.5 跨租户协同（同 SP 内）

```
场景：商户需要使用「租户 B」提供的结汇/换汇服务

收单 PF（租户 A）                    换汇服务商（租户 B）
┌──────────────┐                    ┌──────────────┐
│ 商户 X 账户   │       A2A          │ 商户 X 账户   │
│ (SP-A 内)    │ ──────────────→    │ (同一 SP-A)  │
│              │   即时到账           │ → 换汇/结汇  │
└──────────────┘                    └──────────────┘

资金始终在 SP-A 内，未出金，即时完成
```

## 八、前置流程

> 整体顺序：**PF 入网 → Sandbox 联调 → 商户批量入网 → 业务上线**

### 8.1 收单 PF 入网（KYB）

```
1. PF 向 EurewaX 提交 KYB 材料
   └── 提交营业执照、法人信息、业务说明
   └── SP 审核 KYC/KYB
   └── Webhook 通知审核结果

2. 审核通过 → SP 开立 PF 主账户 + VA
   └── PF 获得 VA 账号，提供给收单行作为结算账户
   └── 开通 A2A 转账、商户管理等产品
```

### 8.2 商户批量入网

```
PF 通过 EurewaX API 批量推送商户信息
→ 审核 KYC/KYB → 为商户开立账户
→ 商户获得独立账户，可接收 A2A 结算款
→ 商户可使用换汇/结汇/提现等能力
```

### 8.3 开通 Sandbox 测试环境

```
步骤 1 → 开通 Sandbox，获取测试账号、测试域名
步骤 2 → 获取 APP ID、平台公钥、AES Key
步骤 3 → 生成 RSA 密钥对（SHA256withRSA, 2048位）→ 上传公钥
步骤 4 → 配置 Webhook 回调地址
步骤 5 → 完成签名验签 + AES 加解密联调
```
