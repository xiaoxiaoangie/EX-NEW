方案概述

背景、核心思路与方案亮点

背景

收单平台（PF）通过收单行（Acquirer）为旗下商户提供收单服务。收单行定期将结算款打给 PF，PF 需要再与各商户进行分账结算。收单行结算到 PF 有两种方式：① 直接结算到 PF 在 A 开立的 VA；② 结算到 PF 自有银行账户，PF 再通过同名充值到 A 的 VA。

核心思路

① PF 在 A（法币 SP）入网，开立 VA 接收收单行结算款；② PF 的商户在 A 入网，建立主子账户关系；③ PF 同名充值到 A 的 VA，资金入账 PF 主账户；④ PF 通过 A2A 转账，即时分发给各商户子账户；⑤ 商户自助操作：在 A 内换汇/结汇，或通过 B 做 OnRamp（法币→数币）。OffRamp 一期暂不开放。

⚡
即时结算
A2A 同 SP 体内完成，秒级到账，零银行手续费
🏦
账户体系
PF 主账户 + 商户子账户，主子关系清晰
📤
灵活下发
支持 API 批量结算，也支持上传结算记录表导入
🔧
商户自助
商户到账后可自主换汇、结汇、B 侧 OnRamp
💰
租户定价
换汇/结汇/OnRamp 费率由租户加价报价
🔄
返点分佣
商户在 B 做 OnRamp 业务归属租户体系，支持返点分佣

收单 PF 的结算困境

跨境场景下的 6 大痛点
💸
结算成本高

SWIFT 单笔 $25+，小额结算利润被吃掉。月结算量大时手续费成本高昂。

⏳
到账时效慢

SWIFT T+2~5，叠加收单行结算 T+3，商户实际 T+5~8 到账。

💱
换汇不透明

中间行换汇加点 0.5-1.5%，商户到手金额不可预期。

📑
合规文件繁琐

中国商户结汇需还原订单明细，商户操作成本高。

🐢
入网效率低

逐个为商户开立收款账户，人工操作，运营成本高。

🚫
缺乏账户体系

无法给商户开账户，每次结算都是一次性打款，无法冻结商户资金。

🎯
核心需求：一套系统 解决 VA 开户 + 即时结算 + 换汇结汇 + 批量管理。

解决方案 — EurewaX 如何帮助收单 PF

VA + A2A + 换汇结汇 + 行内即时调拨

🏦 开 VA 接收结算款

PF 通过 API 开立 VA（虚拟账户），收单行可直接打款到 VA（方式①），或打到 PF 银行账户后同名充值到 VA（方式②）。资金入账后 Webhook 实时通知。

⚡ A2A 即时结算

PF 收到结算款后，通过 A2A 转账即时结算给商户。同服务商内转账 = 记账操作，秒级到账、零手续费，支持批量指令。

💱 商户自助换汇/结汇

商户可自主换汇（USD→其他币种）、结汇（USD→CNY 自动匹配额度）、提现到绑定银行账户，全程 API 完成。

🔗 扩展能力 · 行内即时调拨

如需使用 OnRamp（法币→数币）等更多产品能力，资金可通过行内转账即时调拨到 B 侧，无需出金再入金，高效利用每一分资金。OffRamp 一期暂不开放。

🔐
资金安全：资金全程在持牌服务商体系内流转。合规/风控/KYC 由持牌服务商执行，对您完全透明。

账户结构与角色

主子账户体系 · A/B 双 SP 架构

A 侧 — 法币 SP 账户体系

PF 租户（如 WB）
├─ PF 主账户（VA + 余额账户）
│ ├─ VA 账户 — 接收收单行结算款
│ │ 方式①：收单行直接结算到 VA
│ │ 方式②：收单行 → PF 银行账户 → 同名充值到 VA
│ └─ 余额账户 — A2A 转出给商户
├─ 商户 A（子账户）— 余额账户
├─ 商户 B（子账户）— 余额账户
└─ 商户 N（子账户）— 余额账户
🔗
主子关系：PF 主账户持有 VA 和余额，负责接收收单行结算款（两种结算方式）；子账户接收 PF 的 A2A 转账，资金归商户自主管理。
💡
为何选择 WB 而非 Bonbillhk：Bonbillhk 仅提供法币业务，没有数币业务；WB 的客户有数币需求，且 WB 需要通过分佣模式获取收益。

B 侧 — 承兑 SP 与角色关系

B（承兑 SP） — OnRamp：A 法币 → B 数币
├─ 商户 A — 数币钱包（USDT）
└─ 商户 B — 数币钱包（USDT）
OffRamp（数币→法币）一期暂不开放

五方角色
① 收单 PF — 结算发起方
② 商户 — 结算接收方
③ A — 法币 SP（VA/A2A/换汇）
④ B — 承兑 SP（OnRamp，法币→数币）
⑤ EX — 技术编排层
⚡
跨 SP 协同：A 法币 ↔ B 承兑通过「同名提现/充值」完成，资金先收后做，禁止垫资。
💱
汇率机制：A/B SP 提供批发汇率作为基准，租户（PF）在此基础上加价报价给商户，差价归租户。

核心业务流程

端到端：入网 → 收款 → 结算 → 下发
  1. PF 入网（KYB） — PF 向 EurewaX 提交 KYB 材料 → 审核通过 → 开立 PF 主 VA + 开通产品
  2. 商户批量入网 — PF 通过 API 批量推送商户 → KYC 审核 → 为商户开立独立账户
  3. 收单行结算 → PF — 方式①：收单行直接打款到 PF VA；方式②：收单行打到 PF 银行账户 → PF 同名充值到 VA → Webhook 通知到账
  4. PF 结算给商户（A2A） — PF 通过 API 提交批量结算指令 → A2A 即时完成 → 商户余额增加 → Webhook 通知
  5. 商户自助操作 — 商户在余额中选择：换汇 / 结汇 / 提现到银行账户，全程 API 完成
  6. 扩展能力调拨（可选） — 如需 OnRamp（法币→数币），资金可通过同名提现/充值到 B 侧执行。OffRamp 一期暂不开放。
💡
VA 是持牌服务商开立的真实银行账户，收单行直接打款到该 VA。全程资金在持牌服务商体系内流转。

关键业务规则 & WB 特殊需求

主子账户 · A2A 结算 · 换汇定价 · 自有结汇通道

关键业务规则

规则说明
主子关系PF 主账户自动创建；推送商户入网时，子账户自动挂到 PF 主账户下
A2A 方向只能从 PF 主账户 → 商户子账户单向转入（结算方向）
到账时效同 SP 体内 A2A，即时到账（秒级），零手续费
结算触发PF 主动发起，支持 API 批量或文件上传（CSV/Excel）
换汇定价租户（PF）可自行定价，EX 提供灵活计费配置能力
承兑原则先收后做，B 侧法币到账后才执行 OnRamp，禁止垫资

WB 特殊需求：自有结汇通道

💡
选择 WB 而非 Bonbillhk 的原因：Bonbillhk 仅提供法币业务，没有数币业务；WB 的客户有数币需求，且 WB 需要通过分佣模式获取收益。

🚀 短期方案

WB 将自有结汇通道以 API 方式接入 A 平台。通过路由配置，WB 旗下商户结汇请求自动路由到 WB 自有通道，非 WB 商户仍走平台默认通道。

🏗️ 长期方案

WB 以「结汇 SP」身份入驻 EX 平台，不仅服务自有商户，还可赋能平台上其他租户。WB 结汇能力产品化，成为 EX 基础设施服务。

计费返点 & 风险管控

租户定价 · 承兑返点 · 合规与资金风险

计费与返点分佣

💱
换汇/结汇计费
A/B SP 提供批发汇率作为基准，租户（PF）在此基础上加价报价给商户，差价归 PF。
🔄
承兑业务返点（B 侧）
商户在 B 做 OnRamp 产生的手续费，由 B（承兑 SP）按配置比例直接返点给 PF 租户。如 B 收 1%，其中 30% 直接返点给 WB。OffRamp 一期暂不开放。
📋
A2A 结算计费
一期暂不启用，零手续费促进业务落地；二期支持 PF 自主定价，平台提供计费引擎。

风险与注意事项

🛡️
合规风险
收单行需验证 VA 持有人与 PF 主体一致；多层代开户下 PF 做第一层审核 + A 做最终审核。
⚙️
运营风险
结算记录表支持预审核模式（先预览再执行）；建立定期对账机制，商户新增/注销实时同步。
🔒
资金风险
严格先收后结，禁止垫资；A2A 即时完成无法撤回，大额需二次验证；换汇/承兑支持锁汇模式。
