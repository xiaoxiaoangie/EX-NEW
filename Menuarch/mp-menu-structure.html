Dashboard（工作台）

商户总览 — 余额概览、近期交易、待办事项

📊 MP 菜单设计总览

菜单设计原则：
1. Assets（资产）= 同名充提 + Exchange（统一兑换入口：FX + Buy Crypto/On-Ramp + Sell Crypto/Off-Ramp）
2. Exchange 统一兑换入口（参考 Coinbase 模式）：FX/买U/卖U 本质都是 Conversion，统一到一个入口。用户选择兑换方向和资金来源/去向，系统自动决定底层是纯 Conversion 还是 Deposit+Conversion / Conversion+Withdraw 编排
3. Transfer In（转账收款）= 收款工具管理（Collection Tools）+ 汇款人维护 + Payins（所有入金）
4. Checkout（收单）= 商户管理收款订单，底层收款方式（VA/地址/在线支付）由系统自动选择
5. Transfer Out（转账付款）= Beneficiary（收款人）+ Payouts（所有出金）+ Remittance Orders（收→换→付一单到底，编排层）
6. Cards（发卡）= 通过卡消费方式付款
7. Payins vs Payouts 对称：Payins = 所有入金，Payouts = 所有出金
8. On-Ramp / Off-Ramp 归属 Exchange（不再作为 Transfer In / Transfer Out 的独立菜单），因为本质是兑换操作，涉及外部入金/出金时由 Exchange 内部编排
SP透明 & 账户结构规则：
1. 底层账户结构：每个SP给商户开独立账户（IPL有USD账户，BB也有USD账户）
2. 商户端不展示SP：商户只看到按币种合计的总余额，SP明细只在租户端展示
3. 充值/提现/付款：商户不选SP，系统根据默认配置和优先级自动选择扣款/入账的SP账户
4. 付款余额不足时：系统自动从多个SP账户扣款（如IPL $5000 + BB $3000 = 总余额 $8000）
5. SP间互转：打通的SP之间（如IPL⇄BB）可互转，在Exchange中操作，对商户透明
6. 交易记录：所有SP的记录混合展示，不区分SP来源

Assets / Fiat Accounts

Fiat Accounts（法币账户）

管理法币账户余额，同名充值与提现 同名操作

职责定义

本菜单只管"自己的法币"：查看余额、同名充值（自己银行→平台）、同名提现（平台→自己银行）。
不管：收别人的钱（→ Transfer In / Checkout）、付给别人（→ Transfer Out）、兑换（→ Exchange）

页面功能

账户余额总览
按币种展示总余额（多个SP账户合计），如 USD $8,000 / EUR €3,000。商户不感知SP，底层可能分属多个SP账户
同名充值（Deposit）
从自己的银行账户转入法币到平台。商户只选币种和金额，系统自动选择入账的SP账户（有默认配置）
同名提现（Withdraw）
从平台提现到自己的银行账户。商户只选币种和金额，系统自动选择扣款的SP账户
交易记录
所有法币变动记录（充值/提现/收款入账/付款扣款等），所有SP混合展示，不区分SP来源

边界说明

账户结构（商户不可见）：
• 底层每个SP独立开户：IPL-USD $5,000 + BB-USD $3,000 = 商户看到 USD $8,000
• 付款时系统根据优先级自动从多个SP扣款，商户只看到总余额
• SP明细仅在租户端（TP）展示
当前阶段（仅 IPL + BB）：
• 法币余额 = 1个合并余额（IPL + BB 合计），商户看到按币种的总余额
• 后续增加其他SP时再评估是否需要调整展示逻辑
Assets / Crypto Wallet

Crypto Wallet（加密钱包）

管理加密资产，同名充币与提币 同名操作

职责定义

本菜单只管"自己的加密资产"：查看余额、同名充币（自己外部钱包→平台托管地址）、同名提币（平台→自己外部钱包）。
不管：收别人的加密转账（→ Transfer In / ）、转给别人（→ Transfer Out ）、法币⇄加密兑换（→ Exchange）

页面功能

钱包余额总览
按币种+链展示总余额（多个SP托管地址合计），如 USDT-TRC20 10,000 / BTC 0.5。商户不感知SP
同名充币（Deposit）
从自己的外部钱包转入加密资产到平台托管地址。系统自动选择SP的托管地址，显示充值地址和二维码
同名提币（Withdraw）
从平台托管地址提币到自己的外部钱包。系统自动选择扣款的SP托管地址
交易记录
所有加密变动记录（充币/提币/收款入账/转出扣款等），所有SP混合展示，含链上TxHash

边界说明

当前阶段（仅 IPL + BB）：
• 数币余额 = 1个余额（仅BB提供加密托管能力），商户看到按币种+链的余额
• 后续增加其他有加密托管能力的SP时再评估合并展示逻辑
Assets / Exchange

Exchange（统一兑换入口）

FX 换汇 / Buy Crypto (On-Ramp) / Sell Crypto (Off-Ramp) — 统一兑换入口，底层按资金来源/去向自动编排

职责定义

统一兑换入口（参考 Coinbase Buy/Sell 模式）：FX + Buy Crypto + Sell Crypto 统一到一个 Exchange 入口，三者本质都是 Conversion（体系内资产互换），只是资产类型不同。

核心逻辑：
• 用户选择兑换方向 → 选择资金来源/去向 → 系统自动决定底层能力组合
• 从余额兑换 = 纯 Conversion（体系内）
• 从外部充值买U（一单到底） = Deposit + Conversion（编排）
• 卖U到外部银行（一单到底） = Conversion + Withdraw（编排）

不管：承兑后要付给第三方的场景（→ Remittance Orders，收→换→付一单到底自动编排）

FX 换汇（法币 ↔ 法币）纯 Conversion

即时换汇
体系内法币账户间换汇，如 USD → EUR。底层能力 = Conversion，不涉及外部资金。系统自动选择SP，商户不感知
换汇记录
历史换汇记录、汇率、手续费详情
场景 S1：USD 账户 → EUR 账户，体系内法币→体系内法币，纯 Conversion
原子能力：Conversion × 1

Buy Crypto / On-Ramp（法币 → 数币）

方式A：从体系内余额买U 纯 Conversion
从自己法币账户余额直接兑换为加密资产（如 USD → USDT），存入加密钱包。不涉及外部资金，钱没出体系，只是换了形态
底层：Conversion × 1
场景 S3：法币账户 USD → 数币钱包 USDT
方式B：从外部银行充值买U（一单到底）编排：Deposit + Conversion
从外部银行充值法币，系统自动将法币兑换为 USDT，最终入账到加密钱包。一步操作，用户不需要分步执行
底层：Deposit（外部入金）+ Conversion（法币→数币）编排
场景 S2：外部银行 → 充值法币 → 自动换成 USDT
Buy Crypto 记录
所有买U记录，含兑换方式（余额/外部充值）、汇率、手续费、到账数量。外部充值模式可追踪全链路状态
用户体验：统一入口，用户选择"资金来源"（体系内余额 / 外部银行账户），系统根据选择决定底层是纯 Conversion 还是 Deposit + Conversion

Sell Crypto / Off-Ramp（数币 → 法币）

方式A：卖U到体系内法币余额 纯 Conversion
从加密钱包余额兑换为法币（如 USDT → USD），存入法币账户。不涉及外部资金，钱没出体系，只是换了形态
底层：Conversion × 1
场景 S5：USDT → 换成 USD 到法币账户
方式B：卖U提现到外部银行（一单到底）编排：Conversion + Withdraw
USDT 兑换为法币，系统自动提现到外部银行账户。一步操作，用户不需要分步执行
底层：Conversion（数币→法币）+ Withdraw（外部出金）编排
场景 S4：USDT → 换成法币 → 提现到外部银行
Sell Crypto 记录
所有卖U记录，含兑换方式（到余额/到银行）、汇率、手续费、到账金额。外部提现模式可追踪全链路状态
用户体验：统一入口，用户选择"资金去向"（体系内法币余额 / 外部银行账户），系统根据选择决定底层是纯 Conversion 还是 Conversion + Withdraw

原子能力模型

四个原子能力：
• Deposit（入金）：外部资金 → 体系内账户
• Withdraw（出金）：体系内账户 → 外部
• Conversion（兑换）：体系内 A 账户 → 体系内 B 账户（含法币↔法币、法币↔数币）
• Transfer（转账）：体系内资金付给非同名第三方

Exchange 内的能力组合：
• FX（USD→EUR）= Conversion
• 余额买U / 卖U到余额 = Conversion
• 充值买U = Deposit + Conversion（编排）
• 卖U提现 = Conversion + Withdraw（编排）

边界说明

Exchange vs Remittance vs 同名充提：
• Exchange = 统一兑换入口，含纯体系内兑换和涉及外部的一单到底编排（On-Ramp/Off-Ramp）
• Remittance = 收→换→付一单到底，钱最终要付给第三方（Deposit + Conversion + Transfer）
• 同名充值/提现 = 纯 Deposit / Withdraw，不涉及兑换（在 Fiat Accounts / Crypto Wallet 中操作）
• SP间互转（如IPL⇄BB）也在Exchange中完成，对商户透明
• Remittance 的子步骤中如果涉及兑换，Exchange 也会产生对应记录，通过关联订单号串联
Transfer In / Collection Tools

Collection Tools（收款工具）

管理所有收款工具：Global Account（VA）和加密收款地址 他人→我（转账）

职责定义

本菜单统一管理"商户主动申请的收款工具"：包括法币收款的 Global Account（VA）和加密收款的链上地址。两者都是商户长期持有的固定收款入口，供客户/买家反复转入。

边界说明

收款工具 vs Payins：
• Collection Tools = 管理收款工具本身（VA账户、加密地址的申请/查看/启停）
• Payins = 查看所有入金交易记录（通过VA收到的法币、通过地址收到的加密、同名充值、充币等）
• 收款工具是"入口"，Payins是"流水"
Transfer In / Remitter

Remitter（汇款人维护）

管理向您汇款的汇款人信息，支持银行账户和链上地址 他人→我

职责定义

本菜单管"谁给我打钱"：维护汇款人（Remitter/Sender）信息，记录其汇款方式（银行账户或链上地址）。
用于入金时自动匹配汇款来源、加速入账确认、满足合规要求（KYC/AML需要知道资金来源方）。
与 Beneficiary 的对称关系：Beneficiary = 我付给谁（Transfer Out），Remitter = 谁付给我（Transfer In）

页面功能

汇款人列表
查看所有已登记的汇款人，支持按类型（个人/企业）、汇款方式（银行/链上）筛选
新增汇款人
登记汇款人基本信息：名称、类型（个人/企业）、国家/地区、证件信息
汇款方式 — 银行账户
为汇款人添加银行账户信息：银行名称、账户名、账号、SWIFT/IBAN、银行地址等
汇款方式 — 链上地址
为汇款人添加链上地址：钱包地址、链、币种。用于匹配链上入金来源
编辑/删除汇款人
修改汇款人信息、管理其汇款方式、删除/恢复
汇款人详情
查看汇款人完整信息、所有汇款方式、关联的入金记录历史

边界说明

汇款人 vs 收款人（对称设计）：
• Remitter（汇款人）在 Transfer In — 记录"谁给我打钱"，一个汇款人可有多种汇款方式（多个银行账户 + 多个链上地址）
• Beneficiary（收款人）在 Transfer Out — 记录"我付给谁"，一个收款人可有多种收款方式
• 入金时系统可自动匹配已登记的汇款人，加速入账确认
• 合规要求：KYC/AML需要明确资金来源方身份

Transfer Out / Payouts

Payouts（所有出金交易）

所有资金流出的交易记录，统一查看 出金

职责定义

本菜单管"所有出金交易记录"：统一展示所有资金流出记录，包括：
1. 法币付款（银行转账付给他人收款人）
2. 链上付币（加密资产转给他人链上地址）
3. 同名提现（平台法币账户→自己银行账户）
4. 同名提币（平台托管地址→自己外部钱包）
不管：需要先承兑/换汇再付款的组合流程（→ Remittance = 收→换→付一单到底）；Checkout退款产生的出金（→ Checkout）
与 Payins 的对称关系：
• Payins = 所有入金（VA收款 + 链上收款 + 同名充值 + 同名充币）
• Payouts = 所有出金（法币付款 + 链上付币 + 同名提现 + 同名提币）
• 两者都是"纯交易记录"，不含编排层（Exchange 内 On-Ramp/Off-Ramp 编排、Remittance 编排是独立菜单）

Trade Documents / Trade Documents

Trade Documents（贸易单据）

贸易单据管理与合规文档

职责定义

本菜单管"贸易背景材料"：管理与收付款相关的贸易单据，用于证明交易的真实贸易背景，满足合规审核要求。
典型单据：商业发票（Invoice）、采购合同（Contract）、提单（Bill of Lading）、装箱单（Packing List）、报关单等

页面功能

单据列表
查看所有贸易单据，按类型（发票/合同/提单/报关单）、状态（待审核/已通过/已拒绝）筛选
上传单据
上传发票、合同、提单等贸易文档，支持批量上传，自动OCR识别关键信息
单据关联
将单据关联到对应的收付款订单、汇款订单，作为贸易背景证明
审核状态
查看单据的合规审核状态，支持补充材料（RFI）
Trade Documents / Order Files

Order Files（订单文件）

管理与交易订单相关的文件材料

职责定义

本菜单管"订单级别的文件"：与具体交易订单绑定的文件材料，包括采购订单、销售合同、物流单据等。
与 Trade Documents 的区别：Trade Documents 是通用贸易单据（可关联多笔订单），Order Files 是订单维度的文件（与单笔订单绑定）

页面功能

订单文件列表
按订单查看关联文件，支持按订单号/时间/文件类型筛选
上传订单文件
为指定订单上传相关文件（采购单、合同、物流单据、报关单等）
文件预览与下载
在线预览订单文件，支持下载原件
文件审核状态
查看订单文件的审核进度，合规要求的补充材料提示
Trade Documents / Shop Management

Shop Management（店铺管理）

管理电商店铺，同步订单数据作为贸易背景

职责定义

本菜单管"电商店铺绑定与订单同步"：商户绑定其电商平台店铺（如Amazon、Shopify、eBay等），系统自动同步店铺订单数据，作为收付款的贸易背景证明。
用途：合规审核时自动匹配订单与收付款记录，减少人工上传单据的工作量

页面功能

店铺列表
查看所有已绑定的电商店铺，含平台类型、店铺名称、绑定状态、最近同步时间
绑定新店铺
选择电商平台 → 授权绑定 → 配置同步规则（同步频率、订单范围等）
订单同步记录
查看从店铺同步的订单数据，含订单号、金额、买家信息、物流状态
店铺设置
管理店铺授权、同步配置、解绑店铺
