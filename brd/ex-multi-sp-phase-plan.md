## 0. 现状梳理：老 EX 平台

### 0.1 两种服务模式

```
老 EX 有两种模式：

模式一：白牌（White Label）
  · 租户使用 EX 提供的白牌界面（MP 商户端）
  · 商户在白牌界面上操作
  · 底层能力：
    - 纯法币租户 → SP = IPL（法币账户、VA 收款、POBO 出款）
    - 数法融合租户 → 法币账户 = IPL，数币账户 = BB，承兑 = BB
  · 商户端不感知 SP

模式二：API
  · 租户通过 API 对接 EX
  · 租户自己有前端系统，EX 只提供后端能力
  · 当前 API 不区分 SP → 余额、交易等接口返回的是"混合"数据
  · 4月20日左右更新：API 按 SP 维度返回数据
    （如：查余额 → 返回各 SP 下的余额，而不是合并余额）
```

### 0.2 老 EX 的 SP 结构（隐式）

```
老 EX 底层已有 SP 概念，但不彻底：

┌──────────────────────────────────────────────────────────────┐
│                      老 EX Platform                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  白牌：                                                       │
│  · 纯法币 → IPL 一个 SP                                      │
│  · 数法融合 → IPL(法币) + BB(数币+承兑)                       │
│  · 商户端看到的是合并视图                                      │
│                                                              │
│  API：                                                        │
│  · 当前：接口不区分 SP（余额/交易混在一起）                    │
│  · 4月20日后：接口按 SP 返回（余额按 SP 拆分）                │
│                                                              │
│  核心问题：                                                    │
│  · SP 概念不完整 — 没有标准化的 SP 注册/管理/能力声明          │
│  · 账户模型不清晰 — 没有明确"每个 SP 独立法币账户"            │
│  · 产品与 SP 绑定不标准化                                      │
│  · 无跨 SP 划转产品                                           │
│  · 无 SP 审核独立流程                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 0.3 老 EX 客户分布

| 客户类型 | 模式 | SP 组合 | 典型租户 |
|----------|------|---------|----------|
| 纯法币 | 白牌 | IPL | 法币收付款租户 |
| 数法融合 | 白牌 | IPL + BB | 需要承兑+法币收付的租户 |
| 纯法币 | API | IPL | API 对接的法币租户 |
| 数法融合 | API | IPL + BB | API 对接的数法融合租户 |
| VCC | API | 发卡SP | VCC 发卡租户 |

## 1. 目标梳理：新 EX 架构

> 详见 ex-multi-sp-architecture-brd-v2.md

```
新 EX 的核心架构特征（从 BRD 推导得出）：

  ❶ 多 SP 共存 — 标准化的 SP 注册/管理/能力声明
  ❷ 数币只在 BB — BB 是唯一数币资金中心
  ❸ 法币按 SP 区分 — 因 nesting，每个 SP 独立法币账户
  ❹ 产品绑定 SP — 每个产品绑定唯一 SP（Phase 1）
  ❺ 跨 SP 划转是独立产品 — 商户自行发起
  ❻ SP 审核独立 — 每个 SP 独立审核商户
  ❼ EX 是编排层 — 不持牌、不碰钱、不做合规审核
  ❽ EX 是渠道对接平台 — 接入的渠道注册为 SP
```

## 2. 差异分析：老 EX vs 新 EX

| 维度 | 老 EX | 新 EX | 差距 |
|------|-------|-------|------|
| **SP 模型** | 隐式，后端有 BB/IPL 概念但不标准 | 标准化 SP 注册/管理/能力声明 | 🔴 需重建 |
| **账户模型** | 白牌合并视图，API 混合返回（4月后按 SP 拆分） | 每个 SP 独立法币账户，数币只在 BB | 🟡 API 4月已改；白牌需调整 |
| **产品-SP 绑定** | 不标准化 | 产品绑定唯一 SP | 🟡 需配置 |
| **跨 SP 划转** | 无 | 独立产品 | 🔴 需新建 |
| **SP 审核** | 不独立 | 每个 SP 独立审核 | 🔴 需重建 |
| **白牌** | 合并视图 | 按 SP 看余额（Phase 1）/ 统一视图（远期） | 🟡 需改造 |
| **API** | 不区分 SP → 4月后按 SP 返回 | 原生按 SP | 🟢 4月已对齐 |
| **合规流程** | EX 自己做 / 不标准 | SP 各自审核，EX 做编排 | 🔴 需重建 |

## 4. Phase 0：老 EX API 多 SP 改造（4月）

> **目标：让老 EX API 按 SP 维度返回数据，为新 EX 上线做数据结构对齐**

### 4.1 改造内容

| 改造项 | 改造前 | 改造后 | 状态 |
|--------|--------|--------|------|
| 余额查询 | 返回合并余额 | 按 SP 返回各 SP 下的余额 | 🔄 4月20日上线 |
| 交易查询 | 不区分 SP | 交易记录带 SP 标识 | 🔄 4月20日上线 |
| 账户信息 | 不区分 SP | 账户带 SP 归属 | 🔄 4月20日上线 |

### 4.2 老 EX 白牌：Phase 0 不动

```
Phase 0 只改 API，不改白牌。原因：

  · 白牌改造涉及前端 UI 变更，工作量大
  · 白牌用户当前体验没有问题（合并视图正常运作）
  · 白牌改造放到 Phase 1 随新 EX 一起做（新前端直接按新架构设计）
  · 避免"改了老的又要迁移到新的"的双重成本
```

### 4.3 API 租户影响

```
对现有 API 租户的影响：

  · 余额接口返回格式变更 → 需要租户适配
  · 建议：新接口并行上线，老接口保留兼容期（如 3 个月）
  · 租户在兼容期内完成接口切换
  · 切换后：租户可以看到各 SP 下的独立余额
    例：BB USD: 5,000  |  IPL USD: 3,000（而不是合并的 8,000）
```

## 5. Phase 1：新 EX 核心上线（白牌优先）

> **目标：新 EX 平台按多 SP 架构上线，白牌优先，API 同步支持**

### 5.1 上线策略：白牌先行

```
为什么白牌优先？

  · 白牌是 EX 控制的前端 → 可以完全按新架构设计 UI
  · 白牌改动不需要租户技术团队配合
  · 新白牌直接体现多 SP 架构的账户模型（数币在 BB，法币按 SP 独立）
  · API 在 Phase 0 已经按 SP 拆分 → Phase 1 API 只需指向新后端

上线顺序：
  1. 新 EX 白牌上线（新租户直接用新白牌）
  2. 新 EX API 同步开放（基于新后端）
  3. 老白牌暂不改动（等 Phase 2 迁移）
```

### 5.2 业务线与 SP 归属（Phase 1）

```
业务线按能力归属 SP：

┌─────────────────────────────────────────────────────────────────────────┐
│                          业务线与 SP 归属                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【法币收付款】— IPL                                                     │
│  · VA 收款（虚拟账户收款）                                               │
│  · POBO 出款（代付）                                                     │
│  · 法币充值/提现（同名）                                                 │
│  → VA 和 POBO 统一归法币收付款业务线                                     │
│                                                                         │
│  【承兑】— BB                                                            │
│  · OnRamp（法币 → 数币）                                                │
│  · OffRamp（数币 → 法币）                                               │
│  · BB 数币钱包（链上充币/提币）                                          │
│  · BB 法币账户（承兑资金落地）                                           │
│  → BB 是唯一数币资金中心                                                 │
│                                                                         │
│  【发卡】— BB                                                            │
│  · VCC 发卡                                                              │
│  · 发卡收款（卡片收款能力）                                               │
│  → BB 有 VCC 发卡能力                                                   │
│                                                                         │
│  【Crypto Checkout】— BB + 其他 SP（Phase 3）                            │
│  · BB 有 Crypto Checkout 能力，但能力不够                               │
│  · 需要接入其他 SP 补充能力                                              │
│  → Phase 1 仅 BB，Phase 3 引入多 SP                                     │
│                                                                         │
│  【跨 SP 划转】— 独立产品                                                │
│  · BB ↔ IPL 法币划转                                                    │
│  · 商户自行发起，不属于任何业务线                                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.3 能力清单（Phase 1）

| 业务线 | 能力 | SP | 白牌 | API | 说明 |
|--------|------|-----|------|-----|------|
| **法币收付款** | VA 收款 | IPL | ✅ | ✅ | 虚拟账户收款 |
| | POBO 出款 | IPL | ✅ | ✅ | 代付 |
| | 法币充值/提现 | IPL | ✅ | ✅ | 同名 |
| **承兑** | OnRamp | BB | ✅ | ✅ | 法币 → 数币 |
| | OffRamp | BB | ✅ | ✅ | 数币 → 法币 |
| | 数币钱包 | BB | ✅ | ✅ | 唯一数币入口 |
| | BB 法币账户 | BB | ✅ | ✅ | 承兑资金落地 |
| **发卡** | VCC 发卡 | BB | ✅ | ✅ | BB 有发卡能力 |
| | 发卡收款 | BB | ✅ | ✅ | 卡片收款 |
| **Crypto Checkout** | Crypto Checkout | BB | ✅ | ✅ | Phase 1 仅 BB |
| **跨 SP** | 划转（BB ↔ IPL） | — | ✅ | ✅ | 独立产品 |
| **平台** | SP 独立审核 | — | ✅ | ✅ | BB/IPL 各自审核 |
| | 按 SP 看余额 | — | ✅ | ✅ | 各 SP 独立余额 |

### 5.4 Crypto Checkout 的多 SP 规划

```
Crypto Checkout 能力分布：

  Phase 1：
    · 仅 BB 提供 Crypto Checkout
    · BB 能力有限，覆盖主流币种/链

  Phase 3：
    · 引入其他 Crypto Checkout SP
    · 补充 BB 做不了的币种/链/场景
    · EX 统一路由，选择最优 SP

  原因：
    · Crypto Checkout 不是 BB 独占能力
    · 不同 SP 有不同的链/币种覆盖
    · 类似法币收付款的多 SP 模式
```

### 5.5 新白牌的关键 UI 变化

```
老白牌（数法融合租户）：
  · 商户看到一个法币余额（IPL+BB 合并显示？）
  · 商户不知道有两个 SP

新白牌：
  · 商户看到：
    - 数币钱包（BB）：USDT 5,000
    - BB 法币账户：USD 7,000
    - IPL 法币账户：USD 3,000
  · 商户知道有两个法币余额，但不一定知道 SP 名称
    （可按业务场景命名：如"承兑账户"和"收付款账户"）
  · 商户可自行发起划转：BB USD ↔ IPL USD

  关键设计决策（待确认）：
  · Q1：商户端是否暴露 SP 名称（BB/IPL），还是用业务名称包装？
  · Q2：如果用业务名称，叫什么？（"承兑账户" / "收付款账户"？）
```

### 5.4 新 EX API 与老 EX API 的关系

```
Phase 1 的 API 策略：

  · 新 EX API = 全新的后端 + 全新的接口定义
  · 老 EX API（Phase 0 改造后）继续运行，服务老租户
  · 新租户直接用新 EX API
  · 老租户在 Phase 2 迁移到新 EX API

  两套 API 并行运行期间：
  · 老 EX API 只做维护，不加新功能
  · 新功能只在新 EX API 上开发
  · 并行期目标：≤ 6 个月
```

## 6. Phase 2：老 EX → 新 EX 迁移

> **目标：将老 EX 的存量租户和商户迁移到新 EX 平台**

### 6.1 迁移分类

```
老 EX 客户分四类，迁移策略不同：

  类型A：老白牌 — 纯法币租户
    · 当前：白牌 + IPL
    · 迁移到：新白牌 + IPL
    · 复杂度：🟢 低（只有一个 SP，账户模型不变）

  类型B：老白牌 — 数法融合租户
    · 当前：白牌 + IPL + BB
    · 迁移到：新白牌 + IPL + BB（账户拆分为两个独立余额）
    · 复杂度：🟡 中（商户端从合并视图变为拆分视图）

  类型C：老 API — 纯法币租户
    · 当前：API + IPL
    · 迁移到：新 API + IPL
    · 复杂度：🟢 低（Phase 0 已按 SP 拆分，接口格式适配即可）

  类型D：老 API — 数法融合租户
    · 当前：API + IPL + BB
    · 迁移到：新 API + IPL + BB
    · 复杂度：🟡 中（租户需适配新接口 + 多 SP 账户逻辑）
```

### 6.2 迁移顺序

```
推荐迁移顺序（从低风险到高风险）：

  第1批：类型A（老白牌纯法币）
    · 最简单，只有 IPL
    · 白牌由 EX 控制，不需要租户技术配合
    · 可作为迁移流程的验证批次

  第2批：类型C（老 API 纯法币）
    · Phase 0 已做 SP 拆分，数据结构已对齐
    · 租户只需切换 API 地址 + 适配新接口格式
    · 提供迁移文档 + 技术支持

  第3批：类型B（老白牌数法融合）
    · 商户端从合并视图变为拆分视图 → 需要商户教育
    · 余额拆分需要数据迁移（原来的"合并余额"拆成 BB 法币 + IPL 法币）
    · 划转功能必须同步上线

  第4批：类型D（老 API 数法融合）
    · 最复杂：租户需要改代码适配多 SP 账户
    · 需要给租户充分的适配期（建议 2-3 个月）
    · 提供详细迁移指南 + sandbox 环境
```

### 6.3 迁移的核心技术问题

| 问题 | 说明 | 方案建议 |
|------|------|----------|
| **余额拆分** | 老 EX 数法融合商户的法币余额是否已按 SP 独立？还是合并存储？ | 如已按 SP 独立 → 只改展示层；如合并 → 需数据迁移 |
| **历史交易** | 老 EX 历史交易是否带 SP 标识？ | 如不带 → 需要补标或接受历史数据不带 SP 标签 |
| **商户 ID** | 老 EX 的 MID 能否直接映射到新 EX？ | 建议保留原 MID，减少租户改动 |
| **审核状态** | 老 EX 已审核通过的商户，是否需要在新 EX 重新审核？ | 建议承认老 EX 审核结果，不重复审核 |
| **在途交易** | 迁移时有未完结的交易怎么办？ | 建议等所有在途交易完结后再迁移（设置冻结窗口） |
| **API 兼容** | 老 API 和新 API 能否并行？ | Phase 0 改造后的老 API 保留兼容期，给租户缓冲 |

### 6.4 白牌迁移：商户感知

```
对白牌商户的影响（数法融合场景）：

  迁移前（老白牌）：
    商户看到：USD 余额 10,000

  迁移后（新白牌）：
    商户看到：
      承兑账户（BB）：USD 7,000
      收付款账户（IPL）：USD 3,000
      划转按钮：BB ↔ IPL

  商户需要理解：
    · "为什么我的余额变成了两个？"
    · "哪个账户用来做什么？"
    · "怎么把钱从一个转到另一个？"

  商户教育方案：
    · 迁移前：站内公告 + 邮件通知（提前 2 周）
    · 迁移时：弹窗引导说明新账户结构
    · 迁移后：帮助中心 FAQ + 客服话术准备
```

## 7. Phase 3：平级 SP + 能力扩展

> **目标：引入 BB 做不了的业务能力，接入平级 SP**

### 7.1 新增能力

| 能力 | 说明 |
|------|------|
| 平级 SP 接入（如鲲鹏） | 人民币结汇等 BB 做不了的场景 |
| 人民币结汇风险转移 | 合规风险直接转移给结汇 SP |
| **Crypto Checkout 多 SP** | 引入其他 Crypto Checkout SP，补充 BB 做不了的币种/链 |
| 划转扩展 | BB ↔ 鲲鹏，IPL ↔ 鲲鹏 |
| 承兑流程中自动跨 SP 划转 | A2/B2 场景 |
| 多 SP 承兑（可选） | EX 统一汇率引擎 |

### 7.2 Crypto Checkout 多 SP 接入

```
Crypto Checkout 多 SP 接入策略：

  现状（Phase 1）：
    · BB 有 Crypto Checkout 能力
    · 但能力有限：覆盖主流币种/链，部分场景不支持

  Phase 3 目标：
    · 引入其他 Crypto Checkout SP（如专业收单机构）
    · 补充能力：
      - 更多币种/链
      - 特殊场景（如高并发、特定行业）
      - 更优费率/成功率

  路由策略：
    · EX 统一路由引擎
    · 根据币种/链/商户配置选择最优 SP
    · 商户端不感知底层 SP

  与承兑的区别：
    · 承兑（OnRamp/OffRamp）= BB 独占（数币钱包唯一）
    · Crypto Checkout = 多 SP（类似法币收付款模式）
```

### 7.3 前提条件

- 新 EX Phase 1 已稳定运行
- 跨 SP 划转的清算和对账基础设施已建成
- 平级 SP 的接入规范和风险转移协议已定义
- 大部分老 EX 客户已迁移完毕（Phase 2）

## 9. 历史客户迁移方案

### 9.1 迁移原则

```
原则❶：不中断业务
  · 迁移过程中商户可以继续交易
  · 如必须停服，窗口 ≤ 2 小时，提前 72 小时通知

原则❷：不丢数据
  · 余额、交易历史、审核状态全部迁移
  · 迁移前后余额必须一致（自动校验）

原则❸：不重复审核
  · 老 EX 已审核通过的商户，新 EX 承认审核结果
  · 避免商户重新提交 KYC/KYB

原则❹：给租户缓冲期
  · API 租户给 2-3 个月接口适配期
  · 老 API 保留兼容期

原则❺：从简单到复杂
  · 先迁纯法币 → 再迁数法融合
  · 先迁白牌 → 再迁 API
```

### 9.2 迁移时间线

```
Phase 0 (4月)          Phase 1 (5-6月)        Phase 2 (7-9月)
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ 老EX API改造  │      │ 新EX上线      │      │ 客户迁移      │
│ 按SP返回数据  │      │ 新白牌+API    │      │               │
│ 4月20日上线   │      │ 新租户入驻    │      │ 第1批:白牌法币 │
│               │      │               │      │ 第2批:API法币  │
│               │      │               │      │ 第3批:白牌融合 │
│               │      │               │      │ 第4批:API融合  │
└──────────────┘      └──────────────┘      └──────────────┘
      │                      │                      │
      │ 老API继续运行         │ 新老并行运行          │ 老EX逐步下线
      └──────────────────────┴──────────────────────┘
                    老EX API兼容期（≤6个月）
```

### 9.3 每批迁移的具体步骤

#### 白牌迁移（第1批/第3批）

```
步骤 1：数据准备
  · 导出老 EX 商户数据（MID、KYC 状态、余额、历史交易）
  · 在新 EX 创建对应的商户记录
  · 余额校验：老 EX 余额 = 新 EX 初始余额

步骤 2：白牌切换
  · 将租户的白牌指向新 EX 前端
  · 老白牌页面显示迁移公告
  · 新白牌页面显示欢迎引导

步骤 3：验证
  · 商户登录新白牌，确认余额正确
  · 执行一笔小额测试交易
  · 确认无误后标记迁移完成

步骤 4：善后
  · 老 EX 该商户数据标记为"已迁移"
  · 老 EX 该商户冻结（不允许新交易）
  · 保留历史数据查询 6 个月
```

#### API 迁移（第2批/第4批）

```
步骤 1：通知租户
  · 发送迁移通知（提前 1 个月）
  · 提供新 API 文档 + 差异对比表
  · 开放 sandbox 环境供测试

步骤 2：租户适配
  · 租户在 sandbox 环境完成接口对接
  · 重点适配：多 SP 余额、SP 标识、新接口格式
  · EX 技术支持协助

步骤 3：切换
  · 租户将生产环境 API 地址切换到新 EX
  · 老 EX API 保留只读访问（查询历史数据）

步骤 4：验证
  · 生产环境执行测试交易
  · 余额校验
  · 确认后关闭老 EX 写入权限
```

### 9.4 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 余额迁移不一致 | 商户投诉 | 迁移前后自动校验 + 人工复核 |
| API 租户适配慢 | 迁移延期 | 延长兼容期，不强制切换 |
| 在途交易未完结 | 数据不完整 | 设置冻结窗口，等在途交易完结 |
| 商户不理解新账户结构 | 客诉 | 提前教育 + 引导 + FAQ |
| 老 EX 下线后需要查历史 | 查询需求 | 历史数据保留 6-12 个月 |

## 10. 贯穿所有 Phase 的不变量

| 不变量 | 说明 |
|--------|------|
| **数币只在 BB** | 无论哪个 Phase，BB 始终是唯一数币资金中心 |
| **法币 SP 间不自动混合** | 因 nesting，法币账户始终按 SP 独立管理 |
| **EX 不持牌** | EX 始终是科技编排层，合规由 SP 负责 |
| **SP 审核独立** | 每个 SP 独立审核商户 |
| **老 API 兼容期** | 任何迁移都给租户充足适配时间 |
| **不丢数据** | 余额+历史交易+审核状态完整迁移 |
