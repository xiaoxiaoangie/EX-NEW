# FIBD 职责与 SP 生命周期管理

> 文档版本：v0.2
> 适用团队：EurewaX / EX - FIBD
> 当前假设：FIBD 暂按 1 人团队设计，先保证流程能跑通，不追求大而全。
> 文档目标：基于 SP 生命周期，明确 FIBD 在每个阶段要做什么、产出什么、找谁协作。

---

## 1. s周期建议拆成 6 个阶段：

```text
1. 需求识别
   ↓
2. SP 寻源
   ↓
3. 准入评估
   ↓
4. 签约接入
   ↓
5. 上线运营
   ↓
6. 风险处置 / 优化退出
```

这 6 个阶段比“准入评估 → 签约接入 → 运营维护 → 风险处置”更完整，因为在准入前需要先明确业务缺口和寻找 SP，运营后也需要有优化、替换或退出机制。

---

## 3. FIBD 最小职责清单

在 1 人团队阶段，FIBD 先聚焦 5 件最重要的事：

1. 找 SP：根据业务缺口寻找潜在 SP。
2. 评估 SP：判断 SP 的能力、价格、时效、合规和稳定性是否适合 EX。
3. 推动签约和接入：协调法务、合规、财务、产品、技术完成签约和上线。
4. 维护 SP 能力：确保 BD、产品、运营知道这个 SP 能做什么、不能做什么。
5. 处理 SP 侧问题：通道异常、费率变化、政策变化、时效问题、风险事件时，负责拉通 SP 和内部团队。

---

## 4. SP 生命周期职责与产出

### 阶段 1：需求识别

目标：先明确 EX 缺什么能力，再去找 SP。

FIBD 要做：

- 收集 BD 的租户需求。
- 收集产品侧的能力缺口。
- 识别当前 SP 覆盖不足的国家、币种、产品或支付方式。
- 判断缺口优先级。

重点判断：

- 是缺国家覆盖，还是缺币种？
- 是缺收款、付款、换汇、收单、卡，还是 Ramp？
- 是现有 SP 成本太高，还是时效不稳定？
- 是客户已经有明确需求，还是战略储备能力？

最小产出：

- SP 能力缺口清单

建议字段：

| 字段         | 说明                                    |
| ------------ | --------------------------------------- |
| 缺口类型     | 国家 / 币种 / 产品 / 通道 / 合规 / 成本 |
| 需求来源     | BD / 产品 / 客户 / 市场观察             |
| 具体需求     | 例如：需要肯尼亚 KES 本地付款           |
| 优先级       | 高 / 中 / 低                            |
| 期望上线时间 | 例如：1 个月内                          |

---

### 阶段 2：SP 寻源

目标：找到可能满足能力缺口的 SP。

FIBD 要做：

- 通过展会、行业资源、客户推荐、竞品观察等方式寻找 SP。
- 初步沟通 SP 的业务范围和合作意愿。
- 收集基础资料和联系人。
- 建立 SP Pipeline。

重点判断：

- 这个 SP 能不能补 EX 当前能力缺口？
- 它是核心能力 SP，还是备用 SP？
- 是否值得进入正式评估？

最小产出：

- SP Pipeline 清单

建议字段：

| 字段       | 说明                                  |
| ---------- | ------------------------------------- |
| SP 名称    | 服务商名称                            |
| 类型       | 银行 / PSP / 收单 / Ramp / 风控等     |
| 覆盖区域   | 国家或地区                            |
| 支持能力   | 收款 / 付款 / 换汇 / 收单 / 卡 / Ramp |
| 当前状态   | 初步接触 / 已会议 / 评估中 / 暂缓     |
| 联系人     | SP 对接人                             |
| 下一步动作 | 例如：等待报价、安排技术会议          |

说明：这里用 **SP Pipeline 清单**，不使用“资源池台账”，避免和财务台账或资金池混淆。

---

### 阶段 3：准入评估

目标：判断 SP 是否值得接入。

FIBD 要做：

- 收集 SP 的牌照、公司主体、产品介绍、覆盖地区、支持币种。
- 收集 SP 的报价、汇率、费率、结算周期、到账时效和限额。
- 收集 SP 的技术文档和接入方式。
- 拉合规判断资质与业务风险。
- 拉产品判断能力匹配度。
- 拉技术判断接入复杂度。
- 输出是否接入和优先级建议。

重点判断：

- 合规上能不能合作？
- 能力上是否真的补足 EX 缺口？
- 价格和汇率是否有竞争力？
- 到账时效是否能对外承诺？
- 接入成本是否值得？
- 是否有备用或替代 SP？

最小产出：

- SP 准入评估表

建议字段：

| 维度     | 需要确认的问题                       |
| -------- | ------------------------------------ |
| 主体资质 | 公司主体、牌照、监管地区             |
| 能力范围 | 国家、币种、产品、支付方式           |
| 成本     | 费率、汇率、固定费用、保证金         |
| 时效     | 入金、出金、结算、退款时效           |
| 限制     | 单笔限额、日限额、禁入行业、禁入国家 |
| 技术     | API、Webhook、沙箱、文档完整度       |
| 合规     | KYC、KYB、AML、制裁筛查、资料要求    |
| 风险     | 资金安全、政策稳定性、历史异常       |
| 结论     | 接入 / 暂缓 / 不接入                 |

---

### 阶段 4：签约接入

目标：把通过评估的 SP 推进到合同签署和技术接入。

FIBD 要做：

- 与 SP 确认商务条款。
- 推动合同审批和签署。
- 将费用、结算、保证金、发票等信息同步给财务。
- 将合同条款和责任边界同步给合规、法务、产品。
- 收集 API 文档、测试账号、技术联系人。
- 协调技术排期和联调。
- 跟进接入问题闭环。

重点判断：

- 合同是否明确责任边界？
- 费用和结算规则是否已同步财务？
- 技术资料是否完整？
- 是否具备测试条件？
- 是否有明确上线范围？

最小产出：

- SP 签约接入 Checklist

建议字段：

| 项目         | 负责人             | 状态                   |
| ------------ | ------------------ | ---------------------- |
| 商务条款确认 | FIBD               | 未开始 / 进行中 / 完成 |
| 合规审核     | 合规               | 未开始 / 进行中 / 完成 |
| 合同审核     | 法务               | 未开始 / 进行中 / 完成 |
| 财务信息同步 | 财务               | 未开始 / 进行中 / 完成 |
| 技术资料收集 | FIBD / 技术        | 未开始 / 进行中 / 完成 |
| 沙箱测试     | 技术 / 产品 / FIBD | 未开始 / 进行中 / 完成 |
| 生产验证     | 技术 / 运营 / FIBD | 未开始 / 进行中 / 完成 |

---

### 阶段 5：上线运营

目标：确保 SP 上线后，内部知道怎么用、边界是什么、异常怎么处理。

FIBD 要做：

- 明确 SP 上线能力范围。
- 向 BD、产品、运营同步 SP 支持能力和限制。
- 维护 SP 的国家、币种、产品、费率、时效、限额、合规限制。
- 跟踪上线初期交易表现。
- 协调 SP 处理异常订单、延迟到账、失败交易等问题。
- 定期确认 SP 能力是否发生变化。

重点判断：

- BD 是否知道这个 SP 能卖什么？
- 运营是否知道异常找谁处理？
- 产品是否知道能力边界和路由规则？
- 费率、汇率、时效、限额是否是最新版本？

最小产出：

- SP 能力卡片

建议字段：

| 字段       | 说明                                  |
| ---------- | ------------------------------------- |
| SP 名称    | 服务商名称                            |
| 上线状态   | 测试中 / 已上线 / 暂停 / 下线         |
| 支持产品   | 收款 / 付款 / 换汇 / 收单 / 卡 / Ramp |
| 支持国家   | 国家或地区                            |
| 支持币种   | 法币 / 数币                           |
| 费率规则   | 成本费率或报价规则                    |
| 汇率规则   | 锁汇、实时汇率、加点方式              |
| 到账时效   | T+0 / T+1 / T+1-3 等                  |
| 限额       | 单笔、日累计、月累计                  |
| 禁止场景   | 禁入行业、主体、国家                  |
| 异常联系人 | SP 与内部对接人                       |

---

### 阶段 6：风险处置 / 优化退出

目标：当 SP 出现问题时，快速降低业务影响，并决定继续使用、限流、暂停、替换或退出。

FIBD 要做：

- 接收并判断 SP 侧异常。
- 第一时间联系 SP 排查原因。
- 同步产品、运营、BD、合规和技术。
- 推动 SP 给出恢复时间和处理方案。
- 必要时建议暂停、限流或切换备用 SP。
- 事件结束后完成简短复盘。

常见风险：

- 通道中断
- 到账延迟
- 汇率异常
- 费率临时变化
- 交易失败率升高
- 合规政策突然变化
- 资金冻结
- SP 无响应
- SP 牌照或监管风险

最小产出：

- SP 风险事件记录

建议字段：

| 字段         | 说明                             |
| ------------ | -------------------------------- |
| 事件时间     | 发生时间                         |
| 涉及 SP      | 服务商名称                       |
| 事件类型     | 通道 / 资金 / 合规 / 技术 / 费率 |
| 影响范围     | 客户、币种、国家、产品           |
| 当前动作     | 排查 / 限流 / 暂停 / 切换        |
| SP 反馈      | 原因和预计恢复时间               |
| 内部同步对象 | BD / 运营 / 产品 / 技术 / 合规   |
| 最终结论     | 恢复 / 观察 / 限流 / 下线        |

## 5. FIBD 与其他团队的简单边界

| 团队         | 负责什么                               | FIBD 需要对接什么                            |
| ------------ | -------------------------------------- | -------------------------------------------- |
| BD           | 找租户、拿客户需求、推动业务成交       | 告诉 FIBD 客户需要什么国家、币种、产品和价格 |
| 商业产品经理 | 定义产品方案、能力包装、报价和路由规则 | 把 SP 能力转成可卖、可配置、可运营的产品能力 |
| 技术         | 系统开发、API 接入、联调、监控         | 判断能不能接、怎么接、多久接完               |
| 合规         | 资质审核、合规判断、禁入规则           | 判断 SP 和业务场景能不能做                   |
| 法务         | 合同审核和法律条款                     | 审合同、明确责任边界                         |
| 财务         | 合同台账、费用台账、结算、付款、发票   | 接收 FIBD 同步的商务条款和结算信息           |
| 运营         | 日常订单处理、异常跟进、客户通知       | 把 SP 规则和异常联系人用起来                 |

## 6. 当前阶段最重要的 4 个文档

考虑 FIBD 目前只有 1 个人，先不铺太多文档。建议先维护 4 个核心文件：

1. `SP能力缺口清单`
2. `SP Pipeline 清单`
3. `SP准入评估表`
4. `SP能力卡片`

这 4 个文件能覆盖最关键的问题：

- 我们缺什么能力？
- 现在有哪些 SP 在谈？
- 哪些 SP 值得接入？
- 已上线 SP 到底能做什么？

## 8. 当前建议

现阶段不要把 FIBD 写成完整部门体系，而是先把它定义成一个轻量闭环：

```text
发现缺口 → 找 SP → 判断值不值得接 → 推动上线 → 维护能力 → 出问题时协调处理
```

只要这个闭环跑顺，FIBD 的职责就基本成立。后续再根据交易量、SP 数量和风险复杂度，把流程逐步做重。
