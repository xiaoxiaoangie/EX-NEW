# B（MSB / VASP）做大方案：通过桥接借用 A 的法币能力

> *B 是数币持牌主体（MSB / VASP），客户有数币 + 跨境法币收付的混合需求，但 B 自身缺法币能力，且 B 的客户多数不满足 A 的直接 KYB 要求。*
>
> *本方案：让 B 的客户在客户视角"在 B 平台一站式收发法币 + U"，在 B 后台借用 T（桥接贸易主体）→ A（法币持牌）的法币能力，让 A 看到的永远是 T 的合规 B2B 贸易客户。*

## 1. 三主体重新定位

| 代号        | 主体                                       | 核心能力                                             | 缺什么                                                              |
| ----------- | ------------------------------------------ | ---------------------------------------------------- | ------------------------------------------------------------------- |
| **B** | 数币持牌主体（MSB / VASP）                 | U 钱包、链上结算、OnRamp / OffRamp、加密资产服务     | **法币 VA、法币 POBO、银行通道弱**；牌照宽松 = 客户准入门槛低 |
| **A** | 法币持牌主体（MSO / MPI）                  | **VA、POBO、银行通道、跨境法币收付**——能力强 | 保守、客户准入严、不接数币背景、不接缺材料客户                      |
| **T** | 桥接贸易主体（独立外贸公司，可信伙伴持有） | 三流齐备、合规材料、独立法人                         | 不持牌、自己不做金融业务，只做"贸易服务"                            |

**三方关系**：

```
       [B 主战场]
       做大 B 的流水 / 客户数
            │
   B 客户的法币收付需求
            │
            ▼
       [T 桥接层]
   翻译成 A 能接的合规 B2B 业务
            │
            ▼
       [A 法币能力]
   提供 VA + POBO + 银行通道
```

**关键洞察**：

- **B 不需要持有法币牌照**——借 A 的法币能力即可
- **A 不需要直接接 B 的客户**——A 只接 T（材料齐的合规商户）
- **T 吸收 B 客户的合规缺口**——T 是可信伙伴的独立主体，T 自身合规义务（KYC 终端、AML、单据）
- **B 客户体验完整**——在 B 后台一站式，不感知 T 和 A

## 2. B 想做大的业务清单

### 2.1 B 主战场业务

| # | 业务                               | 法币侧需求       | 数币侧需求                  |
| - | ---------------------------------- | ---------------- | --------------------------- |
| 1 | **OnRamp**（法币→U）        | 收法币（VA）     | 发 U 给客户钱包（B 自营）   |
| 2 | **OffRamp**（U→法币）       | 付法币（POBO）   | 收 U 进 B 钱包（B 自营）    |
| 3 | **Web3 项目方法币收付**      | VA + POBO        | 项目方部分用 U 结算，B 配合 |
| 4 | **跨境电商 + 数币结算**      | VA + POBO + 兑换 | U 结算部分 B 处理           |
| 5 | **数币背景客户的纯法币业务** | VA + POBO        | 不必要                      |
| 6 | **稳定币代发工资**           | POBO             | 部分员工拿 U                |

### 2.2 这些业务的客户特点

- **不一定有完整 KYB 材料**（A 直接接不了）
- **业务有"数币标签"**（A 当前一刀切拒）
- **客户量大、单笔小到中**（散户 / 中型项目方）
- **客户接受**："我在 B 平台用一站式服务"——不在乎后端结构

### 2.3 B 单独做不了 / 做得弱

| 能力         | B 现状  | 后果            |
| ------------ | ------- | --------------- |
| 法币 VA      | 无 / 弱 | 客户没法收法币  |
| 法币 POBO    | 无 / 弱 | 客户没法付法币  |
| 跨境法币收付 | 弱      | 客户体验差      |
| 银行通道     | 弱      | 单银行 / 限额低 |
| 多币种 FX    | 弱      | 兑换价差大      |

**结论**：B 做大必须借 A 的法币能力。但 B 的客户直接接 A 不可行（A 一刀切拒）。所以需要 T 作为合规翻译层。

## 3. 为什么必须要 T（不能 B 直接对接 A）

### 3.1 直接对接的失败模式

| 模式                                                            | 失败原因                                                                                                       |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| ❌ B 把客户推荐给 A 直接 KYB                                    | A 拒（数币背景 + 缺材料 + 客户体量小）                                                                         |
| ❌ B 自己作为 A 的客户（PI 同业 nesting）+ B 给客户开法币子账户 | A 接受持牌 PI nesting 但要求 EDD + 银行知情；且 B 后续需对每个终端做合规背书；B 替客户兜底法币业务的合规风险大 |
| ❌ B 直接用 A 的 VA + POBO API（白标）                          | A 不愿意承担 B 客户的合规风险                                                                                  |

### 3.2 T 桥接的成功逻辑

```
B 客户       T 桥接                    A 持牌
  │           │                         │
  │ 业务需求    │ 翻译为合规 B2B 贸易服务   │ 标准 KYB 通过
  │ ─────────► │ ───────────────────────► │
  │           │                         │
  │           │ T 提供：                  │ A 看到：
  │           │ - 营业执照、章程            │ - 一家正常贸易公司
  │           │ - 服务合同、发票           │ - 标准 B2B 流水
  │           │ - 三流凭证                 │ - 三流齐
  │           │ - 终端 KYC（自身合规）     │ - 没有数币标签
  │           │                         │
B 看到：       │                         │
- T 是合规法币 │                         │
  服务伙伴      │                         │
- 提供法币 VA  │                         │
  + POBO 服务  │                         │
```

**T 的本质**：把"B 客户的金融需求"翻译成"A 客户的贸易需求"，让 A 看到的全部是合规可解释的贸易行为。

## 4. 完整业务链路（B 视角，4 方向）

### 4.1 链路 1：B 客户 OnRamp（法币 → U）

```
[B 客户的买方]
   │ 法币付款（备注 reference = B 客户号 / 订单号）
   ▼
[T 在 A 的 VA（户名 = T）]
   │ A 看：T 的 B2B 贸易收款
   ▼
[T 在 A 的法币余额]
   │ T 内部账本：归集到 B 客户子账户
   │ T 把客户余额信息同步给 B
   ▼
[B 后台：B 客户余额 +X USD（B 与 T 协议同步）]
   │ B 客户在 B 后台发起 OnRamp
   ▼
[B 自营 OnRamp 引擎：B 发 U 给客户钱包]
   │
   │ T ↔ B 周期性净额结算：
   │   T 法币（在 A）→ B 法币账户
   │   B U 库存抵扣
```

**A 看到**：T 的标准 B2B 收款，三流齐
**B 看到**：自己客户完成 OnRamp，B 流水 +1
**B 客户看到**：在 B 平台收到法币 + 拿到 U

### 4.2 链路 2：B 客户 OffRamp（U → 法币）

```
[B 客户的 U 钱包]
   │ U 发到 B 的 OffRamp 钱包
   ▼
[B 收 U + 链上 KYC + 风控]
   │ B 后台：客户法币应收余额 +X USD
   ▼
[B 通知 T：请代付 USD X 到客户的供应商 / 客户的自名账户]
   │ T 对该笔做内部 KYC + 用途审查
   ▼
[T 在 A 发起 POBO 付款（户名 = T）]
   │ A 看：T 的标准 B2B 付款
   ▼
[供应商 / B 客户自名账户（在 MSB / 银行）收到法币]

[B ↔ T 净额结算：B 把 U 转给 T，T 用未来收款抵扣]
```

**A 看到**：T 的标准 B2B 付款
**B 看到**：自己客户完成 OffRamp + 法币落地
**B 客户看到**：在 B 平台 U → 法币 → 落到指定账户

### 4.3 链路 3：B 客户的纯法币收款（不涉 U）

```
[B 客户的买方]
   │ 法币
   ▼
[T 在 A 的 VA + reference]
   │
   ▼
[T 内部账本：归集到 B 客户]
   │ B 后台同步显示客户余额
   ▼
[B 客户后续：
   - 余额留 B 法币账户
   - 发起 OnRamp（链路 1）
   - 发起付款（链路 4）
   - 提现到自己其他账户（POBO 操作，链路 4）
]
```

### 4.4 链路 4：B 客户的纯法币付款（不涉 U）

```
[B 客户在 B 后台发起付款 USD X 给供应商 ABC]
   │ B 风控审 + 余额扣
   ▼
[B 通知 T：代付 USD X 给 ABC]
   │ T 内部 KYC + 用途审
   ▼
[T 在 A 的 POBO 付款（户名 = T，备注 B2B 服务费 / 货款）]
   │ A 看：T 的标准付款
   ▼
[ABC 收到 T 名义款项]
```

**变种**：客户要求 **自名付款（POBO own）**

- 如果 B 客户接受 T 名义付出 → 用上述链路
- 如果客户必须自己名义付出 →
  - 客户必须在 A 直接有自名账户（少数大客户场景）
  - 或客户在另一持牌主体有自名账户，T 把法币转到该账户后客户自己付（参 `a-leverage-bridge-strategy.md` §5.2.2）

## 5. B、T、A 的法律 + 清算关系

### 5.1 法律关系

| 双方                  | 关系                                                          | 协议                                       |
| --------------------- | ------------------------------------------------------------- | ------------------------------------------ |
| **B ↔ T**      | T 给 B 提供"法币代收代付服务" + B 给 T 提供"数字资产清算服务" | 双向服务协议 + 净额结算协议 + 数据共享协议 |
| **T ↔ A**      | T 是 A 的标准 B2B 商户客户                                    | A 的标准商户协议（无任何关联交易披露）     |
| **B ↔ A**      | 无直接合同，或 B 作为持牌 PI 同业客户在 A 开账户（视情况）    | 若有，则单独 EDD + 银行知情                |
| **B 客户 ↔ B** | B 平台用户协议                                                | B 自身的服务协议                           |
| **B 客户 ↔ T** | B 客户可能完全不知道 T，或知道"B 平台合作的法币服务伙伴"      | 视暴露策略                                 |

### 5.2 资金清算结构

```
日常业务（T 流水）
   │
   ├── T 在 A 的法币流入（来自 B 客户的买方）
   │     ▼
   │  T 在 A 的余额沉淀
   │     │
   │     ├── 用于 T 在 A 的代付（B 客户付款 / OffRamp 落地）
   │     └── 周期性结算给 B（净额）
   │
B 在数字资产端
   │
   ├── B 收 U（来自客户 OffRamp）
   │     ▼
   │  B 用 U 抵扣 T 给 B 的法币结算
   │
   └── B 发 U（OnRamp 给客户）
         ▼
       B 的 U 库存消耗 → 由 T 法币结算补充

净额结算原则：
   - T → B 法币：覆盖 B 给客户的 OnRamp / 法币提现额
   - B → T U 或法币：覆盖 T 给客户付款 / OffRamp 落地额
   - 周期：日 / 周 / 月（视体量 + 信用）
   - 阈值：超过 X 即时结算，避免单边敞口过大
```

### 5.3 单边敞口风险

| 风险                                       | 防控                             |
| ------------------------------------------ | -------------------------------- |
| T 在 A 的余额过多（B 没及时结算给 T 法币） | 设上限 + 强制结算阈值            |
| B 持有过多 U 但 T 没及时给 B 法币          | 同上 + 备付金                    |
| 单方违约 / 跑路                            | 备付金 + 双方账本对账 + 法律协议 |

## 6. B 客户在 B 后台的一站式体验

### 6.1 B 客户看到的功能

| 功能               | 实现                                               |
| ------------------ | -------------------------------------------------- |
| 法币收款 VA        | T 在 A 的 VA + reference 标识 B 客户（B 后台展示） |

| OnRamp（法币→U）  | B 自营 + T 法币收                                  |
| OffRamp（U→法币） | B 自营 + T 法币付                                  |
| 法币兑换           | B 调用 T 的 FX，或 B 自身 FX                       |
| U 钱包             | B 自营                                             |
| 加密资产服务       | B 自营                                             |

### 6.2 B 客户的知情度

| 客户类型          | 知道                               | 不知道                    |
| ----------------- | ---------------------------------- | ------------------------- |
| 大客户 / 关键客户 | "B 平台后端有合作的法币服务伙伴 T" | T 与我方的关系 / A 的存在 |

| 小客户 / 散户     | 在 B 平台一站式                    | 任何后端结构              |

### 6.3 客户合规义务

- B 对自己客户做 **B 自身的 KYC**（按 MSB / VASP 标准）
- T 对 B 客户做 **T 自身的 KYC**（按 T 的贸易服务客户标准）——可与 B 做共享，但各自独立留底
- 双重 KYC 不重复但不缺失

## 7. T 的角色与设计要点

### 7.1 T 在本方案中的定位

T = **法币代收代付服务公司**（独立外贸 / 跨境贸易服务公司）

| 视角     | T 的样子                                               |
| -------- | ------------------------------------------------------ |
| A 视角   | 一家正常的跨境贸易 / 跨境服务公司，做 B2B 业务，三流齐 |
| B 视角   | 一家有 A 法币通道能力的合规合作伙伴                    |
| 监管视角 | 独立法人，自己做合规、自己报送                         |
| 我方视角 | 可信伙伴持有的独立主体，与我方零股权交叉               |

### 7.2 T 的"业务范围"设计

T 的工商业务范围必须能覆盖以下"对外服务"：

- 跨境贸易服务 / 跨境集采服务
- 跨境数字商品分发 / SaaS 经销
- 跨境收付款代理服务（**关键**：让"代收代付"在工商范围内合法）
- 跨境结算咨询 / 外贸顾问

### 7.3 T 的客户池构成（重要）

> 参 `a-leverage-bridge-strategy.md` §6.2，但 B 视角下重点：

| 类型                                            | 占比             | 作用                   |
| ----------------------------------------------- | ---------------- | ---------------------- |
| **真实直接小商户**（自身经营，非 B 客户） | **30-40%** | 防穿透时能展示真实经营 |
| B 客户（数币 / Web3 / 缺材料）                  | 30-40%           | 业务核心               |
| 灰色行业客户                                    | 20%              | 桥接吸收               |
| 其他高频小客户                                  | 10%              | 流水多样性             |

> **底线**：T 的真实直客占比必须 ≥ 30%，否则 T 业务结构看起来全是"B 的衍生"，穿透即破防。

### 7.4 T 的合规材料（A 视角）

| 文件                                       | 谁出                                         |
| ------------------------------------------ | -------------------------------------------- |
| 营业执照 + 章程 + UBO                      | T                                            |
| 业务说明（跨境贸易服务公司）               | T                                            |
| 客户清单（脱敏后）+ 合同模板               | T                                            |
| 服务发票 + 物流单 / 服务交付凭证           | T 按业务出                                   |
| 银行账户（多家）+ 流水                     | T                                            |
| 与 B 的合同（脱敏 / 翻译为"贸易服务合作"） | T 提供给 A 的版本须避免暴露 B 的数币业务标签 |

## 8. 合规 + 红线

### 8.1 B 自身合规（不动）

B 作为 MSB / VASP，遵守：

- BSA / AML / OFAC（MSB）/ 各地 VASP 要求
- 链上 KYC + 风控（Chainalysis / TRM）
- Travel Rule（如适用）
- SAR / STR 报送

### 8.2 T 自身合规（不动）

T 作为独立法人，遵守：

- 主体所在地的工商 / 税务 / 社保
- 对终端客户做 KYC（按贸易服务公司标准）
- 反洗钱内部程序
- 单据保存 ≥ 7 年
- 协助监管穿透检查

### 8.3 A 自身合规（不动）

参 `fle-maximization-strategy.md` §2 + §2.5。

### 8.4 B 视角特有红线

| #   | 红线                                                                            | 后果                               |
| --- | ------------------------------------------------------------------------------- | ---------------------------------- |
| RB1 | ❌ B 把"用 T → A 通道"作为对客户的明示卖点                                     | 暴露架构                           |
| RB2 | ❌ B 客户的资金 / 单据 / 合同上出现 A 的名字                                    | A 牌照风险                         |
| RB3 | ❌ B 给 T 推送的客户清单 100% 全是 B 客户（必须有 T 自身真实客户）              | 桥接结构失防                       |
| RB4 | ❌ B / T / A 三方在同一办公地址 / 同一邮箱域 / 同一员工                         | 关联交易认定                       |
| RB5 | ❌ B 客户中制裁国 / 高风险国家比例过高（一刀切拉黑）                            | 整体合规风险                       |
| RB6 | ❌ B / T / A 任一对监管 / 银行 / 税务的口径不一致                               | 穿透后整体破防                     |
| RB7 | ❌ T 在 A 的余额超过约定上限（防止 T 变成"B 的 nostro 账户"被认定为类银行业务） | MSO 红线                           |
| RB8 | ❌ B 把"我们能做法币 VA + POBO"作为 B 自身的法币能力宣传                        | 持牌虚假宣传风险（B 没有法币牌照） |

### 8.5 灰区诚实声明

本架构灰区性质：T 是一家"真实贸易公司"，但其"代收代付"业务部分实际承接的是 B 的客户金融需求。这部分的合规防御靠：

1. T 的真实独立性 + 真实经营底盘
2. T 自身 KYC + 反洗钱程序
3. T 客户池 ≥ 30% 真实直客
4. 三流凭证真实
5. 监管报送真实
6. 与 B / A 法律 / 治理 / 信息隔离
7. 红线不越

任一造假即整体失防。

## 9. KPI 体系（B 中心）

### 9.1 B 核心 KPI（季度）

| KPI                                                            | 当前  | 一期     | 二期 |
| -------------------------------------------------------------- | ----- | -------- | ---- |
