B-A 联动场景：OnRamp / OffRamp 流程

基于 MSO 牌照监管框架 + 后端渠道（DBS / ICBC / BC 等）"服务直客"硬要求设计的实际业务流程。

📘 文档说明

目的：充分利用 A 的核心能力，承接 B 客户的法币收付需求。

A 的核心价值：

  • 优势大行通道（DBS / ICBC / BC 等）—— 资金到账快、合规背书强、银行评级高
  • 小币种覆盖 —— 大量非主流币种可结汇 / 下发，是 B 客户最稀缺的能力
  • 本地下发能力 —— 本地清算网络（ACH / FPS / SEPA / 各国本地汇路），不走 SWIFT，成本低、效率高、到账快
  • POBO 能力 —— 替客户名义出款（Payment On Behalf Of），客户不用自己开境外账户也能完成跨境付款

当前痛点：A 合规政策保守（材料严、客户分级硬、白名单制），部分 B 客户场景无法走 A 直连。

解决思路：业务分场景设计 ——

  • 正常场景：客户走 A 直连，T 不介入
  • 异常场景：借用 T 桥接（替代收付 / 提供材料），让业务合规可走

🎯 业务背景

  • MSO 牌照监管角度，A 可以接 SRO（B 这类持牌主体）做通道
  • 但 A 后端主要渠道（DBS / ICBC / BC 等）要求：不能做 nesting，必须服务直客
  • 解决方案：B 通过 API 把客户推送到 A，A 对该客户做 full KYC，客户在 A 是直客身份
  • 这样既满足渠道"直客"要求，又让 B 客户能用 A 的能力（VA 收款 / POBO 付款 / 本地下发）
💡 优化方向：full KYC 需要审核时效，影响客户开户体验。最优是说服渠道允许 nesting + 简单 KYC + 实时下号——这是未来争取的战略空间。

一、OnRamp 场景（法币 → U）

按资金来源分两类：客户自己同名充值 / 境外买家付款。

1.1 同名充值（客户001 自己境外账户充值法币上链）

客户001 用自己境外同名银行账户向 A 的 VA 充值，然后在 B 上链拿 U。

✓ 正常场景

正常 客户001 自己同名充值

  4. A 入账审核通过 → 通知 B 到账
  5. B OnRamp 按汇率把 U 发到 客户001 钱包

异常 客户001 从其他承兑机构 USD 进 A 的 VA

🚀 进阶 · 把客户的 OffRamp 业务拿过来一起做
  1. BD 阶段说服客户001 把承兑业务整体迁到我方 B 平台
  2. 客户001 直接在 B 做 OffRamp（U → 法币），资金合规来源完整闭环
  3. 同步开通 OnRamp / 法币收付能力，形成全链路客户
  4. 如客户资质低，引入 T 提供贸易背景服务（参 1.2 异常 2）
价值：B 获得高 GMV 客户（承兑场景客单量大）；A 通过完整链路证据接得下；客户的 OnRamp + OffRamp 同时在我方做。

二、OffRamp 场景（U → 法币）

OffRamp 分为两大类：提现（回自己同名账户）和付第三方（POBO 付给收款人）。核心规则：凡是付给第三方，必须从客户001 在 A 的账户出款（收款行调单会查到 A，需要客户名义 POBO）。

⚠️ OffRamp 核心规则

  • 提现（同名回自己）：需要贸易背景解释"外汇收入从哪来"——什么生意让你有外币收入要提现？
  • 付第三方：需要提供与收款方的凭证（合同/发票/PI）。客户没有的话 T 可以出。
  • 付第三方必须客户名义：收款行可能调单（发邮件到 A 确认），所以只要付第三方都必须从客户001 在 A 的账户出，确保 POBO 名义是客户001。
  • T 名义出款仅限提现：T 直接 POBO 到第三方会导致收款行调单时对不上，运营风险大。

2.1 提现（法币回客户自己同名账户）

客户001 把 U 换成法币后，提现到自己的境外同名银行账户。A 审核重点：外汇收入来源（什么贸易/服务让你有这笔钱）。

正常：客户001 能自己解释收入来源

正常 客户001 有贸易背景解释外汇收入
客户001 做跨境贸易/服务（如出口货物、提供技术服务），有外汇收入的合理来源，材料齐全。
  1. 客户001 在 B 入网 → B API 推送到 A → A full KYC 通过开户
  2. 客户001 在 B 发起 OffRamp，U 转到 B 指定地址
  3. B 卖出 U 得法币 → 转到 客户001 在 A 的账户
  4. 客户001 在 A 发起提现（同名转出）→ 法币到客户001 境外同名银行账户

⚠️ 为什么付第三方不能从 T 账户出？

收款行在处理跨境入账时可能调单（发邮件到付款行/A），核实：
• 汇款人是谁？
• 与收款人是什么关系？
如果汇款人是 T 而不是客户001，收款方会疑问"这是谁付的款？"；A 回复调单时也对不上客户001 的交易。
结论：只要付第三方，必须从客户001 在 A 的账户出（客户名义 POBO）。

五、A 的可见性讨论：客户能否不知道 A 的存在？

📌 OnRamp 场景：客户001 能否不感知 A？

Q：OnRamp 中有没有客户001 不知道 A 存在、但我们利用 A 能力的可能？
结论：没有。
VA 是开在 客户001 名下的（A full KYC 直客要求），客户001 一定知道 VA 来自 A。即使入账时没调单，出款时 A 也可能调单，客户会看到 A 的邮件 / 通知。
除非入账出款全程不调单（A 白名单制），但这不可控，不能作为产品假设。
所以 OnRamp 所有场景中，客户001 对 A 可见。

📌 OffRamp 场景：客户001 能否不感知 A？

Q：OffRamp 非正常 1 中，T 代收 U → OffRamp → T 在 A 账户 POBO 给客户001，客户001 能不知道 A 吗？
结论：技术上可以，但运营上有风险。
在 OffRamp 非正常 1 中：客户001 只与 T 和 B 交互（发 U 到 T 钱包），最终收到的法币是 T 名义 POBO 出款，客户001 看到的汇款人是 T（不是 A）。
但有以下运营问题：
  • 合规审计可追溯 —— A 持有 T 的 KYC 和所有交易记录，监管 / 合规查到 A 一定能查到最终收款人是 客户001
  • 客户查询穿透 —— 客户001 如果对收到的款项有疑问（金额差异 / 到账延迟），找 B 或 T 客服，客服排查时需要查 A 的交易记录，可能暴露 A 的存在
  • 退款 / 争议处理 —— 如果 POBO 出款被银行退回 / 冻结，处理链路会涉及 A，客户001 可能收到 A 侧的通知或 SWIFT 报文中的银行信息
  • 前后台沟通成本 —— B 的前台客服需要理解"背后有 A"的链路，增加培训成本；如果 B 客服直接告诉客户"我们的资金通过 XX 机构处理"，则 A 暴露
建议：不刻意隐藏 A，而是在客户沟通中把 A 定位为"合规资金通道合作伙伴"，降低客户对 A 的关注度但不回避。

📌 OffRamp 非正常 2：客户001 一定知道 A

Q：OffRamp 非正常 2（客户自名 POBO）中客户001 对 A 的感知？
结论：一定知道。
客户001 自己在 A 入网开户 + 自己在 A 发起 POBO，完全感知 A。这个场景下 A 对客户001 是透明的。
