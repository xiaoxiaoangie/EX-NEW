B-A 联动场景：OnRamp / OffRamp 流程

基于 MSO 牌照监管框架 + 后端渠道（DBS / ICBC / BC 等）"服务直客"硬要求设计的实际业务流程。

📘 文档说明

目的：充分利用 A 的核心能力，承接 B 客户的法币收付需求。

A 的核心价值：

  • 优势大行通道（DBS / ICBC / BC 等）—— 资金到账快、合规背书强、银行评级高
  • 小币种覆盖 —— 大量非主流币种可结汇 / 下发，是 B 客户最稀缺的能力
  • 本地下发能力 —— 本地清算网络（ACH / FPS / SEPA / 各国本地汇路），不走 SWIFT，成本低、效率高、到账快
  • POBO 能力 —— 替客户名义出款（Payment On Behalf Of），客户不用自己开境外账户也能完成跨境付款

当前痛点：A 合规政策保守（材料严、客户分级硬、白名单制），部分 B 客户场景无法走 A 直连。

解决思路：业务分场景设计 ——

  • 正常场景：客户走 A 直连，T 不介入
  • 异常场景：借用 T 桥接（替代收付 / 提供材料），让业务合规可走

🎯 业务背景

  • MSO 牌照监管角度，A 可以接 SRO（B 这类持牌主体）做通道
  • 但 A 后端主要渠道（DBS / ICBC / BC 等）要求：不能做 nesting，必须服务直客
  • 解决方案：B 通过 API 把客户推送到 A，A 对该客户做 full KYC，客户在 A 是直客身份
  • 这样既满足渠道"直客"要求，又让 B 客户能用 A 的能力（VA 收款 / POBO 付款 / 本地下发）
💡 优化方向：full KYC 需要审核时效，影响客户开户体验。最优是说服渠道允许 nesting + 简单 KYC + 实时下号——这是未来争取的战略空间。
客户客户001 A · 法币持牌 B · 数币持牌 T · 桥接主体 买家 / 供应商

💰 真实资金流：B-A 定期调拨

真实资金流 vs 信息流（业务视角）的区别

  • 信息流（业务视角）：描述每笔交易在各系统中的逻辑流转路径（客户→A VA→B OnRamp→客户 U 钱包），是订单/记账维度
  • 真实资金流：实际的钱在 B 和 A 之间如何移动 — 不是逐笔实时转账，而是定期批量调拨
💰 B-A 定期调拨机制
核心原则：B 和 A 之间的资金不是逐笔跟随交易实时流转，而是定期（T+1 / 周结 / 自定义周期）批量调拨净额。
A 侧：客户的法币余额是 A 的记账（A 是实际资金持有方）。
B 侧：OnRamp/OffRamp 引擎操作的是 B 在 A 的备付金/保证金池。

OnRamp 方向（法币 → U）真实资金流

  1. 客户法币入账到 A（A 实际持有法币）
  2. B 在 A 的备付金账户中扣减对应金额（记账）
  3. B 发放 U 给客户（B 的 U 库存减少）
  4. 定期调拨（真实资金移动）：A 按约定周期将客户入账汇总净额划转到 B 的结算账户

二、OffRamp 场景（U → 法币）

OffRamp 分为两大类：提现（回自己同名账户）和付第三方（POBO 付给收款人）。核心规则：客户只操作 B 平台，B 通过 API 调用 A 执行出款（客户001 名义 POBO，收款行调单会查到 A）。

⚠️ OffRamp 核心规则

  • 提现（同名回自己）：需要贸易背景解释"外汇收入从哪来"——什么生意让你有外币收入要提现？
  • 付第三方：需要提供与收款方的凭证（合同/发票/PI）。客户没有的话 T 可以出。
  • 付第三方必须客户名义：收款行可能调单（发邮件到 A 确认），所以只要付第三方都必须以客户001 名义 POBO（B 通过 API 调用 A 执行）。
  • T 名义出款仅限提现：T 直接 POBO 到第三方会导致收款行调单时对不上，运营风险大。

2.1 提现（法币回客户自己同名账户）

客户001 把 U 换成法币后，提现到自己的境外同名银行账户。A 审核重点：外汇收入来源（什么贸易/服务让你有这笔钱）。

正常：客户001 能自己解释收入来源

客户001 做跨境贸易/服务（如出口货物、提供技术服务），有外汇收入的合理来源，材料齐全。

五、T 的可见性讨论：A 是否能发现桥接主体 T 的桥接角色？

核心原则：客户只操作 B 平台，对 T 完全无感知（所有 T 相关操作由 B 系统自动完成）。
因此真正的风险不在客户侧，而在A 的合规调单中是否能发现 T 的桥接角色。

📌 不涉及 T 的场景（正常场景 + 异常2）

OnRamp 正常 1（同名充值）、正常 2（买家付款）、异常 2（客户自行找材料）、OffRamp 提现正常、POBO 正常 —— 这些场景中 T 不参与，客户001 自行对接 A，调单链路中没有 T。
结论：不存在 T 可见性问题。

📌 OnRamp 异常 1（T 代收）：客户知道 T，但 A 不知道客户

Q：A 合规调单时能看到 T 背后的客户001 吗？
结论：A 看不到客户001。
A 的直客是 T，调单对象是 T。A 只看到"买家→T 的 VA"这笔入账，不知道 T 背后有客户001。
客户001 感知 T：无感知（系统自动完成，客户只看到 B 平台的 OnRamp 到账）。
T 暴露风险：低 —— 除非 A 深入审计 T 的业务模式并追问"你的下游客户是谁"。

📌 OffRamp 提现非正常（T 代为提现）：客户知道 T，A 可能追问

Q：T 名义出款给客户001，A 会不会追问 T 与收款人的关系？
结论：很可能。
A 看到 T 出款到客户001 的账户，合规可能追问"收款人是谁？和 T 什么关系？"。T 需要有合理解释（如"服务外包费""贸易尾款"等）。
客户001 感知 T：无感知（客户在 B 发起 OffRamp/提现，系统自动经 T 完成，客户只看到法币到账）。
T 暴露风险：中高 —— A 可从收款人信息推断 T 的桥接角色。

📌 OffRamp POBO 非正常（T 提供材料 + 客户自名出款）：T 暴露风险最高

Q：A 能关联 T→客户001 内转 和 客户001→第三方 POBO 两条链路吗？
结论：能。
A 系统中同时存在：① T→客户001 的内转记录 ② 客户001 的 POBO 出款记录。两笔时间、金额高度关联，A 合规很容易串联。
如果 A 追问"客户001 的 POBO 资金从哪来？"→ 发现来自 T 的内转 → 追问"T 为什么转钱给客户001？"→ T 的桥接角色暴露。
客户001 感知 T：无感知（客户在 B 发起付款，系统自动经 T 完成内转 + POBO，客户只看到付款成功）。
T 暴露风险：高 —— A 能看到完整 T↔客户001 资金往来。
