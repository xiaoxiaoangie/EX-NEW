BB-X-SRO-渠道 信息流与资金流详解

按渠道维度拆解每个 OnRamp / OffRamp 场景在各系统中的资金流、信息流、调单链路。
渠道 = MTC / A(IPL) / 智朗（大账户）｜ SRO = 独立 SP 系统（不是渠道）｜ BB = 租户

核心概念

  • BB（租户）：面客营业，承兑报价，客户管理。BB SRO 需要在渠道入网开户
  • SRO（SP 系统）：合规数据视图。不是渠道，不开 VA、不出款。SRO 只看到桥接主体 X 的数据（信息流）
  • 渠道：MTC / A(IPL) / 智朗等，独立提供 VA 开户、入账、出款能力
  • 桥接主体 X：在 SRO 入网的合规客户，与真实客户之间有映射关系。桥接主体 = 供应商，客户 = 买家
  • BB SRO 入网渠道：BB SRO 主体需要在各渠道入网开户（作为渠道的客户/主账户）
  • 桥接主体 T（B-A flow）vs X（B-X-SRO flow）：是否为同一主体 → 见文末讨论

客户端所见

OnRamp VA 场景：客户开 VA → VA 入账后 OnRamp 到客户钱包 → 客户添加提币地址 → 提币

OffRamp：客户发起出款/提现 → 法币到收款人

一、系统结构与入网关系

BB SRO 在渠道的入网关系

  • MTC：SRO 在 MTC 开户入网，作为主账户；桥接主体 X 做 SRO 子账户；客户做 SRO 子账户
  • A (IPL)：BB 已在 A 入网开户（做发卡时已完成）；桥接主体 T(X) 在 A 入网开 VA；客户在 A 入网开 VA。BB 与 A 之间定期调拨，不笔笔调拨
  • 智朗：大账户模式，客户看到一个充值大账户

各系统数据视图

系统 MTC 渠道 A (IPL) 渠道 智朗渠道
渠道系统 SRO 主账户 → 桥接子账户 + 客户子账户 BB 已在 A 入网开户 + 桥接主体 T(X) 的 VA + 客户的 VA（各自入网）
BB 与 A 定期调拨净额		 大账户（客户充值到统一账户）
BB 运营系统 客户和桥接都入网，都开 VA 客户充到 VA，做 OnRamp 客户看到大账户，充值后 OnRamp

关键原则：SRO 系统上只有一个信息流，没有资金流。资金流在渠道层。SRO 看到的永远是桥接主体的数据。

📌 OnRamp 主子账户体系设计问题

Q：是否将桥接主体做成子账户，做一套主子账户体系？
设计方向：所有子账户的入账，在主账户下做一个信息流，都是桥接主体 VA 异名入账。SRO 向桥接主体调单。

主子账户结构：
• 主账户 = SRO（或 BB SRO 主体）
• 子账户 = 桥接主体 X（SRO 系统中的客户）
• 每个真实客户映射到一个桥接主体子账户

信息流规则：无论哪个渠道入账，SRO 系统上统一记录为"桥接主体 VA 入账（异名）"
调单规则：SRO 统一向桥接主体调单

⚠️ 风险分析：异名出款场景

风险低：客户添加的是自己的提现账户
→ 桥接主体提供与客户的关系证明材料（供应商-买家关系）→ 可以解释

风险高：客户添加的是第三方供应商/服务商
→ 桥接主体需要提供与该第三方的关系材料 → 困难，因为桥接主体与该第三方可能没有真实业务关系

风险极高：收款行向付款行发起反向调单
→ 付款行 → 向 A → 向 SRO 发起调单
→ SRO 提供的是桥接主体的证明
→ 问题：收款人（客户的供应商）并不知道桥接主体的存在，无法确认"桥接主体代客户付款"这个关系
→ 收款行可能不接受解释 → 资金冻结风险

四、核心问题讨论

📌 问题 1：BB SRO 需要在渠道入网开户

Q：BB SRO 在各渠道的入网角色是什么？
MTC：SRO 在 MTC 作为主账户入网，桥接主体和客户做子账户
A (IPL)：BB 已在 A 入网开户（做发卡时已完成），双方定期调拨净额，不笔笔调拨
智朗：BB SRO 作为大账户的持有方

核心：BB SRO 是渠道层的"伞"，所有客户和桥接主体都在 BB SRO 下面

📌 问题 2：系统上维护桥接主体和客户的关系

Q：如何在系统上维护映射关系？如果客户满足 SRO 要求怎么办？桥接主体更换怎么办？
基础映射：X001 ↔ Merchant001（一对一映射表）

场景 A：客户满足 SRO 要求
→ 客户自己入网 SRO → 自己映射自己（客户 = 自己的桥接主体）
→ 结构上等于：桥接主体 = 客户本人，无实质桥接
→ SRO 系统直接看到客户数据，无需映射转换
→ 系统设计：映射表中 bridge_entity = customer_id（自映射标记）

场景 B：桥接主体更换（异常场景）
→ 原桥接主体 X001 不能用了（如被 SRO 拒绝、主体注销等）
→ 需要将客户映射到新桥接主体 X002
→ 操作步骤：
  1. 新桥接主体 X002 在 SRO 入网
  2. X002 在渠道开新子账户/VA
  3. 更新映射关系：X002 ↔ Merchant001
  4. 在途交易完成后，关闭 X001 相关账户
  5. SRO 系统中，后续数据归属 X002
→ 风险：SRO 可能会注意到"新客户突然出现大量交易"，需要提前沟通

📌 问题 3：桥接主体 T（B-A flow）和 X（B-X-SRO flow）是否要一样？

Q：T 和 X 用同一个主体好还是不同主体好？
方案 1：T = X（同一主体）
✅ 优点：
  • 一套主体、一套合规材料、一套运营体系
  • 映射关系简单：T(X) ↔ 客户（不需要 T-X 之间再映射）
  • 调单时材料一致：无论是 A 调单还是 SRO 调单，桥接主体是同一个
  • 客户只需知道一个"代收代付"合作方
❌ 缺点：
  • 更难找：需要同时满足 A(IPL) 的入网要求 + SRO 的入网要求
  • 单点故障：如果这个主体出问题，A 和 SRO 两边同时受影响
  • A 和 SRO 的 KYC 标准不同，同一主体需要同时满足两套标准

方案 2：T ≠ X（不同主体）
✅ 优点：
  • 风险隔离：A 那边出问题不影响 SRO，反之亦然
  • 分别找更容易：T 满足 A 的标准即可，X 满足 SRO 的标准即可
  • 灵活更换：换一边不影响另一边
❌ 缺点：
  • 需要搭一套 T-X 映射：系统上不仅要维护 T↔客户、X↔客户，还要知道同一客户在 A 侧用 T、在 SRO 侧用 X
  • 运营复杂度翻倍：两套桥接主体、两套合规材料、两套调单链路
  • A 调单和 SRO 调单时，需要确保两边提供的材料不矛盾
  • 映射难度更大：客户→T、客户→X、T 和 X 之间如何协调？

💡 推荐：T = X（同一主体），但准备备用
  • 主方案：优先找能同时满足 A + SRO 的主体作为 T=X
  • 备用方案：如果找不到，再拆为 T≠X，接受映射复杂度
  • 长期规划：准备 2-3 个桥接主体轮换使用（不论 T=X 还是分开）

📌 问题 4：收款行反向调单风险（OffRamp 最大风险）

Q：如果收款行向付款行发起调单，付款行向 SRO 发起调单怎么办？
调单链路：收款行 → 付款行 → A/渠道 → SRO → 提供桥接主体证明

问题所在：
• SRO 提供的是桥接主体的证明
• 但收款人（客户的供应商/服务商）并不知道桥接主体的存在
• 收款人只认识客户，不认识桥接主体
• 收款行拿到"桥接主体代客户付款"的解释 → 可能不接受

分场景评估：
出款对象风险说明
客户自己的提现账户低桥接主体→客户 = 服务合同关系，容易解释
客户的关联公司中需要桥接主体与关联公司的间接关系证明
客户的第三方供应商高桥接主体与第三方无真实业务，材料难以编造

缓解方案：
  • 方案 A：限制 OffRamp 只能出到客户自己的账户（同名提现），不支持第三方出款
  • 方案 B：第三方出款必须走客户自己的渠道账户（不走桥接主体），接受 SRO 异名
  • 方案 C：出款备注中标注"On behalf of [客户名]"+ 客户提前通知收款方
