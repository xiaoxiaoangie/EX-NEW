B-X-SRO 联动场景：MSB 对外营业 + SRO 通道

B 用 MSB 主体面客，X 桥接到 SRO 通道完成承兑 / 支付，K 作为壳主体替 SRO 在特殊渠道开户。解决 SRO 入网门槛高 + 渠道开户受限的双重问题。

文档背景

本文档描述 B 在 SRO 监管环境下的双主体运营架构：

  • 问题：SRO 要求比以往 VASP 严格得多，B 的历史客户大部分不满足 SRO 入网条件
  • 方案：B 用 MSB 主体对外营业（低门槛入网），SRO 主体作为 B 的底层通道（高合规能力）
  • X：桥接主体，连接 MSB 和 SRO，确保 SRO 看到的都是合规客户/交易
  • K：壳主体，替 SRO/MSB 在无法开户的渠道银行开户（如美国发卡渠道）。K 对 SRO 不可见，通过"物理渠道 vs 运营渠道"分离实现隔离
  • 目标：客户体验不变（在 MSB 做承兑/支付），SRO 合规达标（看到的都是满足要求的交易），渠道覆盖完整（K 补充 SRO 开不了户的渠道）

一、架构总览

核心矛盾

  • SRO 入网门槛高 —— 要求完整 KYB/KYC、UBO 穿透、资金来源证明、行业限制多
  • B 历史客户不达标 —— 大量客户在 VASP 时代入网，材料不完整、行业不合规、无法补充
  • SRO 支付能力强 —— 一旦入网，SRO 的承兑/支付/VA/POBO 能力完善
  • 商业现实 —— 不能因为监管升级丢掉存量客户，需要过渡方案

四、核心问题讨论

📌 同名 VA 问题：VA 开在谁名下？

Q：SRO 要求 VA 开在自己直客名下，但客户不在 SRO 入网，怎么办？
方案 1：VA 开在 X 名下（推荐）
X 是 SRO 的直客，VA 在 X 名下。客户充值时，汇款人是客户、收款人是 X。
• 优点：SRO 只看到合规的 X；客户不需要在 SRO 开户
• 缺点：不是"同名"（汇款人≠收款人），SRO 入账审核时可能要求解释资金来源
• 解决：X 与客户之间有服务合同（集采/代收协议），向 SRO 说明是合同项下收款

方案 2：X 持有多个子 VA（虚拟子账号）
如果 SRO 支持 X 开多个子 VA，每个子 VA 对应一个客户。汇款人名称和子 VA 标签可以匹配。
• 优点：每个客户有独立的收款账号，资金隔离清晰
• 缺点：需要 SRO 支持子账号功能

方案 3：客户在 SRO "轻入网"（仅开 VA，不做全业务）
SRO 给客户做精简 KYC，只开 VA 收款功能，不开 POBO/承兑等高风险功能。
• 优点：真正同名 VA
• 缺点：SRO 是否允许这种"半入网"模式待确认

📌 POBO 出款：收款方看到的汇款人是谁？

Q：客户的供应商收到款，看到汇款人是 X 还是客户？会不会有问题？
场景 B（X 代付）：收款方看到的汇款人 = X
• 如果收款方是客户自己的境外同名账户（同名提现）—— 问题不大，只是汇款方名称是 X 不是客户自己
• 如果收款方是第三方（供应商）—— 供应商可能会问"谁给我付的款"

解决方案：
  • SWIFT/本地汇款附言：在付款备注中标注"代 [客户名] 付款 / On behalf of [客户名]"
  • 客户提前通知收款方：MSB 端提示客户"您的款项将由我们的合作机构代为发出"
  • X 出具付款确认函：如收款方要求，X 可出具付款确认（证明是代客户付款）

📌 SRO 合规穿透：SRO 会不会追问 X 的资金来源？

Q：SRO 审核 X 的交易时，会不会发现背后其实是不合规的客户？
关键设计原则：X 必须有自己的合规逻辑
X 不能是"纯通道"——X 需要真实经营，向 SRO 展示的是 X 自身的业务。

具体操作：
  • X 自身定位为"贸易服务公司 / 供应链公司 / 集采平台"
  • X 与客户之间有真实的服务合同（代收/代付/集采分销）
  • X 向 SRO 提交的材料是 X 自己的贸易材料（不是客户的）
  • SRO 审核的是"X 的业务是否合规"，而不是"X 背后的客户是否满足 SRO 入网标准"
⚠️ 风险点：如果 SRO 做深度穿透（要求 X 披露最终受益客户），则 X 模式可能受限。
缓解：X 披露是"X 的客户"即可（X 自身做了 MSB 级别的 KYC），不等于客户需要满足 SRO 入网标准。

📌 MSB 与 SRO 之间的资金流转

Q：MSB 卖 U 得到的法币，怎么进 X 在 SRO 的账户？
OnRamp 方向（不需要 MSB→SRO 转账）：
客户充值直接到 X 的 SRO VA，资金已经在 SRO 内。MSB 只负责把 U 发给客户。
MSB 自身需要 U 库存 / 从市场买 U，这是 MSB 的资金管理问题。

OffRamp 方向（需要法币从 MSB 到 SRO）：
• 方案 1：MSB 卖 U 得到的法币直接存入 X 在 SRO 的账户（MSB 有 SRO 账户或银行账户，转账到 X 在 SRO 的 VA）
• 方案 2：MSB 和 X 在同一银行有账户，银行内转
• 方案 3：MSB 卖 U 给 SRO（如果 SRO 本身也做承兑），SRO 直接给 X 入账

五、K 主体：渠道侧壳主体

K 不是客户，不在 B 入网，也不能让 SRO 看到。K 是连接 SRO 与特殊渠道银行的桥梁——类似 B-A 场景中 T 连接 A 的角色，但 K 连接的是渠道/银行。

为什么需要 K？

  • 问题：部分渠道银行拒绝给 web3/crypto 相关公司开户（如美国发卡渠道要求美国主体 + 传统行业背景）
  • BB 的 SRO/MSB 都是 crypto 牌照主体，银行直接拒绝开户
  • 解决：BB 用 K（壳主体，如注册在美国的传统公司）去这些渠道开户，实际资金流通过 K 走渠道
  • K 的本质：不是客户，不做业务，只是"渠道开户的马甲"

5.4 风险分析与应对

⚠️ 核心风险：SRO 审计物理渠道时资金对不上

场景：SRO 去查渠道银行 B 的资金流水/交易记录，发现与自己系统的记录对不上
问题拆解：
  • SRO 系统记录：交易走了"物理渠道 = 渠道银行 B"，金额 $100,000
  • 实际情况：资金走了 K 在渠道银行 B 的账户，开户主体是 K 不是 SRO
  • 如果 SRO 直接找渠道银行 B 调数据：银行说"SRO 没在我这开户，K 开了户"→ 穿帮

📌 应对方案

方案 1：K 作为 SRO 的"技术服务商/支付代理"（推荐）
K 与 SRO 签订《渠道代理协议》/ 《支付技术服务协议》：
• K 受 SRO 委托，在特定渠道银行开设运营账户
• K 按 SRO 指令处理交易，资金归属权是 SRO 的
• K 定期向 SRO 出具对账报告

话术：SRO 通过技术服务商 K 接入渠道银行 B，K 是 SRO 的外包合作方，不是独立运营。
优点：有合同基础，能解释为什么开户主体是 K
缺点：SRO 需要知道 K 的存在（有限可见），但 K 不是 SRO 的客户
方案 2：完全隔离，SRO 不知道 K（当前方案）
SRO 后台只看到物理渠道（渠道银行 B），不知道实际通过 K 走。
• 日常运营没问题（交易正常，金额对得上）
• 风险点：SRO 审计时直接联系渠道银行 B 调数据 → 对不上
• 缓解：控制 SRO 的审计权限，渠道银行的对接由 B 运营团队统一管理，SRO 不直接接触渠道银行
• 话术储备：
  • "渠道银行 B 的开户主体是我们的关联运营公司，用于满足银行对主体类型的要求"
  • "交易量和金额与 SRO 系统一致，只是开户法人不同，这是银行端的技术限制"
  • "资金归属权不变，关联公司账户资金全部归属 SRO 业务"
方案 3：混合方案 — 部分渠道 SRO 直连 + 部分走 K
SRO 能开户的渠道银行 → SRO 直连（无风险）
SRO 开不了户的渠道银行 → K 开户（仅限必要场景）

优点：K 的使用范围最小化，降低被穿透的概率
建议：这是实际操作的推荐方式——只在 SRO 确实开不了户的渠道才用 K

⚠️ K 场景的调账/对账风险

  • 日常对账：K 在渠道银行 B 的流水 vs SRO 系统的交易记录 → 需要运营团队每日核对，确保金额一致
  • 调拨记录：SRO 系统显示"资金调拨到渠道银行 B"，但实际是调拨到 K 在渠道银行 B 的账户 → 记录层面需要一致
  • SRO 审计时：如果 SRO 要求看渠道银行 B 的银行流水（statement），需要提前准备 K 账户的 statement，并确保交易明细与 SRO 系统记录吻合
  • 关键原则：金额和交易笔数必须对得上，开户主体不同可以用"关联运营公司"解释，但金额对不上无法解释
