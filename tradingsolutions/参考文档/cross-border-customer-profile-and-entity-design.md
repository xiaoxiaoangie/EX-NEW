## 0. TL;DR — 一页纸看完整个方法论

### 0.1 核心论点（One-liner）

> **钱庄判定标尺作用在「客户」身上，不是作用在「持牌主体」身上。**
> 持牌主体本身是支付公司，**自身不碰货物是天然属性**（XTransfer / 万里汇 / 我们 / 任何持牌支付机构都不碰货）。持牌主体的核心职责是用「KYB + 三流核验」**判定客户是否真是货物流 / 服务流的一环** —— 客户必须以发货人 / 报关人 / 收货人 / 服务交付方身份出现在单据中；客户**只做代收付而不出现在货物 / 服务单据中** = 客户实质是钱庄 = 我方拒绝准入。客户判定（前 3 重）+ 我方自身合规（后 3 重）共同构成「KYB + 三流核验 + 客户独立账户 + 持牌资质 + 监管报送 + 单据存档」**六重体系**。
>
> 主框架只写了一种客户画像（货物贸易商）；本文档把客户画像扩展到 8 类，每类画像有不同的"业务实质 → 法律主体 → 客户在单据中的真实角色 → 业务流程 → 客户说服逻辑"组合。**画像选错 = 客户接受度差 + 单据成本高；主体选错 = 法律隔离失败；客户身份选错（客户不在单据中）= 直接踩钱庄红线。**

### 0.2 为什么不能只做"货物贸易"

| 货物贸易的局限 | 表现 | 后果 |
|--------------|------|------|
| **单据要求最严** | 报关单 / 提单 / 装箱单 / CO / 商检 全套 | 小商户拿不出，准入率低 |
| **客户接受度最差** | 客户嫌单据麻烦、嫌费率高 | 流失到 PingPong / 找换店 |
| **判别客户钱庄风险的成本最高** | 持牌主体本身天然不碰货物（支付公司属性），完全依赖客户单据 + 实地走访 + 第三方 API 反证「客户是货物流真实一环」 | **核验成本与误拒率双高** |
| **市场已被 XTransfer 等占据** | 头部友商深耕 5+ 年 | 拓客难、获客成本高 |
| **行业天花板** | 中国出口贸易增速放缓 | 长期增长空间有限 |

**结论**：货物贸易是必须做的"压舱石"画像，但不能是唯一画像。本文档新增 7 类画像，**重点是单据相对易拿、客户接受度高、行业增速快、与持牌主体牌照匹配**的画像。

## 1. 钱庄问题：为什么纯代收付危险

### 1.1 监管对"地下钱庄"的法律定义

> ⚠️ 以下条文待法务确认最终引用文号（**DC10 待办**），目前依据公开渠道梳理：

**中国（中国人民银行 / 国家外汇管理局）**

- 《中华人民共和国外汇管理条例》第 45 条：禁止非法买卖外汇、变相买卖外汇、倒买倒卖外汇或者非法介绍买卖外汇
- 《最高人民法院关于审理非法从事资金支付结算业务、非法买卖外汇刑事案件适用法律若干问题的解释》（法释〔2019〕1 号）：明确"非法资金支付结算"包括"为他人虚构交易、虚开价格、对敲资金等方式非法支付结算"
- 《刑法》第 225 条 非法经营罪：未经许可经营货币兑换 / 资金支付结算业务情节严重

**中国香港（HKMA / 海关 MSO 监管）**

- AMLO（《打击洗钱及恐怖分子资金筹集条例》）：MSO 持牌人须执行 CDD、保存记录、可疑交易报告
- HKMA 反洗钱指引：对客户尽调与持续监察要求

**新加坡（MAS）**

- Payment Services Act 2019：持牌支付机构反洗钱与客户尽调要求

> 📌 **结论**：监管对钱庄的核心判断不是"是否接触货物"，而是 ——
> 1. **是否有真实交易背景**
> 2. **是否做了客户尽调（KYB）**
> 3. **是否保存了完整记录**
> 4. **资金流是否可追溯**
> 5. **是否持有相应牌照**

### 1.2 钱庄的 7 个识别特征 vs 合规支付的 7 条分界线

| # | 钱庄特征 | 合规支付分界线 | 我方对应落地措施 |
|---|---------|--------------|--------------|
| 1 | **资金来源不明** | KYB 完备 + 资金来源穿透核查 | 主框架 §4.6 KYB SOP / §7.4 KYC 标准 |
| 2 | **名义收付双方与实际不一致** | 三流匹配（合同流 / 物流 / 资金流）严格核验 | 主框架 §6 凭证清单 + §9.2 单笔 SOP |
| 3 | **客户无真实贸易/服务背景**（只做代收付，不参与货物/服务流） | 客户必须是货物流/服务流的真实一环（发货人/报关人/收货人/服务方），单据中能找到客户名 | 本文档每类画像的"业务真实性核验" + KYB 阶段查客户在报关单/提单/服务交付证据中的真实角色 |
| 4 | **资金池化、客户混账** | 客户独立子账户、不混账、不挪用 | §12.1 客户独立子账户体系 |
| 5 | **无监管约束、无审计** | 持牌 + 定期外部审计 + 监管报送 | 主框架 §1.0 + 本文档 §12.3 监管报送 |
| 6 | **批量对私大额、个人到个人** | 法律实体对法律实体（B2B）+ 对私交易严控 | 主框架 §7.3 AML 阈值 + 本文档 §12.2 |
| 7 | **缺乏单据存档、记录不全** | 7-10 年单据全链路存档 | §12.4 + 待 **DC9** 决策最终保留期 |

> 💡 **关键认知 1**：合规支付公司和钱庄的核心差异**不在业务形态**（都做"代收付"），而在**6 重证据链**是否齐全。XTransfer / 万里汇 / 我们 —— 本质上都是"代收付 + 合规中台"，都不是钱庄。
>
> 💡 **关键认知 2**：上述 7 条分界线中，**第 1/2/3/6 条作用在客户身上**（持牌主体用它们审查客户是否钱庄），**第 4/5/7 条作用在持牌主体自身身上**（我方作为合法支付通道的基础合规要求）。前者错 = 我方接了钱庄客户；后者错 = 我方持牌主体本身合规失效。两者缺一不可。

### 1.3 为什么"单纯货物贸易"也存在钱庄风险

货物贸易乍看有报关单 / 提单等"硬单据"，看似最不容易被认定为钱庄。但实际操作中存在以下暗坑（**核心都是「客户在货物流中的真实角色」是否经得起核验**）：

| 暗坑 | 描述 | 后果 |
|------|------|------|
| **客户在货物流中只挂名不实际参与** | 客户既不是发货人也不是收货人，只在资金链路中收付（俗称"借单"） | 客户实质 = 钱庄；我方接了 = 协助钱庄 |
| **客户做了 1039 报关但我方未核验客户身份** | 客户拿到了 1039 报关单，但持牌主体未核验「客户是否真是单据上的发货人」 | 监管查时无法举证"业务真实性核验"，可能被认定为"明知或应知" |
| **跨境贸易中间环节多** | 实物货物可能从 A→B→C→D 走，资金可能从 X→Y→Z 走，两条链不一定能对上 | 三流匹配难度大，需要专门的核验中台 |
| **客户单据可能造假** | 报关单 / 提单存在伪造产业 | 未做交叉验证 = 形式合规但实质失败 |
| **资金池化诱惑** | 多客户共用一个汇兑账户（节省成本） | 直接踩第 4 条钱庄红线（我方系统层失败） |

> 📌 **货物贸易并非"天然合规"**。**核心要看客户是否真是货物流的参与者** —— 客户名出现在报关单 / 提单上（作为发货人 / 收货人 / 报关人）+ 持牌主体 6 重证据链，两者缺一不可。**业务形态不重要，客户身份与证据链才重要。**

### 1.4 同行参照：XTransfer / 万里汇为什么不是钱庄

| 维度 | 钱庄做法 | XTransfer / 万里汇 做法 | 我方应做做法 |
|------|---------|---------------------|------------|
| **客户尽调** | 不做或形式化 | 严格 KYB + 实控人核身 + 持续监察 | 同主框架 §4.6 |
| **业务真实性** | 不核验 | 报关单核验 + 物流追踪 + 行业经验判断 | 同主框架 §6 + 本文档每类画像 |
| **账户隔离** | 资金池化 | 每个客户独立子账户（虚拟账户技术） | §12.1 |
| **资金路径** | 不可追溯 | 全程可追溯，监管 / 银行随时可查 | §12.2 |
| **牌照** | 无 | HK MSO + 内地支付牌照（多地） | 持牌主体已具备（IPL/BB） |
| **监管报送** | 无 | 按月 / 按笔报央行外管局 + HKMA | §12.3 |
| **审计** | 无 | 四大 / 大型审计所定期审计 | 同主框架（继承现有审计） |

### 1.5 持牌主体的「客户钱庄判定 + 自身合规」六重体系

> **说明**：
> - **第 1-3 重**：审查客户用的 —— 持牌主体识别一个客户是否钱庄的核心标尺
> - **第 4-6 重**：持牌主体自身的合规基础 —— 作为合法支付通道的必要条件
>
> 客户判定缺失 → 我方接了钱庄客户（最常见踩雷场景）；
> 我方自身合规缺失 → 我方持牌主体本身合规失效（牌照风险）。

```
┌─────────────────────────────────────────────┐
│ 第 1 重：KYB + 实控人核验                          │
│   ─ 营业执照 + UBO 25%+ 穿透                       │
│   ─ 实控人身份证 + 视频核身                          │
│   ─ 制裁名单全名单筛查（OFAC/UN/EU/HK/客户所在国）    │
└─────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│ 第 2 重：客户在货物流/服务流中的真实角色核验          │
│   ─ 核心问题：客户是货物/服务流的真实一环吗？         │
│   ─ 货物贸易：客户必须是报关单/提单上的发货人/收货人   │
│   ─ 服务贸易：客户必须是合同/验收报告上的服务交付方    │
│   ─ 数字内容：客户必须是版权/IP 持有人或正式被许可方   │
│   ─ 客户只挂名不参与 → 客户 = 钱庄 → 拒绝准入        │
│   ─ ……（每类画像详 §3-§10）                         │
└─────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│ 第 3 重：三流匹配核验中台                             │
│   ─ 合同流 ↔ 物流（或服务交付）流 ↔ 资金流          │
│   ─ 任一不匹配 → 单据被打回                          │
│   ─ 自动核验 + 人工复核双层                          │
└─────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│ 第 4 重：客户独立子账户体系                          │
│   ─ 每个贸易主体客户独立子账户                       │
│   ─ 资金不混池、不挪用                              │
```

## 2. 客户画像分类总览

### 2.1 分类的两个维度

**维度 1：业务实质类型**

- **货物贸易**（实物跨境）：P-A
- **数字 / 服务交付**（无形）：P-B, P-C, P-E, P-H
- **零售型贸易**（B2C 小额）：P-D
- **知识产权 / 版权**：P-F
- **人力 / 薪酬**：P-G

**维度 2：客户成熟度**

- **小商户**（年流水 < 100 万 USD）：分散、单据弱、议价能力低 → 多见于 P-A、P-D
- **中型机构**（年流水 100-1000 万 USD）：组织化、单据较全 → 多见于 P-B、P-C、P-H
- **大型客户**（年流水 > 1000 万 USD）：高度合规要求、机构化 → 多见于 P-E、P-F、P-G

## 3. 客户画像 P-A：货物贸易商（重构，避钱庄）

### 3.1 客户画像

| 子画像 | 典型客户 | 年流水 | 客户特征 |
|--------|---------|--------|---------|
| **P-A1 小商户** | 义乌小老板、白沟箱包、广州小商品、深圳华强北电子 | 50-500 万 USD | 单据弱、组织松散、嫌麻烦、议价能力弱 |
| **P-A2 中型工厂** | 长三角制造、东莞工厂、佛山家具 | 500-3000 万 USD | 单据较全、流程规范、配合度中 |
| **P-A3 大型出口商** | 上市公司外贸子公司、大型集采贸易商 | > 3000 万 USD | 单据齐全、合规要求高、决策慢 |

**典型业务场景**：
- 中国工厂 → 客户主体 → 越南/印尼/中东/非洲 进口商
- 实物货物（服装、家具、电子、家电、机械、原材料、农产品）跨境流转
- RMB 结算或人民币/USD/USDT 跨境闭环

### 3.2 钱庄风险点 & 避钱庄设计

| # | 风险点 | 暗坑机理 | 避钱庄设计 |
|---|--------|---------|----------|
| 1 | **客户必须是货物流的真实一环（发货人/报关人/收货人）** | 客户只做代收付而不在货物单据中出现 = 客户实质钱庄 | **KYB 阶段核验客户在报关单/提单中的角色**（必须是发货人/收货人/报关人之一）+ 强制三流核验，客户不在货物单据中 → 拒绝准入 |
| 2 | **单据可造假** | 报关单 / 提单存在伪造市场 | 单据交叉验证（船公司 API / 海关 API / 第三方数据源）+ 实地走访 A 级客户 |
| 3 | **小商户资金归集** | 多个小商户资金归到同一中间账户 → 池子化 | **每个贸易主体客户独立子账户**，资金路径全程可追溯 |
| 4 | **代理报关存在中介风险** | 报关行可能与客户串通 | 风控保留对客户报关行的"独立选择权审查"（**DC5 待办**） |
| 5 | **结汇阶段被认定为"代客结汇"** | 客户用我方账户结汇而无真实贸易 | 持牌主体只接受**完整三流证据**后才走结汇 |
| 6 | **退税申报涉嫌虚开** | 部分客户为骗退税虚开发票 | KYB 阶段调取客户历史退税记录 + 风控异常监测 |

### 3.3 推荐贸易主体客户的公司主体形式

> ⚠️ 我方**不为客户注册**任何主体（v0.4 边界）。本节是 KYB 阶段对"客户应具备什么主体形式"的**画像筛选标准**，不是我方代办服务。

**双层主体结构（推荐）**

```
   ┌─────────────────────────────┐
   │  HK 贸易公司 (TE-HK)         │
   │  ─ 收外汇（USD/EUR/USDT）     │
   │  ─ 与海外终端买家签合同        │
   │  ─ 持有 HK 银行账户            │
   │  ─ 由客户独立运营             │
   └─────────────────────────────┘
              ↕ 关联贸易合同
   ┌─────────────────────────────┐
   │  境内贸易公司 (TE-CN)        │
   │  ─ 走 1039/9710 报关          │
   │  ─ 与上游中国出口商签合同      │
   │  ─ 收 RMB 结算（境内）         │
   │  ─ 由客户独立运营             │
   └─────────────────────────────┘
```

**单层主体（小商户备选）**

仅 HK 贸易公司，境内通过义乌当地"独立报关行"做 1039 → 客户单据复杂度更低、但 KYB 时需补强境内业务真实性证据。

**禁止的主体结构**

- ❌ 多层 BVI/Cayman 嵌套（穿透困难）
- ❌ 个人独资 + 个人账户混用
- ❌ 与我方员工/管理层/家属任何形式关联

### 3.4 业务流程设计（Step-by-step，避钱庄版本）

```
[Step 1] 终端贸易商 A（中国卖货方，如义乌张三）
        ↓ 货物 + 合同
[Step 2] 贸易主体客户 TE-CN（独立第三方公司）
        ↓ 1039 报关（自行委托报关行）
        ↓ 出口报关单（关键单据）
[Step 3] 海外仓 / 海运 / 空运
        ↓ 物流单据（提单 / 空运单 / 装箱单）
[Step 4] 贸易主体客户 TE-HK 收外汇 USD/USDT
        ↓ 持牌主体核验三流后过账
        ↓ ★★★ 此处发生「代收付」—— 但客户在 Step 2/3 已经是 ★★★
        ↓ ★★★ 报关单/提单上的真实发货人 + 报关人 = 货物流的真实一环 ★★★
        ↓ ★★★ 不是"只收付不碰货"的中间人 = 不是钱庄 ★★★
[Step 5] TE-HK 向终端买家所在地的本地服务商付 USD
        ↓ 本地服务商兑当地货币（VND/IDR）
[Step 6] 终端买家 B（越南进口商）收到当地货币
[Step 7] 全套单据归档（7-10 年）+ 月度对账 + 监管报送
```

> 💡 **关键差异化 vs 钱庄**：
> - Step 2 **不可省略**：必须有真实报关单
> - Step 3 **不可造假**：物流单据需经船公司 API 等独立渠道验证
> - Step 4 **核验前不放款**：合规中台三流匹配通过后才发起资金动作
> - Step 7 **永久可查**：监管随时调取，全链路完整

### 3.5 单据清单与分级要求

| 单据 | A 级要求 | B 级要求 | C 级要求 | 出具方 | 我方核验方式 |
|------|---------|---------|---------|--------|------------|
| 销售合同 | 强制（双语） | 强制 | 强制 | 客户 / 客户与对手方 | 文本核验 + 关键字段交叉 |
| 商业发票 (Commercial Invoice) | 强制 | 强制 | 强制 | 客户 | 金额 / 数量 / 商品名一致性 |
| 装箱单 (Packing List) | 强制 | 强制 | 强制 | 客户 / 货代 | 与发票交叉核验 |
| 提单 (Bill of Lading) | 强制 | 强制 | 强制 | 客户委托的货代 | **船公司 API 验真**（待 **DC6**） |
| 出口报关单（1039/9710） | 强制 | 强制 | 强制 | 客户委托的报关行 | **海关 API 验真**（待 **DC7**） |
| 原产地证 (CO / FORM E) | 强制 | 部分强制 | 可缺 | 贸促会 / 商会 | 文本核验 |
| 商检证书 | 视品类 | 视品类 | 视品类 | 检验机构 | 文本核验 |
| 物流追踪截图 | 强制 | 强制 | 强制 | 客户 / 货代 | 与提单交叉 |
| 客户与买方往来记录（邮件/微信） | 强制 | 部分强制 | 可缺 | 客户 | 抽样核验 |
| 上游供应商付款记录 | 强制 | 部分强制 | 可缺 | 客户 | 与发票交叉 |
| 终端买方收货确认 | 强制 | 部分强制 | 可缺 | 终端买方 | 抽样核验 |

> 📌 **分级原则**：A 级 = 全单据 + 实地走访；B 级 = 关键单据 + 部分补强；C 级 = 关键单据 + 限额低 + 监控密。**任何级别都不能缺"四件套"**（合同、发票、提单、报关单）。

### 3.6 客户说服逻辑（为什么客户会同意流程）

**客户典型反对意见 & 我方回应**

| 反对意见 | 回应（说服话术） |
|---------|--------------|
| **"单据太麻烦了，PingPong / 找换店都不要这些"** | "找换店做的是地下，3 个月内可能被冻账。我方做的合规，单据完整后客户可申报退税，长期成本更低，账户更稳。" |
| **"我没那么多单据，做不了"** | "我方按 A/B/C 分级，C 级允许部分缺失，先小额跑通后升级。" |
| **"我费率太高了"** | "费率与单据复杂度匹配。如果你能提供 A 级单据，享受 A 级费率（≈ 0.5-1%），与友商相当甚至更低。" |
| **"我担心隐私 / 商业秘密"** | "我方持牌主体严格 NDA，单据只用于合规核验、不向第三方披露。监管查时按法定流程配合。" |
| **"我之前用过 XTransfer，没差别"** | "如果你已经习惯 XTransfer 的单据要求，我方流程几乎相同，但我方在【东南亚走廊本地兑付】上有差异化（具体见各走廊本地服务商网络）。" |
| **"我能不能用个人账户先收一笔"** | "禁止。监管对'对私大额'高度敏感，会直接命中钱庄判定。所有收付必须公司账户对公司账户。" |

**正面卖点**

- ✅ 合规凭证齐全 → 客户可申报出口退税 / 服务贸易增值税豁免 → 综合收益高于"低成本但不合规"路径
- ✅ 银行账户不会突然冻 → 长期可持续经营
- ✅ 监管查时有完整证据链 → 客户法定代表人风险低
- ✅ 升级到 A 级后享受最优费率 + 较长账期
- ✅ 我方提供单据辅导（首次申报手把手）

### 3.7 失败案例与拒绝红线

**应该拒绝的子画像**

- ❌ 业务说明含糊、无法描述货物来源 / 去向
- ❌ 实控人有外汇 / 走私 / 洗钱前科
- ❌ 拒绝提供"四件套"中的任何一件
- ❌ 实控人 / 业务地址与我方员工有任何关联（v0.4 红线）
- ❌ 多个不同贸易主体客户共用同一实控人 + 同一办公地（壳公司链路）

**进入后的红线（触发即终止，详主框架 §4.10）**

- 连续 3 月单据齐全率 < 95%
- 单笔单据被海关 / 船公司 API 验真发现造假
- AML 触发月度 ≥ 4 次

## 4. 客户画像 P-B：跨境数字服务 / SaaS

### 4.1 客户画像

| 子画像 | 典型客户 | 年流水 | 客户特征 |
|--------|---------|--------|---------|
| **P-B1 SaaS 出海** | 中国 SaaS 产品出海（CRM / 协同 / 设计工具）、月费/年费订阅 | 100-2000 万 USD | 客户机构化、流水稳定、高合规要求 |
| **P-B2 软件定制开发** | 中国软件外包公司接欧美/东南亚项目 | 50-1000 万 USD | 项目制、单据简单（合同 + 验收） |
| **P-B3 技术咨询/数字化** | 中国数字化转型服务出海 | 200-3000 万 USD | 大客户、单笔大额 |
| **P-B4 游戏/应用出海** | 中国 App / 游戏全球发行收入回流 | 500-1 亿 USD | 平台分成模式（App Store / Google Play / Steam）|

**典型业务场景**：
- 中国 SaaS 公司订阅服务卖给海外企业 → 海外企业月付 USD/EUR → 通过持牌主体回流境内
- 中国软件外包公司接海外项目 → 项目验收后客户付 USD → 通过持牌主体回流境内
- 中国 App / 游戏在 App Store / Google Play 上架 → 平台分成（70%）→ 通过持牌主体回流境内

### 4.2 钱庄风险点 & 避钱庄设计

| # | 风险点 | 暗坑机理 | 避钱庄设计 |
|---|--------|---------|----------|
| 1 | **服务无形，没有物理交付** | 无法用"物流"证明交付 | 用"**服务交付证据链**"替代物流：合同 + 验收报告 + 系统访问日志 + 客户满意度评价 |
| 2 | **小额高频订阅模式** | 海外用户月付小额订阅，资金碎片化 | 平台账单（Stripe/Paddle/Adyen 等）作为汇总凭证 + 实质性的服务上线证据（App 截图、网站可访问性） |
| 3 | **平台分成模式资金来源单一** | 资金来自 App Store / Google Play 一个平台 | 平台对账单 + 平台 API 直连验证 |
| 4 | **客户可能虚构"研发外包"洗钱** | 假合同 + 假交付 | 合规中台需查"客户员工规模 vs 项目交付能力"匹配性 |
| 5 | **个人开发者 vs 公司客户难区分** | 个人独立开发者付款 | 强制要求贸易主体客户为**公司主体**，个人独立开发者通过其他渠道 |

### 4.4 业务流程设计

| 2 | **"版权金"作为洗钱掩护** | 版权登记证书 + 行业平台公开记录交叉验证 |
| 3 | **版权许可期长，资金错位** | 合同与每期付款节点严格匹配 |
| 4 | **影视 / 游戏行业敏感**（监管在内地有审批要求） | 强制要求客户提供广电局 / 文化部审批证明（如适用） |
| 5 | **数字内容易复制易造假** | 版权链上备案（区块链 IP 服务）+ 抽样审查 |

### 8.3 推荐主体形式

```
   ┌─────────────────────────────────┐
   │  HK / BVI / SG IP 控股或许可公司   │
   │  ─ 业务范围：知识产权许可、版权代理 │
   │  ─ 与海外版权方签授权 / 许可合同     │
   │  ─ 持有 IP 登记 / 备案              │
   └─────────────────────────────────┘
              ↕ 关联许可 / 分许可
   ┌─────────────────────────────────┐
   │  境内运营 / 发行公司                │
   │  ─ 业务范围：影视 / 游戏 / 文化运营  │
   │  ─ 取得国内审批（如适用）            │
   └─────────────────────────────────┘
```

> 📌 **特例**：版权类业务可允许 BVI 主体（破例于主框架 §4.2 "BVI 拒绝准入"原则），因为 BVI 是国际 IP 控股惯例。**但必须 UBO 穿透至自然人**（待 **DC14**）。

### 8.4 业务流程

```
[Step 1] 海外版权方与贸易主体客户 HK / BVI 公司签授权 / 许可合同
[Step 2] 版权登记 / 备案
[Step 3] 贸易主体客户按合同节点付版权金 → 海外版权方
        ↓ 资金路径：境内 RMB → 持牌主体核验 → HK 公司 USD → 海外版权方
[Step 4] 持牌主体核验：版权合同 + 版权证书 + 境内审批（如适用） + 行业平台公开记录
[Step 5] 归档（含版权链）
```

|------|------|------|

## 12. 持牌主体侧的"非钱庄"系统设计

### 12.1 客户独立子账户体系

**核心原则**：**每个贸易主体客户一套独立子账户**，资金物理或逻辑隔离，不池子化。

**实现方式（待 DC4 决策）**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 持牌主体内部虚拟子账户** | 持牌主体在自己核心账户内部对每个客户做账务隔离 | 系统简单、上线快 | 监管视角下仍是混账（账户层未隔离） |
| **B. SP 通道商提供子账户** | 通过 SP / Banking-as-a-Service 提供商开虚拟账户 | 账户层隔离、合规强 | 依赖 SP 能力、成本高 |
| **C. 物理账户一一对应** | 每个客户开一个银行物理子账户 | 最干净 | 银行开户负担、扩展性差 |
| **D. 链上钱包**（USDT 通道） | 每客户独立 USDT 地址 + 多签风控 | 链上可追溯 | 仅限 USDT 通道 |

**推荐**：**B + D 组合**（法币用 SP 子账户，USDT 用独立链上钱包），既合规又可扩展。

### 12.2 三流核验中台

**核心：合同流 / 物流（或服务交付）流 / 资金流 三向匹配**，任一不匹配则放款被拦截。

```
┌─────────────────────────────────────────────┐
│  合同流：贸易主体客户上传的合同 / 协议         │
│     ↓ 关键字段抽取（金额、对手方、商品/服务）    │
└─────────────────────────────────────────────┘
              ↕ 必须匹配
┌─────────────────────────────────────────────┐
│  交付流：物流单据（P-A/D）/ 服务交付证据（其他） │
│     ↓ 第三方 API 验真 + 关键字段对照            │
└─────────────────────────────────────────────┘
              ↕ 必须匹配
┌─────────────────────────────────────────────┐
│  资金流：资金路径（收 / 付 / 中转）             │
│     ↓ 链上 + 银行流水 + 客户子账户对账         │
└─────────────────────────────────────────────┘
              ↓
         三向匹配通过 → 放款
         任一不匹配 → 拦截 → 人工复核
```

**核验中台技术栈**（待 PRD 细化）：
- 合同 OCR / NLP 关键字段抽取
- 海关 / 船公司 / 平台 API 集成
- 链上风控（Chainalysis / TRM）
- 规则引擎 + 人工复核工作流

### 12.3 监管报送系统

**按辖区差异化报送**：

| 辖区 | 报送对象 | 报送频率 | 报送内容 |
|------|---------|---------|---------|
| **中国（外管局 / 央行）** | SAFE / PBOC | 月度 + 大额即时 | 跨境收支申报、可疑交易报告 STR |
| **香港（HKMA / 海关）** | HKMA / C&ED | 季度 + 抽查响应 | MSO 合规报告、AML/CFT |
| **新加坡（MAS）** | MAS | 季度 | PSA 合规报告 |
| **其他辖区** | 当地监管 | 按当地要求 | - |

**系统集成要求**：
- 与持牌主体现有监管报送系统集成（不另建）
- 贸易场景专项字段补丁（如 1039 报关单号、平台账单 ID）
- 可疑交易自动识别 → STR 准备

### 12.4 单据存档与可追溯

**存档要求**：

| 类别 | 保留期 | 存储介质 |
|------|--------|---------|
| KYB / KYC 资料 | 客户终止后 7-10 年（待 DC9） | 加密云存储 + 离线备份 |
| 合同 / 单据 | 同上 | 同上 |
| 资金流水 | 同上 | 数据库 + 区块链存证（备选） |
| 风控决策日志 | 同上 | 加密日志系统 |
| 监管报送底稿 | 同上 | - |

**可追溯要求**：
- 任一笔资金动作 → 可在 5 分钟内调取全链路（KYB、合同、单据、风控决策、报送记录）
- 监管抽查响应时间 ≤ 24h

### 12.5 与主框架 §7 控量模型的关系

本文档 §12 关注**质量层**（每笔交易的合规性）；主框架 §7 关注**量层**（限额、集中度、AML 阈值）。两者**互补共建**：
- 质量层失败 → 单笔不放款
- 量层失败 → 客户被限流 / 降级 / 终止
