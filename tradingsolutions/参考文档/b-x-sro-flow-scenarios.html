B-X-SRO 联动场景：BB 租户 + SRO 系统 + 渠道开户

BB 作为租户（Tenant）面客营业；SRO 提供独立 SP 系统（合规数据视图）；客户/桥接主体直接去渠道开户（VA/出款）。
SRO 不是渠道——不会给客户开 VA 或出款，SRO 提供的是一套合规系统。
📎 渠道层桥接主体（K）的方案见独立文档：k-channel-bridge.html

文档背景

本文档描述 BB（租户）在 SRO 监管环境下，通过桥接主体 X 实现合规隔离的方案：

  • BB：租户（Tenant），面客营业、客户管理、承兑报价
  • SRO：独立的 SP 系统，不是渠道。SRO 提供一套合规系统（数据视图），不直接给客户开 VA 或出款
  • 渠道：独立于 SRO，客户或桥接主体可以直接去渠道开户（开 VA、出入款）
  • X（桥接主体）：在 SRO 系统中作为客户入网，X 与真实客户之间有映射关系。渠道层同时给客户和 X 各开 VA，通过信息流同步让 SRO 看到合规数据
  • 问题：SRO 要求比以往 VASP 严格，BB 的历史客户大部分不满足 SRO 入网条件
  • 方案：不满足 SRO 条件的客户 → 映射到桥接主体 X → SRO 系统只看到 X 的数据（信息流）
  • 调单链路：SRO 合规调单 → EX 运营 → B 运营 → B 看到桥接主体 → 桥接主体提供材料
  • K：渠道层桥接主体 → 详见独立文档 k-channel-bridge.html

一、架构总览

核心矛盾

  • SRO 入网门槛高 —— 要求完整 KYB/KYC、UBO 穿透、资金来源证明、行业限制多
  • BB 历史客户不达标 —— 大量客户在 VASP 时代入网，材料不完整、行业不合规、无法补充
  • SRO 提供系统不提供渠道 —— SRO 是独立的 SP 系统（合规数据视图），不直接给客户开 VA 或出款
  • 渠道独立 —— 客户（或桥接主体）直接去渠道银行开户、收付款
  • 商业现实 —— 不能因为监管升级丢掉存量客户，需要过渡方案

四、核心问题讨论

📌 SRO 的定位：系统 vs 渠道

Q：SRO 在这个架构里到底是什么角色？
SRO = 独立的 SP 系统（合规数据视图），不是渠道
• SRO 不直接给客户开 VA，不直接出款
• SRO 提供的是一套合规系统：看到客户/桥接主体的入账、出款数据（信息流）
• 渠道独立于 SRO：客户和桥接主体直接去渠道银行开户

BB = 租户（Tenant），面客营业、承兑报价、客户管理
渠道 = 独立银行/支付机构，提供 VA、入账、出款能力

📌 渠道层 VA 开户：谁去开？

Q：VA 是谁去渠道开的？
直连场景：客户自己去渠道开同名 VA
桥接场景：渠道同时给客户 Merchant001 和桥接主体 X001 各开 VA
• 客户的 VA 用于实际收款
• X001 的 VA 用于信息流映射（SRO 系统看到的入账归属于 X001）

特殊情况：如果客户或 X 无法在某些渠道开户 → 由 K（渠道层桥接主体）代为开户
→ 详见独立文档 k-channel-bridge.html

📌 信息流同步机制

Q：渠道的入账/出款信息如何同步到 SRO 系统？
核心设计：渠道事件 → 映射转换 → SRO 系统
• 渠道通知 Merchant001 VA 入账 → 映射层将其转换为 X001 的入账 → 推送到 SRO 系统
• SRO 系统只看到 X001 的数据，不知道背后是 Merchant001

同步方式待确认：
  • 方案 1：EX 系统做映射转换（自动化）
  • 方案 2：BB 运营手动同步（初期可行，长期不可持续）
  • 方案 3：渠道直接推送给 SRO 系统（需要渠道配合改造）

📌 异名出款问题（OffRamp）

Q：OffRamp 时客户从自己的渠道账户出款，SRO 看到出款人 ≠ X001，怎么处理？
方案 1（推荐）：客户接受桥接主体出款
由桥接主体 X001 名义出款 → 不存在异名问题 → SRO 看到的出款人 = X001

方案 2：客户不接受桥接主体出款
客户从自己的渠道账户出款 → SRO 看到异名 → 需要桥接主体提供关系证明材料
中间步骤：桥接主体 → 客户的内部转账（转账流程根据每个渠道不同而不同）

跟 B-A flow 类似：如果客户可以接受桥接主体出款，就用桥接主体；不能的话，就走桥接主体内部转账

📌 调单流程（SRO 合规调单）

Q：SRO 合规调单时，信息怎么传递？
完整调单链路：
SRO 合规调单 → EX 运营 → B 运营 → B 运营看到的是桥接主体 → 桥接主体提供材料

桥接主体提供的材料类比：
桥接主体 = 供应商，客户 = 买家
• 供应商提供与买家的服务合同 / 采购协议
• 资金来源说明（客户的贸易背景）
• 交易目的说明

关键设计：B 运营在后台看到的是桥接主体（不是客户），桥接主体负责准备合规材料

📌 SRO 合规穿透：SRO 会不会追问 X 的资金来源？

Q：SRO 审核 X 的交易时，会不会发现背后其实是不合规的客户？
关键设计原则：X 必须有自己的合规逻辑
X 不能是"纯通道"——X 需要真实经营，向 SRO 展示的是 X 自身的业务。

具体操作：
  • X 自身定位为"贸易服务公司 / 供应链公司 / 集采平台"
  • X 与客户之间有真实的服务合同（代收/代付/集采分销）
  • X 向 SRO 提交的材料是 X 自己的贸易材料（不是客户的）
  • SRO 审核的是"X 的业务是否合规"，而不是"X 背后的客户是否满足 SRO 入网标准"
⚠️ 风险点：如果 SRO 做深度穿透（要求 X 披露最终受益客户），则 X 模式可能受限。
缓解：X 披露是"X 的客户"即可（X 自身做了 BB 级别的 KYC），不等于客户需要满足 SRO 入网标准。
