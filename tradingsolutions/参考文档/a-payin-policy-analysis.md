# A 入账调单政策特点分析

> 基于 A 实际入账政策（2026 上半年）提炼的核心规律，作为 `a-payout-policy-analysis.md` 的姐妹文档。
>
> 不复述场景细节（看原文档），只提炼"A 的入账调单逻辑特征 + 对 B-T-A 架构的影响"。

---

## 一、A 入账调单的 12 个核心特征

### 特征 1：**PI + 物流 + 网站 = 入账三件套**（标配，绕不开）

几乎所有场景都要齐 3 件：
- **PI / 合同**（双方主体 + 交易信息 + 产品 + 物流方式 + 付款方式 + 收款账户）
- **物流单据**（出境段，多种形式可接受）
- **结汇网站**（必须商户自有、必须与出口品类一致）

> **实战启示**：客户入账前先准备好"三件套"，缺一不可。**网站是 70% 客户被卡的地方**（没有自己的网站 / 网站类目不匹配）。

---

### 特征 2：**场景分类极细，远比出金细**（14+ 场景）

出金有 7 大类收款人；入账场景细分有 **14+ 类**：
标准 B2B / 代付 / 个人付款 / 同名充值 / 关联公司充值 / 一件代发 / B2B2C / EXW / 特殊档口 / 海外仓 / 物流服务 / 物流公司代收货款 / COD / 组货人 / ...

> **实战启示**：业务 BD 必须能在客户来询时**第一时间识别对应场景**，否则材料准备就错方向。**"场景识别"应该产品化为前置问卷工具**。

---

### 特征 3：**同名充值的"时间分级 + 质量分级"双轴系统** ★ 关键设计

A 把同名充值客户分成 3 档：

| 客户档 | 时间标准 | 贸易背景 | 材料要求 | 限额 |
|----|----|----|----|----|
| 老客户 | 20240601 前已走过 | 是 | 宽松 | 标准 |
| 新客户-优质 | 20240601 后才进 | 有可核实 | 中等 | 标准 |
| 新客户-普通 | 20240601 后才进 | 无可核实 | 严格 | **单订单 100 万 / 单笔 60 万** |

> **实战启示**：
> - **20240601 是 A 的"客户分层时间戳"**——这天之后入场的客户、合规材料要求高一档
> - 我方"新客户"想被认定为"优质"，关键是 **可核实的真实贸易背景**——这正是 T 桥接最容易帮忙的地方
> - **"普通新客户"是材料要求最高的群体**——如果我方 B 客户大多是新客 + 弱贸易背景，几乎注定走严格审核

---

### 特征 4：**反"过桥"硬约束（入账侧）**——与出金侧呼应

同名充值明确："不支持持同名外币账户入账后又转出至同名外币账户"。

> **实战启示**：与出金侧规则一起，**A 已经构建了完整的"反过桥"闭环**：
> - 入：同名进同名出 ✗
> - 出：同名进同名出 ✗
> - **B-T-A 架构中所有"客户同名往返"链路都要重新审视**

---

### 特征 5：**代付的"白名单 + 附言 + 证明"三层防御**

针对汇款人 ≠ 买家 / 收货人的代付场景：

**3 个硬禁止**：
- ❌ 付款方是地下钱庄 / 虚拟货币金融 / 支付公司
- ❌ 来自禁止国 / 极高风险国
- ❌ 个人代付

**减免机制**：规范附言可豁免代付材料——附言含 `产品名 + PI No. + 买家名称 + 卖家名称`。

**否则四选一证明**：代付声明邮件 / 代付协议 / 金融业务关系证明 / 代付公司注册证书。

> **实战启示**：
> - **❌ "付款司是支付公司" 这条对 T 极不利**——如果 T 被识别为支付公司（虽然 T 是贸易公司，但要"看起来不是 PSP"是关键）
> - T 的对外品牌、网站、业务范围都不能出现"支付""结算""收付款"等敏感词
> - **规范附言机制要产品化**——给 B 客户的买家提供附言模板，能大幅减少调单

---

### 特征 6：**个人付款是入账高敏区**（多重限制）

| 限制 | 详情 |
|----|----|
| ❌ 个人帮个人代付 | 汇款人与买家都是纯个人、无企业背景 |
| ❌ 个人帮无关企业代付 | 汇款人不来自买家公司 |
| ❌ 占比上限 | 纯个人汇款 ≤ 商户整体贸易额 50% |
| ❌ 年龄上限 | 不支持 60+ 岁个人付款 |
| ❌ 新商户限制 | 商户成立 < 6 个月不支持纯个人款项 |

> **实战启示**：
> - C 端业务（toC 跨境电商，买家全是个人）要管好占比 ≤ 50%
> - **B 客户里"新成立的 HK 公司 + 大量个人 C 端付款"几乎一定踩线**
> - 60+ 岁限制意味着海外退休群体客户（如美国退休消费者）不能作为付款人——这条很特别

---

### 特征 7：**首笔严格、后续松绑**

| 场景 | 首笔 | 非首笔 |
|----|----|----|
| 同名充值 | 5000 USD 试单 | 走分级机制 |
| 海外仓 | 国内段 + 国外段全链路物流 | 仅尾程 |
| 一件代发 | 完整 ERP 截图 + 物流 | 同付款人仅首次 |

> **实战启示**：
> - 首笔通过 = 客户档案"升级"，后续运营成本骤降
> - 业务侧应**特别重视首笔过审**——多投入资源辅导
> - 对应运营策略：**"首笔陪跑服务"**，过审后客户自助

---

### 特征 8：**结汇网站是验证业务真实性的核心抓手**

要求层层细致：
- 必须**实际架设**（不能死链 / 模板站 / 抄袭站）
- 必须**属于商户**（域名归属 / Whois / 备案）
- 产品类目必须**与出口品类一致**
- 服务类客户（物流）需展示**主营物流服务**

> **实战启示**：
> - **没有自有网站的客户 = 入账几乎不可能**
> - 我方可产品化"**入网网站建站服务**"——给客户做合规外贸网站（已有市场，TikTok、Shopify 都做这个）
> - 网站需要常态化维护（产品上新、新闻更新），不能是"死站"

---

### 特征 9：**物流单据形式接受度广但要"可查"**

接受形式：
- 国际段：海运 / 空运正本 / 电放提单 / 快递面单 / 铁路 / 中港车 / 报关单
- 大陆段：商业快递（顺丰）/ 货拉拉 / 闪送 / 满帮 App 截图 / 货代送货单

但必须 4 要素齐：**发件人、收件人、商品、日期**。**草稿件不收**。

> **实战启示**：
> - 物流单据可作假成本相对低（大陆段尤其）
> - **A 主要靠"号可查"+ 信息一致性"做核查**
> - **关键防御点**：A 真去查物流公司确认 = 全盘破，所以材料必须背靠真实物流单（哪怕成本高）

---

### 特征 10：**EXW 的"出境段缺失"补救机制**（A 的灵活性体现）

EXW 场景天然没出境物流凭证（买家自提）。A 没一刀切，提供 3 种替代路径：

- 买家指定国内仓：仓库沟通 + 境内段物流
- 买家上门提货：买家给的出境单号 / 仓库带唛头照片 / 买家签收单
- 国内工厂再加工：境内段物流 + 三方协议（公章）

> **实战启示**：
> - EXW 场景是 **真实大陆贸易商常用的方式**——A 实事求是
> - 我方可重点开发 EXW 客群——这类客户多是真实工厂 / 出口商，材料能落实
> - **替代材料路径要做成 SOP**，业务侧能引导客户准备

---

### 特征 11：**特殊场景需"开户阶段报送风控 TL"**

特殊档口、COD、组货人等非标场景，开户阶段就要风控 TL 审批，且交易期间还有限额：

特殊档口（无 EXW 物流）：
- 企业付款人 + 主营匹配 → 月 < 5 万 USD
- 企业 + 主营不匹配 + 有正规附言 → 月 < 2 万 USD
- 企业 + 主营不匹配 + 无附言 → 月 < 1 万 USD
- 个人付款人主营匹配 → 月 < 2 万 USD
- 个人付款人主营不匹配 + 有附言 → 月 < 1 万 USD

> **实战启示**：
> - 非标场景不是"补材料"能解决，要 **走单独审批通道**
> - 进入这类通道的客户 **限额很硬**——业务模型受限
> - B 客户里"档口型客户"（深圳华强北、广州十三行等批发档口）数量大但流水受限

---

### 特征 12：**风控可疑时的"无限调单权"**

明文："当订单风险无法排查时，需提供**包括但不限于**汇款水单 / 关于本次订单查询的完整沟通截图等材料"。

> **实战启示**：
> - **过审不等于"过关"**，A 随时可二次调单
> - 客户必须长期保留全套材料 + 沟通记录（≥ 6 年）
> - 我方应建立"**客户原始证据档案库**"统一管理，调单时秒查

## 二、对 B-T-A 架构的关键影响

### 2.1 入账侧 T 的 4 个核心价值

| T 帮助点 | 解决 A 政策中的什么问题 |
|----|----|
| **提供"可核实贸易背景"** | 让"新客户-普通"升级为"新客户-优质"，免去严格审核 + 限额 |
| **充当"实际买家"** | 替缺真实买家的 B 客户做 PI 主体（合规度看 T 的真实经营）|
| **代办结汇网站** | T 的网站 / 子站可作为客户的入账网站（需做实控关系材料）|
| **代办出境物流** | T 作为发货 / 收货链条中一方，提供物流单据 |

### 2.2 入账侧"危险信号"（B-T-A 架构容易踩的）

| ⚠️ 风险点 | 解释 |
|----|----|
| T 被识别为"支付公司"或"PSP" | A 的代付规则禁支付公司代付，T 对外不能有任何支付字眼 |
| T 代付 + 无规范附言 + 无证明 | 直接退回，必须四选一证明（最容易的：T 用域名邮箱发代付声明）|
| 同名充值（B 客户 + T）反复来回 | 触发反过桥规则 |
| 个人付款占比 > 50% | 整个商户分类被降级 |
| 商户新成立 < 6 个月 + 大量个人付款 | 直接拒 |

### 2.3 入账场景与 B 业务的对应

| B 业务 | 对应 A 入账场景 | 友好度 |
|----|----|----|
| Web2 跨境电商收款（B2B 直发）| 标准 B2B | ✓ 高 |
| Web2 跨境电商（一件代发 / DropShipping）| 一件代发 | ✓ 高 |
| Web2 跨境电商（海外仓发货）| 海外仓 | ✓ 高 |
| Web2 跨境电商（自有海外店铺）| B2B2C | ✓ 中（要实控关系）|
| OnRamp（B 客户卖家收法币）| 标准 B2B（包装为真实交易）| ⚠️ 看包装 |
| Web3 项目方收 SaaS / 服务费 | 物流服务 / 特殊场景 | ⚠️ 难找对应 |
| 数币 OTC 散单 | **无对应场景** | ✗ 极难 |
| 客户接受桥接代收（COBO）| 代付（T 是付款方）| ⚠️ 需附言或证明 |

> **关键**：B 业务里 **越靠近"真实跨境电商"的越容易过 A 入账**；越靠 **纯数币 / OTC / 散单的越难**。

## 三、A 入账调单的"潜规则"总结

```
1. 三件套硬要求（PI + 物流 + 网站）—— 没网站基本无解
2. 场景识别是材料准备的起点 —— 14+ 场景各有套路
3. 20240601 是新老客分水岭 —— 时间锁定后客户分层
4. 同名往返被反过桥规则禁 —— 入出均封
5. 代付看付款方身份 + 附言规范性 + 证明 —— 三道闸
6. 个人付款是高敏区 —— 多重限制叠加
7. 首笔严格、后续松绑 —— 重视首笔陪跑
8. 网站验真是核心抓手 —— 没站等于没业务
9. 物流单据形式松、内容严 —— 4 要素 + 号可查 + 非草稿
10. EXW 给了灵活路径 —— 真实工厂客户友好
11. 非标场景要单独 escalation —— 直接走风控 TL
12. 风控保留无限调单权 —— 材料长期留存
```

## 四、对我方业务的关键动作

1. **客户入账场景识别产品化**：BD 阶段标准化问卷 → 自动匹配 14+ 入账场景 → 输出"材料清单 + T 是否介入 + 限额预估"
2. **首笔陪跑 SOP**：所有新客户首笔走"陪跑通道"，业务 + 合规人工辅导，提升过审率
3. **结汇网站服务产品化**：给客户做合规外贸网站（建站 + 域名 + 维护）；T 也可以提供"网站模板共享"给小客户
4. **附言模板下发**：所有 B 客户的买家发款时提供"规范附言"模板（产品名 + PI + 买卖双方名）—— 减少代付证明工作量
5. **T 对外品牌严控**：T 的工商范围、网站、对外宣传 0 涉及"支付""结算""收付款"等敏感词
6. **客户档案库**：建立每个客户的"原始材料库"——合同 / 网站快照 / 物流凭证 / 沟通记录，调单秒查
7. **入网客户分层**：把客户按"自身入账可行性"分级：A 类（标准 B2B，材料齐）/ B 类（需 T 部分介入）/ C 类（必须 T 桥接代收）
