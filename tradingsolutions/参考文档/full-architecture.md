# 全局架构：商户 · 租户 · EX · SP（借助 EX 体系）

> **目的**：从 EX 平台架构视角，说明 BB 运营主体（SP）、BBSRO（独立 SP）、X 租户、商户之间的关系，以及渠道的运营渠道/持牌渠道拆分。

---

## 一、EX 平台架构（更新版）

### 核心角色定义

| 角色                   | 身份                        | 说明                                                                    |
| ---------------------- | --------------------------- | ----------------------------------------------------------------------- |
| **BB 运营主体**  | **SP（服务商）**      | 运营 SP，维护 X↔商户主子账户关系，维护持牌渠道与运营渠道关系           |
| **BBSRO**        | **独立 SP（服务商）** | 持牌 SP，只接收 X 的数据（合规视图）；MTC / A / 智朗都是 BBSRO 下的渠道 |
| **X1 / X2 / X3** | **租户（Tenant）**    | 每个 X 是一个独立租户，有自己的网址；X 之间商户不能重复入网             |
| **X（主商户）**  | **主商户**            | X 在自己租户下作为主商户入网                                            |
| **普通商户**     | **子商户**            | 在 X 租户下作为子商户入网                                               |

### 分层结构

```
[ 子商户 M001/M002 ]  ←  普通商户，在某个 X 租户下作为子商户入网
       │
       ▼
[ X（主商户）]        ←  X 自身作为主商户入网（X 租户下）
       │
       ▼
[ X 租户 ]            ←  每个 X 一个租户，一个网址（X1/X2/X3 互相隔离，商户不可重复）
       │
       ├──→ [ BB 运营主体 SP ]  ←  维护 X↔商户主子账户关系 + 持牌/运营渠道关系
       │         │
       │         ▼
       └──→ [ BBSRO SP ]        ←  只接收 X 数据（合规）；渠道 = MTC / A / 智朗
                  │
                  ▼
           [ 渠道 Channel ]      ←  分"运营渠道"和"持牌渠道"两类
```

### 关键规则

1. **X1, X2, X3 之间商户不可重复入网** — 一个商户只能属于一个 X 租户
2. **主子账户都归 BB 运营主体 SP** — X（主商户）和普通商户（子商户）的账户都在 BB 运营主体下
3. **只有 X 的数据给到 BBSRO** — 普通商户数据不上报 SRO，BBSRO 只看到 X 主商户层
4. **BB 运营主体 SP 额外维护**：持牌渠道与运营渠道的映射关系

## 二、X 租户体系详解

### X 作为独立租户

每个 X（X1、X2、X3...）在 EX 上是一个**独立租户**：

| 属性                 | 说明                                                            |
| -------------------- | --------------------------------------------------------------- |
| **独立网址**   | X1.domain.com、X2.domain.com — 各自独立品牌站点                |
| **独立商户池** | X1 下的商户不能同时出现在 X2 下（**跨租户不可重复入网**） |
| **主商户**     | X 自身作为"主商户"入网到自己的租户下                            |
| **子商户**     | 普通商户在 X 租户下作为"子商户"入网                             |

### X 租户下的入网结构

```
┌─ X1 租户（独立网址：x1.domain.com）──────────────┐
│                                                     │
│   X1（主商户）← X 自身入网为主商户                    │
│     ├── M001（子商户）                               │
│     ├── M002（子商户）                               │
│     └── M003（子商户）                               │
│                                                     │
│   ⚠️ M001 不能同时在 X2 租户下入网                    │
└─────────────────────────────────────────────────────┘

┌─ X2 租户（独立网址：x2.domain.com）──────────────┐
│                                                     │
│   X2（主商户）← X 自身入网为主商户                    │
│     ├── M004（子商户）                               │
│     ├── M005（子商户）                               │
│     └── M006（子商户）                               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 核心约束：商户不可跨 X 重复

- **同一商户只能在一个 X 下入网**，跟其他 EX 租户逻辑不同（其他租户间可重复）
- 原因：避免同一商户在不同 X 下产生重复合规数据、重复资金流
- 系统实现：入网时校验商户唯一标识（如注册邮箱、法人证件号），跨 X 租户去重

## 三、双 SP 体系：BB 运营主体 + BBSRO

### SP 1：BB 运营主体

| 维度               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| **身份**     | SP（服务商）                                                 |
| **核心职责** | 维护 X↔商户的主子账户关系；维护持牌渠道与运营渠道的映射关系 |
| **数据范围** | 看到所有 X（主商户）+ 所有普通商户（子商户）的完整数据       |
| **账户归属** | X 的主账户 + 普通商户的子账户 → 都归到 BB 运营主体 SP 下    |
| **渠道定位** | 以**运营渠道**为主                                     |

### SP 2：BBSRO

| 维度               | 说明                                                           |
| ------------------ | -------------------------------------------------------------- |
| **身份**     | 独立 SP（服务商），与 BB 运营主体是两个独立 SP                 |
| **核心职责** | 合规持牌主体，只接收 X 的数据                                  |
| **数据范围** | **只看到 X（主商户）的数据**，看不到子商户 M 的数据      |
| **渠道定位** | 以**持牌渠道**为主；MTC、A(IPL)、智朗都是 BBSRO 下的渠道 |
| **渠道**     | MTC / A / 智朗 — 全部挂在 BBSRO 下                            |

### 数据流向

```
┌─ X1 租户 ──────────────────────┐
│  X1（主商户）+ M001/M002（子商户）│
└────────────┬───────────────────┘
             │
             ▼
┌─ BB 运营主体 SP ──────────────────────────────────────┐
│                                                         │
│  ✅ 看到：X1 + M001 + M002（全量）                       │
│  ✅ 维护：X1↔M001/M002 主子账户关系                      │
│  ✅ 维护：持牌渠道（BBSRO 下）↔ 运营渠道（BB 下）关系      │
│  ✅ 主子账户都在此 SP 下                                  │
│                                                         │
│  渠道方向：运营渠道为主                                    │
└────────────┬───────────────────────────────────────────┘
             │ 只传递 X 的数据（不传 M 数据）
             ▼
┌─ BBSRO SP ────────────────────────────────────────────┐
│                                                         │
│  ✅ 只看到：X1、X2、X3（主商户数据）                      │
│  ❌ 看不到：M001、M002 等子商户                          │
│                                                         │
│  渠道方向：持牌渠道为主                                    │
│  渠道列表：MTC / A(IPL) / 智朗                            │
└───────────────────────────────────────────────────────┘
```

## 四、渠道体系：运营渠道 vs 持牌渠道

### 渠道双维度

| 维度                           | 说明                             | 所属 SP               |
| ------------------------------ | -------------------------------- | --------------------- |
| **运营渠道** | 负责日常运营、商户对接、资金操作 | 主要归 BB 运营主体 SP |
| **持牌渠道** | 负责合规持牌、监管报告、渠道签约 | 主要归 BBSRO SP       |

### 渠道归属

| 渠道              | 归属 SP | 身份     | 说明                                                 |
| ----------------- | ------- | -------- | ---------------------------------------------------- |
| **MTC**     | BBSRO   | 持牌渠道 | SRO 在 MTC 开户入网                                  |
| **A (IPL)** | BBSRO   | 持牌渠道 | BB 已在 A 入网开户（做发卡时完成），BB 与 A 定期调拨 |
| **智朗**    | BBSRO   | 持牌渠道 | 大账户模式                                           |

### BB 运营主体 SP 的额外职责

BB 运营主体 SP 需要维护**持牌渠道与运营渠道的对应关系**：

```
BB 运营主体 SP 维护的映射表：
┌───────────────┬──────────────┬──────────────────┐
│ 渠道           │ 运营渠道归属   │ 持牌渠道归属      │
├───────────────┼──────────────┼──────────────────┤
│ MTC           │ BB 运营主体 SP │ BBSRO SP         │
│ A (IPL)       │ BB 运营主体 SP │ BBSRO SP         │
│ 智朗           │ BB 运营主体 SP │ BBSRO SP         │
└───────────────┴──────────────┴──────────────────┘
```

- 运营渠道维度：BB 运营主体 SP 执行日常操作（OnRamp/OffRamp/出入金）
- 持牌渠道维度：BBSRO SP 承担合规责任（渠道签约、监管报告、调单响应）
- 同一渠道的运营渠道归属和持牌渠道归属可以不同（当前都是 BB/BBSRO 体系内，未来可扩展）

## 五、可见性矩阵

| 观察者                       | 看到 X（主商户） | 看到 M（子商户） | 看到渠道    | 看到主子关系     | 看到持牌/运营渠道关系 |
| ---------------------------- | ---------------- | ---------------- | ----------- | ---------------- | ----------------- |
| **X 租户**             | ✓ 自己          | ✓ 自己的子商户  | ✗          | 部分（自己下的） | ✗                    |
| **BB 运营主体 SP**     | ✓ 全部          | ✓ 全部          | ✓ 运营渠道 | ✓ 全部          | ✓ 维护方             |
| **BBSRO SP**           | ✓ 全部          | ✗               | ✓ 持牌渠道 | ✗               | ✗                    |
| **渠道（MTC/A/智朗）** | 部分             | ✗               | — 自己     | ✗               | ✗                |
| **商户 M**             | ✗               | — 自己          | ✗          | ✗               | ✗                |

### 隔离红线

- **BBSRO 绝对看不到子商户 M** — 只看到 X 主商户数据
- **X 租户之间完全隔离** — X1 看不到 X2 的商户
- **商户看不到 X/SP/渠道** — 商户只看到自己所在的 X 租户品牌
- **渠道看不到子商户** — 渠道只对接 X 或 BB 运营主体

## 七、关键设计原则

### 1. X 租户独立性

- 每个 X 是独立品牌、独立网址、独立商户池
- X 之间商户不可重复 — 这是与普通 EX 租户逻辑的**核心差异**
- X 自身作为主商户入网，体现"代理商"角色

### 2. 双 SP 分工

- **BB 运营主体 SP**：全量数据 + 主子关系 + 持牌/运营渠道映射 = 运营中枢
- **BBSRO SP**：只接 X 数据 + 持牌渠道 = 合规中枢

### 3. 渠道双维度管理

- 运营渠道（日常操作）归 BB 运营主体
- 持牌渠道（合规签约）归 BBSRO
- BB 运营主体负责维护两者的对应关系

### 4. 商户唯一性约束

- 一个商户只能在一个 X 下入网（跨 X 去重）
- 这保证了合规数据不重复、资金流可追溯、调单链路清晰
