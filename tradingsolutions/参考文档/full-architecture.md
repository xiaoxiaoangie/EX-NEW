# 全局架构：商户 · 租户 · EX · SP（含双 SP 签约场景）

> **目的**：从 EX 平台架构视角，说明 BB / BBSRO 在不同 SP 签约关系下的客户映射、租户归属、SP 可见性，确保 BB 能持续展业、同时 SRO 只看到合规的桥接客户。

---

## 一、EX 平台四层架构

EX 平台的标准分层：

```
[ 商户 Merchant ]  ←  最终业务客户（C 端 / B 端）
       │
       ▼
[ 租户 Tenant ]    ←  EX 平台上的运营主体（持牌或非持牌运营机构）
       │
       ▼
[ EX 平台 ]        ←  产品中心 / 客户中心 / 清算中心 / 渠道中心 / 财资中心 / 合规风控
       │
       ▼
[ SP Service Provider ]  ←  实际持牌 + 持有银行/清算渠道的服务商
       │
       ▼
[ 渠道 Channel ]   ←  银行 / 清算网络 / 卡组 / 链上节点
```

### 各层职责

| 层级 | 角色 | 例子 |
|---|---|---|
| **商户** | 真实业务客户，发起交易 | M001、M002（普通客户）；X001、X002（桥接主体客户）；T001（贸易桥接客户） |
| **租户** | EX 上的运营主体，对商户面客 | BB（普通运营主体）、BBSRO（持 SRO 牌照的主体） |
| **EX** | 多租户 SaaS 平台 | 产品/客户/清算/渠道/财资/合规 六大中心 |
| **SP** | 持牌服务提供商 + 渠道集成方 | BBSRO（自有 SRO 牌照）、IPL（HK MSO）、MTC（其他司法持牌） |

## 二、基线示例：BBSRO 作为 SP（自闭环）

> **场景描述**：BB 集团内部由 BBSRO（持 SRO 牌照的兄弟主体）作为 SP，BB 作为租户在 EX 上对外营业。

### 实体映射

| EX 层级 | 实体 | 说明 |
|---|---|---|
| 商户 | M001、M002（普通客户）<br>X001、X002（桥接客户） | M001/M002 是 BB 的存量历史客户，材料不达 SRO 入网标准 |
| 租户 | **BB** | BB 作为运营主体，承接所有面客业务 |
| SP | **BBSRO** | BB 的关联持牌主体，持有 SRO 牌照 + 渠道签约 |
| 映射规则 | M001 → X001<br>M002 → X002 | BB 内部把 M 客户的交易"穿透"为 X 桥接主体的交易，再交给 BBSRO 处理 |

### 可见性

```
┌─ BB（租户）─────────────────┐
│  全量看见：M001、M002、X001、X002        │
│  执行映射：M001↔X001、M002↔X002          │
└──────────┬──────────────────┘
           │ 仅传递 X001 / X002 的交易
           ▼
┌─ BBSRO（SP）────────────────┐
│  只看到：X001、X002                       │
│  M001、M002 对 SRO 不可见                 │
└──────────────────────────────┘
```

### 关键点

- **SRO 只看到桥接主体 X**，所有交易在 SRO 视角下都是合规交易
- **BB 内部维护 M↔X 映射表**，承担合规过滤责任
- **资金流闭环**：BBSRO 通过自有渠道完成 VA / POBO / 清算
- **没有外部 SP 介入**，集团内部完成业务

## 三、场景 1：BB 直接跟渠道签约（非 BBSRO 签约）

> **背景**：BB 想用其他持牌 SP（如 IPL / MTC）的渠道能力，签约主体是 BB 而非 BBSRO。

### 实体映射

| EX 层级 | 实体 | 说明 |
|---|---|---|
| 商户 | M001、M002（普通客户）<br>X001、X002（桥接客户） | 商户层结构与基线一致 |
| 租户 | **BB** | BB 作为面客运营主体 |
| SP | **IPL / MTC** | **合作主体 = BB**，不是 BBSRO |
| 映射规则 | M001 → T001（IPL 视角下的桥接客户）<br>M002 **可能直接满足 IPL 要求**，无需映射 | IPL/MTC 的合规标准与 BBSRO 不同，部分 M 客户原生达标 |

### 可见性

```
┌─ BB（租户）────────────────────────┐
│  全量看见：M001、M002、X001、X002、T001          │
│  按 SP 不同执行不同映射策略：                     │
│    - 走 BBSRO 路：M001→X001、M002→X002           │
│    - 走 IPL/MTC：M001→T001、M002 直连            │
└──────────┬──────────────────────────┘
           │ IPL/MTC 路只传 T001 + M002
           ▼
┌─ IPL / MTC（SP）─────────────────┐
│  签约对手：BB                                  │
│  看到：T001（M001 的桥接）、M002（直接客户）    │
└─────────────────────────────────┘
```

### 关键点

- **BB 直接持有 SP 合同**，IPL/MTC 把 BB 当作机构客户对接
- **映射规则按 SP 不同灵活配置**：
  - 不达标客户 → 桥接为 T 主体上链
  - 达标客户 → 直接以本名上链（M002 直连 IPL）
- **BB 要在 EX 上维护多套映射表**（针对不同 SP 的合规标准）
- **优势**：BB 可以横向接入多家 SP，扩展渠道覆盖
- **劣势**：BB 自己承担与每家 SP 的合规对接成本

## 四、场景 2：BBSRO 直接跟渠道签约

> **背景**：BBSRO 用自己的 SRO 牌照与外部 SP（IPL / MTC）签约，租户层换成 SRO 主体直接面客。

### 实体映射

| EX 层级 | 实体 | 说明 |
|---|---|---|
| 商户 | X001、X002、T001、T002（**全部是桥接主体**） | 不再有 M001、M002 这类原始客户身份 |
| 租户 | **SRO**（即 BBSRO 自身充当租户） | BB 作为租户的角色让位 |
| SP | **IPL / MTC** | **合作主体 = BBSRO**，不是 BB |
| 映射规则 | M001 → X001、M002 → X002（在 BB 侧已完成映射） | BB 把 M 客户穿透为 X 桥接主体后，再交给 SRO 租户 |

### 可见性

```
┌─ BB（隐含运营层，非 EX 租户）─────────┐
│  原始接客：M001、M002                          │
│  完成映射：M001→X001、M002→X002                │
└──────────┬─────────────────────────────┘
           │ 只把 X001 / X002 注入 SRO 租户
           ▼
┌─ SRO（EX 租户，即 BBSRO）─────────┐
│  看到：X001、X002、T001、T002（全部桥接主体）│
│  M001、M002 已经不在 SRO 视野里             │
└──────────┬─────────────────────────┘
           │ SRO 持有合同，把交易传给 SP
           ▼
┌─ IPL / MTC（SP）──────────────────┐
│  签约对手：BBSRO                               │
│  看到：X001、X002、T001、T002                   │
└─────────────────────────────────┘
```

### 关键点

- **租户层从 BB 变为 SRO**：合规等级抬高，SRO 持有所有合同关系
- **SRO 视野里只有桥接主体**，看不到底层 M 客户
- **BB 退到 SRO 后端**：负责接客 + 映射，但不出现在 EX 租户层
- **SP 视角更"干净"**：所有交易都是合规桥接主体发起，符合 IPL/MTC 的入网要求
- **优势**：SRO 牌照背书 → SP 接受度高 → 渠道议价能力强
- **劣势**：所有客户必须做"穿透映射"，BB 端运营成本高

## 五、三种场景对比

| 维度 | 基线（BBSRO 自闭环） | 场景 1（BB 签约外部 SP） | 场景 2（BBSRO 签约外部 SP） |
|---|---|---|---|
| **EX 租户** | BB | BB | SRO（即 BBSRO） |
| **EX SP** | BBSRO | IPL / MTC | IPL / MTC |
| **SP 合作对手** | （集团内部） | **BB** | **BBSRO** |
| **商户层身份** | M + X 共存 | M + X + T 共存 | 全部 X / T（桥接） |
| **SP 看到的客户** | X001、X002 | T001、M002 | X001、X002、T001、T002 |
| **M 是否对 SP 可见** | ✗ | M002 可见（达标）<br>M001 不可见 | ✗ 全部不可见 |
| **映射执行层** | BB 租户 | BB 租户（多套映射） | BB 后端（注入 SRO 前完成） |
| **合规背书强度** | SRO 自有牌照 | BB 主体 | SRO 牌照 |
| **适用客户** | SRO 合规客户 | IPL/MTC 标准客户 | 所有需 SRO 背书 + 外部渠道的客户 |

## 六、关键设计原则

### 1. **BB 持续展业**

无论 SP 是 BBSRO 还是 IPL/MTC，BB 都保留"接客 + 映射"的能力：

- 历史 M 客户继续在 BB 入网（低门槛）
- 通过映射穿透为合规桥接主体（X / T）后，再走任意 SP 通道
- BB 不会因为 SP 切换而失去存量客户

### 2. **SP 视野隔离**

不同 SP 看到不同集合的客户：

- **BBSRO**：只看 X 桥接主体，确保 SRO 合规水位
- **IPL / MTC**：根据签约主体（BB or BBSRO）决定看到 M+T 还是 X+T

### 3. **桥接主体职责**

| 桥接类型 | 用途 | 适用场景 |
|---|---|---|
| **X**（SRO 桥接） | 把 M 客户穿透为 SRO 合规客户 | BBSRO 路线（基线 + 场景 2） |
| **T**（贸易桥接） | 提供贸易背景材料 / 替代收付 | 外部 SP 路线（场景 1 + 场景 2） |

### 4. **映射表维护**

BB / SRO 需要在 EX 客户中心维护：

- M ↔ X 映射（BBSRO 路）
- M ↔ T 映射（外部 SP 路）
- 多对多关系 + 时效管理（一个 M 可能在不同 SP 路下映射不同桥接主体）
