## 1. 问题陈述

### 1.1 现状诊断

| 维度 | 现状 | 痛点 |
|------|------|------|
| A 的牌照 | 真实持有 MSO（HK）/ MPI（SG）/ MSB（US）等 | 强 |
| A 的银行通道 | 多家本地银行账户、VA 能力 | 中-强（个别银行近期收紧）|
| A 的合规团队 | 制度齐、人员到位 | **过度保守**：拒绝率高、RFI 滥用、一刀切 |
| A 的客户结构 | 头部 + 中部为主 | 长尾接不住、转化率低 |
| A 的产能利用率 | 估计 30-50% 之间 | **大量本来能接的客户被劝退** |
| A 与商户的体验 | 准入慢、补料反复 | 客户流失到 PingPong / XTransfer / WorldFirst |

### 1.2 一句话问题

> **A 不是合规能力不够，是合规风险偏好过保守 + 流程未优化 + 团队 KPI 错位，导致大量本可合规接入的客户被自己劝退。**

### 1.3 不是什么问题

- ❌ 不是"A 牌照不够用" —— 牌照覆盖足够
- ❌ 不是"A 合规要求太高" —— MSO 客观要求就在那，绕不开
- ❌ 不是"市场没需求" —— 同行都在吃这块
- ✅ 是"A 在 MSO 框架内本可以接住更多客户，但当前的流程 / 偏好 / KPI 让它接不住"

### 1.4 目标

**在不动 MSO/MPI 任何合规底线的前提下，把 A 的月活商户数 / 月流水做到当前的 2-3 倍。**

## 2. MSO/MPI 合规边界（不可越线）

> 本节是后续所有动作的"天花板"。任何放量动作不得突破本节任何一条。

### 2.1 香港 MSO（AMLO + Customs 监管）

| 类别 | 必做 | 不得做 |
|------|------|--------|
| **KYC/CDD** | 客户身份核实、UBO 穿透、PEP 筛查、制裁名单筛查 | ❌ 简化版 KYC 用于非低风险客户 |
| **EDD** | 高风险客户加强尽调（来源国 / 行业 / 流水）| ❌ 跳过 EDD 接入高风险客户 |
| **三流匹配** | 资金流 + 货物流 / 服务流 + 单据流 一致 | ❌ 接受"账户对账户"无业务背景资金过桥 |
| **来源 / 用途** | 资金来源 (SoF) + 财富来源 (SoW) + 资金用途说明 | ❌ 仅凭客户口头说明放行大额 |
| **交易监控** | 持续监控、异常告警、阈值规则 | ❌ 一次准入终生免审 |
| **STR** | 可疑交易 30 日内向 JFIU 报送 | ❌ 知情不报 / 拖延报送 |
| **记录保存** | KYC + 交易记录 ≥ 6 年 | ❌ 自行销毁 / 不可追溯 |
| **Nesting** | 客户限于终端商户 **或持牌机构**；服务持牌 PI 须银行知情 + EDD + 持续监控 | ❌ 为 **无牌** 支付机构 / 钱庄做"二级账户"（= 协助无牌经营）|
| **制裁合规** | UN / HKMA / OFAC（视客户结构）筛查 | ❌ 接收制裁国 / 制裁个人资金 |
| **持续培训** | 合规人员 + 业务人员定期受训 | ❌ 制度上墙不培训 |

### 2.2 新加坡 MPI（PSA + MAS 监管）

与 HK MSO 相似，**额外**强约束：

- **MAS Notice PSN01 / PSN02**：AML/CFT 标准更细；技术风险管理 (TRM) 要求更严
- **客户尽调分级**：Simplified / Standard / Enhanced 三档明确
- **跨境汇款限额**：单笔 / 月度阈值合规报送
- **Travel Rule**（涉及加密时）：超 SGD 1500 须随附发起人 / 受益人信息

### 2.3 美国 MSB（FinCEN 监管，如适用）

- BSA/AML 合规计划、CIP（Customer Identification Program）
- SAR / CTR 报送（CTR 单笔 ≥ USD 10,000）
- 各州 MTL 牌照对应州内业务

### 2.4 红线一句话清单（A 永远不做）

| 红线 | 后果 |
|------|------|
| ❌ 接 **无牌** 支付机构 / 钱庄做"二级账户" | MSO 牌照吊销 + 协助无牌经营刑责 |
| ❌ 接钱庄 / 无业务背景过桥 | 刑事追责 |
| ❌ 接制裁国 / 制裁主体 | OFAC 罚款 + 刑事 |
| ❌ 跳过 KYB 接客 | 监管处罚 + 牌照风险 |
| ❌ 三流不匹配硬放行 | STR 漏报 + 牌照风险 |
| ❌ 给客户"合规担保" / "合规 by 我们" | 法律追责 |
| ❌ 在银行不知情情况下做 nesting / PI-as-customer 业务 | 银行账户被关 + 信任崩塌 |

> **本文档 §3 起所有放量动作，必须在本节之上。** 触红线 = 立即停止，无任何例外。

### 2.5 Nesting 的正确边界（澄清：不是一刀切的红线）

> 这里要把"nesting 红线"拆开讲清楚，避免合规团队把"持牌 PI 客户"误判为红线一刀切拒绝。

| 场景 | MSO 法规层面 | 银行政策层面 | 是否可做 |
|------|-----------|-----------|--------|
| A 服务**无牌**金钱服务经营者 / 地下钱庄 | ❌ 协助无牌经营，AMLO + MSO Ordinance 红线 | ❌ 银行必拒 | **绝对不做** |
| A 服务**持牌 PI**（HK MSO / SG MPI / EU EMI / US MTL 等），但银行不知情 | ⚠️ 法规不绝对禁，但若被穿透到无牌主体则违规 | ❌ 多数银行账户协议明确禁"在未披露情况下为其他 PI 提供服务" | **不做**（银行风险太大）|
| A 服务**持牌 PI**，**银行知情 + 同意 + 加强 KYC + 持续监控** | ✅ FATF R.13 + AMLO 框架内可做（视为 Correspondent / FI 客户）| ✅ 部分 PI-friendly 银行（如 DBS Treasures、ZA、Airwallex partner banks）允许，但要披露 + 报告 | **可做**，需 EDD + 银行专项审批 |
| A 服务**持牌 PI 的下游商户**（B2B2B，且 A 直接 KYB 商户）| ✅ 等同于直接服务商户 | ✅ 多数银行可接受 | **可做** |

**关键判断点**：

1. **下游主体是否持牌**：持牌 → 法规可做；无牌 → 一律不做
2. **银行是否知情且同意**：未披露 → 不做（银行风险）；披露且同意 → 可做
3. **是否做了 EDD 和持续监控**：把 PI 客户当成"金融机构客户"做 EDD（业务模式 / 客户群 / AML 程序 review / 流水监控）
4. **AML 责任划分是否清晰**：A 与下游 PI 的 AML 分工写进协议，A 不替下游 KYC 终端商户但保留对下游 PI 的监控义务

**判定流程**：

```
[来了一个 PI / 类 PI 客户]
       │
       ▼
[查持牌状态] ──无牌──► 直接拒（红线）
       │ 持牌
       ▼
[联系合作银行 RM 披露 + 询问可否] ──银行拒──► 拒，或换银行
       │ 银行同意
       ▼
[启动 EDD：业务模式 / 客户群 / AML 程序 / UBO]
       │
       ▼
[评估单银行 PI 客户集中度] ──超阈值──► 排队 / 换银行
       │ 在阈值内
       ▼
[准入 + 协议明确 AML 分工 + 持续监控]
       │
       ▼
[每季度向银行报告 + 内部抽查]
```

**结论**：

- 持牌 PI 客户 **是 A 的高价值客户类型**（流水大、客户结构清晰、合规可对话）
- 不该被合规团队按"nesting 红线"一刀切拒绝
- 真正的红线是 **无牌 + 银行不知情 + 没做 EDD** 这三件事任一触发

## 3. A 当前"过度保守"的诊断（要修的是这些）

### 3.1 偏好层面

| 偏好 | 表现 | 后果 |
|------|------|------|
| 风险厌恶 | "宁可不接，不能出事" | 接入率低 |
| 责任规避 | 合规团队个人承担拒绝/通过的责任 | 倾向拒绝 |
| 一刀切 | 一个国家有事 → 整个国家拉黑；一个行业有事 → 整个行业拒 | 误伤大量良性客户 |
| 不愿做灰色 | 灰色 ≠ 违规，但合规团队不愿背 | 灰色业务全流失 |

### 3.2 流程层面

| 流程问题 | 表现 |
|---------|------|
| KYB 模板过重 | 小客户也要交大客户的全套材料 |
| RFI 多轮反复 | 一次说清楚的事分 3 轮问，客户流失 |
| 审核时长不可控 | T+? 不透明，客户没耐心 |
| 决策权过度集中 | 小金额也要高级合规拍板 |
| 与业务团队割裂 | 业务团队不知道合规口径，承诺了又被拒 |

### 3.3 KPI 层面

| 现有 KPI | 副作用 |
|---------|-------|
| 合规事件数为 0 | 拒绝优先 → 接入率被牺牲 |
| 不考核接入率 / 流水 | 合规团队没动力放量 |
| 不考核客户体验 | 客户投诉与合规团队无关 |
| 不区分 RFI vs 拒绝 | 用 RFI 拖死客户，KPI 上不算拒绝 |

### 3.4 银行端压力的真实性

银行端确实给 A 施压（怕被穿透），但：

- 部分压力是 **A 合规团队"想象中的"**，未必银行真要求
- 部分压力是 **个别银行**，不代表所有银行
- 部分压力 **可以通过把客户分到不同银行通道** 缓解

> **诊断结论**：A 不需要降低合规标准，需要 **重新校准风险偏好 + 优化流程 + 重设 KPI + 银行通道分层**。

## 4. 在 MSO 约束下放大 A 产能的六条路径

### 4.1 路径 1：客户分层 + 差异化 KYB（最大杠杆）

**原则**：MSO 本身允许 **基于风险的差异化 CDD**（risk-based approach, RBA）。低风险客户用 Simplified CDD，高风险客户用 EDD，不是所有客户都做 EDD。

| 风险等级 | 客户特征 | KYB 强度 | 决策时长 SLA |
|---------|---------|---------|----------|
| **Low** | 持牌机构 / 上市公司 / 公开品牌的大型电商 | Simplified CDD | 3 工作日 |
| **Standard** | 标准外贸 / 服务出口 / 独立站，主体清晰、单据齐 | Standard CDD | 5-7 工作日 |
| **Medium** | 业务真实但材料不全、行业稍敏感 | Standard + 补料 | 7-10 工作日 |
| **High** | 行业敏感（虚拟币周边 / 高汇率国家 / 高现金行业）| EDD（增强尽调）| 10-15 工作日 |
| **Reject** | 红线触发（制裁 / 钱庄 / nesting）| 直接拒 | 1 工作日给明确反馈 |

**杠杆点**：

- 把 60-70% 客户判定为 Low/Standard，避免一刀切走 EDD
- Reject 客户必须 **明确反馈拒绝原因**（在合规允许的范围内），让客户去走桥接架构或友商，而不是吊死在 RFI

### 4.2 路径 2：业务类型矩阵 + 三流匹配模板化

**原则**：不同业务类型的"三流"不一样，应该有对应模板，而不是用货物贸易的模板套服务贸易。

| 业务类型 | 资金流凭证 | 货物 / 服务流凭证 | 单据流凭证 |
|---------|---------|-------------|---------|
| 货物贸易 (B2B) | 银行回单 | 报关单 / 提单 / 运单 | PI + 合同 + 商业发票 |
| 货物贸易 (B2C 跨境电商) | 平台账单 + 银行回单 | 平台订单 + 物流号 | 平台合同 + 自报关 |
| 服务贸易 (SaaS) | 银行回单 | **交付凭证**（账号开通邮件 / API 调用日志 / 后台截图）| 服务协议 + 月度账单 |
| 服务贸易 (设计 / 咨询) | 银行回单 | **交付物**（成果文件 / 验收单）| SOW + 发票 |
| 跨境广告投流 | 平台流水 | 广告主合同 + 投放报告 | 媒介采购合同 + 发票 |
| HR / 外包工资代发 | 银行回单 | 员工花名册 + 个税 / 社保记录 | 服务合同 + 月度结算单 |
| App Store / Steam 收款 | 平台结算单 | 平台开发者协议 + 上架证明 | 开发者主体证明 |

**杠杆点**：

- 每类业务有 **预定义模板包**，业务团队拿到客户立刻给到对应清单
- 客户 **一次性交齐**，避免 N 轮 RFI
- 合规团队按模板审核，**审核时长大幅压缩**

### 4.3 路径 3：RFI 标准化 + 决策权下沉

**原则**：RFI 不是"再问一遍"的工具，是"补齐关键缺失"的工具。一次 RFI 要把所有问题问完。

| 改造前 | 改造后 |
|-------|-------|
| RFI 多轮，每轮问 1-2 个 | **RFI 一次性清单**，把所有缺失项一次给到客户 |
| RFI 内容笼统："请补充业务说明" | **结构化模板**：缺项 + 具体要什么 + 提供示例 |
| RFI 必须高级合规审 | **简化 RFI 由初级合规审**，复杂由高级 |
| 客户回复后又起新一轮 | **回复后 SLA 24h 内给最终结论** |

**决策权下沉**：

| 客户金额 / 风险 | 现状 | 改造后 |
|------|------|-------|
| 月流水 < USD 50K 标准客户 | 高级合规审 | 初级合规审 + 抽样复核 |
| USD 50K-500K | 高级合规审 | 中级合规审 |
| ≥ USD 500K 或高风险 | 合规总监 | 不变 |
| 红线触发 | 合规总监 + 法务 | 不变 |

### 4.4 路径 4：银行通道分层 + 单银行集中度管理

**原则**：A 的不同银行账户对客户类型容忍度不同。把客户分配到合适的银行，而不是把所有客户塞同一个银行。

| 通道类型 | 容忍度 | 适合客户 |
|---------|-------|--------|
| Tier 1 主流银行（HSBC / DBS / Citi）| 低 | 头部大客户、纯白名单业务 |
| Tier 2 商业银行 / 数字银行 | 中 | 标准外贸 / 服务客户 |
| Tier 3 持牌虚拟银行 / 跨境收单方 | 中-高 | 中小客户、新兴业务 |
| VA 通道（Wise / Airwallex / Currenxie 等）| 中 | 收款型客户 |

**杠杆点**：

- 单银行集中度 ≤ 30-40%（避免单点失败）
- RM 关系定期维护，主动汇报合规故事
- 不要把所有"灰色但合规"业务集中到一个账户

### 4.5 路径 5：合规话术 / 合规故事统一口径

**原则**：A 对客户讲什么、对银行讲什么、对监管讲什么，必须 **一致**。三方故事不一致 = 自掘坟墓。

**对客户**：

- 我们是持牌 MSO/MPI
- 我们的合规要求来自监管，不是我们刻意为难
- 给材料示例 + 解释为什么要这些材料
- 拒绝时给明确理由（在合规允许范围内）

**对银行**：

- 客户结构（行业 / 国家 / 流水规模）定期主动汇报
- 异常事件（STR / 大额）主动告知
- 不藏不掖，让 RM 帮我们讲故事

**对监管**：

- 标准 KYB 档案 + 三流凭证 + 交易监控记录
- 合规手册版本化、培训记录可查
- STR 该报必报、CTR 该报必报

### 4.6 路径 6：合规团队 KPI 重设

> **这个不改，前面五条都是空的。**

| 维度 | 现有 KPI | 改造后 KPI |
|------|--------|----------|
| 风险类 | 合规事件数 = 0 | 重大合规事件 = 0 + STR 漏报 = 0 |
| 效率类 | （无）| 准入审核 SLA 达成率 ≥ 90% |
| 覆盖类 | （无）| 月度新增准入数（按风险等级加权）|
| 质量类 | （无）| 抽样复核通过率 ≥ 95% |
| 体验类 | （无）| 准入客户 NPS / 投诉率 |
| 协同类 | （无）| 业务团队对合规协作满意度 |

**核心是引入"覆盖率 KPI"，让合规团队和业务团队一起背量。**

## 5. 客户准入分层细则（A 直接接的客户）

### 5.1 哪些客户 A 能直接接

| 类型 | 画像 | 优先级 |
|------|------|------|
| **持牌机构客户** | 其他持牌支付机构 / 银行 / 数币持牌 | ⭐⭐⭐⭐⭐ |
| **大型品牌外贸 / 电商** | 年流水 USD 1M+，主体清晰，材料齐 | ⭐⭐⭐⭐⭐ |
| **中型规模化商户** | 年流水 USD 100K-1M，独立站 / SaaS / 服务出口 | ⭐⭐⭐⭐ |
| **标准化小商户**（材料齐）| 年流水 USD 10K-100K，主体真实、单据可补全 | ⭐⭐⭐ |
| **行业聚焦类客户** | 单一垂直行业内的标准玩家（如游戏 / SaaS / 跨境广告）| ⭐⭐⭐⭐ |

### 5.2 哪些客户 A **不**直接接（→ 走桥接）

| 类型 | 原因 |
|------|------|
| 月流水 < USD 5K 的散户 | KYB 边际成本 > 收入，让贸易主体集合服务 |
| 主体真实但单据补不全 | 材料缺口大，让贸易主体提供集合 KYB 包装 |
| 行业 A 当前不接（特定高风险行业）| 走贸易主体的合规分层 |
| 业务类型 A 模板化不到的 | 等模板成熟再回收 |

### 5.3 灰区处理：A 想接但需要专项评估

| 灰区客户 | 评估机制 | 决策周期 |
|---------|--------|--------|
| 行业敏感但客户合规可信 | 合规专项评估 + 试运行（月流水阈值）| 2-3 周 |
| 高汇率国家但单据齐 | EDD + 银行通道选 Tier 3 | 2 周 |
| 创新业务（NFT / Web3 服务）| 合规 + 法务联合 + 试点白名单 | 1 月 |

## 6. 与桥接架构的客户分流规则

### 6.1 入口统一，分流在合规层

```
            [所有意向客户]
                  │
                  ▼
        [统一意向登记 + 初筛]
        （业务团队负责 + 标准化问卷）
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
  [候选 A 直接接]      [候选桥接]
        │                   │
   走 §5.1                  走桥接架构（dual-license-bridge）
   不再回流              贸易主体 BD 池
```

**关键约束**：

- 一个客户在 **同一时点只走一条路** ：要么 A 直接接，要么走桥接，不能两条都走
- A 直接接 → 后续不再降级到桥接（除非主动退出）
- 桥接客户 → 不能"以小商户身份找 A 直接接"绕过桥接合规

### 6.2 分流判定矩阵

| 维度 | 走 A | 走桥接 |
|------|------|------|
| 月流水 | ≥ USD 10K | < USD 10K |
| 主体材料 | 齐全 | 不齐 |
| 单据能力 | 三流可匹配 | 单据缺口大 |
| 业务类型 | 在 §5.1 列表内 | 不在列表 |
| 客户态度 | 配合 KYB | 不愿走 KYB |
| KYB 边际成本 | < 预期收入 | ≥ 预期收入 |

> **任何一个维度落 "走桥接" → 走桥接**。 全部 "走 A" → 走 A。

## 9. 风险与红线

### 9.1 不能因为追量而……

| 红线 | 触发即停 |
|------|--------|
| ❌ 降低 KYB 标准 | 立即回滚到原标准 |
| ❌ 跳过 EDD 接高风险客户 | 重新审 |
| ❌ 三流不匹配硬放行 | 立即拦截 + STR |
| ❌ 让业务团队代替合规拍板 | 撤销决策 + 追责 |
| ❌ 与桥接客户混流 | 重新分流 |
| ❌ 银行 RM 不知情突击放量 | 主动告知 + 沟通 |

### 9.2 与桥接架构红线的对齐

| 桥接架构红线 | 本文档对应 |
|----------|----------|
| A 不接 nesting | 本文档 §2.4 同样禁 |
| A 与贸易主体保持独立 | 本文档分流规则确保不混流 |
| A 员工不参与贸易主体 | 本文档不与之冲突（员工只服务 A 自身客户）|
| 信息暴露红线（不暴露 CLE / 一期 BP）| 本文档同样适用 |
