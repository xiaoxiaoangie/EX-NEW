# 法币持牌主体（A）能力最大化方案：正常通道 + 桥接通道双轨架构

> *为了"活下去"，把 A 在 MSO 框架内的法币收付能力用到极限，让 Web2 + Web3 客户在客户视角"正常用 A"，在监管和银行视角"完全合规"。*

## 1. 背景与战略动因

### 1.1 三个事实

1. **必须"活下去"**：BP 一期 KPI 是流水 + 客户数，达不到则没有二期融资
2. **A 严重保守**：尤其 **数币背景业务**——不止"直接接触数币"被拒，**作为数币承兑商收付通道也被拒**；老板判断 **1-2 年内 MSO 将开放成为数币承兑通道是大势**，现在保守是阶段性偏好不是法规要求
3. **A 能力很强**：在 Web2 + Web3 双场景下，A 的法币收付通道是稀缺资源；如果连 A 都用不动，整个 Web2+Web3 大愿景基本无解

### 1.2 三个目标

| 目标 | 达成标志 |
|------|--------|
| **客户视角** | "我在正常用 A 的服务"——同样的入网、账户、收付体验 |
| **监管 / 银行视角** | A 提交的客户结构 + 三流凭证 + 报送材料齐备、口径一致、可经得起穿透 |
| **业务视角** | A 服务 Web2 直客 + Web3 收付通道客户 + 混合客户，月活 + 月流水做到当前 3 倍以上 |

### 1.3 不在本文档范围

- ❌ A 自身合规体系改造（A 的合规底线不动）
- ❌ CLE 牌照运营
- ❌ BP 财务模型

## 2. 行业参考：X 公司模式

```
[监管 / 银行]
     ▲ 只看到
     │
[X 持牌主体（外向展业）]
     ▲ 表面客户
     ├──[直客 A: 大客户，材料齐]
     ├──[桥接 B: 贸易主体，材料齐]
     └──[桥接 C: 贸易主体，材料齐]
            │
            ▼ 桥接吸收
        [终端小客户 / 缺材料 / Web3 客户]
```

**关键设计**：

1. 持牌主体只展示合规客户：直客 + 桥接主体
2. 桥接主体吸收"问题客户"：缺材料、Web3、散户、灰色
3. **比例平衡**：监管要看到足够"正常客户"，业内观察 **直客:桥接 ≥ 1:2** 是底线
4. 桥接提供监管想要的全套材料：合同 / 发票 / 报关 / 物流 / 服务凭证
5. 桥接 ≠ 我方控股：必须法律意义上的独立第三方

**X 模式合规边界**：

| 做 | 不做 |
|----|------|
| ✅ 桥接是真实独立法人 + 真实经营 | ❌ 空壳 / 我方代持 |
| ✅ 桥接提供真实贸易凭证 | ❌ 凭证全部伪造 |
| ✅ 桥接自己 KYC 终端客户 | ❌ 桥接不审任何人 |
| ✅ 桥接涉案则切割 | ❌ 与我方混同 |

> 本文档采纳 X 模式 + 多两层保护：可信伙伴硬约束 + 与 A 极致法律 / 治理 / 信息隔离。

## 3. A 当前能力盘点

### 3.1 A 实际能力（强）

| 能力 | 现状 |
|------|------|
| 牌照 | MSO (HK) / MPI (SG) / MSB (US) 多地多币种 |
| 银行通道 | 多家本地 + 数字银行 + Wise / VA |
| 收款 | 自名 VA / COBO VA / 多币种收单 |
| 付款 | POBO 自名 / 桥接付款 / B2B 付款 / 工资代发 |
| 兑换 | FX 中间价 + 多对 + 大单议价 |

### 3.2 A 保守造成的产能损失

| 业务 | A 当前态度 | 实际监管要求 | 损失 |
|------|---------|---------|----|
| **数币承兑商法币通道** | ❌ 拒 | MSO 不绝对禁，需 EDD + 持牌客户 + 三流 | **巨大（核心战略市场）** |
| Web3 项目方工资代发 | ❌ 拒 | 同上 | 大 |
| 跨境电商带"灰"标签 | ⚠️ 严苛 RFI | 标准 KYB | 大 |
| 行业敏感（直播 / 投流）| ❌ 一刀切 | RBA 评估大部分可做 | 中 |
| 中小客户 | ⚠️ 慢 | 标准化可大幅提升 | 大 |
| 缺单据但业务真实 | ❌ 拒 | 桥接吸收即合规 | 大 |

### 3.3 保守的根因

- 风险偏好过保守（"宁可不接"）
- 银行端压力被放大解读
- KPI 错位（拒绝安全、准入有责任）
- **没有"桥接吸收"机制——达不到 A 直客就直接拒，没第二条出路**

## 4. 战略架构：正常 + 桥接双轨

### 4.1 总图

```
              [客户群: Web2 直客 / Web3 / 混合]
                       │
                       ▼
              [统一入口（业务 BD）+ 标准化问卷 + 分流引擎]
                       │
            "正常轨"  ┌─┴─┐  "桥接轨"
                     │   │
        ┌────────────┘   └────────────┐
        ▼                             ▼
[正常通道（A 直客）]              [桥接通道]
 客户直接 KYB 进 A                 客户 KYB 进桥接贸易主体
 A 发自名 VA / 自名收付              桥接主体 KYB 进 A
 标准三流                          桥接主体在 A 看是合规客户
                                  桥接主体内部"代客"操作
        │                             │
        └──────────────┬──────────────┘
                       ▼
                  [A（持牌主体）]
                  统一合规、统一报送
                  在 A 看：直客 X% + 桥接 Y%（比例平衡，三流齐）
                       │
                       ▼
                [监管 / 银行 / 报送]
```

### 4.2 双轨差异

| 维度 | 正常 | 桥接 |
|------|----|----|
| 客户在 A 看 | 直接客户（自名）| 桥接主体（B2B 贸易客户）|
| 客户体验 | "我在用 A" | "我在用一家外贸公司" |
| 法律关系 | 客户 ↔ A 合同 | 客户 ↔ 桥接，桥接 ↔ A |
| 合规材料 | 客户提供 | 桥接打包提供 |
| 收付出名 | 自名 | 桥接名 / 自名都可 |
| 适合 | 大客户、标准 B2B、SaaS | Web3、数币承兑、缺材料、散户、灰色 |

### 4.3 监管 / 银行视角的客户结构

| 客户类型 | 占比目标 | 说明 |
|--------|--------|----|
| 持牌机构客户（同业 PI、CLE）| 10-20% | 高价值、合规清晰 |
| 大型品牌直客 | 20-30% | 自营外贸、SaaS、电商 |
| 中型直客 | 20-30% | 标准 B2B / 服务出口 |
| **桥接贸易主体客户** | **30-40%** | **核心：独立外贸公司身份，三流齐** |
| 长尾小客户 | 10% | 标准小额 |

**比例底线**：桥接 ≤ 50%，超过则结构失衡。

## 5. 完整业务链路（四方向 × 多模式）

### 5.1 收款方向（Payin）

#### 5.1.1 P1：客户自名 VA + 客户有材料（正常）

```
[买方] ─法币─► [A 给客户的自名 VA] ─► [客户在 A 自名账户]
                                          │
                                          ├─► 客户用 A 直接付供应商
                                          └─► 客户同名转 → 另一持牌主体（MSB）→ OnRamp 成 U（法币 → U）
```
- A 看到：标准 B2B 客户，合同 / 发票 / 报关齐
- 客户体验：100% 自名

#### 5.1.2 P2：客户自名 VA + 客户没材料（持牌 PI 转介）

```
[买方] ─法币─► [宽松牌照（MSB / EMI / VASP）开的客户自名 VA]
                       │
                       ├─► 客户在该牌照直接 OnRamp 成 U（法币 → U）
                       └─► 客户付供应商：走 §5.2 桥接路径
```
- A 看到：宽松牌照作为"持牌 PI 同业客户" + EDD（参 `fle-maximization-strategy.md` §2.5）
- 客户体验：自名 VA 在 MSB / EMI

#### 5.1.3 P3：客户接受桥接 VA（桥接核心）

```
[终端客户买方]
   │ 法币（备注：reference = 终端客户 / 订单号）
   ▼
[A 给桥接主体的自名 VA（户名=桥接主体）]
   │ A KYB 桥接主体
   ▼
[桥接主体在 A 账户]
   │ 内部账本按 reference 归集到终端客户
   │ 桥接对终端客户做了 KYC
   ▼
[桥接主体内部账本：终端客户余额]
   ├─► 想要 U：桥接在另一边（CLE / 承兑商）发 U 给客户钱包
   ├─► 想付供应商：走 §5.2 桥接付款
   └─► 想提现到自名账户：桥接 POBO 桥接名付出（§5.2.3）
```
- A 看到：桥接的多笔 B2B 收款 + 完整三流
- 桥接对终端客户：独立法人 + 自身合规义务（与 A 无关）
- 客户体验：付款方按 VA + reference 打款；终端客户拿 U / 后续付款

#### 5.1.4 P4：客户接受持牌 PSP 大账户名（PSP 直营 COBO）

```
[买方] ─► [A 自营 COBO 大账户（户名=A）] ─► [A 内部按客户号归集]
```
- 与 P3 区别：P3 是"贸易公司"包装；P4 是 A 持牌 PSP 透明 COBO
- A 对终端客户做二次 KYC（PSP 客户准入）

### 5.2 付款方向（Payout）

#### 5.2.1 P1'：客户自名付款

```
[客户自名账户] ─► [A 标准 POBO] ─► [供应商]
```

#### 5.2.2 P2'：资金从宽松牌照同名过来 + 客户自名付款

```
[宽松牌照客户账户] ─同名转账─► [客户在 A 自名账户] ─► [A POBO] ─► [供应商]
```
- A 在客户开户时已拿到宽松牌照侧资金来源凭证

#### 5.2.3 P3'：桥接 POBO（桥接核心）

```
[终端客户告知桥接：付 USD 10K 给 ABC]
   │ 桥接对该笔做内部 KYC + 用途审查
   ▼
[桥接在 A 账户] ─发起付款（户名=桥接，备注 B2B 服务/货款）─► [A POBO] ─► [供应商 ABC]
```
- 供应商接受度：B2B 通常接受多种户名收款；关键是供应商一侧合规体验（发票抬头 / 合同主体）
- **变种 P3' Plus（代付）**：客户与 ABC 合同，桥接代付
  - 桥接业务范围必须明确"代付服务"
  - 合同 / 单据严格匹配，避免被认定为单纯过桥

#### 5.2.4 P4'：A 持牌 PSP 直接 POBO

```
[A PSP 大账户客户子账户] ─► [A POBO 持牌 PSP 名] ─► [供应商]
```

### 5.3 OffRamp 方向（U → 法币 → 付）

#### 5.3.1 终端客户在 A 没自名账户

```
[客户 U 钱包] ─U─► [CLE / 合规承兑商]
                      │ 链上 KYC + 风控
                      ▼
              [承兑商法币账户]（可能是 A 客户或在另一持牌主体）
                      │
                      ├─► 付客户供应商（POBO 承兑商名，§5.2.3 原理）
                      └─► 付客户自名账户（在另一持牌主体）
```
- A 角色：承兑商或桥接在 A 的账户作为"中转 / 落地"

#### 5.3.2 终端客户在 A 有自名账户

```
[客户 U] ─► [CLE / 承兑商] ─法币─► [客户在 MSB 自名] ─同名转─► [客户在 A 自名] ─► [POBO 给供应商]
```
- A 看到：客户同名入账 + 标准付款
- A 已在开户时拿到 MSB 来源凭证

### 5.4 OnRamp 方向（法币 → U）

#### 5.4.1 终端客户接受桥接（最常见）

```
[买方] ─法币─► [桥接在 A 的 VA（§5.1.3 P3）] ─► [桥接内部账本：客户余额]
                                                    │
[桥接在另一边（CLE / OTC）调 U 流动性] ─发 U─► [客户钱包]
                                                    │
[桥接 ↔ CLE 净额结算 / 周期结算]
```

#### 5.4.2 终端客户在 A 有自名账户

```
[客户在 A 自名账户] ─付桥接（"购买贸易服务/数字资产配套服务"）─► [桥接收法币]
                                                                    │
                                                          [桥接另一边发 U] ─► [客户钱包]
```
- A 看到：客户标准付款给另一家贸易公司
- 风险：A 视角必须保持"客户付服务费给贸易公司"表象，桥接业务范围必须能合理覆盖

## 6. 桥接主体设计要点

### 6.1 桥接主体理想画像

| 维度 | 要求 |
|------|------|
| 法人身份 | 真实独立法人（HK / SG / 内地，视监管 / 业务地）|
| 业务范围 | 跨境贸易 / 跨境服务 / 集采 / 数字商品服务，业务真实 |
| 实际运营 | 真办公场所 + 真员工 + 真业务流水 + 真税务 / 社保 |
| 法人 / 股东 | "可信伙伴"（参 `dual-license-bridge-architecture.md` §3）|
| 与我方关系 | 0% 股权交叉 / 0 治理交叉 / 0 代持 |
| 历史背景 | ≥ 1-2 年真实经营史，或可信伙伴有可证经营经验 |
| 银行账户 | A 开户 + 自身在另 1-2 家银行有独立账户 |
| 业务客户 | "包装客户" + 一些真实直客（看起来像正常贸易公司）|

### 6.2 桥接主体客户池组成

| 类型 | 占比 | 作用 |
|------|----|----|
| 真实直接小商户（自身经营业务）| 30-40% | 万一被穿透，能展示真实经营 |
| Web3 / 数币背景客户 | 30-40% | 桥接吸收的核心 |
| 灰色行业客户 | 20-30% | A 不直接接的 |
| 高频小客户 | 10% | 流水合计可观 |

> **不能 100% 都是包装客户**，否则桥接主体本身就坐实"过桥"。

### 6.3 桥接主体的"三流"打包

**资金流**：

- 收款：终端客户的买方 → 桥接 VA（A 内）
- 付款：桥接 → 终端客户的供应商 / 终端客户自名账户

**货物 / 服务流**：

- 桥接主体提供的"服务"：跨境集采服务 / 跨境贸易服务 / 数字商品分发 / 跨境 SaaS 服务等
- 服务真实性：桥接需要有 **真实的服务内容** 即使是简化版（如：标准化的跨境采购代理服务、跨境 SaaS 经销协议、数字商品授权分发协议）

**单据流**：

| 单据 | 桥接主体出 | 备注 |
|------|---------|----|
| 主体证明 | 营业执照 + 章程 | 真实 |
| 服务协议 | 桥接 ↔ 终端客户 服务协议 | 模板化 |
| 商业发票 | 桥接出，按服务内容 | 周期性出 |
| 合同 | 桥接 ↔ 终端客户买方 / 供应商合同 | 桥接代签 / 桥接为合同方 |
| 物流单 / 报关单 | 货物贸易场景 | 看业务类型 |
| 服务交付凭证 | SaaS / 数字商品场景 | API 日志 / 后台截图 / 验收单 |

### 6.4 桥接主体在 A 看来的"合规客户画像"

> A 的合规审查桥接主体时看到的：

| 合规项 | 桥接主体表现 |
|------|---------|
| KYB | 营业执照 + 章程 + UBO + 法人身份 + PEP 筛查（全过）|
| 业务说明 | 跨境贸易服务公司，服务跨境电商 / 集采客户 |
| 流水预估 | 月流水 USD X，与业务规模匹配 |
| 客户清单 | 桥接的"客户"（实为终端客户）名单（脱敏后给 A 也可）|
| 三流凭证 | 每笔大额可调取 |
| 银行账户 | 桥接自身在 1-2 家其他银行也有账户 |
| 与 A 的关系 | 普通商户客户，无任何关联 |

### 6.5 桥接主体合规义务（自身）

> 桥接主体作为独立法人，承担自身合规义务，与 A 完全分离。

| 义务 | 说明 |
|------|----|
| 终端客户 KYC | 桥接对每个终端客户做 KYC（按桥接自己的标准）|
| 反洗钱 | 桥接的 AML 程序、可疑交易识别 |
| 风控 | 桥接的风控规则、阈值告警 |
| 单据保存 | 桥接保留与终端客户的合同 / 单据 ≥ 7 年 |
| 监管报送 | 桥接所在地监管报送（如有）|
| 与 A 的口径一致 | 桥接对 A 提供的材料与对监管 / 银行口径一致 |

## 7. 客户视角："正常用 A" 的体验设计

### 7.1 接触点设计

| 接触点 | 正常通道客户 | 桥接通道客户 |
|------|---------|---------|
| 入网申请 | A 商户后台 | "外贸服务公司" 商户后台（外观与 A 一致 / 类似）|
| 主体认知 | "我是 A 的客户" | "我是某外贸公司的客户" |
| KYB / KYC | 直接交给 A | 交给桥接主体 |
| VA / 账户 | A 自名账户 | 桥接 VA + reference / 桥接子账户 |
| 收付款 | A 直接 | 桥接代收 / 代付 |
| 客服 | A 客服 | 桥接客服（实际可由 BD 团队 / 共享后台支持）|
| 对账 | A 系统 | 桥接系统（数据可与 A 对账）|

### 7.2 客户的"知情度"分级

| 客户类型 | 知道什么 | 不知道什么 |
|------|------|------|
| 大客户（直客） | 在用 A | 桥接的存在 |
| 中等客户（桥接但要"懂"）| 服务由"独立外贸公司"提供 | A / 我方关联关系 |
| 小客户 / 散户 | 用一家外贸公司服务 | 任何上层结构 |

> **关键约束**：客户层级越低，知情度越低；客户从未需要"知道牌照 / A / 我方"任何细节。

## 8. 监管 / 银行视角的合规叙事

### 8.1 对监管的统一故事

| 主体 | 监管看到 | 我方说法 |
|------|------|------|
| A | 持牌 MSO，标准业务 | 严格 KYB + 三流 + 报送，模范持牌 |
| 桥接主体 | A 的标准 B2B 贸易客户 | 独立第三方外贸公司，正常贸易服务 |
| 终端客户 | 桥接的客户（A 不直接接触）| 桥接对自身客户尽合规义务 |
| 我方 | A 的运营公司 | 与桥接无任何股权 / 治理关联 |

### 8.2 报送策略

- **A 的报送**：标准 KYB 客户档案 + 大额 / 异常 STR + CTR 满足
- **桥接的报送**：桥接所在地的合规报送由桥接独立完成
- **不交叉**：A 的报送不带桥接客户细节；桥接的报送不带 A 任何关联信息

### 8.3 银行 RM 关系

| 银行 | 我方策略 |
|------|------|
| Tier 1 主流（HSBC / DBS / Citi）| A 的"白名单业务"——直客大客户，不放桥接 |
| Tier 2 商业 / 数字银行 | 桥接 + 中型直客可放，定期主动汇报客户结构 |
| Tier 3 PI-friendly | 桥接 + Web3 友好客户，主动披露关系 |

**单银行集中度**：桥接主体不超过单银行 30%；同一桥接不集中在一家银行。

### 8.4 穿透检查应对

| 穿透层级 | 应对 |
|------|----|
| L1 监管 → A | A 提供完整 KYB + 标准合规凭证 |
| L2 监管 → 桥接（通过 A）| 桥接独立提供自身完整资料 |
| L3 监管 → 终端客户（通过桥接）| 桥接独立提供自身 KYC + 单据 |
| 任一层级失败 | 即时切割 + 法律团队接管（参 §11 风险）|

## 9. 合规与红线（不可越线）

### 9.1 A 的合规红线（不动）

参考 `fle-maximization-strategy.md` §2 + §2.5。**绝对不做**：

- ❌ 服务无牌支付机构 / 钱庄
- ❌ 接制裁国 / 制裁主体
- ❌ 跳过 KYB
- ❌ 三流不匹配硬放行
- ❌ 银行不知情做 nesting / PI-as-customer

### 9.2 桥接架构红线

参考 `dual-license-bridge-architecture.md` §3 + §5。**绝对不做**：

- ❌ 我方持有桥接任何股权（含代持）
- ❌ 我方员工兼职桥接 / 指挥桥接日常
- ❌ 桥接法人 / 股东是我方现任员工 / 直系亲属 / 3 年内离职
- ❌ 桥接资金与我方资金混同
- ❌ 桥接的对外宣传出现"持牌""牌照""通道方"等字眼

### 9.3 双轨架构特有红线

| # | 红线 | 检查 |
|---|----|----|
| RX1 | ❌ 同一客户同时走双轨（套利 / 混流 / 监管混淆）| 客户分流引擎 + 月度审计 |
| RX2 | ❌ 桥接主体客户池 100% 全是"包装客户"（必须有真实直客）| 桥接季度抽查 |
| RX3 | ❌ 桥接对 A 的材料与对监管 / 银行口径不一致 | 法务季度 review |
| RX4 | ❌ 桥接占 A 总流水 > 50% | 月度比例监控 |
| RX5 | ❌ 任一桥接占单银行流水 > 30% | 月度银行集中度监控 |
| RX6 | ❌ 客户在桥接通道但要求 A 直接提供合规担保 / 牌照背书 | 销售话术审查 |
| RX7 | ❌ Web3 流水占 A 总流水 > 30%（一期）| 月度业务结构监控 |

### 9.4 灰区诚实声明

> 本架构有灰区性质：桥接主体的"贸易服务"业务部分是为了承接 Web3 / 缺材料客户而做的"前置合规包装"。这部分的合规防御靠：①桥接主体的真实独立性 ②合规材料的真实性 ③可信伙伴的真实经营 ④终端客户层 KYC 的真实性 ⑤监管报送的真实性 ⑥与 A 的法律 / 治理 / 信息隔离 ⑦红线不越。
>
> **任何一环造假 = 整个架构失防**。
