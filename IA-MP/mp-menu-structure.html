Dashboard（工作台）

商户总览 — 余额概览、近期订单、待办事项

📊 MP 菜单设计总览

EX 平台定位：Web2 + Web3 支付能力聚合平台
• IPL（香港 MSO）= 法币世界：法币账户、FX 换汇、VA 收款、跨境汇款
• BB（BlancBlock）= 数币世界：数币托管、承兑 On/Off-Ramp、链上收付、Crypto Checkout
• 商户端不展示 SP，只看到统一的能力和按币种合计的余额
菜单设计原则：
1. 按资金流向分组：Assets = 自己的钱 / Transfer In = 别人给我转账 / Checkout = 客户按订单付款 / Transfer Out = 我付给别人
2. Exchange = 纯体系内兑换（钱包→钱包 / 账户→账户），不涉及外部资金进出
3. 涉及外部的"一单到底"操作放在各账户页面（充值买U → Crypto Wallet / 卖U提现 → Crypto Wallet）
4. Payins / Payouts = 全量订单主表，Assets 账户页面的订单记录是筛选视图
5. 数据唯一真相源：VA 在 Collection Tools 管理，其他页面引用展示；订单在 Payins/Payouts 管理，账户页面筛选展示
SP透明 & 账户结构规则：
1. 底层账户结构：每个SP给商户开独立账户（IPL有USD账户，BB也有USD账户）
2. 商户端不展示SP：商户只看到按币种合计的总余额，SP明细只在租户端展示
3. 充值/提现/付款：商户不选SP，系统根据默认配置和优先级自动选择扣款/入账的SP账户
4. 付款余额不足时：系统自动从多个SP账户扣款（如IPL $5000 + BB $3000 = 总余额 $8000）
5. SP间互转：打通的SP之间（如IPL⇄BB）可互转，对商户透明
6. 订单记录：所有SP的记录混合展示，不区分SP来源
数据源与引用模型：
• Collection Tools → VA/加密地址的唯一管理入口，Fiat Accounts 充值时引用展示 VA，Crypto Wallet 充币时引用展示地址
• Payins → 全量入金订单主表，Fiat Accounts / Crypto Wallet 订单记录 = 按账户筛选的视图
• Payouts → 全量出金订单主表，同上
• Exchange → 兑换订单主表，账户页面订单记录中涉及本账户的兑换也展示
• 账户页面订单记录 = 该账户所有余额变动（入金+出金+兑换+结算，不区分同名/他人）

Assets / Fiat Accounts

Fiat Accounts（法币账户）

管理法币账户余额、充值提现、查看所有余额变动 我的钱

职责定义

本菜单管"我的法币账户"：查看余额、同名充值、同名提现、查看该账户所有余额变动记录。
操作入口：充值/提现操作在此发起
订单记录：该账户所有余额变动（不只是充值提现，包括 VA 收款入账、付款扣款、兑换等所有导致法币余额变化的订单）
不管：收款工具本身的管理（→ Collection Tools）、付款发起（→ Payouts）、兑换（→ Exchange）

页面功能

账户余额总览
按币种展示总余额（多个SP账户合计），如 USD $8,000 / EUR €3,000。商户不感知SP，底层可能分属多个SP账户
同名充值（Deposit）
从自己的银行账户转入法币到平台。点击充值后引用展示 Collection Tools 中的 VA 信息（银行名/账号/SWIFT/IBAN），商户照此去银行转账。VA 的管理仍在 Collection Tools，此处仅引用展示
同名提现（Withdraw）
从平台提现到自己的银行账户。商户只选币种和金额，系统自动选择扣款的SP账户
订单记录（该账户所有余额变动）
展示所有导致法币余额变动的订单，不区分同名/他人，包括：
• 同名充值 自己银行 → 平台
• 同名提现 平台 → 自己银行
• VA 收款入账 他人汇款 → VA → 入账
• 法币付款扣款 平台 → 他人银行
• FX 换汇 其他币种 → 本币种 / 本币种 → 其他币种
• Buy Crypto 扣款 法币 → 数币（Exchange）
• Sell Crypto 入账 数币 → 法币（Exchange）
• Checkout 结算入账 收单订单结算
数据来源：Payins + Payouts + Exchange 中涉及本法币账户的记录（筛选视图，非独立数据）

边界说明

账户结构（商户不可见）：
• 底层每个SP独立开户：IPL-USD $5,000 + BB-USD $3,000 = 商户看到 USD $8,000
• 付款时系统根据优先级自动从多个SP扣款，商户只看到总余额
• SP明细仅在租户端（TP）展示
数据引用关系：
• 充值时展示的 VA 信息 → 引用自 Collection Tools（唯一管理入口）
• 订单记录 → 筛选自 Payins / Payouts / Exchange（全量订单主表）
• 本页面不产生独立数据，所有数据均来自其他模块
Assets / Crypto Wallet

Crypto Wallet（加密钱包）

管理加密资产、充币、提币/提现到银行、充值买U、查看所有余额变动 我的钱

职责定义

本菜单管"我的数币钱包"：查看余额、同名充币、提币/提现到银行、充值买U、查看该钱包所有余额变动记录。
操作入口：充币 / 提币（或提现到银行）/ 充值买U 操作在此发起
订单记录：该钱包所有余额变动（不只是充币提币，包括链上收款入账、Exchange 兑换等）
不管：收款地址本身的管理（→ Collection Tools）、链上付币给他人（→ Payouts）、体系内兑换（→ Exchange）

页面功能

钱包余额总览
按币种+链展示总余额（多个SP托管地址合计），如 USDT-TRC20 10,000 / BTC 0.5。商户不感知SP
同名充币（Crypto Deposit）
从自己的外部钱包转入加密资产到平台托管地址。点击充币后引用展示 Collection Tools 中的充币地址（地址+二维码+链+币种），地址的管理仍在 Collection Tools，此处仅引用展示
提现（Withdraw）— 统一入口，选择提币或提现到银行
商户点击"提现/Withdraw"后，选择目标：
• 提币到外部钱包：同名提币，链上转账到自己的外部钱包地址
• 提现到银行账户：卖U提现（一单到底），数币→承兑为法币→提现到自己银行账户。底层编排 = BB 承兑 + 跨SP转账 + IPL 提现，对商户透明
提现到银行 = P2
充值买U（Deposit & Buy Crypto）— 一单到底
从外部银行充值法币，系统自动承兑为 USDT，入账到数币钱包。一步操作，商户不需分步执行
底层编排：外部银行 → 充值法币到 IPL → BB 承兑为 USDT → 入 BB 数币钱包，对商户透明
P2
订单记录（该钱包所有余额变动）
展示所有导致数币余额变动的订单，不区分同名/他人，包括：
• 同名充币 自己外部钱包 → 平台
• 同名提币 平台 → 自己外部钱包
• 链上收款入账 他人链上转账 → 收款地址 → 入账
• 链上付币扣款 平台 → 他人链上地址
• Buy Crypto 入账 法币 → 数币（Exchange）
• Sell Crypto 扣款 数币 → 法币（Exchange）
• 充值买U 入账 外部银行→法币→USDT→入钱包（编排）
• 卖U提现 扣款 USDT→法币→提现到银行（编排）
含链上 TxHash（如适用）
数据来源：Payins + Payouts + Exchange 中涉及本钱包的记录（筛选视图，非独立数据）

边界说明

当前阶段（仅 IPL + BB）：
• 数币余额 = 1个余额（仅BB提供加密托管能力），商户看到按币种+链的余额
• 后续增加其他有加密托管能力的SP时再评估合并展示逻辑
数据引用关系：
• 充币时展示的地址 → 引用自 Collection Tools（唯一管理入口）
• 订单记录 → 筛选自 Payins / Payouts / Exchange（全量订单主表）
• 本页面不产生独立数据，所有数据均来自其他模块
Assets / Exchange

Exchange（兑换 / Convert）

统一兑换入口：选 source → target 币种，系统自动判断方向（FX / 买币 / 卖币），本质都是 Conversion

职责定义

Exchange = 纯体系内 Convert：只做钱包→钱包 / 账户→账户的资产形态转换，钱不进不出体系。

统一操作模型：用户只需选择 From（源币种）→ To（目标币种），系统自动识别兑换类型：
• 法币 → 法币 = FX 换汇（底层：IPL Spot FX）
• 法币 → 数币 = Buy Crypto（底层：BB On-Ramp 承兑）
• 数币 → 法币 = Sell Crypto（底层：BB Off-Ramp 承兑）

不需要分 tab，因为三种方向本质都是同一个 Convert 操作，只是币种对不同。

不包含：涉及外部资金的操作：
• 充值买U（外部银行→USDT 一单到底）→ Crypto Wallet 页面发起
• 卖U提现（USDT→外部银行 一单到底）→ Crypto Wallet 提现入口发起
• 收→换→付编排 → Remittance Orders

页面功能

Convert（兑换）
选择 From 币种 → To 币种 → 输入金额 → 查看实时汇率/费用 → 确认兑换
UI 形态：类似 Wise / Coinbase 的双币种选择器，无需用户关心底层是 FX 还是 On-Ramp / Off-Ramp
支持的方向：
• USD ↔ EUR / GBP / HKD ... （法币↔法币，底层 IPL FX）
• USD / EUR / HKD → USDT / USDC ... （法币→数币，底层 BB 承兑）
• USDT / USDC → USD / EUR / HKD ... （数币→法币，底层 BB 承兑）
跨SP自动完成：法币账户在 IPL，数币钱包在 BB，SP间互转对商户透明
兑换订单
所有兑换订单列表，含源币种/目标币种、汇率、手续费、到账金额/数量。支持按方向（FX/买币/卖币）、币种对、时间筛选
兑换方向自动识别标签（用于订单展示）：
• FX 法币 → 法币（如 USD → EUR）
• Buy Crypto 法币 → 数币（如 USD → USDT）
• Sell Crypto 数币 → 法币（如 USDT → USD）
涉及外部资金？不在这里：
• 充值买U（一单到底）→ Crypto Wallet → 充值买U P2
• 卖U提现到银行（一单到底）→ Crypto Wallet → 提现 → 提现到银行 P2

原子能力模型

Exchange 只用 Conversion 一个原子能力：
• FX（USD→EUR）= IPL Conversion
• Buy Crypto（USD→USDT）= BB Conversion（On-Ramp 承兑）
• Sell Crypto（USDT→USD）= BB Conversion（Off-Ramp 承兑）

涉及外部资金的编排不在 Exchange（需要组合多个原子能力）：
• 充值买U = Deposit + Conversion → Crypto Wallet 发起
• 卖U提现 = Conversion + Withdraw → Crypto Wallet 发起
• 收→换→付 = Deposit + Conversion + Transfer → Remittance Orders

边界说明

Exchange vs 其他：
• Exchange = 纯体系内 Convert（单一 Conversion 原子），钱包→钱包 / 账户→账户
• Crypto Wallet = 涉及外部资金的一单到底编排入口（充值买U / 卖U提现）
• Remittance Orders = 收→换→付一单到底，钱最终要付给第三方
• 同名充值/提现 = 纯 Deposit / Withdraw，不涉及兑换（在 Fiat Accounts / Crypto Wallet 中操作）
• SP间互转（如IPL⇄BB）在 Convert 内部自动完成，对商户透明
Transfer In / Collection Tools

Collection Tools（收款工具）

VA 和加密收款地址的唯一管理入口 他人→我（转账）

职责定义

本菜单是"收款工具的唯一管理入口"（数据真相源）：包括法币收款的 Global Account（VA）和加密收款的链上地址。两者都是商户长期持有的固定收款入口，供客户/买家反复转入。

数据引用关系（其他页面引用本菜单数据）：
• Fiat Accounts → 充值时引用展示本菜单中的 VA 信息（银行名/账号/SWIFT/IBAN）
• Crypto Wallet → 充币时引用展示本菜单中的充币地址（地址+二维码+链+币种）
• 其他页面仅"引用展示"，VA/地址的申请、启停、管理只在本菜单操作

不管：Checkout中系统自动分配给订单的临时VA/地址（对商户透明，不在此管理）

边界说明

收款工具 vs Payins：
• Collection Tools = 管理收款工具本身（VA账户、加密地址的申请/查看/启停）
• Payins = 查看所有入金订单（通过VA收到的法币、通过地址收到的加密、同名充值、充币等）
• 收款工具是"入口"，Payins是"订单"
Transfer In / Remitter

Remitter（汇款人维护）

管理向您汇款的汇款人信息，支持银行账户和链上地址 他人→我

职责定义

本菜单管"谁给我打钱"：维护汇款人（Remitter/Sender）信息，记录其汇款方式（银行账户或链上地址）。
用于入金时自动匹配汇款来源、加速入账确认、满足合规要求（KYC/AML需要知道资金来源方）。
与 Beneficiary 的对称关系：Beneficiary = 我付给谁（Transfer Out），Remitter = 谁付给我（Transfer In）

页面功能

汇款人列表
查看所有已登记的汇款人，支持按类型（个人/企业）、汇款方式（银行/链上）筛选
新增汇款人
登记汇款人基本信息：名称、类型（个人/企业）、国家/地区、证件信息
汇款方式 — 银行账户
为汇款人添加银行账户信息：银行名称、账户名、账号、SWIFT/IBAN、银行地址等
汇款方式 — 链上地址
为汇款人添加链上地址：钱包地址、链、币种。用于匹配链上入金来源
编辑/删除汇款人
修改汇款人信息、管理其汇款方式、删除/恢复
汇款人详情
查看汇款人完整信息、所有汇款方式、关联的入金记录历史

边界说明

汇款人 vs 收款人（对称设计）：
• Remitter（汇款人）在 Transfer In — 记录"谁给我打钱"，一个汇款人可有多种汇款方式（多个银行账户 + 多个链上地址）
• Beneficiary（收款人）在 Transfer Out — 记录"我付给谁"，一个收款人可有多种收款方式
• 入金时系统可自动匹配已登记的汇款人，加速入账确认
• 合规要求：KYC/AML需要明确资金来源方身份
Transfer In / Payins

Payins（全量入金订单）

全量入金订单主表，所有资金流入订单的唯一完整视图 所有入金

职责定义

本菜单是“全量入金订单主表”（数据真相源）：统一展示所有资金流入订单，包括：
1. 他人通过 VA 银行转账入金
2. 他人通过链上地址转入加密资产
3. 同名充值（自己银行→平台法币账户）
4. 同名充币（自己外部钱包→平台托管地址）
5. 充值买U（外部银行→法币→USDT 一单到底编排）P2

数据引用关系：
• Fiat Accounts 订单记录 = 本表按“法币账户”筛选的视图
• Crypto Wallet 订单记录 = 本表按“数币钱包”筛选的视图
• 本表是入金订单的唯一完整来源

不管：Checkout 收单产生的收款（→ Checkout / Online Payment 等）

数据引用关系：
• Fiat Accounts 订单记录 = 本表按“法币账户”筛选的视图

• 本表是出金订单的唯一完整来源

不管：需要先承兑/换汇再付款的组合流程（→ Remittance = 收→换→付一单到底）；Checkout退款产生的出金（→ Checkout）
与 Payins 的对称关系：
• Payins = 全量入金（VA收款 + 链上收款 + 同名充值 + 同名充币 + 充值买U）
• Payouts = 全量出金（法币付款 + 链上付币 + 同名提现 + 同名提币 + 卖U提现）
• 两者都是“全量订单主表”，Assets 账户页面的订单记录是筛选视图

Transfer Out / Remittance Orders

Remittance Orders（汇款订单）

收→换→付一单到底，系统自动编排全链路 收→换→付

职责定义

本菜单管"收→换→付一单到底的编排订单"：商户发起一笔汇款，系统自动编排全链路：
1. 收款（可能是外部法币入金 或 体系内余额）
2. 兑换/承兑（法币→法币 FX / 法币→数币 / 数币→法币）
3. 付款（法币付给收款人银行 / 链上付给收款人地址）

典型场景：
• 商户收到客户的 USDT → BB 承兑为法币 → IPL 跨境汇款到供应商银行
• 商户法币账户 USD → IPL FX 换为 EUR → IPL 付款到收款人欧洲银行

底层编排对商户透明：商户只需指定"从哪出钱→付给谁→多少钱"，系统自动决定最优路径（选择SP、兑换策略、付款方式）

与 Payouts 的区别：
• Payouts = 单步出金（余额直接付出去）
• Remittance Orders = 多步编排（收→换→付一单到底）

页面功能

创建汇款订单
选择资金来源（余额/外部入金）→ 选择收款人 → 输入金额/币种 → 系统自动计算汇率和费用 → 确认提交。系统自动编排全链路
汇款订单列表
所有汇款订单，支持按状态（处理中/已完成/已失败）、金额/时间/收款人筛选。展示全链路进度（收款→兑换→付款 各步骤状态）
订单详情
全链路详情：资金来源、兑换汇率、中间步骤状态、最终付款信息、收款人信息、手续费明细
编排模型（对商户透明）：
• 原子能力组合 = Deposit（可选）+ Conversion（可选）+ Transfer
• 跨SP协作：如 BB 承兑 + IPL 跨境汇款，系统自动完成SP间资金调拨
• 子步骤会在各自模块产生对应记录（Payins/Exchange/Payouts），通过关联订单号串联
