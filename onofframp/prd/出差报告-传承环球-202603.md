## 一、客户背景

### 1.1 公司概况

**传承环球**是一家总部位于福建的综合性贸易集团，体系内拥有上市公司，主营耳机产品，以日本为主战场，在亚马逊（Amazon Japan）、雅虎（Yahoo! Shopping）、乐天（Rakuten）等主流电商平台均有布局，具备成熟的跨境电商运营体系。

### 1.2 支付现状与诉求

| 维度               | 情况                                                                                                 |
| ------------------ | ---------------------------------------------------------------------------------------------------- |
| **自有平台** | 已搭建支付聚合平台，后端对接多家支付服务商（Kun、Tazapay、Dogpay、我司等），具备多渠道比较和切换能力 |
| **牌照资质** | 已在澳洲获得支付牌照，走自有澳洲渠道非常通畅                                                         |
| **业务意向** | 老板最开始有意向做交易所方向，但尚不清楚具体路径和落地方式                                           |
| **核心诉求** | 安全、合规、时效、丝滑度——按此优先级排列                                                           |

### 1.3 客户特征分析

- **地域背景**：基于福建，天然具备丰富的跨境贸易和产业资源，周边灰色/黑色产业链条丰富
- **但合规意识高**：上市公司合规要求严格，主营业务以合规业务为主
- **行业认知深**：对 Web3 理解的理解基本来自于对接渠道和我司销售沟通
- **客户画像**：符合我们定义的目标大客户——独立性强、接受点位报价、注重合规和服务质量、粘性高

## 二、会议核心议题

### 2.1 传承环球：客户选择与市场定位（已明确）

| 客户类型                         | 决策        | 依据                                                                                 |
| -------------------------------- | ----------- | ------------------------------------------------------------------------------------ |
| **散客**                   | ❌ 放弃     | 月交易额低于几十万，人员服务成本高、投入产出比低                                     |
| **比价客户**               | ❌ 放弃     | 洽谈周期长、价格竞争激烈（如2000万美金客户在不同报价中选择），千1～千2费率无利润空间 |
| **大客户（如省广类型）**   | ✅ 重点开发 | 独立性强、接受点位报价、注重安全合规和时效，开发周期长但粘性高                       |
| **海外华人客户（餐饮等）** | 🔄 长期关注 | 福清/长乐等地海外餐饮群体业务量大但分散，需解决税务规避和整合问题                    |
| **新兴市场客户**           | 🔄 逐步拓展 | 如西班牙游乐场客户，目前业务量不大但有发展潜力                                       |

**结论：专注服务大客户，不做散客，不比价格比服务品质。** 借鉴盈科外服模式——为大企业提供海外法律、结汇等全案打包服务，满足出海企业"安全、合法、合规"一站式需求。

### 2.2 产品与服务问题清单

#### ① 时效性问题（紧急 🔴）

- 客户最关注资金到账时效，**T+1 是底线要求**
- 客户从**打款开始计算时间**，而非从我方处理开始——认知错位
- 当天内解决可接受，超过第二天客户即认为服务慢
- 即便其他平台贵千1，客户也会因时效选择竞品

#### ② 汇率波动问题（关注 🟠）

- 小币种汇率波动高达**千4～千5**，与之前约定的万6左右差距巨大
- 印尼盾等小币种客户对汇率极为敏感
- 当前波动值达到千1～千2，远超可接受范围
- 无稳定汇率报价机制

#### ③ 白标后台功能缺失（严重 🔴）

- 无法给客户设置调价等**基本操作权限**
- 调手续费需一天且要多部门审查，流程繁琐
- 产研报告处理速度非常缓慢
- 需要的是让服务商在不超约定红线的情况下**可自主配置**

#### ④ 通知机制缺失（严重 🔴）

- 美国手机号不能注册等变更**无提前通知**
- 客户进件时才知道限制，造成信任度下降
- 数字货币客户有被骗经历，需更高透明度

#### ⑤ 反馈处理机制（问题 🔴）

- **反馈问题未处理是最大毛病**
- 微信群反馈无人回复，沟通效率低
- 需建立"问题不过夜"的处理机制
- 建议领导进群，提高问题处理优先级

### 2.3 竞争环境

| 竞争对手/合作方  | 动态                                                                          | 启示                                 |
| ---------------- | ----------------------------------------------------------------------------- | ------------------------------------ |
| 光子易           | 报出千1.5的低价抢市场                                                         | 避免参与价格战，利润低且耗时耗力     |
| tazapay          | 放接口力度大，接入费从1万降至5000美金，有全球多地V及本地收单，预计3-4月放接口 | 关注其覆盖范围，可评估为潜在合作 SP  |
| **Kun**    | 数据穿透问题严重——申请业务后数据瞬间外流到业务员                            | **数据安全是选择合作方的红线** |
| **Paypal** | 也在放接口，但产品评价一般                                                    | 品牌大≠产品好，关注实际能力         |

### 2.4 合规与监管

- **合规灵活处理**：不能一刀切，需根据各国法律灵活操作，纯做 B2B 业务量受限
- **中国监管趋紧**：预计今年会有"杀鸡儆猴"行动，涉及证监会、银行、外企等
- **外企风险**：外企在国内宣传过于激进存在风险，华人员工出事概率增大，牌照多已外流
- 需加强对中国监管政策的解读和研究，提前做好应对

### 2.5 人员与沟通

- 部分人员专业能力与岗位不匹配，影响沟通效率和业务推进
- 外聘人员权限大但未入职主体公司，由外包人力公司聘用，出事由公司担责，风险大
- 群聊信息回复不及时，需按规则沟通，领导进群提高响应优先级

## 三、关键洞察：Web3 vs Web2 体验落差

### 3.1 传承环球的核心判断

传承环球明确指出：

> **"你们最大的问题是用 Web2 的要求来做 Web3，但客户正是因为 Web2 不好用才来用 Web3。"**

展会上其他客户也持类似观点：Web3 业务（充值 OnRamp、OffRamp、同名提现等纯 Web3 流程）就应该比 Web2 **更快、更丝滑**——这是客户选择 Web3 的根本原因。

### 3.2 现实链路分析

```
客户期望:   Web3 充值/提现 → 秒级或分钟级到账，极致丝滑

实际链路:   Web3 交易 → 合规审核 → 渠道清算（银行端 = Web2）→ 到账
                          ↑                    ↑
                      合规瓶颈            渠道瓶颈（银行清算速度）
```

链上部分确实可以很快，但 OnRamp/OffRamp/同名提现最终落地到银行体系，**依然是 Web2 的清算速度和合规审核**。客户不理解也不关心后端链路，只看到前端是 Web3 就期望 Web3 的速度。

### 3.3 这是一个行业性的认知 Gap

**大部分客户对 Web3 支付的理解并不正确。**

这意味着：

- **拓客有教育成本**：需在售前阶段管理客户预期，解释"链上秒到 ≠ 法币秒到"
- **谁先缩短 Gap 谁赢**：不可能消除银行段延迟，但谁能将其压缩到最短、体验做到最丝滑，谁就有竞争壁垒
- **产品透明度是关键**：让客户实时看到资金在哪一步，管理等待焦虑，好过让客户在黑箱中等待

## 四、核心反思：我们是否准备好了展业？

### 4.1 当前能力版图

我们有两家公司（IPL 和 BB），通过 EX 平台聚合对外提供服务：

| 公司          | 定位                               | 能力                                               | 短板                                                                         |
| ------------- | ---------------------------------- | -------------------------------------------------- | ---------------------------------------------------------------------------- |
| **IPL** | 纯 Web2 法币收付（Payins/Payouts） | 法币收付成熟，持牌合规，渠道稳定                   | 不做 Web3 业务，需保护不被卷入灰色风险                                       |
| **BB**  | Web3 承兑（OnRamp/OffRamp）        | 承兑引擎、数币钱包、合规/风控                      | **渠道能力不足**——银行通道少、币种覆盖窄、汇率波动大、到账时效不稳定 |
| **EX**  | 底层平台（聚合多 SP 能力）         | 多 SP 架构已设计完成，商户端 SP 透明，支持智能路由 | 目前仅 IPL + BB 两个 SP，能力池有限                                          |

**产品能力 = 渠道能力 × 合规能力 × 产研能力。** 三条腿的问题拆解如下：

| 能力维度       | 具体问题                                              | 责任方  | 严重程度    |
| -------------- | ----------------------------------------------------- | ------- | ----------- |
| **渠道** | BB 银行通道少，小币种汇率波动千4～千5，到账时效不稳定 | 渠道/BB | 🔴 致命     |
| **渠道** | 部分地区/币种无备选通道，单点故障无冗余               | 渠道/BB | 🟠 严重     |
| **渠道** | IPL 能力成熟但需保护，不能暴露于 Web3 风险            | 战略层  | 🟡 约束条件 |
| **合规** | 合规审核一刀切，限制市场业务开展                      | 合规    | 🟠 严重     |
| 合规           | 不支持美国主体入网，不及时通知，客户入网后才通知      | 合规    | 🟠 严重     |
| **合规** | 合规审核前置时间长，影响客户进件效率                  | 合规    | 🟡 中等     |
| **产研** | 白标后台功能严重不足，基础操作需人工跨部门处理        | 产研    | 🟠 严重     |
| 运营           | 汇率变更未及时通知                                    | 运营    | 🟠 严重     |

### 4.2 结构性矛盾

```
                    客户选择 Web3 的原因
                    ───────────────────
                    "Web2 不好用，所以来用 Web3"
                    期望：更快、更丝滑、更透明
                            │
                            ▼
                ┌───────────────────────┐
                │   EX 平台能力现状      │
                │                       │
                │  IPL = Web2 ✅ 成熟    │ ← 需保护，不参与 Web3
                │  BB  = Web3 ❌ 渠道弱  │ ← 银行通道少、汇率波动大
                │  SP池 = 只有2个       │ ← 能力池太浅
                │                       │
                └───────────────────────┘
                            │
                            ▼
                  核心矛盾：客户要 Web3 速度
                  我们的 Web3 链路卡在 Web2 段
                  而 Web2 能力（IPL）又不能随意暴露
```

**一句话总结：客户奔着 Web3 来，但我们的 Web3 交付能力受限于 Web2 渠道短板；而我们成熟的 Web2 能力（IPL）又需要战略保护。这就是当前业务"磕磕绊绊"的根源。**

### 4.3 解题方向：两条腿走路

解决渠道瓶颈不能只押一条路。EX 本质上是一家科技公司，对接新 SP 的周期不一定比 BB 自身拓展渠道更短——技术对接、合规审核、联调测试都需要时间。因此必须**两条腿并行**：

```
                        解题方向
                ┌──────────┴──────────┐
                │                     │
        腿1: BB 自身增强          腿2: EX 对接新 SP
        (渠道+合规)              (能力池扩充)
                │                     │
     · 拓展银行通道           · 引入新 SP 补充覆盖
     · 优化汇率报价机制       · 智能路由到最优通道
     · 合规分级处理           · 商户端完全无感
     · 提升到账时效           · 竞争性冗余备份
                │                     │
                └──────────┬──────────┘
                           │
                    渠道能力达到展业标准
```

#### 腿1：BB 自身渠道与合规能力增强

BB 是自家公司，增强自身能力是**最可控、最直接**的路径：

| 维度                 | 当前状态                       | 目标                                        |
| -------------------- | ------------------------------ | ------------------------------------------- |
| **银行通道**   | 通道少，部分地区/币种无覆盖    | 主力币种≥2条通道，关键地区有冗余备份       |
| **汇率稳定性** | 小币种波动千4～千5             | 建立稳定汇率报价机制，波动控制在千1以内     |
| **到账时效**   | 不稳定，无 SLA 承诺            | 定义各链路标准处理时间，T+1 可承诺          |
| **合规效率**   | 一刀切，不支持美国主体等       | 按国家/业务类型分级处理，进件审核有明确 SLA |
| **合规通知**   | 规则变更不通知，客户进件才知道 | 变更提前通知，建立合规政策透明化机制        |

**优势**：可控性强、不依赖外部、能力沉淀在自己手里
**风险**：拓展周期可能较长，受限于 BB 自身资源和谈判能力

#### 腿2：EX 平台对接新 SP

EX 平台架构天然支持多 SP 接入（SP 套餐体系已设计完成、商户端 SP 完全透明）：

```
当前:  EX ← IPL (Web2, 受保护) + BB (Web3, 渠道弱)
            ↓
           能力池浅，展业磕磕绊绊

目标:  EX ← IPL (Web2, 受保护)
           + BB (Web3, 自身增强后)
           + 新SP-A (补银行通道/地区覆盖)
           + 新SP-B (补币种/汇率稳定性)
           + ...
            ↓
           能力池深，智能路由到最优通道
```

| 维度               | 要求                                                              |
| ------------------ | ----------------------------------------------------------------- |
| **补什么**   | BB 暂时覆盖不到的地区/币种/银行通道，以及竞争性冗余               |
| **怎么接**   | EX 已有 SP 套餐体系（基础版/完整版），按标准流程对接              |
| **商户端**   | 完全无感——商户不知道后面是哪个 SP，系统根据配置和优先级自动路由 |
| **保护 IPL** | IPL 继续做纯 Web2 法币收付，不混入 Web3 业务链路，风险隔离        |

**优势**：能力互补、竞争冗余、降低单一 SP 依赖风险
**风险**：EX 是科技公司，对接周期不一定短（技术联调+合规审核+测试验证）；外部 SP 的数据安全和合作稳定性存在不确定性

**选择新 SP 的标准：**

| # | 标准                 | 说明                                | 前车之鉴                            |
| - | -------------------- | ----------------------------------- | ----------------------------------- |
| 1 | **渠道互补**   | 覆盖 BB 薄弱的地区/币种/银行通道    | —                                  |
| 2 | **数据安全**   | 不能有数据穿透问题                  | Kun：申请业务后数据瞬间外流到业务员 |
| 3 | **资质合规**   | 持牌、良好市场口碑、无诈骗/跑路记录 | 需背景审查                          |
| 4 | **合作稳定性** | 有明确对接人和 SLA                  | 杭州 Uki：资料上交一年无回复        |
| 5 | **接入成本**   | API 对接成本可控，接入周期合理      | Tazapay：接入费5000美金             |

#### 两条腿的关系

| 维度     | 腿1：BB 自身增强            | 腿2：EX 对接新 SP         |
| -------- | --------------------------- | ------------------------- |
| 可控性   | ✅ 高，自家公司             | 🟡 中，依赖外部合作方     |
| 见效速度 | 🟡 取决于 BB 资源和谈判能力 | 🟡 取决于 SP 对接周期     |
| 能力沉淀 | ✅ 沉淀在自己手里           | 🟡 依赖外部，有被替换风险 |
| 覆盖广度 | 🟡 受限于单一公司           | ✅ 多 SP 互补，覆盖更广   |
| 风险分散 | ❌ 单点依赖                 | ✅ 竞争性冗余             |

**结论：两条腿缺一不可。** BB 增强自身能力是根基（能力在自己手里最踏实），EX 对接新 SP 是扩展（快速补充覆盖、建立冗余）。不能只等 EX 接新 SP，也不能只靠 BB 慢慢拓——**并行推进，谁先跑通谁先上线。**

### 4.4 更深层的问题：EX 作为 SaaS 的价值是什么？

上述两条腿解决的是"渠道能力不足"的战术问题。但此次出差暴露了一个更根本的战略问题——

#### 现状：EX ≈ 渠道对接层

当前客户对接 EX，实质是在对接渠道（IPL/BB）。商务和技术是同一波人，客户感知不到 EX 作为独立平台的存在。即便未来 EX 接入更多 SP：

```
客户对接模式（当前 vs 未来）

当前：
    传承环球 ──商务+技术──→ BB/IPL（= EX 团队）
    客户感知：EX = BB 的门面

未来（即便有更多 SP）：
    传承环球 ──商务──→ SP（直接谈费率、合规、商务条款）
              ──技术──→ EX（仅 API 对接）
    客户感知：EX = 技术中间件
```

#### 致命挑战：技术对接层的护城河正在消失

Vibe Coding / AI 辅助开发的发展速度远超预期。1-2年后：

| 能力              | 当前                       | 1-2年后（AI 辅助开发成熟）      |
| ----------------- | -------------------------- | ------------------------------- |
| API 对接          | 需要专业开发团队，数周     | AI 生成对接代码，数天甚至数小时 |
| 多 SP 聚合        | 有一定技术门槛             | 变成标准化工程，门槛大幅降低    |
| 前端白标系统      | 需要完整产研团队           | AI 可快速生成，定制成本极低     |
| 数据报表/对账     | 开发周期长                 | AI 生成 + 低代码平台可替代      |

**如果 EX 的价值仅停留在"技术对接层"，那它的护城河会在 AI 浪潮中被迅速抹平。** 任何一个有技术能力的租户（比如传承环球自己），都可以用 AI 快速搭建自己的聚合层。

#### EX 的价值必须超越代码

EX 要有存在意义，必须提供**租户自己做不了或不愿意做的事**——这些事不是代码能解决的：

```
                    EX 价值层级模型

    ┌─────────────────────────────────────┐
    │  第4层：生态网络效应（终极护城河）    │  ← 交易量越大，数据越多，
    │  智能路由 / 风控模型 / 费率议价权    │    路由越优，费率越低
    ├─────────────────────────────────────┤
    │  第3层：合规基础设施（高门槛）        │  ← 不能 vibe code
    │  跨司法管辖区 KYB/KYC/AML           │    需要牌照+法律+持续维护
    │  合规政策引擎 / 风控引擎            │
    ├─────────────────────────────────────┤
    │  第2层：清结算能力（金融级复杂度）    │  ← 不能 vibe code
    │  多 SP 清算对账 / 资金调拨 /        │    需要金融专业+审计合规
    │  差错处理 / 结算分账                │
    ├─────────────────────────────────────┤
    │  第1层：技术对接（正在被 AI 抹平）   │  ← 当前 EX 主要在这层
    │  API 聚合 / 白标前端 / 报表         │    1-2年后门槛极低
    └─────────────────────────────────────┘
```

| 价值层级         | 能否被 Vibe Code 替代 | EX 当前建设程度 | 战略重要性   |
| ---------------- | --------------------- | --------------- | ------------ |
| **技术对接** | ✅ 1-2年内可替代      | 🟡 在建         | 低（衰减中） |
| **清结算**   | ❌ 不能               | 🟡 在建         | 🔴 高        |
| **合规基础设施** | ❌ 不能           | ❌ 未建         | 🔴 极高      |
| **生态网络效应** | ❌ 不能           | ❌ 未建         | 🔴 终极      |

#### "数据网络效应"的冷启动困境

上面第4层"生态网络效应"看起来很美，但必须正视一个残酷的鸡生蛋问题：

```
        数据网络效应的死循环

    要有数据 → 需要交易量
         ↑              ↓
    需要更多 SP ← 但目前只有 IPL + BB
         ↑              ↓
    吸引 SP 需要商户量 ← 商户量上不来因为渠道弱
```

更尖锐的问题是：**客户自己技术对接其他 SP 并不难。** 传承环球已经同时接了 Kun、Tazapay、Dogpay 等多个平台，他们有自己的聚合系统。在 Vibe Coding 时代，一个有技术能力的租户，AI 辅助几天就能完成一个 SP 的 API 对接。

那么问题变成：**如果客户自己能对接 SP，为什么要通过 EX？EX 作为中间层的价值在哪里？**

#### 关键转折：客户到底在为什么付费？

上面的分析如果停留在"技术 vs 非技术"的框架里，会得出一个悲观结论：大客户不需要 EX。但此次深入讨论后，发现这个结论是**错的**——因为我们漏掉了最关键的一层。

**事实：传承环球走的是代理模式——用 BB 的牌照，BB 承担合规责任。**

这意味着传承环球付费买的不是技术对接，而是：

```
客户真正购买的东西

    ┌──────────────────────────────────┐
    │     合规保护伞 = 牌照 + 合规运营能力     │
    │                                          │
    │  BB 持牌 → BB 承担监管责任               │
    │  BB 做 KYB/KYC → 审核商户进件            │
    │  BB 做 AML → 监控可疑交易                │
    │  BB 应对监管检查 → 客户不用直面监管       │
    │                                          │
    │  客户买的是"安全感"和"责任转移"        │
    └──────────────────────────────────┘
```

**传承环球自己有澳洲牌照，甚至能买到更多牌照——但他们不知道怎么做合规运营。** 牌照是门票，合规 know-how 才是护城河。

#### 牌照 vs 合规 Know-How：区分真假护城河

| 维度               | 牌照                       | 合规运营能力（Know-How）                     |
| ------------------ | -------------------------- | -------------------------------------------- |
| 获取方式           | 可以买到                   | 需要长期积累，买不到                         |
| 被替代风险         | 🟡 中（有钱就能拿）       | 🟢 低（经验和体系不可复制）                  |
| 包含什么           | 一张许可证                 | KYB/KYC 审核体系、AML 监控规则、风控模型、   |
|                    |                            | 监管报告流程、可疑交易处理、应对检查经验     |
| 客户感知           | "你有资质"               | "你能帮我不出事"                           |
| 定价能力           | 🟡 中（有牌照的不少）     | 🟢 强（真正能做好的很少）                    |

**大客户（如传承环球）最怕的是什么？**
- ❌ 怕下游商户连累自己（涉灰、洗钱）
- ❌ 怕资金在中间环节被冻结、扣押
- ❌ 怕因合规问题被监管处罚、吊销牌照

**他们什么都怕——所以愿意付钱给能帮他们"不出事"的人。** 这就是 EX/BB 的合规保护伞价值。传承环球清楚知道自己在为合规保护付费，不是为技术付费。

#### 修正后的客户分层：合规保护伞视角

| 客户类型                               | 技术自建能力 | 对合规保护伞的需求 | EX 价值    | 付费模式                  |
| -------------------------------------- | ------------ | ------------------ | ---------- | ------------------------- |
| **有牌照+有技术+有渠道**（如传承环球） | 强           | 🔴 强（怕风险）    | 🟢 **强** | SaaS 订阅 + 交易费        |
| **有牌照+无技术**                      | 弱           | 🔴 强             | 🟢 极强    | SaaS 订阅 + 交易费        |
| **无牌照+有技术**                      | 强           | 🔴 刚需           | 🟢 极强    | 交易费（不愿付订阅费）    |
| **无牌照+无技术**（大量中小机构）      | 弱           | 🔴 刚需           | 🟢 极强    | 交易费（不愿付订阅费）    |

**之前的分析错在：只从"技术"维度评估 EX 价值，忽略了"合规保护伞"这个最核心的维度。** 修正后发现：**EX 对所有客户类型都有价值——因为所有客户都怕风险。** 大客户怕风险所以愿意付 SaaS 费；小客户也怕但不愿付订阅费，通过交易抽佣变现。

#### EX 的三层价值模型

**EX 不是技术对接公司，而是合规运营平台。** 其价值分三层，第1层是核心：

```
            EX 价值模型（合规为核心）

    ┌─────────────────────────────────────┐
    │  第1层：合规运营平台（核心护城河）    │ ← EX 的存在理由
    │                                     │
    │  把 BB/IPL 的合规能力产品化：        │
    │  · 统一进件审核流程                  │ ← 不管后面是哪个 SP
    │  · 合规规则引擎（可配置，不一刀切）  │ ← 按地区/业务类型分级
    │  · 风控决策中台（跨 SP 全局视角）    │ ← 比单 SP 更精准
    │  · 合规状态透明化                    │ ← 客户能看到审核进度
    │  · 监管报告自动化                    │ ← 多司法管辖区
    ├─────────────────────────────────────┤
    │  第2层：清结算平台（高价值辅助）      │ ← 金融级复杂度
    │                                     │
    │  · 多 SP 统一清算对账                │
    │  · 资金归集与智能调拨                │
    │  · 差错处理 / 结算分账               │
    ├─────────────────────────────────────┤
    │  第3层：商户运营平台（入口，非壁垒）  │ ← 护城河在衰减
    │                                     │
    │  · 白标门户 / API 聚合               │
    │  · 智能路由 / 报表分析               │
    └─────────────────────────────────────┘
```

| 价值层级             | 核心能力                           | 能否被 AI 替代 | EX 当前建设 | 客户付费意愿 |
| -------------------- | ---------------------------------- | -------------- | ----------- | ------------ |
| **合规运营平台** | 合规 know-how 产品化、风控中台     | ❌ 不能        | ❌ 未建      | 🟢 最高      |
| **清结算平台**   | 多SP清算、资金调拨、差错处理       | ❌ 不能        | 🟡 在建      | 🟢 高        |
| **商户运营平台** | 白标门户、API聚合、智能路由        | ✅ 1-2年可替代 | 🟡 在建      | 🟡 中（衰减）|

#### 商业模型：谁付什么钱？

| 客户类型     | 付费方式            | 买的是什么                                       |
| ------------ | ------------------- | ------------------------------------------------ |
| 大客户       | SaaS 订阅 + 交易费  | 合规保护伞（责任转移）+ 清结算 + 运营平台        |
| 小客户       | 仅交易费（抽佣）    | 合规保护伞（必须走持牌方）+ 基础运营能力         |

**大客户是利润来源（SaaS 订阅费 = 高毛利），小客户是规模来源（交易量 = 数据积累 + 抽佣收入）。** 两者缺一不可。

#### "数据网络效应"的冷启动：先有合规，后有数据

"交易数据网络效应"不能靠画饼，必须认清因果关系：

```
    合规能力做好 → 客户信任 → 交易量上来 → 数据沉淀 → 风控更精准 → 更多客户信任
         ↑                                                              ↓
         └──────────────────────────────────────────────────────────────┘
                                正向飞轮
```

**起点不是数据，而是合规。** 把合规做好，客户就会来；客户来了，交易量就有了；交易量有了，数据就沉淀了。

冷启动的现实路径：

| 阶段   | 数据来源                 | 说明                                               |
| ------ | ------------------------ | -------------------------------------------------- |
| 冷启动 | IPL + BB 的存量交易数据  | 唯一可控的起步数据源，IPL 有成熟的 Web2 交易量     |
| 第二步 | 中小 TP 的交易流水       | 无牌照中小机构必须走 EX，交易数据自然沉淀          |
| 第三步 | 新 SP 带来的增量         | 每多一个 SP，多一个数据维度                        |
| 长期   | 大客户清算数据           | 如果清算中心做得好，大客户对账数据也会沉淀到 EX    |

#### 终极问题的回答

> **EX 凭什么存在？**
>
> 不是凭代码——代码 AI 能写。
> 不是凭 SP 数量——客户自己能对接。
> 不是凭数据——数据是结果不是起点。
>
> **EX 凭的是合规运营能力——把 BB/IPL 的牌照和合规 know-how 产品化，让客户可以"买到"合规保护伞。** 牌照可以买到，但知道怎么做合规运营、怎么审核进件、怎么监控风险、怎么应对监管——这些 know-how 是长期积累的结果，不能 vibe code，不能 AI 生成，不能一夜建成。
>
> 客户（无论大小）做展业最怕的就是合规和风控出事。大客户有牌照但不知道怎么做合规，小客户连牌照都没有。**EX 的角色是：把合规这件最难的事变成一项可购买的服务。**
>
> 清结算是第二层价值，技术平台是第三层。**合规是 1，清结算和技术是后面的 0。没有合规这个 1，后面多少个 0 都是零。**
>
> 但必须正视：**合规恰恰是 EX/BB 当前最被客户诟病的短板**（一刀切、不通知、不支持美国主体等）。核心价值主张和当前交付能力之间的 gap，是 EX 最紧迫的战略课题。

## 六、行动计划

### 阶段一：止血（已在进行，但无法根治）

目前已在执行的运营层面措施：

| # | 事项                                                                                                  | 责任方    | 现状           |
| - | ----------------------------------------------------------------------------------------------------- | --------- | -------------- |
| 1 | **建立"问题不过夜"机制**：领导进客户群，指定第一响应人，设定响应 SLA                          | 全公司    | ✅ 已在做      |
| 2 | **汇率波动临时方案**：对高波动小币种提供固定汇率窗口或提前告知                                  | BB 渠道   | ✅ 已在做      |
| 3 | **产品变更提前通知**：规则/功能变更提前通知，建立变更公告机制                                    | 产研+合规 | ✅ 已在做      |

> **但必须正视：这些措施治标不治本。**
>
> 止血措施解决的是沟通效率和信息透明度问题，但客户最核心的痛点——**时效**——不是靠运营能力能解决的。根本原因是 Web2 的银行清算基础设施在支撑 Web3 的业务场景，无论沟通多及时、通知多透明，**银行到账就是要 T+1 甚至更久，汇率就是会波动**。
>
> 这就是为什么止血措施已经在做，但客户体感没有本质改善——**问题是结构性的，不是运营性的。** 真正的解法在阶段二：通过渠道能力建设（BB 增强 + EX 接新 SP）从根上缩短链路、增加通道冗余、稳定汇率。

### 7.2 现实矛盾：续费在即，但核心问题短期无解

| 问题                 | 能否短期解决 | 原因                                       |
| -------------------- | ------------ | ------------------------------------------ |
| 到账时效慢           | ❌ 不能       | Web2 银行清算速度是结构性瓶颈              |
| 汇率波动大           | ❌ 不能       | BB 渠道能力不足，通道少无法对冲            |
| 白标后台功能缺失     | ❌ 不能       | 产研开发周期至少1-2个月                    |
| 合规一刀切           | 🟡 部分可以   | 可以针对传承环球做专项合规方案             |
| 沟通响应慢           | ✅ 已改善     | 已建立响应机制                             |
| 变更通知不及时       | ✅ 已改善     | 已建立通知机制                             |

**续费谈判的核心难题：客户知道我们的问题，我们也知道自己的问题，承诺无法兑现只会加速信任崩塌。**

### 7.3 续费合作策略：坦诚 + 分阶段 + 给利益

**原则：不画饼、不承诺做不到的事。用坦诚换信任，用让利换时间，用阶段性成果换长期合作。**

#### ① 坦诚沟通当前能力边界

- **直接告知**哪些能做到、哪些暂时做不到、预计什么时候能做到
- 传承环球是行业老手，画饼只会适得其反；坦诚反而能建立信任
- 给出明确的能力提升时间表（与阶段二渠道建设计划对齐）

#### ② 续费方案：降低门槛，绑定长期

| 策略               | 具体方案                                                                                     |
| ------------------ | -------------------------------------------------------------------------------------------- |
| **费率让利** | 在当前费率基础上给予阶段性优惠，作为对体验不足的补偿；待渠道能力提升后恢复正常费率           |
| **灵活续费** | 缩短续费周期（如按季度），降低客户决策压力；用短期续费争取渠道建设时间                       |
| **专属服务** | 指定专属对接人，传承环球的问题优先级最高，承诺明确的响应 SLA                                 |
| **合规专项** | 针对传承环球的业务类型和主体结构，提供定制化合规方案，而非一刀切                             |

#### ③ 找到当前就能提供价值的合作点

传承环球不是只有一种业务，不是所有业务都卡在时效问题上：

| 业务场景             | 是否受时效瓶颈影响 | 合作策略                                                     |
| -------------------- | ------------------ | ------------------------------------------------------------ |
| **澳洲渠道业务** | 🟢 不受（自有牌照通畅） | 探索在澳洲渠道上的合作空间，发挥他们的渠道优势               |
| **合规业务为主的客户** | 🟡 部分受影响     | 合规客户对时效容忍度相对高，以合规业务为切入点维持业务量     |
| **大额低频交易** | 🟡 可接受         | 大额客户更关注安全合规而非秒级时效，匹配我方当前能力         |
| **小币种高频交易** | 🔴 严重受影响     | 暂时坦诚告知不建议走我方通道，避免体验差导致整体信任崩塌     |

#### ④ 把传承环球变成"共建伙伴"

传承环球对行业认知深、自身也搭了支付聚合平台，与其被动等客户挑毛病，不如主动拉他们参与：

- **产品共建**：邀请参与白标后台功能定义，让他们的需求直接进入产品路线图
- **渠道共享**：传承环球有澳洲牌照和自有渠道，探索是否有渠道合作空间（他们的渠道也可以成为 EX 的 SP）
- **信息互通**：定期同步我方渠道建设进展和产品迭代成果，让客户看到改善是真实在发生的

### 7.4 风险预案

| 风险场景               | 应对                                                           |
| ---------------------- | -------------------------------------------------------------- |
| 传承环球不续费         | 保持关系，承诺渠道能力达标后优先回流；客户不会只用一家         |
| 续费但业务量下降       | 正常，不强推体验差的场景；维持合规业务基本盘                   |
| 竞品抢夺               | 不打价格战；传承环球看重的是合规和安全，这是我们的长板         |
| 客户在圈内传播负面口碑 | 坦诚沟通 + 持续改善是最好的口碑管理；让客户看到改变在发生       |

## 八、结论

> **此次出差最大的收获，不是发现了渠道短板，而是迫使我们想清楚了一个根本问题：EX 到底卖什么？**
>
> 答案不是技术——代码 AI 能写，客户自己几天就能对接一个 SP。
> 答案不是渠道聚合——客户自己有聚合系统，传承环球已经同时接了 Kun、Tazapay、Dogpay。
> 答案不是数据——数据是结果不是起点，鸡生蛋问题短期无解。
>
> **答案是合规保护伞。** 传承环球走的是代理模式，用 BB 的牌照，BB 承担合规责任。客户买的不是 API 对接，而是"安全感"和"责任转移"。传承环球自己有澳洲牌照，甚至能买更多牌照——但他们不知道怎么做合规运营。**牌照是门票，合规 know-how 才是护城河。**
>
> 这也解释了商业模型：大客户（如传承环球）愿意付 SaaS 订阅费 + 交易费，因为他们买的是合规保护；小客户不愿付订阅费但接受交易抽佣，因为他们必须走持牌方。**大客户是利润来源，小客户是规模来源。**
>
> 所以 **EX 不是技术对接公司，而是合规运营平台**——把 BB/IPL 的牌照和合规 know-how 产品化，变成可配置、可交付、可规模化的服务。合规是 1，清结算和技术平台是后面的 0。没有合规这个 1，后面多少个 0 都是零。
>
> **但讽刺的是：合规恰恰是当前客户诟病最多的短板。** 一刀切、不通知、不支持美国主体——核心价值主张和当前交付能力之间的 gap，才是 EX 最紧迫的战略课题。
>
> 具体行动：
> 1. **合规能力产品化**（最高优先级）：统一进件审核、合规规则引擎（可配置不一刀切）、风控中台、合规状态透明化
> 2. **渠道两条腿并行**：BB 增强自身渠道 + EX 对接新 SP，谁先跑通谁先上线
> 3. **传承环球必须留住**：坦诚能力边界、费率让利换时间、聚焦合规业务场景、以续费谈判锚定在"合规保护伞"价值而非"技术对接"价值
>
> 每一个因为体验差流失的客户，不仅是客户损失，更是在圈子里形成负面口碑。**但比留住客户更重要的，是想清楚自己卖什么。这次我们想清楚了：EX 卖的是合规运营能力，不是代码。接下来要做的，是把这个能力真正建起来。**
