# EX Platform - 交易安全验证策略

> **版本**：v2.2
> **更新日期**：2026-04-02
> **关联文档**：MP-register.md / mp-personal-profile.html
> **文档定位**：定义 EX 平台所有操作的安全验证策略，覆盖资金操作、安全设置、敏感管理三类场景

---

## 文档概述

本文档定义 EX 平台中**所有需要二次验证的操作**的安全验证策略。

**核心设计原则：**

```
1. 任何资金操作必须经过至少一次身份验证，不存在"无验证直接执行"
2. 验证方式由两个维度决定：授权模式（Self / Designated）× 操作人安全设置状态
3. 安全设置变更 = 最高可用凭证验证（2FA > 当前凭证）
4. 所有回退路径最终指向 OTP，确保任何情况下都有可用验证方式

Phase 1 边界：
5. 登录 = 密码 + OTP（不引入 TOTP 登录验证）
6. 交易验证 = PIN > OTP（每次只验一种，不叠加；无 TOTP）
7. 2FA(TOTP) 一期不做 — 整个 Authenticator 能力推到二期

Phase 2 扩展（规划）：
8. 登录引入 TOTP（密码 + 2FA）
9. 交易验证 = PIN > TOTP > OTP（TOTP 作为中间层）
10. 风控触发时可叠加验证（如 PIN + TOTP）
11. 商户可自定义安全策略（阈值/白名单/加验规则）
```

## 1. 行业对标

### 1.1 主流平台验证策略对比

| 维度                        | Binance                    | Airwallex    | Stripe       | **EX（我们）**      |
| --------------------------- | -------------------------- | ------------ | ------------ | ------------------------- |
| **平台类型**          | C 端交易所                 | B2B 支付     | B2B 支付     | B2B 支付 + 承兑           |
| **登录**              | 密码 + 2FA                 | 密码 + 2FA   | 密码 + 2FA   | 密码 + 2FA（可选）        |
| **常规交易**          | 不额外验证（登录态）       | OTP          | API Key 签名 | **PIN > TOTP > OTP** |
| **高风险操作**        | 2FA + Email 双重验证       | OTP + 审批流 | Re-auth      | Phase 2：风控加验         |
| **提币/出金到新地址** | 2FA + Email + 24h 白名单锁 | OTP          | API 签名     | Phase 2：新收款人加验     |
| **安全设置变更**      | 2FA 验证                   | OTP 验证     | Re-auth      | 2FA > 当前凭证 + 新凭证   |
| **2FA 用于交易**      | ✅ 提币/出金必验 2FA       | ❌ 走审批流  | ❌ API 模式  | ✅ 有 TOTP 则交易验 TOTP  |
| **团队成员控制**      | 子账户权限                 | 角色 + 审批  | 角色权限     | **Designated 模式** |
| **交易密码**          | ❌ 无                      | ❌ 无        | ❌ 无        | ✅ Payment PIN            |

### 1.2 行业 2FA 使用范围对比

| 平台 | 登录 | 交易/提币 | 安全设置变更 | 2FA 类型 |
| ---- | ---- | --------- | ----------- | -------- |
| **Binance** | ✅ 2FA | ✅ 每笔提币都要 2FA | ✅ 2FA | Authenticator / SMS / Email（可叠加多种） |
| **Coinbase** | ✅ 2FA | ✅ 发币要 2FA | ✅ 2FA | Authenticator / SMS |
| **OKX** | ✅ 2FA | ✅ 提币要 2FA（可叠加） | ✅ 2FA | Authenticator / SMS / Email |
| **Kraken** | ✅ 2FA | ✅ 单独的 Trading 2FA | ✅ 2FA（Master Key） | Authenticator |
| **Stripe** | ✅ 2FA | ❌ API Key + Webhook 模式 | ✅ 2FA | Authenticator / SMS |
| **Airwallex** | ✅ 2FA | ❌ 走审批流 | ✅ 2FA | Authenticator / SMS |
| **EX（我们）** | ✅ 2FA | ✅ 有 TOTP 则验 TOTP | ✅ 2FA | Authenticator（Phase 1） |

> **行业结论：** Crypto 行业（Binance/Coinbase/OKX/Kraken）2FA 全场景使用（登录+交易+设置）；传统支付（Stripe/Airwallex）2FA 主要用于登录+设置。我们是 B2B + 承兑（crypto 相关），对齐 crypto 行业标准。

### 1.3 EX 的策略选择

```
我们是 B2B 平台 + 承兑（crypto 相关），对齐 crypto 行业 2FA 标准

取 Binance/Coinbase 的：
  · 2FA(TOTP) 全场景可用：登录 + 交易 + 安全设置变更
  · 安全设置变更需要高等级验证

取 Airwallex 的：
  · 团队成员控制机制（我们的 Designated 模式更强）
  · OTP 作为基础验证手段

我们独有的：
  · Payment PIN（类似支付宝交易密码）→ 国内用户熟悉，体验好
  · 交易验证优先级 PIN > TOTP > OTP（每次只验一种，不叠加）
  · Designated 模式 → 资金操作必须经过指定人确认，比审批流更轻量
  · Phase 2 风控加验 → 大额/新收款人等场景按需加验

与 Binance 的区别：
  · Binance 提币时可能要求 2FA + Email 双重验证（叠加）
  · 我们每次交易只验一种（PIN 或 TOTP 或 OTP），不叠加 → 体验更轻
  · 我们多了 Payment PIN 作为最快验证方式
```

## 2. 安全凭证体系

### 2.1 三类验证凭证

| 凭证                           | 类型                   | 用途                          | 层级        | 状态                                |
| ------------------------------ | ---------------------- | ----------------------------- | ----------- | ----------------------------------- |
| **Payment PIN**          | 知识因子（你知道的）   | 资金操作快速验证              | Identity 级 | 未设置 / 已设置                     |
| **OTP 验证码**           | 持有因子（你有的）     | 资金操作兜底 + 安全设置验证   | Identity 级 | 始终可用（注册时强制绑定手机/邮箱） |
| **2FA（Authenticator）** | 持有因子（你有的）     | **Phase 2：** 登录 + 交易 + 安全设置变更 | Identity 级 | Phase 1 不支持；Phase 2 未开启/已开启 |
| **Designated Phone**     | 持有因子（指定人有的） | Designated 模式下资金操作验证 | MID 级      | 未设置 / 已设置                     |

### 2.2 凭证使用范围

```
Phase 1（一期）：
                    登录        资金操作        安全设置变更      敏感管理操作
                    ────        ────────        ──────────        ──────────
Password            ✅ 第一因素   ❌               ❌                ❌
Payment PIN         ❌            ✅ Self 优先     ❌                ❌
OTP                 ✅ 登录验证   ✅ Self 兜底     ✅ 始终参与        ✅
Designated Phone    ❌            ✅ Designated    ❌                ❌

Phase 2（二期新增）：
2FA (TOTP)          ✅ 第二因素   ✅ Self 中间层   ✅ 如果已开启      ❌
```

**Phase 1 关键决策：交易验证 = PIN > OTP（每次只验一种，不叠加）**

```
Phase 1 设计逻辑：
  · 有 PIN → 验 PIN（最快，< 2 秒，知识因子）
  · 无 PIN → 验 OTP（兜底，始终可用）
  · 一期无 TOTP 能力

Phase 2 演进（规划）：
  · 引入 TOTP 后优先级变为 PIN > TOTP > OTP
  · 风控触发时可叠加验证（如 PIN + TOTP）
  · 商户可自定义验证策略

为什么一期不做 TOTP：
  · 一期聚焦核心流程，PIN + OTP 已满足安全基线
  · TOTP 需要 Authenticator App 绑定/管理/恢复等完整能力，开发量大
  · 二期配合风控系统上线，TOTP 作为风控加验手段价值更大
```

### 2.3 OTP 发送优先级

| 场景                          | 发送目标                       | 可否切换                  |
| ----------------------------- | ------------------------------ | ------------------------- |
| Self 模式 — 用户已设默认凭证 | 按默认凭证发送（手机号或邮箱） | ✅ 可切换到其他已验证凭证 |
| Self 模式 — 未设默认凭证     | 手机号优先；无手机号则发邮箱   | ✅ 可切换                 |
| Designated 模式               | 固定发到 Designated Phone      | ❌ 不可切换               |

> 用户至少有一个已验证凭证（注册时强制），OTP 始终有可用发送目标。

## 3. 两种授权模式

### 3.1 模式定义

| 模式                 | 含义             | 验证码发给谁                                         | 谁来配置                                |
| -------------------- | ---------------- | ---------------------------------------------------- | --------------------------------------- |
| **Self**       | 操作人自己验证   | 操作人自己的手机号/邮箱                              | Owner 为成员配置（Owner 自己默认 Self） |
| **Designated** | 发到指定安全凭证 | Phase 1：Designated Phone；Phase 2：手机/邮箱/验证器 | Owner 配置 + 为成员配置模式             |

### 3.2 模式对比

| 维度                     | Self                   | Designated                      |
| ------------------------ | ---------------------- | ------------------------------- |
| **验证发给谁**     | 操作人自己             | Designated 凭证持有人           |
| **PIN 是否可用**   | ✅ 已设置则优先用 PIN  | ❌ 不可用                       |
| **操作人体验**     | 自主完成，无需等待他人 | 需线下联系持有人获取验证码      |
| **控制强度**       | 操作人自主             | 指定人把关                      |
| **适用场景**       | 信任成员自主操作       | 需要财务总监/Owner 审核资金操作 |
| **Owner 是否知情** | 否                     | 取决于 DP 是否是 Owner 号码     |

### 3.3 Designated 凭证未设置时

| 场景                            | 系统行为                                                             |
| ------------------------------- | -------------------------------------------------------------------- |
| 模式为 Designated 但未设置凭证  | **阻断操作**，提示："请联系 Account Holder 设置安全验证方式" |
| Designated 凭证无法接收验证      | 操作人联系 Owner 更新配置后重试                                      |

> **Phase 1：** Designated 仅支持手机号（Designated Phone）。
> **Phase 2：** 扩展支持邮箱 OTP 和 Authenticator App，Owner 可选择指定哪种凭证类型。

## 4. 验证策略矩阵（核心）

### 4.1 Self 模式 — 按用户配置状态

#### Phase 1 设计决策

```
一期登录：密码 + OTP（无 TOTP/Authenticator）
一期交易验证：PIN > OTP（每次只验一种，无 TOTP）
```

同一用户既能密码登录也能 OTP 登录，攻击者会选最弱路径。**安全等级 = 该配置下的最弱可攻击路径**，与当次登录方式无关。

#### 登录方式说明（Phase 1）

| 登录方式                              | 验证因子              | 备注                                    |
| ------------------------------------- | --------------------- | --------------------------------------- |
| **密码登录**                    | 知识因子（密码）      | Phase 1 唯一的知识因子登录方式          |
| **验证码登录**（手机/邮箱 OTP） | 持有因子（手机/邮箱） | 便捷登录，无需密码                      |

#### Phase 1 安全等级矩阵

交易验证优先级：PIN > OTP（每次只验一种）

| #  | PIN | 交易验证 | PIN 锁定后 | 密码登录链路                | OTP 登录链路（最弱路径）    | 实际安全等级       |
| -- | --- | -------- | ---------- | --------------------------- | --------------------------- | ------------------ |
| C1 | ❌  | OTP      | —         | 密码 `知识` → OTP `持有` | OTP `持有` → OTP `持有` | ⚠️ ★☆☆ 单因子 |
| C2 | ✅  | PIN      | OTP        | 密码 `知识` → PIN `知识` | OTP `持有` → PIN `知识` | ★★☆ 跨因子      |

> **Phase 1 核心结论：PIN 是交易安全的分水岭。**
> - **C1（无 PIN）** = ⚠️ ★☆☆ — 攻击者走 OTP 登录 + OTP 交易 = 同源单因子
> - **C2（有 PIN）** = ★★☆ — 无论怎么登录，交易都要 PIN（知识因子），最弱路径也是跨因子

#### Phase 1 安全链路可视化

```
C1（无 PIN）→ 交易验 OTP：
  密码登录: [密码] ──────────── → [OTP] → 操作     知识+持有 ★★☆
  OTP 登录: [OTP] ────────────── → [OTP] → 操作     持有+持有(同源) ⚠️★☆☆ ← 最弱

C2（有 PIN）→ 交易验 PIN：
  密码登录: [密码] ──────────── → [PIN] → 操作     知识+知识 ★★☆
  OTP 登录: [OTP] ────────────── → [PIN] → 操作     持有+知识 ★★☆ ← 跨因子
```

#### Phase 1 系统建议策略

| 用户配置状态 | 实际安全等级       | 系统引导                                                    |
| ------------ | ------------------ | ----------------------------------------------------------- |
| 无 PIN（C1） | ⚠️ ★☆☆ 单因子 | 🔴 **强烈建议设置交易密码**（当前为单因子，风险最高）  |
| 有 PIN（C2） | ★★☆ 跨因子      | 🟢 安全设置完善 ✓                                          |

---

#### Phase 2 安全等级矩阵（规划 — 引入 TOTP 后）

> Phase 2 引入 2FA(TOTP) 后，交易验证优先级变为 PIN > TOTP > OTP，矩阵扩展为 C1-C4。

| #  | 2FA | PIN | 交易验证 | PIN 锁定后 | OTP 登录链路（最弱路径）         | 实际安全等级        |
| -- | --- | --- | -------- | ---------- | -------------------------------- | ------------------- |
| C1 | ❌  | ❌  | OTP      | —         | OTP `持有` → OTP `持有`     | ⚠️ ★☆☆ 单因子  |
| C2 | ❌  | ✅  | PIN      | OTP        | OTP `持有` → PIN `知识`     | ★★☆ 跨因子       |
| C3 | ✅  | ❌  | TOTP     | —         | OTP `持有` → TOTP `独立持有` | ★★☆ 异源跨持有   |
| C4 | ✅  | ✅  | PIN      | TOTP → OTP | OTP `持有` → PIN `知识`     | ★★☆ 跨因子       |

> Phase 2 额外能力：
> - 风控触发时可叠加验证（如高风险操作要求 PIN + TOTP = ★★★）
> - OTP 登录不要求 2FA（2FA 仅在密码登录时作为第二因素）
> - 商户可自定义验证策略

#### 2FA(TOTP) 的价值（Phase 2 规划）

```
Phase 2 引入 TOTP 后，凭证定位：
  · PIN   = 最快的交易验证（知识因子，< 2 秒）
  · TOTP  = 独立的交易验证（独立持有因子，不依赖运营商）
  · OTP   = 兜底方案（持有因子，始终可用但依赖运营商/邮箱）

TOTP 的价值：
  1. 登录保护 — 防密码泄露后被登录（防撞库/钓鱼）
  2. 交易保护 — 无 PIN 时，TOTP 作为独立持有因子验证交易（C3）
  3. PIN 锁定兜底 — 有 PIN 但锁定时，TOTP 优先于 OTP 作为回退（C4）
  4. 安全设置变更 — 作为最高身份验证手段
  5. 风控叠加 — 高风险操作时 PIN + TOTP 双重验证

关键区分：OTP vs TOTP
  · OTP（SMS/Email）= 依赖运营商/邮箱服务，SIM 劫持可攻破
  · TOTP（Authenticator App）= 本地生成，不经网络传输，SIM 劫持无效
```

### 4.2 Designated 模式

```
Designated 模式下，不看操作人的 PIN / 2FA 设置，统一走：

  Phase 1：
  成员发起操作 → OTP 发到 Designated Phone → 操作人获取验证码并输入 → 执行

  Phase 2（扩展）：
  Owner 可选择 Designated 凭证类型：
    a. 手机号 → OTP 发到指定手机号（与 Phase 1 相同）
    b. 邮箱   → OTP 发到指定邮箱
    c. 验证器 → 指定人在 Authenticator App 中获取 TOTP

  · PIN 不适用（PIN 是操作人个人的，Designated 的目的是让指定人控制）
  · 操作人的 2FA 不参与（Designated 模式由指定凭证持有人验证，不走操作人个人的 TOTP/PIN）
```

### 4.3 安全设置变更 — 验证策略

安全设置变更属于 Identity 级，不走 Self/Designated 模式，不走 PIN，固定使用最高可用凭证：

| 操作                                 | 有 2FA                       | 无 2FA                        |
| ------------------------------------ | ---------------------------- | ----------------------------- |
| **设置 Payment PIN**           | 2FA 验证 + 设置新 PIN        | 登录密码验证 或 OTP           |
| **修改 Payment PIN**           | 2FA 验证 + 当前 PIN + 新 PIN | 当前 PIN + OTP + 新 PIN       |
| **重置 Payment PIN**（忘记了） | 2FA 验证 + 设置新 PIN        | OTP 验证                      |
| **修改登录密码**               | 2FA 验证 + 新密码            | 当前密码 + OTP + 新密码       |
| **修改手机号**                 | 2FA 验证 + 新手机号 OTP      | 当前手机号 OTP + 新手机号 OTP |
| **修改邮箱**                   | 2FA 验证 + 新邮箱 OTP        | 当前凭证 OTP + 新邮箱 OTP     |
| **开启/关闭 2FA**              | 当前 2FA 验证                | OTP 验证                      |
| **设置/修改 Designated Phone** | OTP（仅 Owner 可操作）       | OTP（仅 Owner 可操作）        |

**核心逻辑：**

```
有 2FA → 验证 2FA + 新凭证
无 2FA → 验证当前凭证 + 新凭证

原因：
  · 2FA 是最高等级的身份证明（持有物理设备）
  · 有 2FA 时用 2FA 验身份，比 OTP 更安全（OTP 有 SIM swap 风险）
  · 无 2FA 时用当前凭证验身份，确保是本人在操作
  · 新凭证验证 = 确认新凭证有效且归本人所有
```

## 5. 验证流程

### 5.1 主流程

```
用户发起操作
  │
  ├── 资金操作？
  │     │
  │     ├── 该成员授权模式 = Designated？
  │     │     │
  │     │     ├── Designated Phone 已设置 → OTP 发到 DP → 输入验证码 → 执行
  │     │     └── Designated Phone 未设置 → 阻断，提示联系 Owner
  │     │
  │     └── 该成员授权模式 = Self？
  │           │
  │           ├── 已设置 PIN → 输入 PIN → 正确 → 执行
  │           │                        → 错误 ≥ 5 次 → PIN 锁定 → 回退 OTP
  │           │
  │           └── 未设置 PIN → OTP 发到操作人凭证 → 输入验证码 → 执行
  │
  ├── 安全设置变更？
  │     │
  │     ├── 有 2FA → 验证 2FA + 新凭证
  │     └── 无 2FA → 验证当前凭证 + 新凭证
  │
  ├── 敏感管理操作？
  │     └── OTP 发到操作人凭证（固定，不走 PIN，不走 Designated）
  │
  └── 普通操作？
        └── 登录态即可，无需二次验证
```

### 5.2 PIN 锁定与回退机制

```
PIN 错误 1-2 次 → 提示"交易密码错误，请重试"
PIN 错误 3-4 次 → 提示"还剩 N 次机会，连续错误 5 次将锁定"
PIN 错误 ≥ 5 次 → PIN 锁定 30 分钟
  → 自动切换到 OTP 验证码
  → 提示"交易密码已锁定，已切换到验证码验证"

PIN 锁定期间：
  · 所有 Self 模式的资金操作自动走 OTP
  · 30 分钟后 PIN 自动解锁
  · 不影响 Designated 模式（本来就不用 PIN）
```

## 6. 操作分级

### 6.1 分级原则

| 类型                    | 判断标准                          | 验证机制                           | OTP 有效期       |
| ----------------------- | --------------------------------- | ---------------------------------- | ---------------- |
| **💰 资金操作**   | 导致账户余额结构变化（金额/币种） | Self/Designated 模式（PIN 或 OTP） | **2 分钟** |
| **🔐 安全设置**   | 修改 Identity 级凭证或安全配置    | 固定：2FA > 当前凭证 + 新凭证      | 5 分钟           |
| **⚠️ 敏感管理** | 不涉及资金，但影响权限/数据安全   | 同上                               | 5 分钟           |
| **📋 普通操作**   | 查看、配置类，不涉及资金/权限     | 登录态即可                         | —               |

> 换汇（USD→HKD）虽然资金未离开平台，但余额结构变了（USD↓ HKD↑），属于资金操作，必须验证。

## 7. OTP 与 PIN 规则

### 7.1 OTP 规格

| 参数             | 资金操作                          | 安全设置 / 敏感管理 |
| ---------------- | --------------------------------- | ------------------- |
| 长度             | 6 位数字                          | 6 位数字            |
| **有效期** | **2 分钟**                  | **5 分钟**    |
| 发送间隔         | 同一凭证 60 秒内只能发 1 次       | 同上                |
| 每日上限         | 同一凭证每天 20 次                | 同一凭证每天 10 次  |
| 错误上限         | 连续 5 次错误，冻结该操作 30 分钟 | 同上                |

> 资金操作 OTP 有效期更短（2 分钟）：验证码泄露后可利用窗口更小。

### 7.2 PIN 规格

| 参数       | 值                                        |
| ---------- | ----------------------------------------- |
| 长度       | 6 位数字                                  |
| 错误上限   | 连续 5 次错误 → 锁定 30 分钟             |
| 锁定后行为 | 自动回退到 OTP 验证                       |
| 适用范围   | 仅 Self 模式下的资金操作                  |
| 不适用范围 | Designated 模式 / 安全设置变更 / 敏感管理 |

## 8. 分期实现

### 8.1 Phase 1 — 核心验证能力

**目标：** 覆盖所有操作的基础验证，确保资金安全。

| 功能            | 说明                                       | 优先级 |
| --------------- | ------------------------------------------ | ------ |
| Self 模式       | PIN > OTP 完整流程                         | P0     |
| Designated 模式 | OTP 发到 Designated Phone                  | P0     |
| Payment PIN     | 设置 / 修改 / 重置 / 锁定回退              | P0     |
| OTP 发送        | 手机号 / 邮箱，优先级规则，有效期/频率限制 | P0     |
| 安全设置验证    | 当前凭证 + 新凭证（无 2FA）              | P0     |
| 敏感管理验证    | 固定 OTP                                   | P0     |
| 操作分级        | 四类操作分级 + 对应验证规则                | P0     |
| 验证弹窗        | PIN 输入 / OTP 输入 / 错误提示 / 锁定提示       | P0     |

**Phase 1 验证策略总结：**

```
                Self 模式                    Designated 模式
                ─────────                    ───────────────
资金操作        PIN > OTP                    OTP → DP
安全设置        当前凭证 + 新凭证          不适用（Identity 级）
敏感管理        OTP（固定）                  不适用（Identity 级）
普通操作        无需验证                     无需验证
```

### 8.2 Phase 2 — 风控增强与体验优化

**目标：** 基于风险等级动态调整验证强度，提升安全性和体验。

| 功能                   | 说明                                              | 触发条件                       |
| ---------------------- | ------------------------------------------------- | ------------------------------ |
| **2FA(TOTP) 登录** | 密码登录后追加 Authenticator TOTP 验证        | 用户开启 2FA                   |
| **TOTP 交易验证**  | 交易验证优先级升级为 PIN > TOTP > OTP           | 用户已开启 2FA                 |
| **风控加验**     | 高风险操作可叠加验证（如 PIN + TOTP）             | 大额交易 / 新收款人 / 异常设备 |
| **新收款人加验** | 首次向新 Beneficiary 付款时加验（即使有 PIN）     | 收款人不在历史记录中           |
| **大额交易加验** | 超过阈值的交易额外验证（即使有 PIN）              | 单笔金额 > 阈值（可配置）      |
| **受信设备**     | 常用设备 + 小额交易免验证                         | 设备指纹 + 金额 < 阈值         |
| **生物识别**     | Face ID / 指纹作为 PIN 的替代输入方式             | 用户设备支持 + 用户开启        |
| **白名单**       | 白名单收款人/地址免加验                           | 用户设置白名单 + 冷却期后生效  |
| **验证时间窗**   | N 分钟内重复操作免验证                            | 同类型操作 + 时间窗口内        |
| **异常告警**     | 异常登录/操作推送通知                             | 新设备/新 IP/异常时间          |
| **Designated 扩展** | Designated 凭证支持邮箱 OTP / Authenticator App   | Owner 配置指定凭证类型         |
| **客户自定义规则** | Owner 可配置 MID 级验证策略（见下方详细说明） | Owner 在 Security 页面配置           |

**Phase 2 风控加验逻辑（Self 模式）：**

```
Phase 1（基础）：
  有 PIN → 输入 PIN → 执行
  无 PIN → OTP → 执行

Phase 2（风控增强 + TOTP）：
  常规操作 + 有 PIN → 输入 PIN → 执行                      ← 体验不变
  常规操作 + 无 PIN + 有 TOTP → 输入 TOTP → 执行           ← TOTP 作为中间层
  高风险操作 + 有 PIN + 有 TOTP → PIN + TOTP → 执行        ← 风控叠加验证
  高风险操作 + 有 PIN + 无 TOTP → PIN + OTP → 执行         ← 风控叠加验证
  受信设备 + 小额 → 免验证 → 执行                          ← 减验（体验提升）

高风险操作定义：
  · 首次向新收款人付款
  · 单笔金额 > 可配置阈值
  · 新设备登录后的首笔操作
  · 24 小时内累计出金 > 可配置阈值
  · 提币到新链上地址
```

**Phase 2 受信设备逻辑：**

```
用户在某设备上完成 2FA 验证登录后，可选择"信任此设备"
受信设备 + 小额操作 → 可免验证（或仅 PIN，不需要 OTP）
受信设备有效期：30 天（可配置）
受信设备管理：用户可在 Security 页面查看/移除受信设备

对标：Binance "受信设备管理"；Stripe "remembered device"
```

**Phase 2 白名单逻辑：**

```
用户可将常用收款人/链上地址加入白名单
白名单地址付款 → 不触发"新收款人加验"
白名单新增后有冷却期（如 24 小时），冷却期内不生效

对标：Binance "提币白名单"（开启后只能提到白名单地址）
```

**Phase 2 客户自定义验证规则：**

```
Owner 可在 Security 页面为 MID 配置验证策略，系统提供可配置项：

1. Designated 凭证类型：
   · 手机号（默认，Phase 1 已支持）
   · 邮箱
   · Authenticator App

2. 资金操作验证策略（MID 级）：
   · 单笔加验阈值：Owner 设置金额阈值，超过则额外加验 OTP
   · 日累计加验阈值：24h 内累计出金超过阈值，后续操作加验
   · 新收款人是否加验：开/关
   · 白名单是否启用：开/关

3. 成员级配置：
   · Owner 可为不同成员设置不同的授权模式（Self / Designated）
   · Phase 2 可按成员设置不同的加验阈值（如财务 50,000 免加验，运营 10,000 就加验）

对标：
  Airwallex "审批流"（多级审批 + 金额触发）
  Binance "提币白名单" + "单笔/日累计限额"
  我们的方案更轻量：不做审批流，而是让 Owner 配置规则，系统自动执行加验
```

### 8.3 Phase 1 vs Phase 2 对比

```
                        Phase 1                     Phase 2
                        ───────                     ───────
登录验证                密码 + OTP                  密码 + 2FA(TOTP)
交易验证                PIN > OTP                   PIN > TOTP > OTP
风控叠加                无                          高风险操作 PIN + TOTP
验证强度                固定（按设置组合）          动态（按风险等级）
大额交易                与小额同等验证              额外加验
新收款人                与老收款人同等验证          额外加验
受信设备                无                          小额免验
生物识别                无                          PIN 替代方式
白名单                  无                          免加验
Designated 凭证       仅手机号                    手机/邮箱/验证器
客户自定义规则        无                          Owner 配置阈值/白名单/加验规则
异常告警                无                          推送通知
商户自定义设置        无                          Owner 可配置验证策略

安全水位：
  Phase 1 = 统一安全基线（PIN > OTP，所有操作同等对待）
  Phase 2 = TOTP 能力 + 风险自适应（高风险加验 + 低风险减验） + 客户自治（Owner 定义规则）
```

## 9. 2FA(TOTP) 与 Payment PIN 的关系（Phase 2 规划）

### 9.1 一句话区别

```
PIN   = 保险箱密码（最快，进门后保护钱）  → 用于交易（优先级最高）
TOTP  = 指纹锁（独立设备，不依赖运营商）  → 用于登录 + 交易 + 设置变更
OTP   = 门卫打电话确认（兜底方案）        → 登录/交易/设置 都可用
```

### 9.2 全维度对比

| 维度                 | 2FA (TOTP)                       | Payment PIN                      | OTP (SMS/Email)                  |
| -------------------- | -------------------------------- | -------------------------------- | -------------------------------- |
| **保护什么**   | 登录入口 + 交易 + 设置变更       | 资金操作（防登录态下未授权转账） | 兜底验证（全场景）               |
| **触发时机**   | 登录（密码后）+ 交易（无PIN时）  | 每次资金操作时（最高优先级）     | 无 PIN 且无 TOTP 时              |
| **验证方式**   | Authenticator App 6位动态码      | 6 位数字（本地输入）             | 短信/邮箱验证码                  |
| **安全特性**   | 本地生成，不经网络，SIM劫持无效  | 知识因子，纯脑记忆               | 依赖运营商/邮箱，SIM劫持可攻破  |
| **是否强制**   | 否，可选                         | 否，可选                         | 始终可用（注册时强制绑定）       |
| **未设置回退** | 登录仅凭密码；交易降级为 OTP     | 交易降级为 TOTP → OTP            | —（兜底方案本身）               |
| **交易优先级** | 第二优先（PIN > **TOTP** > OTP） | 最高优先                         | 最低（兜底）                     |
| **适用模式**   | Self 模式                        | 仅 Self 模式                     | Self + Designated                |
| **用户体验**   | 打开 App 查看动态码（~5秒）      | 直接输入（< 2秒，最快）          | 等待发送（5-30秒，最慢）         |

### 9.3 攻击场景分析

```
场景一：只有 2FA(TOTP)，没有 PIN（C3）
  密码泄露 → 2FA 阻止登录 ✅
  SIM 劫持 → OTP 登录成功 → 发起转账 → 需要 TOTP → 攻击者没有 Authenticator App → 资金安全 ✅
  → TOTP 独立于 SIM，异源保护

场景二：只有 PIN，没有 2FA（C2）
  密码泄露 → 直接登录（无 2FA）⚠️ → 账户被入侵可查看数据
  → 发起转账 → 需要 PIN → 不知道 PIN → 资金安全 ✅
  SIM 劫持 → OTP 登录 → 需要 PIN → 资金安全 ✅

场景三：2FA + PIN 都有（C4，推荐）
  密码泄露 → 2FA 阻止登录 ✅
  SIM 劫持 → OTP 登录 → 需要 PIN → 不知道 PIN → 资金安全 ✅
  即使 PIN 锁定 → 需要 TOTP → 攻击者没有 App → 资金安全 ✅
  → 三重独立保护：密码(知识) + TOTP(独立持有) + PIN(知识)
```

## 10. 完整场景速查表

### 10.1 按用户配置状态 — Phase 1（Self 模式）
