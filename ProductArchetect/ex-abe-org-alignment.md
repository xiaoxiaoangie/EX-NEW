## 1. 问题：架构正确，不代表人会照做

定位文档（`ex-abe-positioning-strategy.md`）已经论证清楚：ABE 的价值不在通道费，而在**资金沉淀在 A/B + 数据沉淀在 EX + 未来的资金收益（float/信贷/理财）**。整套架构的经济逻辑成立的前提是——**流量尽量走 A/B，绕不开的才用外部 SP**。

但战略文档默认了一件不成立的事：**「架构画对了，执行的人就会照着做」**。现实是：

```
战略层（对）：A/B 优先，外部 SP 补位，资金/数据沉淀自家
      ↓  （这里断裂）
执行层（现状）：BD 知道有更便宜的外部渠道，一联系 FIBD 就走了外部 SP，
              没经过 A/B —— 每个人都在做「对自己 KPI 最优」的选择
```

这不是人的问题，是**组织设计的问题**：当 ABE 是三个不同的组织架构（A/B/E 各有团队、各有 KPI），而协同机制缺位时，「局部最优」必然打败「全局最优」。

> ⚠️ 本文不谈「要求大家有大局观」这种空话。大局观靠规则、激励、流程来保证，不靠觉悟。

## 2. 根因分析：三个「局部最优」如何毁掉「全局最优」

用户提出的三个现象，本质是同一个结构性问题的三个切面：

### 现象①：FIBD 自己签约自己的渠道、自己用自己的能力

```
FIBD 的局部动机：多签渠道 = 我的业绩；渠道便宜 = 我好谈客户
但对全局的破坏：
  · 绕过 A/B → 资金不沉淀 → float/信贷/理财的根基没了
  · 绕过 EX 编排/账本 → 数据不沉淀 → 智能风控/授信没了
  · 每个 FIBD 各签各的 → 渠道碎片化，集团议价权被稀释
```

### 现象②：A/B 定制价格比外部贵，EX 租户就用外部的

```
表面矛盾：A/B 贵 → 用外部便宜的，对客户不是更好吗？
但这是「只算通道费这一层账」的误判：
  · 通道费是 ABE 最不该赚的钱（定位文档 §3.1：越做越薄）
  · A/B 贵一点，但资金沉淀带来的 float + 数据 + 后端资金收益
    远大于那点价差 —— 全局账是正的
  · 外部便宜，但资金/数据都留在别人家 —— 全局账是负的
→ 结论：A/B 需要「价格保护」，但价格保护要有规则、有边界，
  不能变成「A/B 随便定价，反正强制走」（否则 A/B 没有降本动力）
```

### 现象③：BD 与 FIBD 小团队协作，抄近路走外部 SP

```
BD（销售）动机：成交快、客户价格低 = 我的 KPI
FIBD 动机：签渠道、渠道能用 = 我的 KPI
两个小团队一拍即合 → 走了最便宜的外部 SP → 没有 A/B
  → 谁都没错（各自 KPI 都达成），但集团输了
```

**共同根因：KPI 只考核「局部指标」（签约数、成交额、通道成本），没有人对「资金沉淀率 / 数据沉淀率 / 全局利润」负责。** 当考核不对齐战略，再正确的架构也会被一线的理性选择架空。

## 3. 第一性原则：先讲清「为什么必须站在一起」

要让人遵守，先要让人**认同**。对内要能用一句话讲清楚：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  我们卖的不是通道，是「资金 + 数据 + 未来的资金收益」。            │
│  通道费谁都能压到很低，那不是我们的生意。                         │
│  只有当钱沉淀在 A/B、数据沉淀在 EX，                              │
│  我们才能做 float、做信贷、做理财 —— 那才是真正赚钱、             │
│  资本市场才给高估值的东西。                                       │
│                                                                 │
│  所以：走外部便宜渠道省下的那点通道费，                           │
│         是在偷走公司未来最值钱的资产。                            │
│                                                                 │
│  这不是「A/B 部门的私利」，是整个 ABE 的生死线。                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

三个必须对齐的共识（建议做成全员宣讲材料）：

- **共识一：ABE 是一家公司，不是三家。** A/B/E 的 KPI 最终要合并到集团一张利润表。FIBD 签外部渠道省的钱，如果让 A/B 的资金沉淀归零，集团是亏的。
- **共识二：通道费是入口，资金收益是利润。** 定位文档 §3 已论证：纯通道估值低、越做越薄；沉淀 + 数据 + 信贷/理财才是高估值故事。为省通道费牺牲沉淀，是捡芝麻丢西瓜。
- **共识三：路由权在 EX，不在个人。** 走哪个 SP 不是 BD 或 FIBD 个人拍板，而是由规则（默认路由 SLA，见定位文档 §4.3）决定。个人不背这个决策，也就没有「抄近路」的空间。

### 3.1 为什么资金要优先落到 A/B：从“通道路由”升级为“清算网络”



ABE 的核心不是简单连接多个 SP，而是把本地支付、跨境收付、法币兑换、数币承兑、账本风控和清算网络组合成一套可运营的跨境资金基础设施。因此，资金不能长期只停留在外部 SP 侧，而应在合规、成本、时效和客户体验允许的前提下，优先归集到 A/B。

资金落到 A/B 的原因有六点：

第一，资金在哪里，能力就在哪里。外部 SP 只能提供局部收付能力；A/B 才承载 ABE 自有的跨境账户、换汇、On/Off Ramp、数币账本、余额管理和清算能力。如果资金不进入 A/B，EX 只能做技术路由和状态查询，无法真正控制后续换汇、承兑、出金、退款、冻结和对账。

第二，资金落到 A/B，ABE 才能从“连接器”变成“网络”。如果资金始终留在外部 SP，客户可以绕过 EX 直接连接该 SP，ABE 的价值会被压缩成系统费或路由费。资金进入 A/B 后，客户使用的是一个跨境资金账户，而不是单一通道。

第三，A/B 承接的是本地 PSP 难以补齐的能力。本地收款、本地付款可以交给本地 SP；但跨境收款、多币种结算、法币换汇、法币与 U 的承兑、链上 KYT、数币清算和统一账本，应优先由 A/B 承接。这不是抢本地段，而是把 ABE 的核心能力沉淀在跨境段和兑换段。

第四，资金落 A/B 才能形成完整数据资产。EX/E 需要沉淀交易、资金、风控和对账数据。若资金只在外部 SP，EX 拿到的是碎片化订单状态；若资金进入 A/B，ABE 才能形成连续资金流水、商户画像、风控记录和后续授信基础。

第五，资金落 A/B 才能形成账户粘性。客户在 A/B 中形成余额后，可以继续完成换汇、出金、U 转法币、法币转 U、供应商付款、统一对账和余额管理。客户依赖的对象从“某条通道”升级为“跨境资金账户”，绕开动机会下降。

第六，资金落 A/B 是未来收益的前提。余额管理、资金效率、提前结算、商户贷、供应链金融、理财代销和稳定币储备收益，都依赖可识别、可归属、可对账、可管理的资金流水和余额。但这些收益必须在牌照、客户资金隔离、利息归属和客户授权清晰后逐步推进，不能在第一阶段作为默认收入。

因此，ABE 的资金策略不是“所有资金都必须进 A/B”，而是：

本地闭环资金留给本地 SP；
跨境资金优先进入 A；
数币和 On/Off Ramp 资金优先进入 B；
A/B 暂不覆盖的币种、国家和支付方式由外部 SP 补位；
所有交易、风控、对账和资金状态回流 E/EX，形成统一账本和数据资产。

一句话：外部 SP 解决触达，A/B 承接清算，E/EX 沉淀账本和风控。资金落到 A/B，是 ABE 从支付通道变成跨境资金网络的关键。

## 4. 组织规则：什么必须遵守、什么可以灵活

把架构变成**明文的、有约束力的组织规则**。区分「红线（必须遵守）」和「灰度（可灵活）」，避免一刀切扼杀一线灵活性。

### 4.1 红线规则（Must，违反需上报/问责）

```
红线①：路由决策权归属 EX 编排层，不归个人
  · 走 A/B 还是外部 SP，由「默认路由 SLA」规则引擎判定（§6）
  · BD/FIBD 不得私下承诺客户「走某个外部渠道」

红线②：数币段必经 B（结构性红线，无例外）
  · 所有 On/Off Ramp 的数币环节一律经 BlancBlock
  · 对应定位文档锁③，这一段不存在「更便宜的外部选择」一说

红线③：FIBD 不得自行签约并直接对客户放渠道
  · 新渠道接入必须进 EX 渠道池 + 走能力差集规则（§4.3 定位文档）
  · A/B 已覆盖且达标的能力，外部同类渠道默认不上架

红线④：资金/数据不落 EX 账本的业务，不予放行
  · 绕过 EX 编排/账本的交易 = 数据资产流失 = 不允许
  · 「不落账本就没发生」——考核与结算都以 EX 账本为准
```

### 4.2 灰度空间（Can，鼓励一线灵活）

```
灰度①：A/B 确实不达标时，可走外部 SP（但走 EX 编排）
  · 「默认路由 SLA」六项指标（价格/成功率/时效/限额/合规/对账）
    A/B 未达标 → 外部 SP 合理补位，这是架构允许的（定位文档 §4.3②）
  · 关键区别：是「规则判定 A/B 不行」而非「个人觉得外部便宜」

灰度②：企业客户可通过「高级路由配置」显式选外部
  · 客户知情并主动选择时，允许 —— 但必须显式、留痕、经 EX 账本

灰度③：一线可发起「A/B 能力缺口」工单
  · BD/FIBD 发现 A/B 某能力长期不达标 → 正式提报，
    推动 A/B 补齐，而不是私自绕开（把绕开动机转成改进动力）
```

### 4.3 价格保护规则（回应现象②）

A/B 需要价格保护，但保护要有边界，否则 A/B 会失去降本动力、变成「反正强制走」的懒惰垄断：

```
价格保护三原则：
  ① 保护带（Price Band）：A/B 价格在市场基准的 +x% 以内时，
     默认走 A/B（价差被沉淀收益覆盖，全局账为正）
  ② 触发切换：A/B 价格超出保护带、且外部明显更优 →
     进入灰度①流程，允许外部补位（防止 A/B 躺平）
  ③ A/B 有降本承诺：享受价格保护的同时，A/B 需给出
     成本下降 roadmap（与定位文档 Q7「A 通道竞争力时间表」对齐）
→ 保护是为了「争取沉淀，等 A/B 追上」，不是「永久强制」
```

## 5. 激励对齐：让「照做」比「绕开」更划算

规则靠激励落地。**只要 KPI 不改，红线就守不住。** 核心是把考核从「局部指标」改为「战略指标」。

### 5.1 KPI 重构（从局部到全局）

| 角色                   | 现状 KPI（导致绕开）       | 建议增加/调整的 KPI                                                                             |
| ---------------------- | -------------------------- | ----------------------------------------------------------------------------------------------- |
| **BD（销售）**   | 成交额、客户数、客户价格低 | 增加「**A/B 资金沉淀率**」「客户走 EX 账本比例」                                          |
| **FIBD（渠道）** | 签约渠道数、渠道成本低     | 增加「**A/B 能力覆盖贡献**」「外部渠道仅在 A/B 缺口处签约」；砍掉「单纯签约数」的直接激励 |
| **A/B 团队**     | 自身通道量                 | 增加「**沉淀余额规模**」「对内 SLA 达标率」「降本 roadmap 兑现」                          |
| **EX 编排**      | 系统可用性                 | 增加「**全局沉淀率**」「路由合规率」「数据完整率」                                        |

> 核心动作：**给每个相关角色都加一条「资金/数据沉淀」的共同 KPI**，让 A/B/E 三方在同一个指标上绑在一起——这是「站在一起」最硬的机制。

### 5.2 集团级共享指标（把三家绑成一家）

```
设一个跨 A/B/E 的「集团沉淀与收益」共享池指标：
  · 沉淀率 = 走 A/B 的资金量 / 全平台资金量
  · 数据完整率 = 落 EX 账本的交易笔数 / 全平台交易笔数
  · 全局 take（含 float/spread/未来资金收益）而非仅通道费
→ 三个团队的奖金部分挂钩共享指标：
  谁绕开导致沉淀下降，三家一起少拿 → 内部自然形成相互监督
```

### 5.3 消除「绕开的理性」

```
绕开之所以发生，是因为绕开对个人有利、对集团有害。
对策 = 让绕开对个人也不再有利：
  · 走外部 SP 省下的通道费，不计入 BD 的价格竞争力考核
    （即：省这点钱不加分）
  · 走 A/B 带来的沉淀，计入 BD/FIBD 的正向考核（照做才加分）
  · 私自绕开（违反红线）→ 该笔业务业绩不认 + 问责
```

## 6. 流程与治理：路由决策不靠个人良心

把「走哪个 SP」从**人的自由裁量**变成**系统 + 流程的默认结果**，这样即使 BD 和 FIBD 想抄近路也抄不了。

### 6.1 路由决策流程（系统兜底）

```
客户下单 → EX 路由引擎按「默认路由 SLA」自动判定：
   ├─ A/B 达标 → 默认走 A/B（个人无法手动改走外部）
   ├─ A/B 未达标 → 系统提示可用外部 SP（仍经 EX 编排/账本）
   └─ 客户显式选外部（高级路由）→ 留痕、经账本
→ 关键：默认路径由系统决定，不给个人「私下走外部」的操作入口
```

### 6.2 新渠道接入治理（管住 FIBD 签约）

```
FIBD 想签一个新的外部渠道，必须走审批流：
  ① 说明该渠道对应的能力 X
  ② 检查：A/B 是否已覆盖 X 且达标？
     · 是 → 默认不接（除非作为备份/容灾，需理由）
     · 否 → 允许接入，进 EX 渠道池 + 差集展示
  ③ 接入后：该渠道的资金/数据必须落 EX 账本
→ 审批权不在 FIBD 个人，建议设跨 A/B/E 的「渠道委员会」
```

### 6.3 治理机构：渠道与路由委员会

```
建议设一个跨 A/B/E 的「渠道与路由委员会」（轻量、定期）：
  · 成员：EX 编排/产品 + A 代表 + B 代表 + 风控/合规
  · 职责：
     – 定期评审「默认路由 SLA」阈值与价格保护带
     – 审批新渠道接入（红线③的把关人）
     – 处理「A/B 能力缺口」工单，推动 A/B 补齐
     – 裁决绕开争议、复盘沉淀率异常
  · 意义：把「个人拍板」变成「集体规则」，责任从人转到机制
```

### 6.4 透明与复盘

```
· 沉淀率/数据完整率做成看板，A/B/E 三方可见（阳光是最好的约束）
· 定期复盘「本期走了哪些外部 SP、为什么、A/B 缺口在哪」
· 异常预警：某团队外部 SP 占比突然升高 → 自动提示委员会核查
```

## 7. 各角色的具体要求

### 7.1 BD（销售 / 获客）

```
必须做：
  · 向客户传递「默认能力（A/B）+ 其他能力（外部）」的产品叙事，
    不主动推销外部渠道的低价
  · 成交后业务走 EX 账本，接受「资金沉淀率」考核
禁止做：
  · 私下承诺客户走某个外部便宜渠道
  · 为了成交绕过 A/B / 绕过 EX 编排
可以做（灰度）：
  · 提报「A/B 价格/能力不达标」工单，推动改进
  · 客户显式要求外部渠道时，走高级路由（留痕）
```

### 7.2 FIBD（渠道 / SP 拓展）

```
必须做：
  · 新渠道接入走审批流（渠道委员会），进 EX 渠道池
  · 优先补 A/B 的能力缺口，而不是签同质竞品
禁止做：
  · 自行签约并直接对客户放渠道（红线③）
  · 签约 A/B 已覆盖且达标的同类能力（左右互搏，稀释沉淀）
考核调整：
  · 从「签约数」改为「A/B 缺口覆盖贡献 + 沉淀友好度」
```

### 7.3 A / B 团队（持牌资金方）

```
必须做：
  · 对内提供达标的 SLA（价格/成功率/时效/限额/合规/对账）
  · 享受价格保护的同时，兑现降本 roadmap（不许躺平）
  · 及时响应「能力缺口」工单，补齐覆盖
考核调整：
  · 增加「沉淀余额规模」「对内 SLA 达标率」「降本兑现」
```

### 7.4 EX（编排 / 产品 / 数据）

```
必须做：
  · 把路由规则、价格保护带、差集展示做进系统（个人无法绕过）
  · 维护统一账本，保证数据完整率
  · 运营渠道与路由委员会的看板与复盘
考核调整：
  · 增加「全局沉淀率」「路由合规率」「数据完整率」
```
