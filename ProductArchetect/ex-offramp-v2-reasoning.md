## 1. 起点：EX 一期的约束条件

```
约束❶：客户（租户的商户）不应感知背后是哪个 SP
        → 原因：防止租户绕过EX直接找SP签约，EX前期不具备多SP生态吸引力
        → 客户端只看到"余额100U"，不看到"SP-1余额100U"

约束❷：EX 一期 SP 数量有限
        → 不可能有很多SP可选，但仍然需要 ≥2 个SP保证可用性

约束❸：EX 仍然用 SP 架构
        → SP 提供付款能力（法币出金渠道）
        → EX 做编排/路由

约束❹：客户选"渠道类型"，不选具体SP
        → 例如选"费率高/资料少" vs "费率低/资料多"
        → 系统在类型内部路由到具体SP
```

---

## 2. 正常流程走通

先看没问题的情况：

```
客户选择"费率高/资料少"的渠道类型
    → EX 路由到 SP-1（客户不知道是SP-1）
    → 客户充 100U
    → 客户看到"余额 100U"
    → 客户发起 OffRamp（卖U→提法币）
    → SP-1 执行法币付款
    → 成功 ✅
    → 客户余额变为 0
```

一切正常，没有矛盾。

## 3. 核心矛盾：SP付款失败需换SP

问题出在异常路径：

```
客户充了 100U
    → 路由到 SP-1
    → 100U 现在在 SP-1 处
    → 客户端显示"余额 100U"（不暴露SP）

客户发起 OffRamp
    → SP-1 付款失败（退票/限额满/渠道维护/合规拦截）
    → 客户余额回退到 100U
    → 系统需要换 SP-2 重试

问题来了：
    → 客户的 100U 在 SP-1 的账户/钱包里
    → SP-2 处没有这 100U
    → SP-1 和 SP-2 是独立机构，资金不互通
    → EX 无法指令 SP-1 把钱转给 SP-2
    → 无法切换 ❌
```

**矛盾的本质：**

```
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  "客户不感知SP" + "资金在某个SP处" = 切换SP时资金转不过去     │
│                                                               │
│  这是一个结构性矛盾：                                          │
│  · 不暴露SP → 客户不知道钱在哪 → 不能让客户自己转            │
│  · SP间独立 → EX控制不了SP间资金流动                          │
│  · 要保证体验 → 必须能无缝切换SP                              │
│                                                               │
│  三个条件不能同时满足（除非引入新的角色）                       │
└───────────────────────────────────────────────────────────────┘
```

## 4. 穷举解法

### 解法A：让 SP-1 把钱转给 SP-2

```
SP-1 收到EX指令 → 把 100U 转给 SP-2 的地址
```

**问题：**

- SP-1 和 SP-2 是独立机构，没有义务配合
- SP-1 为什么要帮你把客户的钱转给竞争对手？
- 即使愿意，转账需要时间（链上确认/法币转账）
- EX 无法强制要求，取决于SP合作意愿

**结论：不可控，不可行 ❌**

### 解法B：客户自己从 SP-1 提走，再充到 SP-2

```
客户从 SP-1 提 100U → 充到 SP-2 → 重新发起 OffRamp
```

**问题：**

- 客户不知道有SP-1和SP-2（约束❶）
- 让客户自己操作，体验极差
- 等于暴露了底层SP架构

**结论：违反约束❶，不可行 ❌**

### 解法C：EX 提前在所有SP处垫资

```
SP-1 处有钱（客户的U）→ 付款失败
SP-2 处也提前有钱（EX/某方提前垫的）→ 用垫的钱给客户付
```

**问题：**

- 谁来垫？EX 自己没有资金
- 垫的钱放在SP处，面临冻结/跑路风险
- 每个SP都要垫 → 资金占用 = SP数量 × 每个SP的备付额度
- 客户的100U还在SP-1处，这笔钱怎么回收？

**结论：可以解决切换问题，但引出"谁垫"和"资金效率"的新问题 ⚠️**

### 解法D：客户的U不直接进SP，而是先进一个"中间方"

```
客户充 100U → 不进SP → 进一个中间方（设为X）
X 统一持有所有客户的U
付款时，X 选择任意SP执行
SP-1 失败 → X 换 SP-2，资金不需要移动（因为从来就不在SP处）
```

**分析：**

- 完美解决切换问题 ✅
- 客户完全不感知SP ✅
- 资金始终在X处，不分散在各SP处，资金效率高 ✅
- 关键问题：X 是谁？X 用什么方式让SP执行付款？

**结论：架构上最优，核心问题是 X 的身份和 X→SP 的资金流 ⚠️**

## 5. 深入分析：U充到BB方案

### 5.1 解法D 中，X = BB

```
理由：
· BB 有数币钱包（能收U）
· BB 有承兑能力（能把U换成法币）
· BB 是 EX 体系内的核心角色
· BB 和各SP有签约关系（或可以建立）
→ X = BB 是最自然的选择
```

### 5.2 客户充U → BB → 然后呢？

U到了BB之后，BB要让SP执行法币付款。此时出现一个新问题：

```
BB 持有客户的 100U
BB 需要让 SP-1 付 100 USD 给收款人
问题：SP-1 凭什么替BB付钱？SP-1 的钱从哪来？

可能性①：BB 先把 100U 转给 SP-1 → SP-1 自己承兑+付款
         = "垫U"模式（前提：SP-1 有承兑能力）

可能性②：BB 自己承兑(100U→100USD) → 用法币指令 SP-1 付款
         = BB 需要在 SP-1 处有法币备付金
         = "垫法币"模式

可能性③：BB 自己承兑+自己付款（BB自己就有付款渠道能力）
         = BB 既是承兑方又是渠道
         = 不需要外部SP（但违反约束❸：用SP架构）
         = 如果BB自己能力有限（地区/币种/额度），还是需要外部SP补充
```

### 5.3 SP 到底在这里扮演什么角色？

```
回到原始问题：EX 用SP架构

SP 在 OffRamp 中的能力 = 法币付款能力
  · SP-1 可以付 USD 到美国银行账户
  · SP-2 可以付 HKD 到香港银行账户
  · SP-3 可以付 EUR 到欧洲银行账户

SP 的价值 = 法币出金渠道的多样性和覆盖面

那 BB 跟 SP 的关系是什么？
```

## 6. BB 的角色推导

### 6.1 三种可能的 BB 角色

```
角色①：BB = 资金池 + 承兑方，SP = 纯付款渠道

  客户U → BB钱包
  BB承兑(U→法币)
  BB在各SP处有法币备付金
  BB指令SP用备付金付款
  SP只是执行付款指令

  BB 的身份：承兑方 + 垫资方
  SP 的身份：纯执行渠道（类似银行通道）

────────────────────────────────────────────

角色②：BB = 资金池 + 垫U方，SP = 有承兑能力的OffRamp SP

  客户U → BB钱包
  BB不承兑，按笔转U给SP
  SP收到U后自己承兑+付法币
  
  BB 的身份：资金池 + 垫U方（不承兑）
  SP 的身份：OffRamp服务商（收U+承兑+付款）

────────────────────────────────────────────

角色③：BB = 资金池 + 承兑方 + 部分有自己的付款能力

  客户U → BB钱包
  BB承兑(U→法币)
  BB自己能付的 → BB直接付（不需要外部SP）
  BB自己不能付的 → 指令外部SP付款

  BB 的身份：承兑方 + 主要渠道 + 垫资方
  SP 的身份：BB能力不足时的补充渠道
```

### 6.2 逐个分析

#### 角色①：BB承兑 + SP纯渠道

```
优势：
  · SP 准入门槛低（只需能收发法币，不需要懂数币）
  · SP 选择面广（传统银行/汇款公司/PSP 都可以）
  · BB 统一承兑，汇率一致

劣势：
  · BB 必须在每个SP处存法币备付金 ← 资金占用大
  · 备付金面临冻结/跑路风险
  · 切换SP时：BB在SP-2处也要有备付金 ← 需要多处备付
  · BB 承担汇率风险

切换SP的资金流：
  客户 100U 在BB → BB承兑 → BB指令SP-1付款 → 失败
                                                ↓
  BB指令SP-2付款（BB在SP-2也有备付金）→ 成功 ✅
  
  关键：客户的U在BB，切换SP只是BB换一个付款指令对象
       前提是BB在SP-2也有法币备付金 → BB需要在多个SP处都预存
```

#### 角色②：BB垫U + SP有OffRamp能力

```
优势：
  · BB 不需要在SP处存法币（不存法币=不怕冻结）
  · BB 按笔转U给SP，资金效率高
  · 汇率风险转给SP

劣势：
  · SP 必须有承兑能力（能收U、换法币、付款）← SP准入门槛高
  · 这样的SP = 本身就是OffRamp服务商 ← 数量少
  · SP自己承兑，汇率不受EX控制 ← 客户体验不一致
  · SP收了U不办事 → 履约风险

切换SP的资金流：
  客户 100U 在BB → BB转U给SP-1 → SP-1不办事/失败
                                    ↓
  问题：U已经转给SP-1了，怎么换SP-2？
  → SP-1 需要退回U → 时间不确定
  → 或者 BB 用自己的U再给SP-2转一笔 → BB需要额外的U流动性
  
  ⚠️ 切换SP时资金流比角色①更复杂
     因为U已经转出去了，不像角色①法币备付金是预存的
```

#### 角色③：BB自己付 + SP做补充

```
优势：
  · 大部分交易BB自己处理，不依赖外部SP
  · 减少SP依赖和垫资风险
  · 客户完全不感知SP

劣势：
  · BB自己的付款能力有限（地区/币种/额度/牌照）
  · 不是所有地方BB都能直接付款
  · 本质上：BB能覆盖的用角色③，BB不能覆盖的回退到角色①或②

实际上：角色③ = 角色①的优化版
  BB能自己付的 → 不需要SP → 最安全
  BB不能付的 → 用SP → 需要垫资 → 回到角色①或②的问题
```

### 6.3 推理汇总

```
┌─────────────────────────────────────────────────────────────────────┐
│  推理链条：                                                          │
│                                                                     │
│  约束：客户不感知SP + SP间资金不通                                    │
│    ↓                                                                │
│  必然结论：客户的U不能直接进SP → 必须进一个中间方                     │
│    ↓                                                                │
│  中间方 = BB（有钱包+有承兑能力+体系内核心角色）                      │
│    ↓                                                                │
│  BB 持有客户U后，怎么让SP执行付款？                                  │
│    ↓                                                                │
│  ├── 角色①：BB承兑+在SP处存法币备付金 → SP纯执行                    │
│  │   · 可行 ✅  切换SP无障碍（BB在多处有备付金）                     │
│  │   · 代价：法币长期存在SP处，有冻结风险                            │
│  │                                                                   │
│  ├── 角色②：BB按笔转U给SP → SP自己承兑+付款                        │
│  │   · 可行 ✅  不需要预存法币（资金效率高）                         │
│  │   · 代价：SP门槛高 + 切换时U已转出难追回                         │
│  │                                                                   │
│  └── 角色③：BB自己付大部分 + SP补充小部分                           │
│      · 可行 ✅  最安全（减少SP依赖）                                 │
│      · 代价：BB付款能力有限，不能覆盖所有场景                        │
│                                                                     │
│  实际上可能是组合：角色③为主 + 角色①/②为补充                        │
└─────────────────────────────────────────────────────────────────────┘
```

## 7. 三种 BB 角色方案对比

| 维度                     | 角色①：BB承兑+垫法币 | 角色②：BB垫U        | 角色③：BB自己付+SP补充 |
| ------------------------ | --------------------- | -------------------- | ----------------------- |
| **客户U去哪**      | BB钱包                | BB钱包               | BB钱包                  |
| **谁承兑**         | BB                    | SP                   | BB                      |
| **BB在SP处存什么** | 法币备付金            | 不存（按笔转U）      | 部分SP存法币            |
| **切换SP**         | ✅ 简单（换指令对象） | ⚠️ 复杂（U已转出） | ✅ 简单                 |
| **SP准入门槛**     | 低（纯法币渠道即可）  | 高（需有承兑能力）   | 低                      |
| **BB资金占用**     | 🔴 高（多SP预存）     | 🟢 低（按笔转出）    | 🟡 中                   |
| **冻结风险**       | 🔴 高（法币在SP处）   | 🟢 低                | 🟡 中                   |
| **汇率控制**       | ✅ BB统一定价         | ❌ SP各自定价        | ✅ BB统一定价           |
| **客户体验一致性** | ✅ 高                 | ⚠️ SP汇率不同      | ✅ 高                   |

### 7.1 角色②的致命问题：切换SP

```
角色②（垫U）看起来资金效率高，但有一个致命问题：

  BB 转 100U 给 SP-1 → SP-1 收到U → SP-1 执行中... → 失败
  
  现在：
  · 100U 已经在 SP-1 手里了
  · BB 要换 SP-2
  · BB 需要再拿 100U 给 SP-2 → BB 的USDT池被多占用一笔
  · SP-1 那笔 100U 何时退回？不确定
  
  如果并发多笔都失败：
  · SP-1 有 100U（待退回）
  · SP-2 有 100U（正在处理）
  · SP-3 有 100U（正在处理）
  · BB 的 USDT 池快速消耗

  vs 角色①（垫法币）：
  · BB 在 SP-1 有备付金 → 指令付款 → 失败 → 备付金回退到SP-1账户
  · BB 换 SP-2 → SP-2 也有备付金 → 指令付款 → 成功
  · 不需要追回资金，因为备付金始终在各SP的备付账户里

  角色①的切换是"换指令"
  角色②的切换是"追回资金+重新转出" ← 慢且不确定
```

### 7.2 角色①的核心问题：资金占用和冻结

```
角色①的切换体验最好，但代价是：

  BB 必须在每个SP处都预存一笔法币备付金
  · SP-1 处：200万 USD
  · SP-2 处：100万 USD
  · SP-3 处：150万 USD
  · 总计：450万 USD 长期锁在外部SP处

  风险：
  · SP-1 被冻结 → 200万 USD 可能拿不回来
  · SP-2 跑路 → 100万 USD 直接损失
  
  这是"用资金安全换客户体验"的 trade-off
```

### 7.3 角色③的现实考量

```
角色③ = BB自己先上，覆盖不了的再用外部SP

问题：BB 自己目前有什么付款能力？
  · BB 自己有哪些地区/币种的法币出金能力？
  · BB 自己有银行账户能直接转账吗？
  · BB 的付款额度能覆盖多少交易量？

如果 BB 自己的付款能力足够（至少覆盖80%+的交易）：
  → 角色③ 是最优解
  → 外部SP只是少量补充
  → 垫资风险极小

如果 BB 自己的付款能力有限（只能覆盖20%）：
  → 角色③ 退化为角色①（大部分还是靠外部SP）
  → 垫资风险没有实质减少
```

## 8. OnRamp 反向验证

上面分析的是 OffRamp（卖U提法币）。反过来看 OnRamp（充法币买U）是否有同样的问题：

```
OnRamp 正常流程：
  客户汇法币 → SP-1 收到 → SP-1 通知BB → BB给客户充U

OnRamp 异常流程：
  客户汇法币到SP-1 → SP-1 处理失败
  → 需要换SP-2？
  → 不行：客户的法币已经汇到SP-1了，不在SP-2
  → 跟 OffRamp 是同一个问题！

OnRamp 的"U充到BB"对应方案：
  → 客户的法币不直接进SP → 先进BB的法币账户？
  → 但问题是：客户汇款时必须有一个收款账户
  → 这个收款账户是BB的还是SP的？
  → 如果是BB的 → BB需要在各地区有收款账户 → BB覆盖面有限
  → 如果是SP的 → 又回到"钱在SP处"的问题

OnRamp 比 OffRamp 更难：
  · OffRamp：U在链上，可以先进BB → BB选SP → 可控
  · OnRamp：法币汇款要有收款账户，客户汇出去就定了 → 不可控

  这可能意味着：
  · OnRamp 的 SP 切换只能在"下一笔"实现，不能对"这一笔"切换
  · 或者 OnRamp 必须用 BB 的收款账户（VA）→ BB 需要有法币收款能力
```

### 8.1 OnRamp 的启示

```
OnRamp 验证了一个结论：

  如果要客户不感知SP + 能无缝切换SP
  → 客户的资金必须先进BB（不管是U还是法币）
  → BB 需要同时有：收U的能力（数币钱包）+ 收法币的能力（VA/银行账户）

  这强化了"BB 是中心"的结论
  但也暴露了BB的能力瓶颈：BB 自己的法币收款能力有多强？
```

## 10. “BB自己付”的真相——所有渠道都是BB的渠道

### 10.1 推理续接：角色①②③的边界模糊

第7章把 BB 分成三种角色：

- 角色①：BB承兑+在SP处垫法币
- 角色②：BB垫U给SP
- 角色③：BB自己付款

但仔细想：

```
BB “自己付款” 是怎么付的？
  · BB 直接从自己银行账户转账？→ 银行 = BB的渠道
  · BB 通过某个PSP付款？→ PSP = BB的渠道
  · BB 通过JRL付款？→ JRL = BB的渠道

→ “BB自己付”的背后也是渠道
→ 跟“BB通过SP付”没有本质区别
→ 角色③ 不是一个独立的方案，它只是角色①的一种特例（BB自有渠道 = BB控制力最强的SP）

**结论：角色①②③ 不是三种互斥方案，而是一个统一模型下的不同渠道类型。**

### 10.2 统一视角：一切都是“BB的渠道”
```

BB 的所有付款能力 = BB 名下的渠道池

渠道池里有：
├── 渠道-A：BB自有银行账户（直接转账）
├── 渠道-B：BB已经对接的某PSP
├── 渠道-C：通过EX新接入的SP-1
├── 渠道-D：通过EX新接入的SP-2
└── 渠道-E：JRL（BB体系内MSO）

对客户来说：全部不可见
对BB来说：都是“我的渠道”——每个渠道有不同的能力、费率、风险、资金安排

```

### 10.3 渠道的分类维度（不是按“谁的”，而是按“特性”）

| 维度 | 分类 | 说明 |
|------|------|------|
| **资金安排** | ① 预存法币 | BB在该渠道处预存法币备付金，指令付款时从备付金扣 |
| | ② 按笔转U | BB按笔转USDT给渠道，渠道自己承兑+付款 |
| | ③ 不需预存 | BB自有银行账户，直接转账（资金始终在BB名下） |
| **BB控制力** | 高 | BB自有银行账户、JRL |
| | 中 | 通过EX对接的SP（有合同约束） |
| | 低 | 第三方PSP（合作关系较松散） |
| **承兑能力** | 有 | 渠道能收U自己换法币（OffRamp SP） |
| | 无 | 渠道只能收发法币（纯付款渠道） |
| **EX可见性** | EX可见 | 通过EX对接的渠道，在EX上注册为SP |
| | EX不可见 | BB自有渠道（银行账户等），BB内部管理 |

### 10.4 为什么要做区分（即使本质相同）
```

实际情况：所有渠道都是EX帮助BB对接的
但架构上先做区分，原因：

1. BB未来可能独立发展
   · BB自己对接新渠道，不通过EX
   · 如果今天不区分，未来拆分很痛苦
2. 风险管理不同
   · BB自有银行账户：资金在BB名下，冻结风险低
   · 外部SP结算账户：资金在SP名下，冻结/跑路风险高
   · 风控策略不同（敢口限额、监控频率等）
3. 结算流程不同
   · BB自有渠道：BB内部对账
   · 外部SP：需要跟SP轧差结算
   · 分开管理更清晰
4. 费率结构不同
   · BB自有渠道：BB内部成本
   · 外部SP：SP收费 + BB加价

```

### 10.5 架构区分方式
```

分两层：

第一层：BB 内部渠道（BB Channel）
  · BB自有银行账户
  · BB已有的合作方（如JRL）
  · 特点：BB内部管理，EX视角看到的是“BB Channel”这一个整体
  · EX不关心BB内部用哪个银行/合作方

第二层：EX 平台SP（EX SP）
  · 通过EX平台新接入的外部SP
  · 在EX上注册、管理、对账
  · BB在这些SP处预存备付金（或按笔转U）
  · EX做路由、监控、风控

两层对客户完全不可见。
两层对租户也不可见。
只有BB后台和EX后台能看到区分。

```

## 12. SP 模型扩展：不是所有 SP 都在 BB 下面

> 第11章建立了"所有渠道都是BB的渠道"的统一模型。但有一个真实场景打破了这个假设。

### 12.1 引发问题的场景：人民币结汇

```
场景：客户想把 USDT 变成 CNY（人民币）

问题：
  · 数币相关业务在中国不完全合法
  · BB 不能直接做 USDT → CNY 的承兑/结汇
  · 但有些 SP（如易宝等中国跨境支付公司）可以做人民币结汇

这意味着：
  · 这笔业务不能走 BB 承兑 → BB 不是这个场景的"资金中枢"
  · 必须由特定 SP 来做 → 而且客户必须知道钱要去哪（合规要求）
  · 客户需要自己操作把钱转到该 SP → SP 必须暴露给客户
```

### 12.2 跟现有模型的冲突

```
现有模型的三个核心假设：

  假设①：客户不感知 SP（约束❶）
    → 在结汇场景下：客户必须感知 SP（看到托管钱包）❌

  假设②：所有客户资金先进 BB
    → 在结汇场景下：资金需要从 BB 钱包 → 客户操作 → SP 钱包 ❌
    → BB 不是这笔业务的执行方

  假设③：BB 是资金中枢，SP 是 BB 的渠道
    → 在结汇场景下：SP 不是 BB 的下级渠道
    → SP 跟 BB 是平级的——客户在两边都有钱包 ❌

三个假设全部被打破。说明现有模型不完整。
```

### 12.3 推理：为什么会出现平级 SP

```
根本原因：BB 的能力边界

  BB 的能力 = 数币钱包 + 承兑（USDT ↔ USD/HKD/EUR 等）+ 法币付款
  BB 的能力不包括：
    · 人民币结汇（合规限制）
    · 某些地区/币种的特殊支付能力
    · 某些需要特定牌照才能做的业务

  当客户需要的能力超出 BB 的边界时：
    · BB 不能做 → 不能走"BB承兑→BB选渠道"的路径
    · 需要一个有这种能力的 SP 来做
    · 而且因为 BB 做不了，资金不能先进 BB 再出去
    · 资金需要直接从客户到这个 SP

  → 这个 SP 不是 BB 的渠道，而是跟 BB 平级的独立服务提供方
```

### 12.4 两类 SP 关系

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  类型一：BB 的渠道（现有模型，大部分场景）                            │
│                                                                     │
│    客户 → BB → SP 执行                                              │
│    · 客户不感知 SP                                                   │
│    · BB 做资金中枢，SP 是 BB 的执行方                                │
│    · EX 自动路由，客户无需操作                                       │
│    · 例：USD/HKD/EUR 等常规 OffRamp                                  │
│                                                                     │
│  类型二：平级 SP / 托管钱包（新场景，特殊能力）                       │
│                                                                     │
│    客户 → 自己操作转账 → SP                                          │
│    · 客户感知 SP（页面上看到"托管钱包"）                             │
│    · SP 跟 BB 平级，不是 BB 的下级                                   │
│    · 客户自己选择何时、是否转账                                      │
│    · 例：人民币结汇（客户主动把 USD 转到易宝托管钱包 → 易宝结汇 CNY）│
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| 维度                 | 类型一：BB 的渠道          | 类型二：平级 SP（托管钱包）               |
| -------------------- | -------------------------- | ----------------------------------------- |
| **客户感知**   | ❌ 不感知                  | ✅ 感知（页面看到托管钱包）               |
| **资金路径**   | 客户 → BB → 渠道         | 客户自己操作转账 → SP                    |
| **谁做路由**   | EX 自动路由                | 客户自己选                                |
| **与 BB 关系** | BB 下级（BB 的渠道）       | BB 平级（独立 SP）                        |
| **为什么需要** | 扩大 BB 的地区/币种覆盖    | BB**做不了**的业务（合规/牌照限制） |
| **承兑方**     | BB 或 SP（取决于资金安排） | SP 自己                                   |
| **EX 的角色**  | 路由引擎自动选渠道         | 提供托管钱包入口 + 转账工具               |
| **典型场景**   | OffRamp USD/HKD/EUR        | 人民币结汇、特殊地区法币                  |

### 12.5 托管钱包的具体运作方式

```
以人民币结汇为例：

前提：
  · 客户在 EX 上有 BB 钱包（USD 余额）
  · 某 SP（如易宝）有人民币结汇能力
  · 客户在该 SP 处有一个"托管钱包"（页面上可见）

流程：
  1. 客户在 BB 钱包中有 1000 USD（BB 承兑 USDT 得来的）
  2. 客户在页面上看到"托管钱包 - 易宝（人民币结汇）"
  3. 客户主动发起转账：BB 钱包 → 易宝托管钱包（1000 USD）
  4. 资金到达易宝 → 易宝执行 USD → CNY 结汇
  5. 客户在易宝托管钱包中看到 CNY 余额
  6. 客户发起提现 → CNY 到国内银行卡

关键特征：
  · 步骤 3 是客户自己发起的（不是系统自动路由）
  · 客户明确知道钱去了"易宝"（SP 暴露）
  · BB 在这个过程中只是"资金来源"，不是"执行方"
  · 这是一个体系外的账户，但在 EX 页面上有入口

为什么客户必须自己操作：
  · 合规要求：涉及中国人民币，数币相关操作敏感
  · BB 不能代替客户做这个决定
  · 客户需要知情同意"我要把钱转到这个 SP"
```

### 12.6 对统一模型的修正

```
原来的统一模型（第11章）：
  BB = 资金中枢，名下有 N 个渠道
  所有客户资金先进 BB
  EX 做路由，选渠道执行

修正后的统一模型：
  BB = 主要资金中枢（覆盖 BB 能力范围内的业务）

### 12.7 对约束❶的重新理解

```
原始约束❶：客户不感知 SP
  → 目的：防止客户绕过 EX 直接找 SP

修正后的理解：
  · 在 BB 能力范围内：客户不感知 SP ✅（保持不变）
  · 在 BB 能力范围外：客户必须感知特定 SP ✅（合规+业务要求）

这不是对约束❶的违反，而是约束❶的边界条件：
  · 约束❶ 适用于"BB 能做的业务"→ SP 是 BB 的渠道 → 不暴露
  · 约束❶ 不适用于"BB 做不了的业务"→ SP 是独立服务方 → 必须暴露

为什么暴露平级 SP 不会导致"客户绕过 EX"：
  · 平级 SP 提供的是特殊能力（如人民币结汇），不是通用 OffRamp
  · 客户不能只用平级 SP 完成全部业务（还需要 BB 做 USDT → USD 承兑）
  · BB + 平级 SP 是互补关系，不是替代关系
  · 客户还是需要 EX 作为入口和管理平台
```
