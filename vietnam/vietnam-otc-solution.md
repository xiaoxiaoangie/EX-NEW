# 越南 OTC 资金运营与风控解决方案

> 文档类型：OTC 专项解决方案 / 讨论稿  
> 适用市场：越南  
> 目标客户：非持牌 OTC、货代、大代理、具备自有账户和钱包的跨境资金运营方  
> 关联文档：`ProductArchetect/ABE-positioning-strategy.MD`、`ProductArchetect/ex-three-layer-solution.md`、`vietnam/vietnam-payment-web3-report.md`

---

## 0. TL;DR

越南 OTC 客户的核心问题不是“缺一个 KYT 工具”，而是缺一套能把参考汇率展示、收款、付款、打币、头寸、风控、对账全部线上化的轻量资金运营系统。

OTC 方案的产品主张：

> 把 OTC 从“人工盯账 + 微信报价 + 人工打币 + 靠经验判断风险”，升级为“可风控、可展示参考汇率、可看头寸、可自动入账、可调拨、可对账”的资金运营台。

方案由四层能力组成：

1. **风控托管层**：A/B 托管 KYB/KYT/AML 判断，按轻量版、标准版分层。
2. **订单运营层**：参考汇率展示、交易试算、入账识别、风控判断、出币/付款、订单状态流。
3. **资金管理层**：多 PSP、多银行、多钱包余额聚合，头寸预警，调拨建议，备付金管理。
4. **渠道连接层**：连接法币 VA、越南本地银行/PSP、交易所钱包、三方托管钱包、B 数币钱包、A/B 承兑和跨境通道。

## 1. 客户画像

### 1.1 目标 OTC 客户

适合以下类型客户：

| 类型 | 典型特征 | 主要需求 |
| --- | --- | --- |
| OTC / 承兑商 | 已经在做 USDT 与法币兑换，有自有钱包和若干法币账户 | 风控、参考汇率展示、头寸、自动入账、打币和对账 |
| 货代 | 以物流和贸易服务切入，帮客户处理跨境收付或结算 | 不懂数币风控，希望有人告诉他能不能做 |
| 大代理 | 掌握客户资源，通常不以牌照主体展业，也不具备完整系统 | 白牌运营、客户管理、订单管理、资金安全 |
| 有自有渠道的贸易服务商 | 已在宝付、光子易、Airwallex、本地银行、交易所开账户 | 把分散账户变成一个资金运营台 |

不放在 OTC 方案中的对象：

- **持牌 MSB / 持牌支付机构**：只要它用牌照展业、对外提供资金服务，就应归入 PSP/持牌租户方案，而不是 OTC 方案。
- **屯牌照但不用牌照展业的主体**：空有牌照但不基于牌照提供资金服务，不按持牌 SP 处理；如果它只是做资源或代理，也不作为 OTC 方案的核心目标客户。
- **需要完整自定义风控体系的成熟持牌机构**：应进入 PSP/持牌机构方案，由其按自身牌照责任承接风控和资金服务。

### 1.2 客户已有基础

OTC 通常不是从零开始，他们已经有一些“能跑业务”的资产：

- 法币账户：本地银行账户、支付公司 VA、跨境支付公司账户；
- PSP 账户：宝付、光子易、Airwallex、iPayLinks、其他跨境支付机构；
- 数币钱包：交易所钱包、三方托管钱包、自有热钱包/冷钱包；
- 客户资源：贸易客户、货代客户、电商客户、代理客户；
- 人工 SOP：微信/Telegram 报价牌、人工盯账、人工确认、人工打币、Excel 对账。

### 1.3 核心痛点

| 痛点 | 当前做法 | 风险 / 成本 |
| --- | --- | --- |
| 资金安全 | 靠经验看客户、看地址、看交易对手 | 脏钱、冻卡、链上高风险地址、被动涉案 |
| 风控能力弱 | 买一个 KYT 工具也看不懂分数含义 | 工具不能转化为决策，仍然不知道能不能做 |
| 入账靠人工 | 人盯银行后台、PSP 后台、交易所钱包 | 时效慢，漏单，错配，夜间无人处理 |
| 参考汇率展示靠人工 | 人在微信/Telegram 发买卖价，客户到处比价后再决定跟谁做 | 无法标准化，客户体验差，价格更新不及时，容易因库存和市场波动亏点 |
| 头寸不可视 | 不知道各账户有多少钱、可用多少钱 | 付款时才发现余额不足，影响时效 |
| 调拨混乱 | 微信/Telegram 通知财务调拨 | 无凭证、无状态、无对账、责任不清 |
| 对账靠 Excel | 订单、银行流水、链上流水分散 | 对账耗人，差错难定位 |
| 合规留痕不足 | 缺客户资料、地址风险、交易背景记录 | 出问题时无法解释这单为什么可做 |

## 2. 方案定位

### 2.1 EX 给 OTC 的定位

EX 不应对 OTC 只卖“系统”或“通道”，而应卖一套运营能力：

> OTC 资金运营台 + 托管风控服务 + 多账户/多钱包/多 PSP 聚合 + A/B 承兑和跨境能力。

对 OTC 的一句话：

> 你继续经营客户和业务，EX 帮你把风控、参考汇率展示、入账、打币、头寸、调拨、对账系统化。

### 2.2 A/B/E 分工

| 角色 | 在 OTC 方案中的职责 |
| --- | --- |
| **EX** | 系统、账本、参考汇率/报价牌、订单、风控结果展示、对账、调拨建议、渠道适配 |
| **A / iPayLinks** | 法币 VA、跨境收付、POBO、FX、中国通道、多币种结算 |
| **B / BlancBlock** | 数币钱包、KYT、USDT 承兑、链上监控、On/Off Ramp、数币清算 |
| **OTC 自有账户** | OTC 自己在 PSP、银行、交易所、钱包中的账户和资金 |
| **外部 PSP/银行/钱包** | 作为法币或数币执行渠道，承接入金、出金、余额和流水 |

### 2.3 资金主体边界

OTC 方案必须明确：

- EX 不持有客户资金；
- A/B 只在合规范围内作为资金主体或承兑主体；
- OTC 自有银行账户、PSP 账户、交易所钱包中的资金属于 OTC 或其对应资金主体；
- 系统可做统一视图，但不能把不同资金主体的余额合并成一个无边界余额；
- 跨主体调拨必须形成明确交易、凭证、状态和责任记录。

## 4. 风控产品分层

### 4.1 分层原则

OTC 风控要分版本，但不做过重的专业风控工作台。OTC 客户通常不适合自己配置完整策略，也不应承担类似持牌机构的风控责任；系统重点是让 A/B 托管判断，并把结果转化成 OTC 能执行的操作。

| 版本 | 定位 | 决策权 | 适合客户 | 客户看到什么 |
| --- | --- | --- | --- | --- |
| **轻量版** | 托管风控 | A/B 决策 | 货代、小代理、刚接触数币的 OTC | 可做 / 不可做 / 待人工确认 |
| **标准版** | 托管 + 工具解释 | A/B 主决策，客户理解原因 | 有 OTC 经验但风控能力弱的客户 | 风险等级、分数含义、基础原因、处理建议 |

一句话：

> 轻量版卖“省心”，标准版卖“看得懂”；更专业的持牌机构风控能力放到 PSP/持牌租户方案中。

### 4.2 轻量版：托管风控

#### 定位

轻量版适合不具备专业风控能力的 OTC、货代、大代理。客户不需要理解 KYT 分数，也不需要自己判断风险，只需要知道这单能不能做。

#### 产品形态

客户在订单中提交：

- 客户名称或基础身份信息；
- 交易金额；
- 币种；
- 钱包地址；
- 交易方向：U → 法币、法币 → U；
- 收付款账户；
- 必要时提交交易背景说明。

系统返回：

- `可做`；
- `不可做`；
- `待人工确认`。

#### 功能清单

| 功能 | 说明 |
| --- | --- |
| KYT 后台托管 | A/B 后台调用 KYT，客户不直接看复杂报告 |
| 基础名单筛查 | 制裁、黑名单、高风险地址基础筛查 |
| 交易规则判断 | 金额、频率、地址类型、客户风险等级 |
| 人工复核入口 | 高风险或不确定订单进入 A/B 人工复核 |
| 简化结果展示 | 只展示能不能做，不展示复杂链上路径 |
| 风控留痕 | 系统保留判断结果、时间、调用记录、处理人 |

#### 不展示内容

- 不展示完整链上路径；
- 不展示详细风险实体；
- 不让客户自定义复杂规则；
- 不让客户覆盖 A/B 的拒绝判断；
- 不展示容易被绕开的具体规则阈值。

#### 价值主张

> 把风控判断托管给 A/B，OTC 只负责经营客户和订单。

### 4.3 标准版：托管风控 + 基础工具

#### 定位

标准版适合已经有 OTC 经验，希望理解“为什么可做/不可做”，但仍不想自己搭建完整风控团队的客户。

#### 产品形态

系统返回：

- 风险等级：低 / 中 / 高；
- KYT 分数区间；
- 分数含义；
- 最小化风险原因；
- 处理建议。

#### 功能清单

| 功能 | 说明 |
| --- | --- |
| KYT 分数解释 | 把供应商分数转化成业务可理解语言 |
| 风险原因展示 | 制裁命中、混币器关联、高风险交易所、诈骗地址关联、资金路径异常等 |
| 地址风险等级 | 地址维度风险标签 |
| 客户风险等级 | 客户维度风险标签 |
| 订单风险等级 | 订单综合风险等级 |
| 处理建议 | 放行、补材料、人工复核、拒绝 |
| 基础报表 | 高风险订单数、拒绝率、人工复核量 |

#### 控制边界

客户可以理解原因，但不能随意覆盖底层风控：

- A/B 保留最终拒绝权；
- 高风险订单必须人工复核；
- 部分规则阈值不对客户透明；
- 客户可看到归因，不看到全部模型细节。

#### 价值主张

> 不只告诉你能不能做，还告诉你为什么，以及下一步怎么处理。

## 5. 订单产品功能

### 5.1 订单生命周期

```text
登录查看参考汇率 / 报价牌
→ 选择交易方向和金额
→ 系统试算可得金额、费用和预计到账
→ 客户确认交易
→ 等待入金
→ 入账识别
→ 风控判断
→ 付款 / 出币
→ 完成
→ 对账归档
```

异常状态：

```text
参考汇率刷新
交易试算失效
入账金额不符
付款人不匹配
链上未确认
KYT 高风险
人工复核中
付款失败
出币失败
退款中
订单取消
```

### 5.2 参考汇率 / 报价牌

OTC 的客户通常会同时看多家 OTC 的价格，谁的价格好、到账快、可信度高，就跟谁做。因此这里不设计逐单人工沟通和人工价格承诺流程，而是提供一套登录后即可查看的参考汇率 / 报价牌。

报价牌展示内容：

- 交易方向：买 U、卖 U、U 换 VND、VND 换 U、USD 换 U 等；
- 买入价；
- 卖出价；
- 参考汇率；
- 汇率更新时间；
- 最小交易金额；
- 最大可交易金额；
- 当前可用库存；
- 支持的收付款方式；
- 预计到账时效；
- 手续费或加点说明；
- 风险提示。

系统动作：

- 读取市场参考价；
- 读取 OTC 自有加点规则；
- 读取当前库存和可用头寸；
- 读取渠道成本和预计时效；
- 按客户等级展示可见价格；
- 按币种、方向、金额展示可交易区间；
- 定时刷新价格，不做人工固定价格承诺。

### 5.3 交易试算

客户看到参考汇率后，可以直接输入金额做交易试算。

试算字段：

- 交易方向；
- 输入金额；
- 输入币种；
- 收款方式；
- 付款方式；
- 钱包地址；
- 预计到账时效。

试算结果：

- 参考汇率；
- 预计可得金额；
- 手续费；
- 渠道成本；
- 预计到账时间；
- 本次交易可用余额是否充足；
- 是否需要人工复核；
- 风险提示。

重要口径：

- 参考汇率不是最终成交价；
- 客户确认交易时，以提交交易瞬间的系统价格为准；
- 若市场价格或库存变化导致试算失效，系统提示客户重新试算；
- 真正的承兑交易由现有承兑产品执行，OTC 系统只提供价格展示、订单和运营管理能力。

### 5.4 入账识别

入账来源：

- PSP VA；
- 本地银行账户；
- A 的法币账户；
- 交易所钱包；
- 托管钱包；
- B 数币钱包；
- 自有钱包。

匹配规则：

- 金额匹配；
- 订单号 / 备注匹配；
- 付款人匹配；
- 时间窗口匹配；
- 钱包地址匹配；
- 链上 TxID 匹配。

异常池：

- 金额不符；
- 无订单入账；
- 付款人不一致；
- 重复入账；
- 链上到账但确认数不足；
- 入账账户不符合订单指定账户。

### 5.5 付款 / 出币

付款动作：

- 法币付款；
- 同 PSP A2A；
- 银行转账；
- POBO；
- USDT 出币；
- 内部账户划转。

控制规则：

- 金额超过阈值需要审批；
- 高风险订单不能自动出款；
- 新客户首单需要人工确认；
- 钱包地址首次使用需要二次确认；
- 出款前再次检查余额和风控状态；
- 敏感操作需要 2FA。

### 5.6 收款业务能力

OTC 收款能力分两种模式：**OTC 代收**和**客户自有 VA / 地址收款**。

#### 5.6.1 OTC 代收

OTC 代收是指：OTC 把自己已经在其他机构开好的 VA、银行账户、收款码或钱包地址配置到系统中，让 OTC 的客户直接向这些账户或地址汇款。

可配置的收款载体：

- OTC 在其他支付机构开的 VA；
- OTC 在银行开的账户；
- OTC 在银行或钱包体系生成的收款码；
- OTC 在交易所或托管钱包中的充值地址；
- OTC 自有热钱包 / 冷钱包地址；
- B 或其他数币渠道提供的钱包地址。

系统需要支持：

- 收款载体配置：账户名、账号/VA、银行、币种、收款码、钱包地址、所属渠道；
- 收款载体标签：OTC 自有、A/B、外部 PSP、银行、交易所、托管钱包；
- 收款载体状态：启用、停用、维护中、限额不足、风控暂停；
- 对客户展示可用收款方式；
- 客户按订单向指定 VA / 账户 / 码 / 地址汇款；
- 系统通过 Webhook、轮询、流水导入或链上监听识别入账；
- 入账后自动匹配订单、更新余额、触发风控和后续付款/出币流程。

适用场景：

- OTC 希望先用自己的账户体系承接客户入金；
- 客户没有自己的 VA 或钱包地址；
- OTC 希望统一管理客户入账和后续承兑执行；
- 渠道暂不支持给客户单独开 VA 或独立地址。

#### 5.6.2 客户自有 VA / 地址收款

客户自有 VA / 地址收款是指：如果 OTC 已经对接了某个渠道，且该渠道支持给客户开 VA、收款账户或钱包地址，则客户可以直接在系统中申请自己的 VA / 地址，用自己的收款载体收款。

可申请的收款载体：

- 客户名下 VA；
- 客户名下银行收款账户；
- 客户专属收款码；
- 客户专属钱包地址；
- 客户在托管钱包或交易所体系下的子地址。

系统需要支持：

- 客户提交 VA / 地址申请；
- OTC 或渠道完成客户资料审核；
- 调用已对接渠道创建 VA / 地址；
- 将 VA / 地址绑定到客户和币种；
- 入账直接映射到客户；
- 展示客户自己的收款流水、余额和对账报表；
- 当渠道不支持自动开户时，支持人工录入客户 VA / 地址并留痕。

适用场景：

- OTC 已经对接的 PSP / 银行 / 钱包渠道支持客户级 VA 或地址；
- 客户希望用自己名义或专属标识收款；
- 需要提升入账识别准确率，减少 OTC 大账户下的流水匹配压力；
- 客户交易频率高，需要独立账务和报表。

### 5.7 付款业务能力

OTC 付款能力同样分两种模式：**OTC 代付**和**客户自有 POBO / 出款能力**。

#### 5.7.1 OTC 代付

OTC 代付是指：OTC 把自己已经对接好的付款渠道、银行账户、PSP 账户、钱包和头寸接入系统，系统获取这些渠道的余额和可用头寸，并通过这些渠道为订单执行出款。

可配置的付款载体：

- OTC 自己的银行账户；
- OTC 在 PSP 的付款账户；
- OTC 已开通的 POBO 付款通道；
- OTC 的交易所提现能力；
- OTC 的托管钱包出币能力；
- OTC 自有钱包；
- A/B 或其他能力方提供的付款/出币能力。

系统需要支持：

- 获取渠道余额和可用头寸；
- 按币种、国家、金额、收款人类型筛选可用出款渠道；
- 发起银行付款、同 PSP A2A、POBO、钱包出币；
- 出款前检查订单状态、风控状态、余额和限额；
- 大额或高风险出款进入审批；
- 出款状态回传：处理中、成功、失败、退回；
- 出款流水与订单自动对账；
- 出款失败后支持重试、换渠道或退款。

适用场景：

- 客户只关心到账，不要求自己发起 POBO；
- OTC 希望统一使用自己的渠道头寸完成付款；
- 需要根据余额、限额、成本、时效选择出款渠道；
- OTC 需要把多个渠道的头寸统一管理。

#### 5.7.2 客户自有 POBO / 出款能力

客户自有 POBO / 出款能力是指：如果客户自己已经在某个渠道下开通了 POBO 或出款能力，且 OTC 已经完成该渠道对接，则客户可以通过系统请求该渠道直接发起 POBO 出款。

系统需要支持：

- 客户申请开通 POBO / 出款产品；
- 收款人管理：收款人名称、账号、银行、国家、币种、用途；
- 付款用途和材料上传；
- 调用已对接渠道的 POBO 出款接口；
- 展示付款状态和失败原因；
- 客户级付款报表；
- 渠道级余额、限额和手续费展示；
- 客户自有 POBO 与 OTC 代付能力分开记账、分开对账。

适用场景：

- 客户已经有自己的 POBO 权限或出款账户；
- 客户希望自己发起供应商付款、工资付款、平台付款；
- OTC 只是提供渠道连接和系统能力；
- 需要保持客户资金主体和 OTC 自有资金主体分离。

关键边界：

- OTC 代付使用 OTC 自己的渠道和头寸；
- 客户自有 POBO 使用客户自己的渠道权限和资金主体；
- 两类余额不能混同展示；
- EX 只做系统、路由、账本和状态记录，资金动作由对应渠道或资金主体授权执行。

### 5.8 订单对账

每笔订单应形成完整对账链路：

```text
订单
→ 参考汇率 / 交易试算
→ 入账流水 / 链上 TxID
→ 风控结果
→ 付款流水 / 出币 TxID
→ 手续费
→ 汇率
→ 利润
→ 操作记录
```

报表：

- 日订单报表；
- 客户交易报表；
- 币种汇总报表；
- 渠道入账报表；
- 钱包出币报表；
- 利润报表；
- 异常订单报表；
- 风控拦截报表。

## 6. 资金管理产品功能

### 6.1 余额视图

余额视图需要按四个维度拆：

| 维度 | 示例 |
| --- | --- |
| 资金主体 | OTC 自有、A、B、外部 PSP、银行 |
| 币种 | VND、USD、CNY、USDT、USDC |
| 账户 | VA、银行账户、交易所钱包、托管钱包、自有钱包 |
| 状态 | 可用、冻结、在途、待结算、已锁定 |

前台可展示：

- 总资产视图；
- 法币余额；
- 数币余额；
- 各账户余额；
- 可用于出款的余额；
- 已被订单锁定的余额；
- 待确认入账；
- 待结算余额。

### 6.2 头寸管理

头寸管理要回答三个问题：

1. 现在有多少钱？
2. 哪些钱可以马上用？
3. 哪些币种/账户需要补？

功能：

- 币种头寸；
- 账户头寸；
- 当日预计收款；
- 当日预计付款；
- 订单锁定头寸；
- 可垫付额度；
- 头寸缺口；
- 补仓建议。

### 6.3 调拨建议

调拨场景：

- VND 付款账户余额不足；
- USDT 钱包余额不足；
- 某 PSP 入账多但付款少，另一个 PSP 付款多但余额少；
- 大额订单即将执行，需要提前准备头寸；
- A/B 承兑需要在法币和数币之间调平。

调拨建议字段：

- 来源账户；
- 目标账户；
- 币种；
- 金额；
- 预计费用；
- 预计到账时间；
- 是否需要换汇；
- 是否需要 KYT/KYB 复核；
- 调拨原因；
- 操作人和审批人。

### 6.4 资金主体隔离

系统可以做统一资金视图，但展示和账务必须区分：

- OTC 自有资金；
- OTC 客户资金；
- A 客资；
- B 数币余额；
- 外部 PSP 余额；
- 银行余额；
- 在途调拨资金。

跨主体调拨必须形成结算交易，不允许只做后台数字搬移。

## 7. 渠道能力要求

### 7.1 法币渠道能力

| 能力 | 说明 | 优先级 |
| --- | --- | --- |
| OTC 代收账户配置 | 配置 OTC 在其他机构开的 VA、银行账户、收款码 | 高 |
| 客户 VA 申请 | 若 OTC 已对接渠道支持开户，客户可申请自己的 VA / 收款账户 | 高 |
| 入账通知 | PSP/银行入账后 Webhook 或轮询通知 | 高 |
| 余额查询 | 查询账户可用余额、冻结余额、待结算余额 | 高 |
| 流水查询 | 拉取入账、出款、退款、手续费流水 | 高 |
| OTC 代付 | 使用 OTC 已对接渠道和头寸发起银行付款、同 PSP A2A、POBO | 高 |
| 客户 POBO 出款 | 客户如有自己的 POBO 权限，系统请求已对接渠道发起 POBO 出款 | 高 |
| 付款状态 | 成功、失败、处理中、退回 | 高 |
| 退款 | 原路退回或指定账户退款 | 中 |
| 换汇 | VND/USD/CNY 等换汇、参考汇率获取 | 中 |
| 账户校验 | 收款人姓名、账号、银行校验 | 中 |

### 7.2 数币渠道能力

| 能力 | 说明 | 优先级 |
| --- | --- | --- |
| OTC 钱包地址配置 | 配置 OTC 自有钱包、交易所钱包、托管钱包地址 | 高 |
| 客户钱包地址申请 | 若渠道支持客户级地址，客户可申请自己的收币地址 | 高 |
| 链上监听 | 监听入金、确认数、TxID | 高 |
| KYT | 地址和交易风险识别 | 高 |
| OTC 代出币 | 使用 OTC 自有钱包、托管钱包或 B 的能力发起 USDT/USDC 出币 | 高 |
| 客户自有钱包出币 | 客户如有自己的钱包/托管权限，系统记录或请求渠道发起出币 | 中 |
| 手续费估算 | 估算链上 gas 或手续费 | 中 |
| 热/冷钱包管理 | 热钱包余额、冷钱包补充、限额 | 中 |
| 交易所钱包接入 | 交易所充值/提现状态同步 | 中 |
| 托管钱包接入 | 三方托管钱包余额、流水、出入金 | 中 |

### 7.3 渠道接入方式

| 接入方式 | 适用场景 | 说明 |
| --- | --- | --- |
| API | PSP、托管钱包、交易所、A/B 能力 | 最优方式，支持自动化 |
| Webhook | 入账、付款状态、链上到账 | 用于实时通知 |
| 轮询 | 无 Webhook 的 PSP/银行 | 作为兜底 |
| 文件导入 | 银行流水、Excel 对账 | 早期兼容 |
| 手工录入 | 暂不开放 API 的账户 | MVP 可支持，但要留痕 |

### 7.4 渠道评分

每个渠道需要有评分，供参考汇率展示、路由和调拨参考：

- 成本；
- 到账时效；
- 成功率；
- 单笔限额；
- 日限额；
- 风控稳定性；
- 冻卡风险；
- 对账质量；
- API 稳定性；
- 客服和异常处理能力。

## 9. 典型业务流程

### 9.1 法币买 U

```text
客户登录查看参考汇率 / 报价牌
→ 输入买 U 金额并完成交易试算
→ 客户确认交易
→ 客户向指定 VA / 银行账户付款
→ 系统识别入账
→ A/B 托管风控判断
→ 风控通过
→ B 或 OTC 钱包出 USDT
→ 系统记录 TxID
→ 订单完成并对账
```

### 9.2 U 换法币

```text
客户登录查看参考汇率 / 报价牌
→ 输入卖 U 金额并完成交易试算
→ 客户确认交易
→ 客户向指定钱包地址转 USDT
→ 链上监听到账
→ KYT 检查地址和交易
→ 风控通过
→ 系统发起法币付款
→ PSP/银行回传状态
→ 订单完成并对账
```

### 9.3 大额订单提前备付

```text
客户预约大额交易
→ 系统检查当前 VND/USD/USDT 头寸
→ 发现付款账户余额不足
→ 生成调拨建议
→ 财务确认调拨
→ 审批人审批
→ 执行调拨
→ 到账后锁定头寸
→ 客户确认交易后执行
```

## 11. 分期落地

### 一期：MVP

目标：先解决“能不能做、有没有到账、有没有出币、能不能对上账”。

范围：

- OTC Risk Lite；
- 基础订单管理；
- 参考汇率 / 报价牌展示；
- 交易试算；
- 手工/半自动账户余额；
- PSP/钱包入账通知；
- KYT 托管判断；
- 出款/出币手动确认；
- 基础对账报表。

### 二期：标准化运营

目标：让 OTC 能稳定线上运营。

范围：

- OTC Risk Standard；
- KYT 分数解释；
- 客户风险等级；
- 自动入账匹配；
- 多账户余额视图；
- 参考汇率规则；
- 头寸预警；
- 异常订单池；
- 利润报表。

### 三期：资金管理深化

目标：把 OTC 运营升级为更完整的资金管理系统，但不开放过重的专业风控工作台。

范围：

- 调拨建议；
- 备付金管理；
- 三方对账；
- 渠道评分；
- 大额订单审批；
- 人工复核队列；
- 自动化付款/出币试点。

### 四期：Agent 化运营

目标：降低人工运营成本。

范围：

- 参考汇率 Agent；
- 对账 Agent；
- 调拨 Agent；
- 风控复核 Agent；
- 客服通知 Agent。

## 12. 风险与待确认事项

### 12.1 风险

- OTC 业务天然靠近 AML 和冻卡风险，必须坚持 A/B 底线风控；
- 轻量版不能让客户误以为“系统通过就绝对无风险”；
- 标准版展示风险原因时，需要避免把规则暴露过细；
- OTC 不提供持牌机构级别的完整自定义风控工作台；持牌 MSB/支付机构应归入 PSP/持牌租户方案；
- 多账户聚合可能受限于 PSP/银行 API 开放程度；
- 交易所钱包、三方托管钱包接入可能受限于权限和合规要求；
- 调拨功能必须先定义资金主体和责任边界。

### 12.2 待确认

- 首批目标 OTC 客户名单；
- 客户已有账户和钱包类型；
- 宝付、光子易、Airwallex 等渠道 API 可用性；
- 越南本地银行/PSP 是否支持 API 或流水导出；
- KYT 供应商价格和可转售条款；
- KYB 供应商价格和越南主体覆盖；
- OTC 对月费、KYT 单价、KYB 单价的接受度；
- A/B 可托管风控的责任边界；
- USDT 出币由 B 执行还是 OTC 自有钱包执行的场景划分。

## 13. 对外一句话

> EX OTC 方案不是卖一个 KYT 工具，而是帮 OTC 把风控、参考汇率展示、交易试算、入账、出款、打币、头寸和对账变成一套可运营的线上系统。
