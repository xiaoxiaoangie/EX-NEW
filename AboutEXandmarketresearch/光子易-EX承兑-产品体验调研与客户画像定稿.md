# 光子易产品体验调研 & EX 客户画像+体验方案定稿

> **文档目的**：以「光子易（PhotonPay）」为对标，调研 **法币系统搬入数币能力** 的产品体验；对比 EX/BB 现状；并 **定稿 EX（含 BB）的客户画像与产品定位**，据此重新定义菜单与产品架构。

---

## 一、客户画像定稿（含 B）

### 1.1 画像

**核心客户 = 做跨境支付的客户**，覆盖：

- **货物贸易**（进出口、跨境电商备货/收款）
- **服务贸易**（SaaS、广告、佣金、咨询、留学教育等）
- **B2B**（企业对企业的跨境结算）
- **B2C**（平台对海外消费者的收/付）

### 1.2 定位（一句话）

> **跨境支付平台；crypto 只是跨境支付的一种方式（一种通道/清算媒介），不是目的。**

### 1.3 由画像推导的体验基础

- 所有体验围绕跨境的 **「收、付、兑」** 三个动作组织，而非围绕「数币 vs 法币」的技术分类；
- 客户心智是 **"我要跨境收一笔款 / 付一笔款 / 把 A 币换成 B 币"**，底层用法币通道还是 crypto 通道 **由系统编排、对客户透明**；
- crypto（USDT/USDC 等）与法币在同一收付兑框架内 **平权呈现**，不单独割裂成一个"数字资产"孤岛。

## 二、光子易（PhotonPay）产品体验调研

> 见截图 2（JINHONG TRADE 账户视图，PHOTONPAY 品牌）。

### 2.1 菜单结构

| 分区     | 菜单项                                                                             |
| -------- | ---------------------------------------------------------------------------------- |
| 顶部     | 首页、概览、报表                                                                   |
| 产品总览 | 账户、**数字资产**、汇率看板、换汇、人民币业务、付款、往来、发卡、请款、收单 |
| 底部     | 开发者                                                                             |

### 2.2 体验特点

- **底座是法币系统，把数币能力"搬进来"**：整体信息架构、账户模型、收付逻辑均沿用成熟法币产品；
- **数币被收敛在「数字资产」一个入口**：除「数字资产」承载 crypto（USDT/USDC 充值/提现/持有）外，**其余菜单（付款、往来、发卡、请款、收单、换汇、人民币业务）几乎都是法币语境**；
- **优点**：法币侧体验成熟、完整、专业（换汇、人民币业务、请款、收单齐全），客户上手无障碍；
- **短板（对 EX 的启示）**：数币是"附加模块"而非"一等公民"，crypto 与法币是 **两套并行体验**，没有真正把 crypto 融进"收付兑"主线——**这恰是 EX 差异化的机会点**。

## 三、EX / BB 现状体验调研

> 见截图 1（EX/BB 商户端菜单）。

### 3.1 菜单结构

| 分区       | 菜单项                                                         |
| ---------- | -------------------------------------------------------------- |
| 资产       | 资产列表、充币、提币、法币充值、法币提现、收款人管理、收款工具 |
| 交易       | 买入数币、卖出数币                                             |
| 开发者管理 | API 申请                                                       |
| 我的卡片   | 卡片列表、卡订单、卡账单                                       |
| 数币收单   | 收单交易查询、结算单查询、退款单查询、支付配置                 |
| 设置中心   | 个人信息                                                       |

### 3.2 现状问题

- **有资产、有买入/卖出、有收付款，但没有清晰的"画像定位"**：菜单是按 **能力/模块** 罗列（资产、交易、卡片、数币收单），不是按 **客户心智（收/付/兑）** 组织；
- **crypto 与法币割裂**：充币/提币 与 法币充值/法币提现 并列但分散；"买入数币/卖出数币"独立成"交易"区，未与"兑"统一；
- **"数币收单"命名把技术形态暴露给客户**：客户要的是"收单/收款"，不关心底层是数币还是法币；
- **缺少跨境支付客户真正需要的主线**：如统一的收款产品矩阵（VA/本地钱包/扫码/web3 地址）、统一付款矩阵，目前散落或缺失。

## 四、对标结论

| 维度         | 光子易               | EX/BB 现状          | EX 目标                                   |
| ------------ | -------------------- | ------------------- | ----------------------------------------- |
| 底座         | 法币系统 + 数币模块  | 数币系统 + 法币补充 | **跨境支付统一底座，法币/数币平权** |
| 组织逻辑     | 按产品模块           | 按能力模块          | **按客户心智：收/付/兑**            |
| crypto 定位  | 附加"数字资产"       | 交易(买入卖出)独立  | **跨境支付的一种通道，融入主线**    |
| 客户心智契合 | 中（法币强、数币弱） | 弱（无清晰画像）    | **强（围绕跨境收付兑）**            |

**核心洞察**：光子易证明了"法币成熟体验 + 数币模块"可行，但其数币仍是孤岛；EX 的机会 = **以跨境支付客户画像为纲，把 crypto 与法币融进统一的"收付兑"主线**，做到客户不感知底层币种/通道差异。

## 五、建议：基于客户画像重定义菜单与产品

> 原则：**按客户心智（账户 → 收 → 付 → 卡 → 收单）组织，法币/数币在每个产品内平权呈现，通道对客户透明。**

### 5.1 基础产品矩阵

| # | 产品               | 内容                                                                  |
| - | ------------------ | --------------------------------------------------------------------- |
| 1 | **账户**     | 法币账户 + 数币钱包；**充值 / 提现（同名充提，法币&数币统一）** |
| 2 | **收款产品** | web3 地址收、VA 收、本地钱包收、扫码收                                |
| 3 | **付款产品** | 付 web3 地址、付本地行、扫码付、付本地钱包                            |
| 4 | **发卡**     | VCC（虚拟卡）、实体卡                                                 |
| 5 | **收单**     | 聚合收银台、Invoice Payment（发票收款）、循环订阅（Subscription）     |

> 说明：**兑（换汇 / OnRamp / OffRamp / 币种互换）** 建议作为贯穿三大动作的能力，既可独立入口（Exchange），也在收/付流程中内嵌自动编排。

### 5.2 建议菜单结构（对客户呈现）

| 分区           | 菜单项                                                                |
| -------------- | --------------------------------------------------------------------- |
| **账户** | 资产总览、法币账户、数币钱包、充值、提现、兑换（换汇/OnRamp/OffRamp） |
| **收款** | 收款工具（web3 地址 / VA / 本地钱包 / 扫码）、收款人管理、收款记录    |
| **付款** | 付款（web3 地址 / 本地行 / 扫码 / 本地钱包）、收款人、付款记录        |
| **发卡** | 卡片列表、卡订单、卡账单（VCC / 实体卡）                              |
| **收单** | 聚合收银台、Invoice Payment、循环订阅、订单/结算/退款查询             |
| **其他** | 报表、开发者、设置中心                                                |

### 5.3 与现状的关键差异

- **取消"数币收单""买入/卖出数币"这类暴露技术形态的命名**，统一为"收单""兑换"；
- **法币充值/提现 与 充币/提币 合并为账户的"充值/提现"**，按币种切换而非按系统割裂；
- **收/付从散点整合为两个完整产品矩阵**，覆盖 web3 地址 / 本地行 / 本地钱包 / 扫码 / VA 全形态；
- **crypto 承兑（OnRamp/OffRamp）内嵌进收付兑主线**，对客户透明（BB 承兑在后台完成）。
