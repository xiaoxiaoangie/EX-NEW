# Cregis 接入：问题分析 - 解决方案 - 风险

> **文档定位**：分析将 **Cregis** 作为 OffRamp 通道接入的 **协议 / 合规问题**，给出可选 **解决方案** 与对应 **风险**，并记录本次 **沟通中的偏差**，供决策。

---

## 一、背景

1. **能力已验证、成本重新可接受**：此前为测试 Cregis 能力，Porsin 开过一个账号，实测流程很顺（丝滑）；当时因 **费率不理想** 未继续。近期 **USDT/USD 汇率波动较大**，综合成本重新算下来 **可以接受**。
2. **有真实客户需求**：最近正好有客户想走交易，计划用 **Vaulta 白牌** 走一笔。
3. **目标链路（本应简单）**：客户入网到 **BB** → 充 U 到 BB → 发起 **OffRamp**；**Cregis 作为 BB 的通道** 完成 OffRamp 出款。
4. **核心障碍 —— 协议不含 OffRamp**：按 BB 现行合规要求，要用 Cregis 的能力必须 **签协议，且协议须包含 OffRamp 功能**；但 Cregis 的标准协议 **明显不包含 OffRamp**（见 [Cregis Standard Terms of Service](https://support.cregis.com/cregis-wallet-guide/zh-cn/terms_and_conditions/cregis-standard-terms-of-service)）。
5. **FIBD 的反馈 —— OffRamp 是衍生功能、不持牌**：
   - OffRamp 是 Cregis 的一个 **衍生产品**，面向 **直客户**，目的是 **吸引更多 U 进来**，服务其自有的收益类产品；
   - 因此该 **OffRamp 平台开户没有专门协议**；若一定要签，只能签上面的 **钱包协议**，而 **不是 OffRamp 协议**；
   - 根本原因：**Cregis 不持牌**（不承接牌照 / 合规责任）。

**问题本质：** BB 合规要求"签含 OffRamp 的协议"，与 Cregis"OffRamp 无独立协议、且不持牌"之间存在 **结构性冲突**——单靠与 Cregis 签约无法满足 BB 的合规口径。

---

## 二、问题分析

- **协议缺口**：Cregis 只提供钱包协议，**无 OffRamp 协议**，无法直接满足 BB"协议须含 OffRamp"的要求。
- **牌照/责任缺口**：Cregis **不持牌**，OffRamp 作为其衍生功能，**不承接合规与出款兜底责任**。
- **责任链断裂风险**：一旦单笔交易出问题（如出款失败），**缺乏协议约束下的责任追溯路径**——租户 / BB / SP / Cregis 之间无明确协议界定处置与赔付。
- **主体选择问题**：谁作为对 Cregis 的开户与业务主体（BB 自身 / EX 某主体 / Porsin），直接决定合规解释路径与责任归属。

---

## 三、解决方案

### 方案一：EX 某主体 / Porsin 作为 EX 自营 SP 去 Cregis 开户

**思路：** 由 **EX 某个主体 / Porsin** 作为 **EX 自营 SP** 在 Cregis 开户并走业务；**BB 与该自营 SP 之间建立类似 Slash 的关系与协议**（BB → 自营 SP → Cregis）。

```mermaid
flowchart LR
    CU["客户"] --> BB["BB 租户"]
    BB -->|"类 Slash 关系与协议"| SP["EX 自营 SP / Porsin"]
    SP -->|"钱包协议"| CREGIS["Cregis 通道 OffRamp"]
```

**需要解决：**

1. **合规 + 研发配合向 SRO 解释**：需向 SRO 说明该结构，**可能有部分功能要开发**（**待评估**）。
2. **责任链风险（详见第四节）**：Cregis 不做背头存，单笔出问题时的责任回溯路径需靠 BB - 自营 SP 之间协议补足。

### 方案二：直接接 Cregis 的钱包与 OffRamp

**思路：** **全部从 Cregis 走**（客户 / 业务直接对接 Cregis 的钱包与 OffRamp），**不牵扯 BB**。

```mermaid
flowchart LR
    CU["客户 / 租户"] -->|"是否需要协议?"| SP["EX 主体 / Porsin"]
    SP -->|"钱包协议"| CREGIS["Cregis 钱包 加 OffRamp"]
```

**需要解决：**

- **资金风险与方案一相同**（Cregis 不吃牌、无 OffRamp 协议），**区别是不牵扯 BB**；
- **待确认**：**EX 主体 / Porsin 与租户之间是否需要签协议？**（明确责任与合规口径）。

### 方案对比

| 维度        | 方案一（EX 自营 SP 开户）                     | 方案二（直接接 Cregis）             |
| ----------- | --------------------------------------------- | ----------------------------------- |
| 链路        | BB → EX 自营 SP → Cregis                    | 租户 / 客户 → Cregis（不经 BB）    |
| 是否牵扯 BB | **牵扯**，需 BB - SP 类 Slash 协议      | **不牵扯 BB**                 |
| SRO 解释    | 需合规 + 研发配合解释，或有功能开发（待评估） | 相对简单，但仍需明确主体与协议      |
| 协议缺口    | 靠 BB - SP 协议补，但 Cregis 侧仍只有钱包协议 | Cregis 侧仍只有钱包协议             |
| 待确认      | 功能开发范围（待评估）                        | EX 主体 / Porsin 与租户是否需签协议 |

## 四、风险

- **责任链无协议兜底**：Cregis **不持牌**，**单笔交易出问题（如钱出不去）时**：
  - 客户 / 租户（BB / 其他租户）→ 找 EX 某主体 / Porsin → 找 Cregis；
  - **各环节间缺协议约束**，处置与赔付路径不清、责任难追溯。
- **协议合规缺口**：Cregis 只有钱包协议、无 OffRamp 协议，**未必满足 BB 合规"协议须含 OffRamp"的要求**。
- **主体与解释成本**：走自营 SP 需向 SRO 解释并可能开发功能（**范围与工作量待评估**）。
