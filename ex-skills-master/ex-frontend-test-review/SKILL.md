# 测试用例评审流程

## 1. 评审准备

收到测试用例评审请求后，按以下步骤执行：

1. 确定评审范围（全量评审或增量评审）
2. 读取测试用例文档：
   - 全量评审：读取 `docs/tests/full-testcases.md`
   - 增量评审：读取任务目录 `docs/requirements/{JIRA}-*/` 下的增量测试用例文档
3. 读取技术方案文档（`docs/designs/full-technical-design.md`，如有）
4. 读取任务拆解文档（`docs/requirements/task-breakdowns/` 目录下最新文档）
5. 读取后端接口文档（`docs/backend-api/` 目录，如有）
6. 理解页面需求、组件设计、交互逻辑和接口定义

## 2. 评审执行

按优先级从高到低依次检查以下维度，详细检查项见 [checklist.md](references/checklist.md)：

**P0 - 阻断级（必须修复）**
- 页面覆盖度：每个页面组件是否至少覆盖 loading、empty、error、success 四种状态
- 交互场景覆盖：每个用户交互（按钮点击、表单提交、筛选、分页）是否有对应测试用例
- E2E 关键流程：关键用户操作路径是否有 E2E 用例覆盖
- 预期结果正确性：测试用例的预期结果是否与需求和接口定义一致

**P1 - 严重级（强烈建议修复）**
- 组件 Props/Emits 覆盖：每个通用/业务组件的 Props 和 Emits 是否有对应用例
- 边界值覆盖：空数组、超长文本、特殊字符、null/undefined 等边界是否覆盖
- 表单校验覆盖：必填校验、格式校验、长度校验是否有对应用例
- 错误处理覆盖：网络错误、业务错误码、超时等异常场景是否覆盖
- 测试分层合理性：组件测试/单元测试/E2E 测试各层测试重点是否正确

**P2 - 建议级（建议修复）**
- 用例编号规范：编号是否符合 TC-{模块序号}{用例序号} 格式
- 用例命名质量：用例名称是否清晰表达测试意图
- 前置条件完整性：前置条件是否描述充分
- 操作步骤明确性：操作步骤是否具体明确而非模糊描述
- 文档结构规范：文档结构是否符合模板要求

## 3. 评审输出格式

对每个发现的问题，按以下格式输出：

```
### [P{级别}] {问题标题}

**位置**: {测试用例编号或文档章节}
**维度**: {评审维度}
**问题描述**: {具体问题说明}
**修复建议**: {如何修复}
**示例**（如适用）:
{修复后的测试用例内容}
```

## 4. 评审总结

评审完成后输出总结报告：

```

**评审范围**: {全量/增量} | {页面数} 个页面 | {组件数} 个组件 | {用例数} 个测试用例

| 维度 | P0 | P1 | P2 |
```

## 5. 各维度评审要点速查

### 页面覆盖度
- 每个页面组件至少覆盖4种状态：loading、empty、error、success
- 每个页面的路由参数变化场景有对应用例
- 权限控制场景（有/无权限）有对应用例
- 页面组件的生命周期关键行为（挂载时数据加载等）有对应用例

### 交互场景覆盖
- 每个按钮点击操作至少1个用例
- 表单提交覆盖正常提交和校验失败两种场景
- 筛选/搜索操作覆盖有结果和无结果两种场景
- 分页操作覆盖翻页、切换每页条数场景
- 弹窗/抽屉的打开、关闭、确认、取消操作有对应用例
- 列表的选择、批量操作场景有对应用例

### 组件 Props/Emits 覆盖
- 每个通用/业务组件的每个 Prop 至少1个用例
- 未传 Props 时使用默认值的场景有用例
- 每个 Emit 事件至少1个触发用例，验证事件参数正确
- 插槽（默认插槽和具名插槽）的渲染有对应用例

### Composables/Store/Utils 覆盖
- 每个 composable 的每个返回值至少1个用例
- composable 的副作用清理（组件卸载后）有对应用例
- Store 的 state 初始值、actions 调用后状态变更、getters 计算结果有对应用例
- 每个工具函数至少1个正常用例 + 1个边界用例

### 表单校验覆盖
- 每个必填字段至少1个空值提交用例
- 有格式要求的字段（邮箱、手机号等）覆盖格式错误用例
- 有长度限制的字段覆盖超限用例
- 联动校验（如确认密码）有对应用例

### 错误处理覆盖
- 网络请求失败（网络错误、超时）的 UI 反馈有对应用例
- 业务错误码的 UI 反馈有对应用例
- 接口返回空数据时的 UI 展示有对应用例

### E2E 关键流程
- 每个关键用户操作路径（如列表→筛选→详情→编辑→保存）有完整 E2E 用例
- 页面导航和路由跳转有对应用例
- 表单完整流程（填写→校验→提交→反馈）有对应用例

### 测试分层
- 组件测试：聚焦渲染、交互、Props/Emits，Mock API 模块和 Router/Store
- 单元测试：聚焦纯逻辑和数据转换，无或最小依赖
- E2E 测试：聚焦完整用户流程和页面导航

### 预期结果规范
- 组件测试：验证 DOM 结构、文本内容、CSS 类名、事件触发
- 单元测试：验证返回值、状态变更、副作用
- E2E 测试：验证页面跳转、数据展示、操作反馈
- 预期结果描述具体明确，不使用模糊表述
