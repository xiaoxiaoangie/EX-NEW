# 测试用例评审流程

## 1. 评审准备

收到测试用例评审请求后，按以下步骤执行：

1. 确定评审范围（全量评审或增量评审）
2. 读取测试用例文档：
   - 全量评审：读取 `docs/tests/full-testcases.md`
   - 增量评审：读取任务目录 `docs/requirements/{JIRA}-*/` 下的增量测试用例文档
3. 读取接口设计文档（`docs/designs/full-api-design.md`）
4. 读取架构设计文档（`docs/designs/full-architecture-design.md`）
5. 读取需求分析报告（`docs/requirements/reports/` 目录下最新报告，要有相同的变更简述的需求分析报告）
6. 读取业务需求文档（`docs/designs/full-prd.md`）：本服务业务设计的全量文档，验证测试用例是否覆盖本服务完整的功能性需求、业务规则和验收标准
7. 理解业务上下文、接口定义和业务规则

## 2. 评审执行

按优先级从高到低依次检查以下维度，详细检查项见 [checklist.md](references/checklist.md)：

**P0 - 阻断级（必须修复）**
- 接口覆盖度：每个接口是否至少有正常场景和异常场景的测试用例
- 业务规则覆盖：每条业务规则是否有对应的违反场景测试用例
- 数据隔离测试：数据隔离场景是否有测试用例覆盖
- 预期结果正确性：测试用例的预期结果是否与接口设计和业务规则一致

**P1 - 严重级（强烈建议修复）**
- 参数校验覆盖：必填、长度、格式、类型校验是否有对应测试用例
- 边界值覆盖：关键边界值（空值、最大最小值、分页边界）是否覆盖
- 并发场景覆盖：并发操作冲突场景是否有测试用例
- 状态流转测试：状态流转的合法路径和非法路径是否覆盖
- 测试分层合理性：Controller/AppService/DomainService 各层测试重点是否正确

**P2 - 建议级（建议修复）**
- 用例编号规范：编号是否符合 TC-{接口序号}{用例序号} 格式
- 用例命名质量：用例名称是否清晰表达测试意图
- 前置条件完整性：前置条件是否描述充分
- 请求参数明确性：请求参数是否具体明确而非模糊描述
- 文档结构规范：文档结构是否符合模板要求

## 3. 评审输出格式

对每个发现的问题，按以下格式输出：

```
### [P{级别}] {问题标题}

**位置**: {测试用例编号或文档章节}
**维度**: {评审维度}
**问题描述**: {具体问题说明}
**修复建议**: {如何修复}
**示例**（如适用）:
{修复后的测试用例内容}
```

## 5. 各维度评审要点速查

### 接口覆盖度
- 每个接口至少1个正常场景用例，验证标准参数的正常请求和预期响应
- 每个接口至少1个异常场景用例
- 查询接口覆盖有数据和无数据两种场景
- 分页查询覆盖首页、中间页、末页场景
- 批量操作接口覆盖空列表、单条、多条场景

### 业务规则覆盖
- 每条业务规则至少1个违反场景用例
- 状态流转规则覆盖所有合法路径和关键非法路径
- 唯一性约束覆盖重复创建场景
- 数据不存在场景有对应用例

### 参数校验覆盖
- 每个必填参数（@NotBlank、@NotNull）至少1个缺失用例
- 有长度限制的参数（@Size）覆盖超限用例
- 有格式要求的参数（@Pattern）覆盖格式错误用例
- 枚举参数覆盖非法枚举值用例

### 数据隔离测试
- 每个查询接口至少1个跨租户/服务商访问用例
- 每个修改接口至少1个跨租户/服务商操作用例
- Header中缺少隔离标识参数的用例

### 边界值覆盖
- 字符串参数：空字符串、null值、最大长度
- 数值参数：最小值、最大值、零值
- 日期参数：跨年、闰年、时区边界
- 分页参数：page=0、pageSize=0、超大页码、超大pageSize

### 测试分层
- Controller层：请求路由、参数校验、DTO转换（Mock AppService）
- AppService层：流程编排、事务边界（Mock DomainService）
- DomainService层：业务规则、领域逻辑（Mock Repository）

### 预期结果规范
- 正常场景：验证 code=0，验证 data 中关键字段值
- 参数校验失败：验证 code0，验证 message 包含字段提示
- 业务规则失败：验证返回对应的业务异常码和错误信息
- 数据隔离失败：验证无法访问其他服务商/租户数据
