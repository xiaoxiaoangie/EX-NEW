# 需求分析规范

本 skill 聚焦于基于 PRD 的需求理解与技术分析，覆盖两种场景：

- **任务单 PRD 已存在**：产品提供的任务单 PRD 是从任务单出发的完整需求描述，可能包括若干个领域。在分析时需提取本服务相关的内容，增量合并到本服务的 `full-prd.md`（本服务业务设计的全量反映，仅包含本服务这个领域），`full-prd.md` 与产品提供的任务单 PRD 是两个东西
- **PRD 不存在**：通过用户简单描述 + 人机多次交互，辅助生成本次任务的 PRD，然后再遵照已有 PRD 的流程进行后续工作

## 分析输入

需求分析基于以下输入材料：

1. **任务单 PRD**（核心输入，二选一）：
   - **产品提供的任务单 PRD**：从任务单出发的完整需求描述，包含功能描述、业务规则、验收标准、交互逻辑，可能包括若干个领域（多个后端服务）
   - **用户简单描述**（PRD 不存在时）：用户提供的需求描述，通过人机多次交互辅助生成本次任务的 PRD
2. **HTML 原型**：任务单 PRD 的组成部分
   - 原型文件位于产品设计产出目录下或由用户提供
   - 从原型中提取：页面字段→数据需求、页面操作→功能需求、页面流转→业务流程
3. **现有项目产出物**（需求变更场景）：
   - docs/designs/full-prd.md（本服务业务设计的全量文档：功能性需求、非功能性需求、核心业务规则和验收标准）
   - docs/designs/full-architecture-design.md（领域模型、业务规则、架构决策）
   - docs/designs/full-api-design.md（接口清单、请求响应定义、错误码）
   - docs/tests/full-testcases.md（测试用例清单）
   - 现有工程代码结构（聚合根、实体、服务、Controller等）

## 分析流程

### 第1步：判定分析场景

根据输入判定当前属于哪种场景：

- **任务单 PRD 已存在**：产品已提供任务单 PRD，直接进入第2步（PRD 验证与技术分析）
- **PRD 不存在**：用户提供简单描述，需先通过人机交互辅助生成本次任务的 PRD（第1.5步）

### 第1.5步：辅助生成本次任务的 PRD（仅 PRD 不存在时）

当产品未提供任务单 PRD 时，通过用户简单描述 + 人机多次交互辅助生成本次任务的 PRD：

1. 提取用户描述中的核心业务目标
2. 主动提问补充关键信息：业务边界、数据归属、隔离要求、核心业务规则
3. 生成 PRD 草稿，包含：功能描述、业务规则、验收标准、交互逻辑
4. **与用户多轮迭代**，逐步完善 PRD 内容直至用户满意
5. 创建任务目录 `docs/requirements/{JIRA}-{YYYYMMDD}-{需求简述}/`，将 PRD 写入 `{JIRA}-prd-{YYYYMMDD}-{需求简述}.md`
6. 如需要，生成 HTML 原型草稿
7. **等待用户确认 PRD** 后，将生成的 PRD 视为任务单 PRD，遵照已有 PRD 的流程继续后续步骤

### 第2步：从任务单 PRD 提取本服务需求并进行技术分析

1. 从任务单 PRD 中提取本服务相关需求：
   - 任务单 PRD 是从任务单出发的完整需求描述，可能包括若干个领域，其他领域的需求由各自服务独立分析
   - 根据本服务的领域边界（参考现有工程代码、README、架构设计文档），从任务单 PRD 中筛选出与本服务相关的功能点、业务规则和交互逻辑
   - 识别与其他服务的交互点（本服务需要调用的外部接口、需要暴露给其他服务的接口）
   - 后续所有步骤和产出物均只包含本服务相关内容
2. 验证提取后的需求完整性：
   - 功能描述：每个功能点的输入、输出、正常流程、异常流程是否明确
   - 业务规则：是否穷举而非概括，是否存在"等等"、"类似的"等模糊表述
   - 验收标准：是否可判定，而非主观描述
   - 交互逻辑和数据约束：字段类型、长度限制、必填/选填、联动关系是否完整
3. 分析 HTML 原型：
   - 页面字段和数据结构 → 识别数据需求
   - 页面操作和按钮 → 识别功能需求
   - 页面流转和状态变化 → 识别业务流程
4. 从技术视角补充分析：
   - 识别关键业务实体和它们之间的关系
   - 识别技术风险和跨模块依赖
   - 评估数据模型影响
5. 对 PRD 中模糊或缺失的部分主动向用户提问：
   - 业务边界：哪些功能属于本服务，哪些属于外部系统
   - 数据归属：数据由谁创建、谁维护、谁消费
   - 隔离要求：多租户/多服务商场景下的数据隔离级别
   - 业务规则：PRD 中未覆盖的边界条件和异常场景

### 第3步：变更影响分析

扫描现有项目产出物，评估变更对各层的影响：

1. **扫描现有产出物**
   - 读取现有文档：
     - docs/designs/full-prd.md
     - docs/designs/full-architecture-design.md（领域模型、业务规则、架构决策）
     - docs/designs/full-api-design.md（接口清单、请求响应定义、错误码）
     - docs/tests/full-testcases.md（测试用例清单）
   - 扫描现有工程代码结构（聚合根、实体、服务、Controller等）
2. **逐层分析影响**
   - **领域模型影响**：是否需要新增/修改聚合根、实体、值对象、领域服务
   - **业务规则影响**：是否需要新增/修改/删除业务规则
   - **接口影响**：是否需要新增/修改/废弃接口，是否有不兼容变更
   - **数据库影响**：是否需要新增/修改表结构、索引
   - **代码影响**：列出需要新增/修改的代码文件
   - **测试影响**：列出需要新增/修改/废弃的测试用例
3. **判定后续阶段执行计划**
   - 根据影响分析结果，判定后续各阶段是否需要执行

### 第4步：生成或更新本服务的 full-prd.md

`docs/designs/full-prd.md` 位于本服务工程的设计文档目录下，是**本服务业务设计的全量反映**，不止是本次任务的增量，而是累积了历次任务后本服务完整的业务需求全景。它**与产品提供的任务单 PRD 是两个独立的文档**：

- **任务单 PRD**：产品提供的从任务单出发的完整需求描述，可能包括若干个领域，是单次任务的输入
- **本服务 full-prd.md**：本服务业务设计的全量文档，每次需求分析时从任务单 PRD 中提取本服务相关内容后增量合并到此文档，始终反映本服务最新最全的业务需求

`full-prd.md` 内容包括但不限于：

- **功能性需求**：本服务负责的全部功能点、业务流程、交互逻辑
- **非功能性需求**：性能、安全、可用性、数据隔离等要求
- **核心业务规则**：业务约束、校验规则、状态流转规则
- **验收标准**：每个功能点的可判定验收条件

#### 首次创建

- 如果 `full-prd.md` 不存在，创建 `full-prd.md`，将本次任务提取的本服务需求写入
- 按功能模块组织需求条目
- 仅写入本服务职责范围内的需求，不包含其他领域的内容

#### 增量更新（后续任务）

- 将本次任务提取的本服务需求增量合并到已有的 `full-prd.md`
- 新增的需求条目追加到对应分类下
- 修改的需求条目直接在原位更新
- 删除的需求条目从文档中移除
- 保留未变更的内容不动
- 确保更新后的 `full-prd.md` 始终是本服务业务设计的完整全量

### 第5步：生成分析报告

在任务目录 `docs/requirements/{JIRA}-{YYYYMMDD}-{需求简述}/` 下生成分析报告。

#### 文件命名

当由开发流程（`ex-backend-develop-workflow`）调度时，会传入 JIRA 单号 `{JIRA}`，文件名加上 JIRA 前缀：

```
{JIRA}-reqanalysis-{YYYYMMDD}-{需求简述}.md
```

- `{JIRA}`：JIRA 单号（由开发流程传入，独立调度时可省略前缀）
- `{YYYYMMDD}`：分析日期
- `{需求简述}`：简短描述

生成前需扫描目录下已有文件，确定当天的下一个可用序号。

### 第6步：等待用户确认

1. 展示分析报告给用户
2. 展示业务需求文档的变更差异（如有更新）
3. **等待用户确认分析报告**
4. 用户确认后，分析报告作为后续架构设计阶段的输入
