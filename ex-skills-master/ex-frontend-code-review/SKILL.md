# 前端代码审查

对页面代码进行全方位审查，确保代码质量。

## 触发规范 Skill

开始代码审查前，必须先触发以下规范 Skill：
1. 触发技术栈规范 Skill（`ex-frontend-tech-stack-spec`）
2. 触发工程结构规范 Skill（`ex-frontend-project-structure-spec`）
3. 触发编码规范 Skill（`ex-frontend-coding-standards-spec`）

## 输入

| 来源 | 说明 |
|------|------|
| 页面代码 | 阶段4的产出物 |
| 技术方案文档 | 阶段2的产出物（如有） |

## 输出

审查报告（直接输出，不生成文件）+ 修复后的代码（直接修改源文件）

## 审查维度

### 1. 技术栈合规

对照 `ex-frontend-tech-stack-spec` 检查：
- 是否使用了禁止的技术或库
- 是否使用 Composition API + `<script setup>`
- 是否使用 TypeScript 且无 `any` 类型
- 依赖引入是否符合规范

**评审标准**：
- P0：使用了禁止的技术（Options API、Vuex、jQuery 等）
- P1：存在 `any` 类型且无注释说明
- P2：未使用推荐的工具库

### 2. 工程结构合规

对照 `ex-frontend-project-structure-spec` 检查：
- 文件放置位置是否正确
- 文件命名是否符合规范
- 组件分类（通用/业务/私有）是否正确
- 目录结构是否符合规范

**评审标准**：
- P0：文件放置在错误的目录
- P1：命名不符合规范
- P2：组件分类可优化

### 3. 编码规范合规

对照 `ex-frontend-coding-standards-spec` 检查：
- Vue SFC 文件结构顺序是否正确
- Props/Emits 是否使用 TypeScript 泛型语法
- 导入顺序是否正确
- 响应式数据使用是否规范
- 模板指令使用是否正确

**评审标准**：
- P0：Props/Emits 无类型定义
- P1：SFC 结构顺序混乱、导入顺序不规范
- P2：代码风格不一致

### 4. 方案一致性

对照技术方案文档（如有）检查：
- 组件架构是否与方案一致
- 数据流是否与方案一致
- 交互实现是否与方案一致

**评审标准**：
- P0：实现与方案严重偏离
- P1：部分实现与方案不一致
- P2：实现方式可优化

### 5. 健壮性

检查代码的健壮性：
- API 调用是否有 loading/error 处理
- 列表是否有空状态处理
- 表单是否有校验
- 是否有必要的边界检查（null/undefined）
- 组件卸载时是否清理了副作用（定时器、事件监听等）

**评审标准**：
- P0：API 调用无错误处理，可能导致页面崩溃
- P1：缺少空状态处理、缺少表单校验
- P2：缺少 loading 状态

### 6. 性能

检查代码的性能：
- 是否有不必要的重复渲染
- 大列表是否使用虚拟滚动
- 组件是否按需加载（路由懒加载）
- 是否有内存泄漏风险
- 计算属性是否合理使用

**评审标准**：
- P1：存在明显的性能问题（大列表无虚拟滚动、频繁不必要的重渲染）
- P2：可优化的性能点

### 7. 安全性

检查代码的安全性：
- 是否有 XSS 风险（v-html 使用是否安全）
- 敏感信息是否暴露在前端代码中
- 路由权限是否正确配置

**评审标准**：
- P0：存在 XSS 风险、敏感信息泄露
- P1：路由权限配置缺失
- P2：安全性可加强

## 审查报告格式

```markdown
## 审查结果：通过 / 不通过

### 问题清单

| # | 级别 | 维度 | 文件 | 问题描述 | 修复建议 | 状态 |
|---|------|------|------|---------|---------|------|
| 1 | P0 | {维度} | {文件路径} | {描述} | {建议} | 待修复/已修复 |

### 审查统计
- P0：{n} 个
- P1：{n} 个
- P2：{n} 个
```

## 自动修复规则

- P0/P1 问题：自动修复代码，修复后重新审查
- P2 问题：展示给用户确认处理方式
- 自动修复循环上限：3 轮
- 审查通过条件：无 P0、无 P1、P2 已由用户确认
