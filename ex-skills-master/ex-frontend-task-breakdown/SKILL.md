# 前端需求拆解

将 PRD/需求描述拆解为可执行的前端开发任务清单。覆盖两种场景：
- **全新需求场景**：从零开始的新页面/新模块开发
- **需求变更场景**：对已有页面/模块的功能修改、新增或删除

## 分析输入

### 输入材料

| 来源 | 路径 | 说明 |
|------|------|------|
| PRD/需求描述 | - | 用户提供的需求文档或口头描述 |
| 业务需求文档 | `docs/designs/full-prd.md`（如已有） | 本服务业务设计的全量文档 |
| 页面原型 | `docs/requirements/prototypes/*.html` | 静态 HTML 原型文件（由 workflow 确认状态） |
| 需求变更描述 | `docs/requirements/{JIRA}-*/{JIRA}-prd-*.md` | 人提供的变更描述（需求变更场景的必要输入） |
| 后端接口文档 | 由 `ex-frontend-develop-workflow` 传入（如有） | 后端接口定义 |
| 现有页面代码 | `src/`（需求变更场景） | 已有的页面组件、路由、Store 等 |

### 原型处理策略

页面原型统一使用静态 HTML 文件格式。根据 workflow 传入的原型状态，采用不同的处理策略：

#### 已有原型（解析模式）

读取 `docs/requirements/prototypes/*.html` 文件，从 HTML 结构中提取信息：
1. **页面字段** → 识别数据需求和类型定义
2. **页面操作和按钮** → 识别交互行为和 API 调用
3. **页面流转和状态变化** → 识别路由结构和状态管理
4. **表单校验规则** → 识别前端校验逻辑
5. **列表和表格** → 识别分页、筛选、排序需求

#### 无原型（AI 生成模式）

AI 根据需求描述和后端接口文档生成 HTML 原型文件：

1. 分析需求描述，确定页面类型（列表页/详情页/表单页/仪表盘等）
2. 生成静态 HTML 原型文件到 `docs/requirements/prototypes/{页面名}.html`
3. 原型文件包含：页面布局结构、字段展示、按钮和操作区域、表格列定义、表单字段和校验提示、状态场景（加载/空数据/错误）
4. 使用内联 CSS 样式，确保浏览器直接打开即可预览
5. **等待用户在浏览器中预览确认后才能继续后续拆解步骤**

#### 需求变更（AI 更新原型模式）

AI 读取变更描述并更新已有的 HTML 原型文件：

1. 读取 `docs/requirements/{JIRA}-*/` 下本次 JIRA 单号对应的变更描述
2. 如无变更描述，询问用户描述变更内容
3. 读取现有 HTML 原型文件，理解当前页面结构
4. 根据变更描述修改 HTML 原型文件，标注变更部分（可用 CSS 高亮新增/修改区域）
5. **等待用户在浏览器中预览确认后才能继续后续拆解步骤**
6. 确认后从更新后的原型中提取变更点：
   - 新增的页面区域/字段/操作
   - 修改的布局/交互/字段
   - 删除的区域/字段/操作

## 分析流程

### 第1步：判定分析场景

根据输入判定当前属于哪种场景：
- **全新需求场景**：项目无现有业务需求文档，或用户明确表示是全新页面/模块
- **需求变更场景**：项目已有业务需求文档和页面代码，用户描述的是对现有功能的修改或新增

判定依据：
1. 检查 `docs/designs/full-prd.md` 是否存在
2. 检查 `src/views/` 下是否存在相关页面组件
3. 用户描述中是否包含"修改"、"调整"、"新增字段"等变更关键词

### 第2步：需求理解与澄清

#### 全新需求场景

1. 提取用户描述中的核心业务目标
2. 根据原型处理策略执行：
   - **已有原型**：解析 HTML 原型文件，提取页面结构和交互信息
   - **无原型**：AI 生成 HTML 原型文件，**等待用户浏览器预览确认后再继续**
3. 识别涉及的页面、路由结构和导航关系
4. 识别关键业务实体和数据结构
5. 列出需要澄清的问题，主动向用户提问：
   - 页面边界：哪些功能属于本页面，哪些属于其他页面
   - 数据来源：哪些数据来自后端接口，哪些是前端本地状态
   - 权限控制：是否有角色/权限相关的 UI 差异
   - 交互细节：加载状态、空状态、错误状态的处理方式

#### 需求变更场景

1. 提取用户描述中的变更内容（新增功能/修改功能/删除功能）
2. 按「需求变更（AI 更新原型模式）」执行：读取变更描述 → 更新 HTML 原型 → **等待用户浏览器预览确认后再继续**
3. 读取现有业务需求文档，理解现有功能和业务规则
4. **扫描现有代码**，评估变更对各层的影响：
   - 扫描 `src/views/` 下相关页面组件的结构
   - 扫描 `src/components/` 下相关组件的 Props/Emits
   - 扫描 `src/api/` 下相关 API 模块
   - 扫描 `src/store/modules/` 下相关 Store
   - 扫描 `src/composables/` 下相关组合式函数
5. 主动提问澄清不明确的变更点

### 第3步：变更影响分析（仅需求变更场景）

逐层分析变更影响：

1. **页面组件影响**：需要新增/修改/删除哪些页面组件
2. **通用/业务组件影响**：需要新增/修改哪些组件，Props/Emits 是否变化
3. **路由影响**：是否需要新增/修改路由配置
4. **API 模块影响**：是否需要新增/修改 API 请求函数
5. **Store 影响**：是否需要新增/修改 Store 的 state/actions/getters
6. **类型定义影响**：是否需要新增/修改 TypeScript 类型
7. **样式影响**：是否涉及全局样式或主题变更
8. **测试影响**：哪些已有测试用例需要更新

### 第4步：页面拆解

对每个涉及的页面进行拆解：

```markdown
### 页面：{页面名称}

**路由**：`/{path}`
**类型**：新增 / 修改
**布局**：DefaultLayout

#### 页面区域划分
1. {区域1}：{描述}
2. {区域2}：{描述}

#### 组件清单
| 组件 | 类型 | 来源 | 说明 |
|------|------|------|------|
| {组件名} | 通用/业务/页面私有 | 新建/已有/改造 | {说明} |

#### 数据需求
| 数据 | 来源 | API | 说明 |
|------|------|-----|------|
| {数据名} | 后端接口/本地状态/路由参数 | {API路径} | {说明} |

#### 交互行为
1. {交互描述}
2. {交互描述}
```

### 第5步：任务清单生成

将页面拆解结果转化为有序的开发任务：

```markdown
## 任务清单

### 基础设施任务
- [ ] {任务描述}（类型定义/API模块/路由配置等）

### 组件开发任务
- [ ] {任务描述}（通用组件/业务组件）

### 页面开发任务
- [ ] {任务描述}（页面组件及其私有组件）

### 联调任务
- [ ] {任务描述}（接口联调/数据验证）
```

### 第6步：依赖分析

分析任务间的依赖关系和建议的开发顺序：

```markdown
## 开发顺序建议

1. **类型定义和 API 模块**：无依赖，优先完成
2. **通用组件**：被多个页面依赖，优先完成
3. **业务组件**：依赖类型定义
4. **页面组件**：依赖上述所有
5. **联调测试**：依赖页面开发完成
```

### 第7步：更新业务需求文档

#### 全新需求场景
- 在 `docs/requirements/` 目录下创建业务需求文档
- 按功能模块组织需求条目，包含功能性需求和交互规则
- 如 `docs/requirements/` 目录不存在则自动创建

#### 需求变更场景
- 增量更新 `docs/designs/full-prd.md`
- 新增的需求条目追加到对应分类下
- 修改的需求条目直接在原位更新
- 删除的需求条目从文档中移除
- 保留未变更的内容不动

## 输出

### 拆解报告

**文件路径**：`docs/requirements/{JIRA}-{YYYYMMDD}-{需求简述}/`

**文件命名**：

{JIRA}-taskbreakdown-{YYYYMMDD}-{需求简述}.md

- `{JIRA}`：JIRA 单号（由开发流程传入）
- `{YYYYMMDD}`：日期
- `{需求简述}`：2-4个中文词概括需求内容

任务拆解文档存放在对应需求任务目录下，与该任务的其他文档（需求描述、测试用例、变更记录等）集中管理。
