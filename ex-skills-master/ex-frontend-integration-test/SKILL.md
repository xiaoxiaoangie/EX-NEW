# 前端联调测试

验证前端页面与后端接口的集成是否正确，生成联调测试报告。

## 输入

| 来源 | 说明 |
|------|------|
| 页面代码 | 阶段4/5的产出物 |
| 后端接口 | 实际运行的后端服务或接口文档 |

## 输出

文件：`docs/requirements/{JIRA}-{YYYYMMDD}-{变更简述}/{JIRA}-integrationtest-{YYYYMMDD}-{变更简述}.md`

### 文件命名

当由开发流程（`ex-frontend-develop-workflow`）调度时，会传入 JIRA 单号 `{JIRA}`，文件名加上 JIRA 前缀：

{JIRA}-testreport-{YYYYMMDD}-{变更简述}.md

- `{JIRA}`：JIRA 单号（由开发流程传入，独立调度时可省略前缀）
- `{YYYYMMDD}`：日期
- `{变更简述}`：简要概括变更内容

## 联调检查清单

### 1. 接口连通性

逐一检查页面涉及的所有 API 调用：

```markdown
| API | 方法 | 路径 | 状态 | 说明 |
|-----|------|------|------|------|
| 获取列表 | GET | /api/{domain}/list | ✅/❌ | {说明} |
| 获取详情 | GET | /api/{domain}/{id} | ✅/❌ | {说明} |
| 创建 | POST | /api/{domain} | ✅/❌ | {说明} |
```

### 2. 数据映射验证

检查后端返回数据与前端类型定义是否匹配：

```markdown
| 字段 | 前端类型 | 后端实际 | 匹配 | 说明 |
|------|---------|---------|------|------|
| id | string | string | ✅ | |
| status | StatusEnum | string | ⚠️ | 需确认枚举值 |
| amount | number | string | ❌ | 需要类型转换 |
```

### 3. 交互流程验证

验证关键用户操作流程：

```markdown
### 流程：{流程名称}

| 步骤 | 操作 | 预期结果 | 实际结果 | 状态 |
|------|------|---------|---------|------|
| 1 | {操作描述} | {预期} | {实际} | ✅/❌ |
| 2 | {操作描述} | {预期} | {实际} | ✅/❌ |
```

### 4. 异常场景验证

验证异常情况的处理：

```markdown
| 场景 | 触发方式 | 预期行为 | 实际行为 | 状态 |
|------|---------|---------|---------|------|
| 网络断开 | 断开网络 | 显示错误提示 | {实际} | ✅/❌ |
| 空数据 | 清空数据 | 显示空状态 | {实际} | ✅/❌ |
| 权限不足 | 切换低权限用户 | 隐藏操作按钮 | {实际} | ✅/❌ |
```

### 5. 兼容性检查

```markdown
| 浏览器 | 版本 | 状态 | 说明 |
|--------|------|------|------|
| Chrome | 最新 | ✅/❌ | |
| Firefox | 最新 | ✅/❌ | |
| Edge | 最新 | ✅/❌ | |
```

## 注意事项

- 如果后端服务未就绪，记录为「待联调」，不标记为失败
- 数据映射不匹配时，明确标注是前端还是后端需要调整
- 联调问题优先在前端侧修复（适配层），避免频繁要求后端改接口
