# 产品设计

辅助产品设计人员将业务需求转化为结构化的 PRD 文档和可预览的页面原型。

基础规范（自动引用，无需单独调度）：

- `ex-product-project-structure-spec`：文档目录结构、文件命名和 full-prd 文档结构规范

## 设计输入

| 输入 | 是否必须 | 说明 |
|------|---------|------|
| 需求描述 | 必须 | 用户描述的本次迭代要做什么 |
| 任务目录 | 自动读取 | `requirements/{JIRA}-{YYYYMMDD}-{需求简述}/` 下的所有文档，包括 BRD（可选）、第三方对接文档、其他辅助说明文档等，作为设计参考 |
| 全量 PRD | 已有时自动读取 | `designs/full-prd.md`，了解产品现状，避免与已有设计冲突 |
| 全量页面原型 | 已有时自动读取 | `prototypes/*.html`，了解现有页面，在此基础上新增或修改 |

## 设计流程

### 第1步：获取 JIRA 单号并读取输入

1. 要求用户输入 JIRA 单号（强制第一步，必须输入，不能临时创建或留空）
   - 如果用户首次输入中包含 JIRA 单号，直接记录为 `{JIRA}`
   - 如果未提供，立即询问并等待用户输入，不进入后续步骤
2. 根据 JIRA 单号在 `requirements/` 下精确匹配以 `{JIRA}-` 开头的任务目录
   - 目录已存在：读取目录下所有文档（BRD、第三方对接文档、辅助说明文档等）作为设计参考
   - 目录不存在：等待用户提供需求描述后创建任务目录
3. 读取已有的全量 PRD 和页面原型（如有），了解产品现状

### 第2步：理解需求

1. 结合任务目录下的文档和用户的需求描述，提取本次迭代的业务目标和功能范围
2. 主动提问补充关键信息，确保所有关键信息都已澄清：
   - 核心使用场景和目标用户
   - 关键业务规则和约束条件
   - 与现有功能的关系（新增模块 / 扩展已有功能 / 修改已有功能）
   - 数据来源和流向
   - 范围边界（本次做什么，不做什么）
3. **⚠️ 必须停下等待用户回答所有澄清问题并确认需求理解正确后，才能进入第3步，不允许跳过**

### 第3步：生成本次迭代的 PRD 和页面原型

基于用户确认后的需求理解，同时生成本次迭代的增量 PRD 和页面原型：

**PRD 生成**：

- PRD 的第一读者是 AI（后续开发阶段的输入），必须结构化、明确、可穷举
- 增量 PRD 文件：`requirements/{JIRA}-prd-{YYYYMMDD}-{需求简述}/{JIRA}-prd-{YYYYMMDD}-{需求简述}.md`
- PRD 文档结构遵循 `ex-product-project-structure-spec` 中增量 PRD 的模板
- 内容要求：
  - 功能描述：每个功能点的输入、输出、正常流程、异常流程
  - 业务规则：穷举而非概括，禁止"等等"、"类似的"等模糊表述
  - 验收标准：可判定的，每个功能点至少一条
  - 数据约束：字段名称、类型、长度、必填、枚举值
  - 交互逻辑：页面间跳转、状态变化、权限控制

**页面原型生成**：

- 直接在全量原型上操作：已有页面直接修改，新页面新建文件
- 原型文件：`prototypes/{页面名}.html`
- 原型要求：
  - 静态 HTML + 内联 CSS，浏览器直接打开即可预览
  - 包含完整的页面布局、字段展示、操作按钮
  - 表单页：字段标签、输入类型、必填标识、校验提示
  - 列表页：表头、示例数据行、筛选条件、分页、操作列
  - 详情页：字段分组、状态展示、操作按钮
  - 多页面间的跳转关系用链接或按钮标注
  - 包含正常态和关键异常态（空数据、加载中、错误提示）
  - 需求变更时，变更部分用醒目样式标注（如背景色高亮）

### 第4步：多轮交互确认

**⚠️ PRD 和原型生成后，必须停下等待用户审阅，不允许跳过直接进入下一步。**

多轮迭代流程：

1. 提示用户在浏览器中预览原型，同时审阅 PRD
2. **必须停下等待用户反馈**，不允许自行假设用户满意
3. 用户反馈修改意见（交互调整、字段增减、规则补充、流程修正等）
4. AI 根据反馈同步更新 PRD 和原型
5. 更新后**再次停下等待用户确认**
6. 重复 1-5，直到用户明确表示确认满意

### 第5步：更新全量文档

**⚠️ 只有用户在第4步明确确认满意后，才能进入第5步，不允许跳过用户确认。**

将本次迭代内容合并到全量文档：

**更新全量 PRD**（`designs/full-prd.md`）：

- 如果不存在，基于本次增量 PRD 创建，按 `ex-product-project-structure-spec` 中 full-prd 的文档结构组织
- 如果已存在，将本次增量内容合并：新增功能追加到对应模块，修改功能原位更新，删除功能移除

**更新全量页面原型**：

- 页面原型在第2步已直接在全量原型上操作，无需额外合并

## 产出物

| 产出物    | 路径                                                                      | 说明               |
| ------ | ----------------------------------------------------------------------- | ---------------- |
| 增量 PRD | `requirements/{JIRA}-{YYYYMMDD}-{需求简述}/{JIRA}-prd-{YYYYMMDD}-{需求简述}.md` | 本次迭代的需求文档        |
| 全量 PRD | `designs/full-prd.md`                                                   | 产品完整的业务需求全景，增量累积 |
| 页面原型   | `prototypes/{页面名}.html`                                                 | 全量页面原型，直接修改      |

## 质量自检

PRD 和原型确认前自检：

- 每个功能点是否有明确的输入、输出、正常流程、异常流程
- 业务规则是否穷举，是否存在"等等"、"类似的"等模糊表述
- 验收标准是否可判定，是否覆盖正向和反向
- 数据约束是否完整（类型、长度、必填、枚举值）
- 页面原型与 PRD 描述是否一致
- 增量 PRD 的内容是否已合并到全量 PRD
