# 后端 Bug 修复流程

## 设计理念

人提供 Bug 现象（描述或日志），AI 完成定位→诊断→修复→验证的全过程，人在关键节点确认。

- 人的职责：提供 Bug 描述/错误日志、确认根因分析、确认修复方案、验收修复结果
- AI的职责：日志分析、代码定位、根因诊断、方案制定、代码修复、回归验证

## 触发规范 Skill

开始 Bug 修复前，必须先触发以下规范 Skill：
1. 触发编码规范 Skill（`ex-backend-coding-standards-spec`）
2. 触发工程结构规范 Skill（`ex-backend-project-structure-spec`）

## 第1步：信息收集与分类

### 接收输入

用户可能提供以下任意一种或多种输入：
- Bug 描述（文字描述的异常现象）
- 错误日志 / 堆栈跟踪（Stack Trace）
- 接口返回的错误信息
- 复现步骤

### 问题分类

根据输入信息判断 Bug 类型：

| 类型 | 特征 | 分析策略 |
|------|------|---------|
| 异常崩溃 | 有明确的 Exception 堆栈 | 从堆栈顶部定位抛出点，逆向追踪调用链 |
| 业务逻辑错误 | 功能行为与预期不符，无异常抛出 | 从接口入口追踪数据流，对比业务规则 |
| 数据问题 | 数据不一致、数据丢失、数据错误 | 检查数据库操作、事务边界、并发控制 |
| 性能问题 | 接口超时、响应慢 | 检查 SQL 查询、循环调用、缓存策略 |
| 集成问题 | 外部服务调用失败、消息处理异常 | 检查 Feign 调用、消息监听、序列化 |

### 补充上下文

如果输入信息不足以定位问题，主动向用户询问：
- 「请提供完整的错误日志（包含堆栈跟踪）」
- 「这个问题在什么场景下出现？是否能稳定复现？」
- 「问题出现前是否有代码变更或配置变更？」
- 「涉及哪个服务/模块？」

## 第2步：定位根因

### 2.1 日志分析（适用于有错误日志的场景）

**异常堆栈分析**：
1. 识别异常类型（NullPointerException、BusinessException、SQLException 等）
2. 从堆栈中提取项目代码的调用链（过滤框架代码）
3. 定位最内层的项目代码抛出点
4. 提取异常消息中的业务标识（orderId、providerId 等）

**日志上下文分析**：
1. 提取错误日志前后的关键日志行
2. 识别请求参数和中间状态
3. 判断是否有并发或时序问题

### 2.2 代码定位

根据分析结果在代码库中搜索：
1. 搜索异常抛出点对应的源代码文件
2. 追踪调用链上下游代码
3. 检查相关的业务规则和数据校验逻辑
4. 检查数据库操作（Repository/Mapper）
5. 检查配置文件和环境变量

### 2.3 根因诊断

综合分析后确定根因，常见根因类别：

| 根因类别 | 典型场景 |
|---------|---------|
| 空指针 | 未做空值判断、Optional 未正确处理、集合操作前未判空 |
| 业务规则缺陷 | 条件判断遗漏、状态流转错误、边界条件未处理 |
| 数据一致性 | 事务未正确管理、并发更新冲突、乐观锁失败未处理 |
| SQL/数据库 | 查询条件错误、索引缺失、字段映射错误 |
| 接口契约 | 请求/响应字段不匹配、序列化/反序列化错误 |
| 配置错误 | 环境配置缺失、参数值错误 |
| 并发问题 | 线程安全、竞态条件、死锁 |

## 第3步：确认根因（★ 等待用户确认）

向用户展示分析结果，格式如下：

```markdown
## Bug 诊断报告

### 问题概述
{一句话描述 Bug 现象}

### 问题分类
{异常崩溃 / 业务逻辑错误 / 数据问题 / 性能问题 / 集成问题}

### 根因分析

**错误位置**：`{文件路径}` 第 {行号} 行
**调用链路**：
  {Controller} → {AppService} → {DomainService/Repository}

**根本原因**：
{详细说明为什么会出现这个问题}

**相关代码**：
{展示问题代码片段，标注问题所在}

### 影响范围
- 受影响的接口：{接口列表}
- 受影响的业务场景：{场景描述}
```

**等待用户确认**：
- 用户确认根因正确 → 进入第4步
- 用户认为根因不对 → 根据用户反馈重新分析，回到第2步
- 用户补充了新信息 → 结合新信息重新分析

## 第4步：修复方案（★ 等待用户确认）

根据确认的根因制定修复方案：

```markdown
## 修复方案

### 修复策略
{简要说明修复思路}

### 变更清单

| # | 文件 | 变更类型 | 变更说明 |
|---|------|---------|---------|
| 1 | {文件路径} | 修改 | {具体修改内容} |
| 2 | {文件路径} | 新增 | {新增内容说明} |

### 修复详情

#### 变更1：{文件路径}
**修改前**：
{原代码}

**修改后**：
{修复后代码}

**修改原因**：{为什么这样改}

### 风险评估
- 影响范围：{修改是否会影响其他功能}
- 兼容性：{是否需要数据迁移或配置变更}
```

**等待用户确认**：
- 用户确认方案 → 进入第5步
- 用户要求调整 → 根据反馈修改方案

## 第5步：执行修复

### 5.1 代码修改

按照确认的修复方案修改代码，遵循以下规范：
- 代码风格遵循 `ex-backend-coding-standards-spec`
- 工程结构遵循 `ex-backend-project-structure-spec`
- 最小化变更范围，只修改必要的代码

### 5.2 自检

修复完成后执行自检：
1. 调度 `ex-backend-code-review` 对变更代码进行审查
2. P0/P1 问题自动修复，循环上限 3 轮

### 5.3 测试验证

1. 运行现有单元测试，确保无回归
   ```bash
   mvn test
   ```
2. 如果现有测试不覆盖本次 Bug 场景，新增测试用例：
   - 新增针对该 Bug 的回归测试（确保 Bug 不再复现）
   - 新增相关边界条件的测试（防止类似问题）
3. 运行全部测试确认通过

### 5.4 编译验证

```bash
mvn compile
```

确保编译通过，无错误和警告。

## 第6步：修复报告

输出修复报告：

```markdown
## Bug 修复报告

**JIRA单号**：{JIRA}（如有）
**Bug描述**：{问题概述}
**根本原因**：{一句话根因}
**修复方案**：{一句话方案}

### 变更文件
| # | 文件 | 变更类型 |
|---|------|---------|
| 1 | {文件路径} | {修改/新增/删除} |

### 测试结果
- 现有测试：{全部通过 / N个通过，M个失败}
- 新增测试：{新增了N个测试用例}
- 回归测试：{Bug场景已覆盖}

### 修复验证
- [ ] 编译通过
- [ ] 单元测试通过
- [ ] Bug 场景回归测试通过
- [ ] 无新增 P0/P1 代码审查问题
```
