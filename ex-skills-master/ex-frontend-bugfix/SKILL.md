# 前端 Bug 修复流程

## 设计理念

人提供 Bug 现象（描述、控制台错误或页面行为），AI 完成定位→诊断→修复→验证的全过程，人在关键节点确认。

- 人的职责：提供 Bug 描述/错误信息、确认根因分析、确认修复方案、验收修复结果
- AI的职责：错误分析、代码定位、根因诊断、方案制定、代码修复、回归验证

## 触发规范 Skill

开始 Bug 修复前，必须先触发以下规范 Skill：
1. 触发编码规范 Skill（`ex-frontend-coding-standards-spec`）
2. 触发工程结构规范 Skill（`ex-frontend-project-structure-spec`）

## 第1步：信息收集与分类

### 接收输入

用户可能提供以下任意一种或多种输入：
- Bug 描述（文字描述的页面异常现象）
- 浏览器控制台错误信息（Console Error / Warning）
- 网络请求错误（Network Error、HTTP 状态码）
- 页面截图或行为描述
- 复现步骤

### 问题分类

根据输入信息判断 Bug 类型：

| 类型 | 特征 | 分析策略 |
|------|------|---------|
| JS 运行时错误 | 有明确的 Error 堆栈（TypeError、ReferenceError 等） | 从堆栈定位源码文件和行号，追踪数据流 |
| 渲染/白屏 | 页面空白、组件不显示、渲染异常 | 检查组件生命周期、响应式数据、条件渲染逻辑 |
| 交互异常 | 点击无响应、表单提交失败、路由跳转异常 | 检查事件绑定、表单校验、路由配置 |
| 样式错误 | 布局错乱、样式不生效、响应式适配问题 | 检查 CSS 选择器、样式优先级、媒体查询 |
| 接口联调 | 接口报错、数据展示异常、请求参数错误 | 检查 API 调用、数据映射、错误处理 |
| 状态管理 | 数据不同步、状态丢失、缓存问题 | 检查 Store 定义、状态更新逻辑、组件通信 |

### 补充上下文

如果输入信息不足以定位问题，主动向用户询问：
- 「请提供浏览器控制台的完整错误信息」
- 「这个问题在什么页面/操作下出现？是否能稳定复现？」
- 「使用的浏览器和版本是什么？」
- 「问题出现前是否有代码变更或依赖更新？」
- 「涉及哪个页面/组件？」

## 第2步：定位根因

### 2.1 错误分析（适用于有错误信息的场景）

**JS 错误堆栈分析**：
1. 识别错误类型（TypeError、ReferenceError、SyntaxError、ChunkLoadError 等）
2. 从堆栈中提取源码文件路径和行号（Source Map 映射后）
3. 定位错误发生的组件和方法
4. 提取错误消息中的关键变量名和属性名

**网络请求分析**：
1. 检查请求 URL、方法、参数是否正确
2. 检查响应状态码和错误信息
3. 检查请求头（Token、Content-Type 等）
4. 判断是前端请求构造问题还是后端返回问题

### 2.2 代码定位

根据分析结果在代码库中搜索：
1. 搜索错误对应的源码文件（组件、Store、API 模块）
2. 追踪数据流：API 调用 → Store → 组件 Props/State → 模板渲染
3. 检查相关的路由配置和权限控制
4. 检查样式文件和 CSS 变量
5. 检查依赖版本和配置文件

### 2.3 根因诊断

综合分析后确定根因，常见根因类别：

| 根因类别 | 典型场景 |
|---------|---------|
| 空值/未定义 | 接口返回 null 未处理、可选链缺失、异步数据未就绪时访问 |
| 响应式丢失 | 解构 reactive 对象、直接替换 ref 值、深层嵌套未触发更新 |
| 生命周期 | onMounted 中访问未渲染的 DOM、组件卸载后异步回调执行 |
| 事件/交互 | 事件绑定错误、阻止冒泡遗漏、防抖/节流配置不当 |
| 路由/导航 | 路由参数丢失、导航守卫拦截、动态路由匹配失败 |
| 样式冲突 | 选择器优先级、scoped 样式穿透、CSS 变量未定义 |
| 接口对接 | 字段名不匹配、数据格式转换错误、分页参数错误 |
| 构建/依赖 | 依赖版本冲突、Tree-shaking 误删、环境变量缺失 |

## 第3步：确认根因（★ 等待用户确认）

向用户展示分析结果，格式如下：

```markdown
## Bug 诊断报告

### 问题概述
{一句话描述 Bug 现象}

### 问题分类
{JS运行时错误 / 渲染白屏 / 交互异常 / 样式错误 / 接口联调 / 状态管理}

### 根因分析

**错误位置**：`{文件路径}` 第 {行号} 行
**所属组件**：`{组件名}`
**数据流路径**：
  {API调用} → {Store} → {父组件} → {子组件}

**根本原因**：
{详细说明为什么会出现这个问题}

**相关代码**：
{展示问题代码片段，标注问题所在}

### 影响范围
- 受影响的页面：{页面列表}
- 受影响的交互：{交互描述}
```

**等待用户确认**：
- 用户确认根因正确 → 进入第4步
- 用户认为根因不对 → 根据用户反馈重新分析，回到第2步
- 用户补充了新信息 → 结合新信息重新分析

## 第4步：修复方案（★ 等待用户确认）

根据确认的根因制定修复方案：

```markdown
## 修复方案

### 修复策略
{简要说明修复思路}

### 变更清单

| # | 文件 | 变更类型 | 变更说明 |
|---|------|---------|---------|
| 1 | {文件路径} | 修改 | {具体修改内容} |
| 2 | {文件路径} | 新增 | {新增内容说明} |

### 修复详情

#### 变更1：{文件路径}
**修改前**：
{原代码}

**修改后**：
{修复后代码}

**修改原因**：{为什么这样改}

### 风险评估
- 影响范围：{修改是否会影响其他页面或组件}
- 兼容性：{是否需要清除缓存或更新依赖}
```

**等待用户确认**：
- 用户确认方案 → 进入第5步
- 用户要求调整 → 根据反馈修改方案

## 第5步：执行修复

### 5.1 代码修改

按照确认的修复方案修改代码，遵循以下规范：
- 代码风格遵循 `ex-frontend-coding-standards-spec`
- 工程结构遵循 `ex-frontend-project-structure-spec`
- 最小化变更范围，只修改必要的代码

### 5.2 自检

修复完成后执行自检：
1. 调度 `ex-frontend-code-review` 对变更代码进行审查
2. P0/P1 问题自动修复，循环上限 3 轮

### 5.3 构建验证

```bash
npm run build
```

确保构建通过，无编译错误和类型错误。

### 5.4 浏览器验证

如果开发服务器可用：
1. 启动开发服务器（如未启动）
2. 在浏览器中打开对应页面
3. 按照 Bug 复现步骤验证问题是否修复
4. 检查浏览器控制台是否有新的错误

## 第6步：修复报告

输出修复报告：

```markdown
## Bug 修复报告

**JIRA单号**：{JIRA}（如有）
**Bug描述**：{问题概述}
**根本原因**：{一句话根因}
**修复方案**：{一句话方案}

### 变更文件
| # | 文件 | 变更类型 |
|---|------|---------|
| 1 | {文件路径} | {修改/新增/删除} |

### 验证结果
- 构建检查：{通过/失败}
- 浏览器验证：{Bug已修复 / 待人工验证}
- 控制台错误：{无新增错误 / 待确认}

### 修复验证
- [ ] 构建通过
- [ ] Bug 场景已修复
- [ ] 无新增控制台错误
- [ ] 无新增 P0/P1 代码审查问题
```
