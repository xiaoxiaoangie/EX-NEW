# 接口设计评审流程

## 1. 评审准备

收到接口设计评审请求后，按以下步骤执行：

1. 确定评审范围（全量评审或增量评审）
2. 读取接口设计文档（`docs/designs/full-api-design.md`）
3. 读取架构设计文档（`docs/designs/full-architecture-design.md`）
4. 读取需求分析报告（`docs/requirements/reports/` 目录下最新报告）
5. 读取业务需求文档（`docs/designs/full-prd.md`）：本服务业务设计的全量文档
6. 触发编码规范Skill（`ex-backend-coding-standards-spec`）
7. 理解业务上下文、领域模型和业务规则

## 2. 评审执行

按优先级从高到低依次检查以下维度，详细检查项见 [checklist.md](references/checklist.md)：

**P0 - 阻断级（必须修复）**
- 架构一致性：接口是否完整覆盖架构设计文档中的所有领域服务方法和聚合根行为
- 业务规则覆盖：每条业务规则违反场景是否有对应的错误码定义
- 数据隔离：需要数据隔离的接口是否在Header中要求隔离标识参数
- 请求响应正确性：请求参数和响应字段是否与架构设计文档中的领域模型一致

**P1 - 严重级（强烈建议修复）**
- 路径与方法规范：路径命名、HTTP方法选择是否符合RESTful规范和编码规范
- 参数校验完整性：必填标注、类型定义、枚举值约束是否完整
- 错误码规范：错误码命名、HTTP状态码映射是否符合规范
- DTO设计规范：DTO命名、字段定义是否符合编码规范
- 分页查询规范：分页参数、响应格式是否符合编码规范
- 幂等性设计：写操作接口是否考虑幂等性

**P2 - 建议级（建议修复）**
- 接口粒度：接口粒度是否合适（不过粗也不过细）
- 命名质量：接口名称、参数名称是否清晰表达业务语义
- 文档完整性：接口描述、参数说明是否充分
- 文档结构规范：文档结构是否符合模板要求
- 变更记录：变更记录是否完整

## 3. 评审输出格式

对每个发现的问题，按以下格式输出：

```
### [P{级别}] {问题标题}

**位置**: {接口编号或文档章节}
**维度**: {评审维度}
**问题描述**: {具体问题说明}
**修复建议**: {如何修复}
**示例**（如适用）:
{修复后的接口定义内容}
```

## 5. 各维度评审要点速查

### 架构一致性
- 架构设计文档中的每个领域服务方法在接口清单中有对应接口
- 架构设计文档中的每个聚合根行为方法在接口清单中有对应接口（通过领域服务调用）
- 接口的请求参数能覆盖领域服务方法的输入参数
- 接口的响应字段能覆盖领域服务方法的输出结果
- 不存在架构设计文档中无对应功能的多余接口

### 路径与方法规范
- 路径前缀符合 /api/{工程代码} 格式
- 资源路径使用小写字母和连字符，名词复数形式
- GET 用于所有查询，POST 用于创建，PUT 用于修改，DELETE 用于删除
- 禁止使用 POST 进行查询操作
- 路径参数用于资源标识，查询参数用于过滤条件

### 请求响应设计
- 请求参数类型与领域模型属性类型一致（金额用BigDecimal、日期用String、时间用Long）
- 响应字段类型与领域模型属性类型一致
- 必填参数标注正确，与业务规则一致
- 枚举参数列出所有合法值
- 列表类型参数说明元素约束（最少/最多个数）

### DTO规范
- 请求DTO命名遵循 {Action}{Resource}Request 格式
- 响应DTO命名遵循 {Action}{Resource}Response 格式
- 仅请求Body和响应Body定义为DTO，Path参数和Query参数不定义DTO
- DTO字段与接口定义的请求/响应参数一致
- 嵌套DTO有独立定义和字段说明

### 错误码设计
- 每条业务规则的违反场景有对应的错误码
- 错误码命名格式：{工程代码}_{明确含义英文}，大写下划线连接
- 错误码与HTTP状态码映射合理（400业务错误、404资源不存在、500系统错误）
- 错误码描述清晰表达错误原因
- 不存在含义重复的错误码

### 分页查询
- 分页请求参数使用 currentPage 和 pageSize
- 分页响应使用标准格式（records、total/totalRecords、currentPage、pageSize）
- 默认每页10条，最大100条，页码从1开始
- 分页查询使用 GET 方法，参数放URL中

### 数据隔离
- 需要数据隔离的接口在Header参数中要求 X-Provider-Id 或 X-Tenant-Id
- 隔离标识参数标注为必填
- 查询接口和修改接口均包含隔离标识
- 批量操作接口的隔离策略明确

### 幂等性
- 创建类接口考虑重复提交场景（如使用referenceId去重）
- 资金操作接口（入金、出金、冻结金额等）有幂等性设计
- 状态变更接口考虑重复调用场景（已处于目标状态时的处理策略）
