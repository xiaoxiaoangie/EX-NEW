## 1. 架构一致性

### 1.1 功能覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| AC-01 | 架构设计文档中的每个领域服务方法在接口清单中有对应接口 | P0 |
| AC-02 | 架构设计文档中的每个聚合根行为方法通过领域服务暴露，在接口清单中有对应接口 | P0 |
| AC-03 | 需求分析报告中的每个功能需求可追溯到至少一个接口 | P0 |
| AC-04 | 不存在架构设计文档中无对应功能的多余接口（每个接口有明确业务场景支撑） | P1 |

### 1.2 参数一致性
| 编号 | 检查项 | 级别 |
|------|--------|------|
| AC-05 | 接口的请求参数能覆盖对应领域服务方法的所有输入参数 | P0 |
| AC-06 | 接口的响应字段能覆盖对应领域服务方法的所有输出结果 | P0 |
| AC-07 | 请求参数的类型与领域模型中对应属性的类型一致 | P0 |
| AC-08 | 响应字段的类型与领域模型中对应属性的类型一致 | P0 |
| AC-09 | 枚举参数的合法值与架构设计文档中值对象的枚举值一致 | P0 |

### 1.3 业务规则覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| AC-10 | 架构设计文档中的每条业务规则在对应接口的错误码中有体现 | P0 |
| AC-11 | 业务规则的校验条件与接口参数的必填/可选标注一致 | P0 |
| AC-12 | 状态流转规则中的非法路径在对应接口的错误码中有体现 | P1 |

## 2. 路径与方法规范

### 2.1 路径规范
| 编号 | 检查项 | 级别 |
|------|--------|------|
| PM-01 | 路径前缀符合 /api/{工程代码} 格式 | P1 |
| PM-02 | 资源路径使用小写字母和连字符（kebab-case），禁止驼峰或下划线 | P1 |
| PM-03 | 资源名称使用名词复数形式（如 /accounts 而非 /account） | P1 |
| PM-04 | 路径层级合理，子资源通过父资源路径嵌套（如 /accounts/{accountId}/records） | P1 |
| PM-05 | 操作类路径使用动词短语（如 /freeze、/unfreeze），放在资源路径之后 | P2 |

### 2.2 HTTP方法选择
| 编号 | 检查项 | 级别 |
|------|--------|------|
| PM-06 | GET 用于所有查询操作（包括列表查询），参数放URL中 | P1 |
| PM-07 | POST 用于创建资源或触发操作（如入金、出金） | P1 |
| PM-08 | PUT 用于修改资源或状态变更操作 | P1 |
| PM-09 | DELETE 用于删除资源（如有） | P1 |
| PM-10 | 禁止使用 POST 进行查询操作 | P1 |

### 2.3 参数位置
| 编号 | 检查项 | 级别 |
|------|--------|------|
| PM-11 | 路径参数（Path Variable）仅用于资源标识（如 {accountId}、{batchId}） | P1 |
| PM-12 | 查询参数（Query Parameter）用于过滤条件和分页参数 | P1 |
| PM-13 | 请求体（Request Body）仅用于 POST 和 PUT 方法 | P1 |
| PM-14 | GET 和 DELETE 方法不使用请求体 | P1 |

## 3. 请求响应设计

### 3.1 请求参数
| 编号 | 检查项 | 级别 |
|------|--------|------|
| RR-01 | 每个请求参数标注类型（String、Integer、BigDecimal、Boolean、List等） | P1 |
| RR-02 | 每个请求参数标注是否必填 | P1 |
| RR-03 | 必填标注与业务规则一致（业务规则要求必填的参数标注为必填） | P0 |
| RR-04 | 枚举类型参数列出所有合法值及说明 | P1 |
| RR-05 | 金额参数使用 BigDecimal 类型 | P1 |
| RR-06 | 日期参数使用 String 类型（yyyy-MM-dd格式），时间参数使用 Long 类型（时间戳） | P1 |
| RR-07 | 列表类型参数说明元素约束（如"至少一个"、"最多N个"） | P1 |
| RR-08 | 条件必填参数说明触发条件（如"accountType=EXTERNAL时必填"） | P1 |

### 3.2 响应参数
| 编号 | 检查项 | 级别 |
|------|--------|------|
| RR-09 | 响应参数类型定义完整且与领域模型一致 | P0 |
| RR-10 | 查询详情接口的响应字段覆盖业务所需的所有信息 | P1 |
| RR-11 | 列表查询接口的响应元素DTO有独立定义 | P1 |
| RR-12 | 创建类接口响应返回新创建资源的ID | P1 |
| RR-13 | 无响应体的接口明确标注"无响应体，成功返回200" | P2 |
| RR-14 | 时间字段在响应中使用 Long 类型（时间戳），日期字段使用 String 类型 | P1 |

### 3.3 Header参数
| 编号 | 检查项 | 级别 |
|------|--------|------|
| RR-15 | Header参数名称符合规范（X-Provider-Id、X-Tenant-Id、X-User-Id等） | P1 |
| RR-16 | 写操作接口包含操作人标识（X-User-Id） | P1 |
| RR-17 | Header参数标注是否必填 | P1 |
| RR-18 | 同一服务的接口Header参数使用一致（不同接口对同一Header的必填要求一致或有合理差异说明） | P2 |

## 4. DTO规范

### 4.1 命名规范
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DTO-01 | 请求DTO命名遵循 {Action}{Resource}Request 格式 | P1 |
| DTO-02 | 响应DTO命名遵循 {Action}{Resource}Response 格式 | P1 |
| DTO-03 | DTO命名中的Action与接口的业务语义一致（如Create、Get、Freeze等） | P1 |
| DTO-04 | 嵌套DTO命名清晰，能表达其用途（如 BatchOpenAccountItemRequest） | P2 |

### 4.2 DTO定义
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DTO-05 | 仅请求Body和响应Body定义为DTO，Path参数和Query参数不定义DTO | P1 |
| DTO-06 | DTO清单中的每个DTO在接口详细定义中有对应使用 | P1 |
| DTO-07 | 接口详细定义中使用的每个DTO在DTO清单中有定义 | P1 |
| DTO-08 | DTO字段与接口详细定义中的请求/响应参数完全一致 | P1 |
| DTO-09 | 嵌套DTO（如列表元素DTO）有独立的字段定义表 | P1 |
| DTO-10 | 不存在未被任何接口使用的多余DTO | P2 |

## 5. 错误码设计

### 5.1 错误码覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| EC-01 | 架构设计文档中的每条业务规则在错误码定义中有对应的错误码 | P0 |
| EC-02 | 每个接口的业务异常场景在该接口的错误码列表中有对应错误码 | P0 |
| EC-03 | 资源不存在场景有对应的错误码（如 XXX_NOT_FOUND） | P1 |
| EC-04 | 状态冲突场景有对应的错误码（如 XXX_ALREADY_FROZEN、XXX_NOT_FROZEN） | P1 |

### 5.2 错误码命名
| 编号 | 检查项 | 级别 |
|------|--------|------|
| EC-05 | 错误码命名格式：{工程代码}_{明确含义英文}，大写下划线连接 | P1 |
| EC-06 | 错误码命名清晰表达错误原因，避免模糊命名（如 INVALID_PARAM） | P1 |
| EC-07 | 不存在含义重复或高度相似的错误码 | P1 |
| EC-08 | 错误码前缀在同一服务内保持一致 | P2 |

### 5.3 HTTP状态码映射
| 编号 | 检查项 | 级别 |
|------|--------|------|
| EC-09 | 参数校验失败映射到 HTTP 400 | P1 |
| EC-10 | 业务规则违反映射到 HTTP 400 | P1 |
| EC-11 | 资源不存在映射到 HTTP 404 | P1 |
| EC-12 | 系统内部错误（如外部系统调用失败）映射到 HTTP 500 | P1 |
| EC-13 | 错误码定义表中包含HTTP状态码列 | P2 |

## 6. 分页查询

### 6.1 请求规范
| 编号 | 检查项 | 级别 |
|------|--------|------|
| PG-01 | 分页请求参数使用 currentPage 和 pageSize 命名 | P1 |
| PG-02 | currentPage 和 pageSize 标注为非必填，有默认值说明（默认 currentPage=1，pageSize=10） | P1 |
| PG-03 | pageSize 有最大值限制说明（最大100） | P1 |
| PG-04 | 分页查询使用 GET 方法，分页参数放在URL查询参数中 | P1 |

### 6.2 响应规范
| 编号 | 检查项 | 级别 |
|------|--------|------|
| PG-05 | 分页响应包含数据列表字段（records 或 items） | P1 |
| PG-06 | 分页响应包含总记录数字段（total 或 totalRecords） | P1 |
| PG-07 | 分页响应包含当前页码字段（currentPage） | P1 |
| PG-08 | 分页响应包含每页大小字段（pageSize） | P1 |
| PG-09 | 分页响应字段命名在同一服务内保持一致 | P2 |

## 7. 数据隔离

### 7.1 隔离标识
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DI-01 | 需要数据隔离的接口在Header参数中要求隔离标识（X-Provider-Id 或 X-Tenant-Id） | P0 |
| DI-02 | 隔离标识参数标注为必填 | P0 |
| DI-03 | 同一服务的所有接口使用一致的隔离标识参数 | P1 |

### 7.2 隔离覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DI-04 | 所有查询接口包含隔离标识参数 | P0 |
| DI-05 | 所有写操作接口包含隔离标识参数 | P0 |
| DI-06 | 批量操作接口的隔离策略明确（批量数据是否必须属于同一隔离维度） | P1 |
| DI-07 | 跨隔离维度的数据访问有明确的禁止策略或权限控制 | P1 |

## 8. 幂等性与并发

### 8.1 幂等性设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| IC-01 | 资金操作接口（入金、出金、冻结金额、解冻金额）使用 referenceId 或类似机制保证幂等 | P1 |
| IC-02 | 创建类接口考虑重复提交场景，有去重机制或幂等设计 | P1 |
| IC-03 | 状态变更接口明确已处于目标状态时的处理策略（返回成功还是返回特定错误码） | P1 |

### 8.2 并发控制
| 编号 | 检查项 | 级别 |
|------|--------|------|
| IC-04 | 需要并发控制的接口在架构设计中有对应的乐观锁或分布式锁方案 | P1 |
| IC-05 | 批量操作接口考虑部分成功场景的处理策略 | P1 |
| IC-06 | 批量操作接口的响应包含成功/失败计数和明细状态 | P1 |

| DOC-04 | 包含错误码定义章节（错误码、描述、HTTP状态码） | P1 |

### 9.2 接口定义完整性
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DOC-06 | 每个接口定义包含HTTP方法和完整路径 | P1 |
| DOC-07 | 每个接口定义包含Header参数表（如有） | P1 |
| DOC-08 | 每个接口定义包含路径参数表（如有） | P1 |

| DOC-10 | 每个接口定义包含请求体参数表（如有） | P1 |
| DOC-11 | 每个接口定义包含响应体参数表 | P1 |
| DOC-12 | 每个接口定义包含业务错误码表（如有业务异常场景） | P1 |

### 9.3 描述质量
| 编号 | 检查项 | 级别 |
|------|--------|------|

| DOC-16 | 条件必填参数在说明中描述触发条件 | P1 |
| DOC-17 | 表格格式统一，内容对齐 | P2 |

### 9.4 接口清单一致性
| 编号 | 检查项 | 级别 |
|------|--------|------|

| DOC-19 | 详细定义章节中的每个接口在接口清单中有对应条目 | P1 |
| DOC-20 | 接口清单中的路径、方法与详细定义中的一致 | P1 |
| DOC-21 | 接口编号连续且无重复 | P2 |
