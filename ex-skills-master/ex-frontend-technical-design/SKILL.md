# 前端技术方案设计

针对复杂需求，设计组件架构、状态管理方案和交互实现方案。

## 触发规范 Skill

开始技术方案设计前，必须先触发以下规范 Skill：
1. 触发技术栈规范 Skill（`ex-frontend-tech-stack-spec`）
2. 触发工程结构规范 Skill（`ex-frontend-project-structure-spec`）

## 输入

| 来源 | 说明 |
|------|------|
| 任务拆解文档 | 阶段1的产出物 |
| 设计稿 | UI 设计稿（如有） |

## 输出

文件：`docs/designs/{JIRA}-techdesign-{YYYYMMDD}-{方案简述}.md`

### 文件命名

当由开发流程（`ex-frontend-develop-workflow`）调度时，会传入 JIRA 单号 `{JIRA}`，文件名加上 JIRA 前缀：

{JIRA}-techdesign-{YYYYMMDD}-{方案简述}.md

- `{JIRA}`：JIRA 单号（由开发流程传入，独立调度时可省略前缀）
- `{YYYYMMDD}`：日期
- `{方案简述}`：2-4个中文词概括方案内容

## 设计流程

### 第一步：组件架构设计

分析页面结构，设计组件树：

```markdown
## 组件架构

### 组件树

{PageName}.vue
├── {SectionA}.vue（页面私有）
│   ├── AppTable（通用组件）
│   └── {StatusTag}.vue（业务组件）
├── {SectionB}.vue（页面私有）
│   ├── AppForm（通用组件）
│   └── {CustomSelect}.vue（页面私有）
└── {ActionBar}.vue（页面私有）

### 组件职责

| 组件 | 职责 | Props | Emits |
|------|------|-------|-------|
| {组件名} | {职责描述} | {主要props} | {主要事件} |
```

### 第二步：数据流设计

设计数据的获取、流转和更新方式：

```markdown
## 数据流设计

### 数据来源

| 数据 | 来源 | 管理方式 | 说明 |
|------|------|---------|------|
| {数据名} | API / Store / Props / Route | ref / store / computed | {说明} |

### 状态管理

#### 需要 Store 的场景
- 多个页面/组件共享的数据
- 需要跨路由保持的状态
- 用户信息、权限信息等全局状态

#### 使用组件本地状态的场景
- 页面内的列表数据、表单数据
- 仅当前组件使用的 UI 状态
- 临时的交互状态（展开/折叠、选中等）

### 数据流向图

用户操作 → 组件方法 → API 调用 → 更新本地状态 → 视图更新
                                  ↘ 更新 Store（如需跨组件共享）
```

### 第三步：交互方案设计

设计复杂交互的实现方案：

```markdown
## 交互方案

### {交互场景1}

**触发条件**：{描述}
**实现方案**：
1. {步骤1}
2. {步骤2}

**关键代码思路**：
```typescript
// 伪代码描述核心逻辑
```

### 异常处理

| 场景 | 处理方式 |
|------|---------|
| 网络超时 | {处理方式} |
| 数据为空 | {处理方式} |
| 权限不足 | {处理方式} |
```

### 第四步：接口适配设计

当后端接口数据结构与前端展示需求不一致时，设计适配层：

```markdown
## 接口适配

### 数据转换

| 后端字段 | 前端字段 | 转换逻辑 |
|---------|---------|---------|
| {后端字段} | {前端字段} | {转换说明} |

### 适配函数位置

- 简单字段映射：在 API 模块中处理
- 复杂业务转换：在 composable 中处理
- 展示格式化：在 utils/format.ts 中处理
```
