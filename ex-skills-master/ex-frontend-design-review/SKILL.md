# 前端技术方案评审

对技术方案文档进行全方位评审，确保方案质量。

## 触发规范 Skill

开始技术方案评审前，必须先触发以下规范 Skill：
1. 触发工程结构规范 Skill（`ex-frontend-project-structure-spec`）
2. 触发技术栈规范 Skill（`ex-frontend-tech-stack-spec`）

## 输入

| 来源 | 说明 |
|------|------|
| 技术方案文档 | 阶段2的产出物 |
| 任务拆解文档 | 阶段1的产出物 |

## 输出

评审报告（直接输出，不生成文件）+ 修复后的技术方案文档（覆盖原文件）

## 评审维度

### 1. 需求覆盖度

检查技术方案是否覆盖了任务拆解中的所有需求：
- 所有页面是否都有对应的组件设计
- 所有交互行为是否都有实现方案
- 所有数据需求是否都有获取方案

**评审标准**：
- P0：遗漏整个页面或核心功能的设计
- P1：遗漏重要交互或数据流设计
- P2：遗漏边缘场景处理

### 2. 组件设计合理性

检查组件架构是否合理：
- 组件粒度是否适当（不过大也不过小）
- 组件职责是否单一
- 组件复用性是否合理（通用/业务/私有分类是否正确）
- Props 和 Emits 设计是否清晰
- 是否遵循 `ex-frontend-project-structure-spec` 的组件目录规则

**评审标准**：
- P0：组件职责混乱，一个组件承担多个不相关职责
- P1：组件粒度不当，或复用分类错误
- P2：Props/Emits 命名不规范

### 3. 数据流合理性

检查数据管理方案是否合理：
- Store 使用是否必要（是否过度使用或不足）
- 数据流向是否清晰，有无循环依赖
- 异步数据的加载/错误/空状态是否考虑
- 数据缓存策略是否合理

**评审标准**：
- P0：数据流设计有循环依赖或状态不一致风险
- P1：Store 使用不当（该用未用或不该用而用）
- P2：缺少加载/错误状态处理

### 4. 技术栈合规性

检查方案是否遵循 `ex-frontend-tech-stack-spec`：
- 是否使用了禁止的技术或库
- 是否遵循 Composition API 规范
- 是否遵循 TypeScript 规范

**评审标准**：
- P0：使用了禁止的技术（Options API、Vuex 等）
- P1：TypeScript 类型定义不完整
- P2：未使用推荐的工具库

### 5. 性能考量

检查方案是否考虑了性能：
- 大列表是否考虑虚拟滚动
- 频繁更新的数据是否考虑防抖/节流
- 组件是否考虑按需加载
- 图片等资源是否考虑懒加载

**评审标准**：
- P1：大数据量场景未考虑性能优化
- P2：缺少防抖/节流等常规优化

## 评审报告格式

```markdown
## 评审结果：通过 / 不通过

### 问题清单

| # | 级别 | 维度 | 问题描述 | 修复建议 | 状态 |
|---|------|------|---------|---------|------|
| 1 | P0 | {维度} | {描述} | {建议} | 待修复/已修复 |

### 评审统计
- P0：{n} 个
- P1：{n} 个
- P2：{n} 个
```

## 自动修复规则

- P0/P1 问题：自动修复技术方案文档，修复后重新评审
- P2 问题：展示给用户确认处理方式
- 自动修复循环上限：3 轮
- 评审通过条件：无 P0、无 P1、P2 已由用户确认
