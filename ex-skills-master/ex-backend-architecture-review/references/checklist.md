## 1. 需求覆盖度

### 1.1 功能需求覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| RC-01 | 需求分析报告中的每个功能需求在设计文档中都有对应的设计方案 | P0 |
| RC-02 | 每个功能需求可追溯到具体的领域服务方法或聚合根行为方法 | P0 |
| RC-03 | 需求中的查询场景在领域服务或仓储接口中有对应方法 | P0 |
| RC-04 | 批量操作场景有独立的聚合或服务方法支持 | P1 |

### 1.2 业务规则覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| RC-05 | 需求分析报告中的每条业务规则在设计文档的业务规则章节中有对应定义 | P0 |
| RC-06 | 每条业务规则明确指定了校验位置（聚合根行为方法或领域服务） | P0 |
| RC-07 | 业务规则的触发条件和影响范围描述完整 | P1 |

### 1.3 非功能需求覆盖
| 编号 | 检查项 | 级别 |
|------|--------|------|
| RC-08 | 数据隔离需求在架构决策中有明确方案 | P0 |
| RC-09 | 并发控制需求有对应的设计方案（乐观锁、分布式锁等） | P1 |
| RC-10 | 外部系统集成需求有防腐层设计 | P1 |

## 2. 领域建模

### 2.1 聚合设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DM-01 | 聚合根拥有独立的生命周期和全局唯一标识 | P0 |
| DM-02 | 聚合根的属性完整，包含所有业务所需字段 | P0 |
| DM-03 | 聚合边界遵循高内聚低耦合原则，同一聚合内的对象在业务上紧密关联 | P0 |
| DM-04 | 聚合之间通过ID引用而非对象引用 | P1 |
| DM-05 | 每个聚合根对应一个仓储接口 | P1 |
| DM-06 | 聚合根包含6个基础属性：version、deleted、createTime、createBy、updateTime、updateBy | P1 |

### 2.2 实体设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DM-07 | 实体有唯一标识且依附于聚合根 | P1 |
| DM-08 | 实体的生命周期由聚合根管理 | P1 |
| DM-09 | 实体不跨聚合引用 | P0 |
| DM-10 | 独立实体（非聚合内实体）包含6个基础属性 | P1 |

### 2.3 值对象设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DM-11 | 值对象无唯一标识，通过属性值判等 | P1 |
| DM-12 | 枚举类型的值对象列出所有枚举值及说明 | P1 |
| DM-13 | 值对象的使用场景和约束条件描述清晰 | P2 |

### 2.4 领域服务设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DM-14 | 领域服务仅处理跨聚合的业务操作或不适合放在聚合根中的操作 | P1 |
| DM-15 | 领域服务方法命名表达明确的业务语义，禁止通用命名（如process、handle） | P1 |
| DM-16 | 领域服务方法的输入输出参数定义清晰 | P1 |
| DM-17 | 领域服务不包含基础设施层的实现细节 | P1 |

### 2.5 聚合根行为方法
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DM-18 | 聚合根行为方法表达明确的业务语义，禁止updateXxx等通用命名 | P1 |
| DM-19 | 行为方法包含必要的业务规则校验 | P0 |
| DM-20 | 行为方法的参数和返回值设计合理 | P2 |
| DM-21 | 状态变更操作通过行为方法完成，而非直接修改属性 | P1 |

## 3. 业务规则

### 3.1 规则定义
| 编号 | 检查项 | 级别 |
|------|--------|------|
| BR-01 | 每条业务规则有唯一编号和明确名称 | P2 |
| BR-02 | 业务规则的触发条件描述完整（何时触发、由谁触发） | P1 |
| BR-03 | 业务规则的校验逻辑描述清晰（校验什么、如何校验） | P0 |
| BR-04 | 业务规则违反时的处理方式明确（抛出什么异常、返回什么错误） | P1 |

### 3.2 规则归属
| 编号 | 检查项 | 级别 |
|------|--------|------|
| BR-05 | 单聚合内的业务规则归属到聚合根行为方法 | P1 |
| BR-06 | 跨聚合的业务规则归属到领域服务 | P1 |
| BR-07 | 输入校验规则（格式、非空等）归属到DTO层 | P2 |

### 3.3 状态管理
| 编号 | 检查项 | 级别 |
|------|--------|------|
| BR-08 | 状态枚举值完整定义，覆盖所有业务场景 | P0 |
| BR-09 | 状态流转规则明确定义所有合法路径（从哪个状态可以转到哪个状态） | P0 |
| BR-10 | 非法状态流转有明确的拒绝策略 | P1 |
| BR-11 | 终态（如注销）不可逆转 | P1 |

### 3.4 唯一性约束
| 编号 | 检查项 | 级别 |
|------|--------|------|
| BR-12 | 唯一性约束明确指定范围（全局唯一还是某个维度下唯一） | P0 |
| BR-13 | 唯一性约束考虑数据隔离维度（如providerId维度下唯一） | P0 |
| BR-14 | 唯一性约束在数据库层有对应的UNIQUE索引支持 | P1 |

## 4. 数据隔离

### 4.1 隔离字段
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DI-01 | 所有聚合根包含隔离字段（providerId） | P0 |
| DI-02 | 需要独立查询的实体包含隔离字段 | P0 |
| DI-03 | 隔离字段在领域模型属性表中有明确定义 | P1 |

### 4.2 隔离策略
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DI-04 | 领域服务的查询方法设计中体现隔离条件 | P0 |
| DI-05 | 仓储接口的查询方法包含隔离参数 | P0 |
| DI-06 | 跨租户数据访问有明确的禁止策略 | P0 |

### 4.3 隔离索引
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DI-07 | 数据库索引设计将隔离字段作为前缀 | P1 |
| DI-08 | UNIQUE约束包含隔离字段 | P1 |

## 5. 防腐层设计

### 5.1 接口隔离
| 编号 | 检查项 | 级别 |
|------|--------|------|
| ACL-01 | 外部系统调用通过防腐层接口隔离，Domain层不直接依赖外部系统 | P0 |
| ACL-02 | 防腐层接口定义在Domain层 | P1 |
| ACL-03 | 防腐层接口方法使用领域语言命名，不暴露外部系统细节 | P1 |

### 5.2 多系统支持
| 编号 | 检查项 | 级别 |
|------|--------|------|
| ACL-04 | 多外部系统场景使用Strategy+Router模式 | P1 |
| ACL-05 | 路由器根据系统编码分发到对应适配器 | P1 |
| ACL-06 | 新增外部系统时无需修改领域层代码 | P1 |

### 5.3 异常处理
| 编号 | 检查项 | 级别 |
|------|--------|------|
| ACL-07 | 外部系统调用异常有明确的处理策略（重试、熔断、降级） | P1 |
| ACL-08 | 外部系统异常转换为领域异常，不泄露外部系统细节 | P1 |

---

## 6. 数据库设计

### 6.1 表结构
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DB-01 | 表结构与领域模型一致，每个聚合根/独立实体对应一张表 | P1 |
| DB-02 | 主键使用业务ID（String类型），非数据库自增 | P1 |
| DB-03 | 包含乐观锁字段（version, Long类型） | P1 |
| DB-04 | 包含逻辑删除字段（deleted, Boolean类型） | P1 |
| DB-05 | 包含审计字段（createTime, createBy, updateTime, updateBy） | P1 |
| DB-06 | 金额字段使用BigDecimal，精度至少6位 | P1 |
| DB-07 | 日期字段使用date类型，时间字段使用timestamptz类型 | P2 |

### 6.2 索引设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DB-08 | 主要查询场景有对应的索引支持 | P1 |
| DB-09 | 索引以隔离字段（providerId）为前缀 | P1 |
| DB-10 | UNIQUE索引包含隔离字段和逻辑删除字段 | P1 |
| DB-11 | 避免过多索引影响写入性能 | P2 |

### 6.3 约束设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DB-12 | 枚举字段有CHECK约束限制合法值 | P1 |
| DB-13 | NOT NULL约束与业务规则一致 | P1 |
| DB-14 | 外键约束不跨聚合（聚合间通过ID关联，不使用外键） | P1 |

---

## 7. 应用层设计

### 7.1 应用服务
| 编号 | 检查项 | 级别 |
|------|--------|------|
| AS-01 | 应用服务方法与领域服务方法一一对应或合理编排 | P1 |
| AS-02 | 应用服务不包含业务规则判断逻辑 | P1 |
| AS-03 | 应用服务方法的输入输出使用DTO | P1 |
| AS-04 | 事务边界在应用服务层声明 | P1 |

### 7.2 DTO设计
| 编号 | 检查项 | 级别 |
|------|--------|------|
| AS-05 | 请求DTO命名遵循{Action}{Resource}Request | P2 |
| AS-06 | 响应DTO命名遵循{Action}{Resource}Response | P2 |
| AS-07 | DTO字段与接口设计文档的请求/响应参数一致 | P1 |
| AS-08 | DTO不包含领域对象的内部实现细节 | P1 |

## 8. 可扩展性

### 8.1 设计弹性
| 编号 | 检查项 | 级别 |
|------|--------|------|
| EX-01 | 枚举值的扩展不需要修改核心业务逻辑 | P2 |
| EX-02 | 新增外部系统集成不需要修改领域层代码 | P2 |
| EX-03 | 新增业务场景（如新的使用场景枚举值）的扩展路径清晰 | P2 |

### 8.2 设计约束
| 编号 | 检查项 | 级别 |
|------|--------|------|
| EX-04 | 不存在过度设计（无明确场景的缓存、消息队列等） | P2 |
| EX-05 | 不存在设计不足（已知场景缺少必要的设计方案） | P1 |

## 9. 文档规范

### 9.1 结构完整性
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DOC-01 | 包含系统概览章节（服务名称、功能描述、服务依赖） | P2 |
| DOC-02 | 包含领域模型设计章节（聚合设计、领域服务） | P1 |
| DOC-03 | 包含业务规则章节 | P1 |
| DOC-04 | 包含架构设计章节（数据隔离、缓存、消息队列等，按需） | P2 |

### 9.2 描述质量
| 编号 | 检查项 | 级别 |
|------|--------|------|
| DOC-05 | 聚合根、实体、值对象的属性表完整（属性名、类型、说明） | P2 |
| DOC-06 | 行为方法的说明清晰表达业务语义 | P2 |
| DOC-07 | 设计说明（如有）解释了关键设计决策的原因 | P2 |
| DOC-08 | 表格格式统一，内容对齐 | P2 |
