# 架构设计评审流程

## 1. 评审准备

收到架构设计评审请求后，按以下步骤执行：

1. 确定评审范围（全量评审或增量评审）
2. 读取架构设计文档（docs/designs/full-architecture-design.md）
3. 读取需求分析报告（docs/requirements/reports/目录下最新报告）
4. 读取业务需求文档（docs/designs/full-prd.md）：本服务业务设计的全量文档
5. 理解业务上下文和设计目标

## 2. 评审执行

按优先级从高到低依次检查以下维度，详细检查项见 [checklist.md](references/checklist.md)：

**P0 - 阻断级（必须修复）**
- 需求覆盖度：设计是否完整覆盖需求分析报告中的所有功能需求和业务规则
- 聚合边界正确性：聚合根识别是否正确，聚合边界是否合理
- 数据一致性：事务边界是否正确，并发控制方案是否到位
- 数据隔离：多租户/多服务商数据隔离方案是否完整

**P1 - 严重级（强烈建议修复）**
- 领域建模合理性：实体、值对象、领域服务的划分是否合理
- 业务规则完整性：业务规则是否完整定义，校验逻辑是否明确
- 防腐层设计：外部系统集成的隔离是否充分
- 数据库设计：表结构、索引、约束是否合理
- 领域服务职责：服务方法是否职责单一，命名是否表达业务语义

**P2 - 建议级（建议修复）**
- 命名规范：聚合根、实体、值对象、服务的命名是否符合规范
- 可扩展性：设计是否考虑未来扩展需求
- 文档完整性：设计文档的描述是否清晰完整
- 设计一致性：设计风格和模式是否在全文保持一致

## 3. 评审输出格式

对每个发现的问题，按以下格式输出：

```
### [P{级别}] {问题标题}

**位置**: {文档章节或设计元素}
**维度**: {评审维度}
**问题描述**: {具体问题说明}
**修复建议**: {如何修复}
**示例**（如适用）:
{修复后的设计内容}
```

## 4. 评审总结

评审完成后输出总结报告：

```
## 架构设计评审总结

**评审范围**: {全量/增量} | {文档章节数}
**发现问题**: P0: {n}个 | P1: {n}个 | P2: {n}个

### 问题分布
| 维度 | P0 | P1 | P2 |
|------|----|----|-----|
| ... | ... | ... | ... |

### 总体评价
{对架构设计质量的整体评价和改进方向}
```

## 5. 各维度评审要点速查

### 领域建模
- 聚合根是否拥有独立生命周期和唯一标识
- 实体是否有唯一标识且依附于聚合根
- 值对象是否无唯一标识且不可变
- 聚合边界是否遵循高内聚低耦合原则
- 领域服务是否仅处理跨聚合的业务操作

### 业务规则
- 每条业务规则是否有明确的触发条件和校验逻辑
- 业务规则是否归属到正确的领域对象（聚合根行为方法或领域服务）
- 状态流转规则是否完整定义所有合法路径
- 唯一性约束是否明确指定范围和条件

### 数据隔离
- 所有聚合根和独立实体是否包含隔离字段
- 所有查询操作是否在设计中体现隔离条件
- 索引设计是否将隔离字段作为前缀

### 防腐层
- 外部系统调用是否通过防腐层接口隔离
- 防腐层接口是否定义在Domain层
- 是否支持多外部系统的路由和适配
- 外部系统异常是否有明确的处理策略

### 数据库设计
- 表结构是否与领域模型一致
- 主键策略是否使用业务ID（非自增）
- 是否包含乐观锁（version）和逻辑删除（deleted）字段
- 索引设计是否覆盖主要查询场景
- 约束（CHECK、UNIQUE）是否与业务规则一致
