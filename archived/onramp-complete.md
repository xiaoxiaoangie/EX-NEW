## 1. 产品概述

### 1.1 定义

**OnRamp = 法币 → 数币承兑**。商户将法币（USD 等）兑换为 USDT/USDC，入账到数币钱包。

### 1.2 核心价值

| 用户类型           | 痛点                                   | OnRamp 解决方案          |
| ------------------ | -------------------------------------- | ------------------------ |
| WEB2 外贸/服贸客户 | 买家需用数币付款，汇率不稳定或外汇管制 | 提供法币→稳定币承兑能力 |
| WEB3 行业客户      | 需要将法币转为数币进入 WEB3 生态       | 一站式法币入金+承兑      |

### 1.3 产品边界

| 范围        | 说明                               |
| ----------- | ---------------------------------- |
| ✅ 本期包含 | OnRamp（法币→数币）全流程         |
| ❌ 本期不含 | OffRamp（数币→法币）— 见独立 PRD |
| ❌ 本期不含 | 纯 FX 兑换（法币↔法币）           |

## 3. 商户配置与开通

> 完整的商户注册、产品开通、租户签约、入网推送流程详见 `productionopening.md`。本节仅列出与 OnRamp 交易直接相关的配置摘要。

### 3.1 一期架构决策：单 SP 承兑策略

> **决策日期**：2026-02-26

#### 3.1.1 问题背景

EX 平台存在多个 SP（如 IPL、BB），理论上商户可能同时持有多个 SP 的法币账户。如果承兑业务允许多 SP 混合，会产生以下问题：

| 问题 | 说明 |
| ---- | ---- |
| **汇率不一致** | 同一币种（如 SGD→USD），IPL 和 BB 的汇率不同，EX 需要做统一汇率引擎来让租户给商户配置统一汇率 |
| **余额合并复杂** | 合并展示一个法币余额，但底层是两个 SP 的账户，合规规则不同（A 可能调单、B 没调单），异常处理极难 |
| **白牌体验差** | 白牌场景下包装统一法币余额非常麻烦，且暴露两个 SP 余额的商户体验很差 |
| **一期复杂度过高** | 统一汇率服务、余额聚合、合规路由等基础设施一期不具备 |

#### 3.1.2 决策：承兑业务一期由 BB 单一承接

**OnRamp/OffRamp 产品在一期绑定单一 SP（BB）提供全部承兑服务。**

- **承兑引擎**：BB
- **承兑法币账户**：BB 法币账户
- **汇率来源**：BB 汇率（无需 EX 做汇率聚合）
- **合规/风控**：BB 的合规规则，无歧义
- **数币钱包**：BB 数币钱包

#### 3.1.3 多 SP 共存规则

商户可以同时使用 IPL 和 BB，但**不同业务线绑定不同 SP**，不会在同一业务内混合 SP 余额：

| 业务线 | SP | 法币账户 | 说明 |
| ------ | --- | -------- | ---- |
| 承兑（OnRamp/OffRamp） | BB | BB 法币账户 | 承兑业务专用 |
| 法币收付款（Payins/Payouts） | IPL（如有签约） | IPL 法币账户 | 收付款业务专用 |

- 商户端会看到两个法币余额，但**各自归属不同业务**，不存在同一业务下的余额合并问题
- 商户端可标注余额的业务上下文（如"收付款余额" vs "承兑余额"）
- **场景 A2（IPL 法币→BB 承兑）** 仍然支持：商户 IPL 法币账户通过中间户划转到 BB 后承兑，汇率只用 BB 的

#### 3.1.4 二期演进

```
一期：OnRamp 产品 → 绑定 BB → 用 BB 的账户/汇率/合规
二期：OnRamp 产品 → EX 汇率引擎（聚合比价）→ 智能路由到最优 SP
```

二期需要的基础设施：
- EX 统一汇率引擎（聚合多 SP 报价，支持租户配置统一汇率）
- 余额聚合展示层（底层多 SP 账户，商户端合并展示）
- 合规路由（根据 SP 合规规则智能选择）

### 3.2 商户配置组合

本期仅两种配置，承兑引擎固定在 BB，不存在仅 IPL 的情况：

| 配置模式           | 包含产品                            | 收款账户选项                                  | 适用场景             |
| ------------------ | ----------------------------------- | --------------------------------------------- | -------------------- |
| **仅 BB**    | BB 承兑 + BB 数币钱包 + BB 法币账户 | BB 代收付账户（可能含 XPAY VA，但对商户透明） | 通用客户             |
| **BB + IPL** | 上述全部 + IPL 法币账户             | BB 代收付账户 + IPL 同名 VA 账户（多银行）    | 有贸易背景的合规客户 |

## 4. 三单模型

### 4.1 模型概览

OnRamp 交易采用三层单据模型，每层对应不同的系统层级和可见性：

```
┌─────────────────────────────────────────────────────────────┐
│  商户单（Merchant Order）                                     │
│  · 归属：EX HUB（编排层）                                     │
│  · 可见性：MP ✅  TP ✅  PP ✅  SA ✅                          │
│  · 职责：面向商户的订单，聚合所有交易单状态                      │
│  · 1个商户单 → 1~N 笔交易单                                   │
├─────────────────────────────────────────────────────────────┤
│  交易单（Transaction）                                        │
│  · 归属：交易引擎（Transaction Engine）                        │
│  · 可见性：MP ❌  TP ✅  PP ✅  SA ✅                          │
│  · 职责：按 SP 拆分的具体交易（BB 交易单 / IPL 交易单）         │
│  · 每笔交易单归属一个 SP                                      │
│  · 包含：计费、汇率、风控、记账                                │
├─────────────────────────────────────────────────────────────┤
│  渠道单（Channel Order）                                      │
│  · 归属：SP 内部渠道服务                                      │
│  · 可见性：MP ❌  TP ❌  PP ✅（仅该 SP 的 PP）                │
│  · 职责：SP 调用底层渠道的执行记录                              │
│  · BB 的渠道单对 EX 透明（因为 EX 帮 BB 对接渠道）              │
│  · IPL 的渠道单仅 IPL PP 可见                                 │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 各场景的单据结构

| 场景 | 配置   | 支付方式          | 商户单 | 交易单                            | 渠道单     | 触发方式        |
| ---- | ------ | ----------------- | ------ | --------------------------------- | ---------- | --------------- |
| A1   | 仅 BB  | 法币钱包余额      | 1      | 1 (BB 承兑)                       | 0          | 商户即时发起    |
| A2   | BB+IPL | 法币钱包余额(IPL) | 1      | 2 (IPL 提现 + BB 承兑)            | 0          | 商户即时发起    |
| B1   | 仅 BB  | 银行转账(BB)      | 1      | 2 (BB 收款 + BB 承兑)             | 1 (BB PP)  | 预约单+渠道触发 |
| B2   | BB+IPL | 银行转账(IPL VA)  | 1      | 3 (IPL 收款 + IPL 提现 + BB 承兑) | 1 (IPL PP) | 预约单+渠道触发 |

### 5.2 场景A1：即时单 — BB法币账户直接承兑（仅BB）

**场景：** 商户BB法币账户已有USD余额，直接承兑为USDT到BB数币钱包。

**特点：** 纯承兑，无外部收付款，无渠道单。

**单据结构（1笔交易单）：**

```
商户单 M001 (On-Ramp: USD→USDT)  ← 仅BB
    └── 交易单 T001 (BB): 承兑 — BB USD→USDT（内部账户划转）

单据数：1 商户单 + 1 交易单 + 0 渠道单
资金流：商户 BB USD 账户 → 商户 BB USDT 钱包
```

**业务流程：**

```
商户下单 → 业务校验 → 校验通过 → 创建商户单+交易单 → 冻结余额 →
计费+汇率 → 风控 → 扣款+入账 → 交易完成 → 通知商户
```

**关键节点：**

1. 商户选择数币（USDT/USDC）、链（TRC-20/ERC-20/BEP-20）、法币金额
2. 选择"法币钱包余额"支付
3. 系统校验余额充足、产品启用、限额合规（**校验失败直接拒绝，不创建任何单据**）
4. 校验通过后创建商户单+交易单，冻结法币余额
5. 实时获取汇率+计费
6. 风控通过 → 确认扣款 → 数币入账
7. 商户单状态更新为 SUCCESS

**说明：**

- **纯BB内部承兑**：BB法币账户 → BB数币钱包，内部账户划转
- **无需渠道单**：不调用外部渠道，无渠道单
- **有BB承兑风控**：冻结余额后、执行划转前，需通过BB承兑风控检查（商户风险等级、交易频率、金额异常等）
- **先校验再建单**：业务校验通过后才创建商户单和交易单（校验失败直接拒绝请求，不创建任何单据）
- **先计费后冻结**：计费+汇率确定后才冻结余额，确保冻结金额准确
- **手续费扣取方式（产品级配置）**：
  - **内扣**：冻结=输入金额，手续费从承兑金额中扣，到账USDT=(输入金额-手续费)/汇率
  - **外扣**：冻结=输入金额+手续费，到账USDT=输入金额/汇率，手续费另行扣除
- **先冻结后风控**：冻结余额 → 风控检查 → 通过后确认扣款并入账；风控拒绝则解冻退回
- **1个商户单，1笔交易单**

**异常处理：**

> 场景A1 所有异常均为**解冻退回**，不产生退款单，不收费。因为资金未离开商户BB法币账户。

| 异常环节       | 触发条件                                     | 处理方式                                               | 商户感知                       |
| -------------- | -------------------------------------------- | ------------------------------------------------------ | ------------------------------ |
| 业务校验失败   | 余额不足/产品未启用/超限额/货币对不支持      | 直接拒绝，不创建商户单                                 | 下单失败，返回错误原因         |
| 冻结余额失败   | 余额校验通过但冻结时被其他交易占用（并发）   | 商户单=FAILED                                          | 提示"余额不足，请稍后重试"     |

### 5.4 场景B1：预约单 — BB代收付/XPAY VA收款→承兑（仅BB）

**场景：** 外部汇款人通过BB的代收付账户或XPAY VA汇入USD，收款到BB法币账户后，自动承兑为USDT到BB数币钱包。

**特点：** 收款（渠道触发）+ 承兑，2笔交易单串行。

**单据结构（2笔交易单）：**

```

    ├── 交易单 T001 (BB): 收款 — 外部USD入到BB法币账户（渠道通知触发）
    │       └── 渠道单 C001 (BB PP): BB 代收付/XPAY 渠道执行记录
    └── 交易单 T002 (BB): 承兑 — BB USD→USDT

单据数：1 商户单 + 2 交易单 + 1 渠道单（仅 BB PP 可见）

**业务流程：**

```

商户线下汇款 → 渠道入账通知 → 匹配预约单 → 业务校验 →

**关键节点：**

1. 商户选择数币、链、法币金额
2. 选择"银行转账"支付

6. 校验通过 → 创建商户单 → 自动创建交易单链 → 承兑 → 数币入账

## 6. 中间户机制（跨SP场景）

### 6.1 适用场景

当商户配置为 BB + IPL 时，资金需要在两个 SP 之间流转，通过中间户完成：

```
BB 和 IPL 各自维护对方的中间户：
· BB 侧维护"IPL 中间户(在 BB)"
· IPL 侧维护"BB 中间户(在 IPL)"

资金通过中间户完成跨 SP 划转：
· IPL → BB：商户 IPL 账户 → BB 中间户(在 IPL) → IPL 中间户(在 BB) → BB 承兑
· BB → IPL：BB 承兑 → IPL 中间户(在 BB) → BB 中间户(在 IPL) → 商户 IPL 账户

中间户是记账手段，BB 和 IPL 之间有清算协议定期轧差
```

### 6.2 IPL 侧同名收款

```
IPL 作为持牌机构，将每笔跨 SP 入账视为"同名收款"（Collection）：
· 同名收款 = 付款人和收款人是同一商户
· 产生完整的收款交易记录
· 走 IPL 收款合规流程（AML 检查）
· 可在 IPL 清算/对账体系中追踪
```

## 7. 状态机

> **重要说明**：OnRamp 涉及两种单据实体——**预约单**和**商户单**，它们是不同的对象，有各自独立的状态机。
>
> - **预约单（Reservation Order）**：商户选择"银行转账"时创建，用于等待和匹配外部汇款。预约单是"前置单"。
> - **商户单（Merchant Order）**：业务校验通过后创建，用于执行承兑交易。即时单场景下单即创建；预约单场景在匹配成功+校验通过后才创建。
> - **关系**：预约单 MATCHED → 业务校验通过 → 创建商户单。两者通过 `reservation_id` 关联。
> - **商户视角**：商户在 MP 端看到的是**统一的订单视图**，系统将预约单和商户单的状态合并展示（详见 7.5）。

### 7.1 预约单状态机（仅银行转账场景 B1/B2）

```
┌──────────┐
│ CREATED  │  商户下预约单
└────┬─────┘
     ▼
┌──────────────┐
│AWAITING_FUNDS│  等待汇款到账（展示收款账户信息）
└──────┬───────┘
       │
       ├──────────────────────────────────────┐
       │                                      │
  渠道入账通知                           72h 超时未收款
       │                                      │
       ▼                                      ▼
┌──────────┐                           ┌──────────┐
│ MATCHED  │  收款匹配成功             │ EXPIRED  │
└────┬─────┘                           └──────────┘
     │
     ├── 业务校验通过 → 创建商户单，预约单状态保持 MATCHED
     │
     └── 业务校验失败 → 不创建商户单，资金留在法币账户
                        预约单状态 = MATCH_FAILED

另外：商户可在 AWAITING_FUNDS 状态主动取消 → CANCELLED
```

**预约单状态说明：**

| 状态           | 说明           | 触发条件                                     |
| -------------- | -------------- | -------------------------------------------- |
| CREATED        | 预约单已创建   | 商户选择银行转账下单                         |
| AWAITING_FUNDS | 等待汇款到账   | 预约单创建后，展示收款账户信息               |
| MATCHED        | 收款已匹配     | 渠道入账通知匹配到预约单（自动或人工）       |
| MATCH_FAILED   | 匹配后校验失败 | 匹配成功但业务校验不通过，资金留在法币账户   |
| EXPIRED        | 过期           | 超时未收到汇款（默认 72h）                   |
| CANCELLED      | 已取消         | 商户主动取消（仅 AWAITING_FUNDS 状态可取消） |

### 7.2 商户单状态机

> 商户单在**即时单**场景下单即创建；在**预约单**场景匹配成功+校验通过后才创建。
> 商户单**不包含** AWAITING_FUNDS / MATCHED / EXPIRED 等预约阶段状态。

```
┌──────────┐
│ CREATED  │  业务校验通过后创建
└────┬─────┘
     ▼
┌──────────┐
│PROCESSING│  开始执行交易单链
└────┬─────┘
     │
     ├─────────────────┐
     ▼                 ▼
┌──────────┐     ┌──────────┐
│ SUCCESS  │     │  FAILED  │
└──────────┘     └────┬─────┘
                      │
                      ▼
                ┌──────────┐
                │ REFUNDING│  （仅跨SP场景：资金在中间户需逆向回滚）
                └────┬─────┘
                     ▼
                ┌──────────┐
                │ REFUNDED │
                └──────────┘
```

**商户单状态说明：**

| 状态       | 说明       | 触发条件                                       |
| ---------- | ---------- | ---------------------------------------------- |
| CREATED    | 订单已创建 | 即时单：下单校验通过；预约单：匹配+校验通过后  |
| PROCESSING | 处理中     | 开始执行交易单链（计费、汇率、风控、记账）     |
| SUCCESS    | 成功       | 所有交易单完成，数币入账                       |
| FAILED     | 失败       | 任一交易单失败（风控拒绝/执行异常）            |
| REFUNDING  | 退款中     | 跨SP场景T001成功但T002失败，逆向回滚中间户资金 |
| REFUNDED   | 已退款     | 退款完成，资金退回商户法币账户                 |

### 7.3 交易单状态机

```
CREATED → PROCESSING → SUCCESS
                    → FAILED → REFUNDING → REFUNDED
```

### 7.4 三者关系图

```
即时单场景（A1/A2）：
  商户下单 → 校验通过 → 创建【商户单】→ PROCESSING → SUCCESS/FAILED

预约单场景（B1/B2）：
  商户下单 → 创建【预约单】→ AWAITING_FUNDS → 收款到账 → MATCHED
                                                          ↓
                                              业务校验通过 → 创建【商户单】→ PROCESSING → SUCCESS/FAILED
                                              业务校验失败 → 预约单 MATCH_FAILED，不创建商户单
```

### 7.5 商户端统一视图（MP 展示逻辑）

> 商户在 MP 端看到的是**一个统一的订单**，系统根据底层单据状态映射为商户可见的展示状态：

| 底层状态                      | 商户端展示状态   | 商户可执行操作     |
| ----------------------------- | ---------------- | ------------------ |
| 预约单 AWAITING_FUNDS         | Awaiting Payment | 查看收款账户、取消 |

## 9. 风控规则

### 9.1 风控分层

| 交易类型     | 风控方   | 检查内容                   |
| ------------ | -------- | -------------------------- |
| BB 收款      | BB 风控  | 汇款人 AML、交易金额、频率 |
| BB 承兑      | BB 风控  | 承兑金额、商户风险等级     |
| IPL VA 收款  | IPL 风控 | 汇款人 AML、贸易背景合规   |
| IPL 同名提现 | IPL 风控 | 出款合规                   |

### 9.2 关键规则

- **同名转账**：银行转账必须为同名转账，非同名转账进入人工确认队列（运营联系商户后决定退款或继续）
- **Purpose Code**：银行转账备注必须包含 Purpose Code，缺失时进入人工确认队列（运营联系商户补录或退款）
- **限额**：单笔 100 ~ 50,000 USD（可配置）
- **超时**：预约单 72h 未收到汇款自动过期

### 9.3 异常退款设计原则与汇总

> **设计原则**（与 `refund.md` 一致）：
>
> - **商户端一致性**：商户只看到1笔退款/失败，内部多笔交易单是内部记账，商户不可见
> - **清算先行**：涉及手续费退还的退款，先由清算处理完再执行退款
> - **退款不再收承兑手续费**：承兑后失败退回法币账户时，不再收承兑手续费
> - **退票 = 新渠道单**：渠道退票时生成一笔新的退款渠道单，关联原渠道单

**OnRamp 退款汇总表：**

| #  | 场景 | 失败环节             | 资金位置       | 退回目标          | 退款单     | 收费       | 商户感知         |
| -- | ---- | -------------------- | -------------- | ----------------- | ---------- | ---------- | ---------------- |
| 1  | A1   | 业务校验失败         | 未扣款         | 直接拒绝，不建单  | ❌         | 不收       | 下单失败         |
| 2  | A1   | 冻结/风控/承兑失败   | 已冻结         | 解冻退回BB法币    | ❌         | 不收       | 订单失败         |
| 3  | A2   | 业务校验失败         | 未扣款         | 直接拒绝，不建单  | ❌         | 不收       | 下单失败         |
| 4  | A2   | 冻结/IPL风控失败     | 未扣款或已冻结 | 解冻退回IPL法币   | ❌         | 不收       | 订单失败         |
| 5  | A2   | BB承兑风控/执行失败  | 中间户         | 逆向回滚→IPL法币 | ✅(RT内部) | 不收       | 订单退款         |
| 6  | B1   | 收款风控拦截         | 冻结在BB       | 退回原汇款人      | ✅         | 清算确认   | 退款通知         |
| 7  | B1   | BB承兑风控/执行失败  | 商户BB法币     | 留在BB法币账户    | ❌         | 不收       | 订单失败，可重试 |
| 8  | B2   | IPL收款风控拦截      | 冻结在IPL      | 退回原汇款人      | ✅         | 清算确认   | 退款通知         |
| 9  | B2   | IPL出款风控/提现失败 | 商户IPL法币    | 留在IPL法币账户   | ❌         | 不收       | 订单失败，可重试 |
| 10 | B2   | BB承兑风控/执行失败  | 中间户         | 逆向回滚→IPL法币 | ✅(RT内部) | 不收承兑费 | 订单退款         |

**核心原则总结：**

1. **先校验再建单** → 业务校验通过后才创建商户单，校验失败直接拒绝请求，不创建任何单据（API和白牌统一）
2. **资金未离开商户账户** → 解冻/回滚即可，不产生退款单
3. **资金在中间户**（跨SP场景） → 逆向回滚，创建内部退款交易单，不收承兑手续费
4. **收款风控拦截** → 退回原汇款人，参考 `refund.md` 2.2 流程，清算确认手续费
5. **预约单收款成功但承兑失败** → 资金留在商户法币账户，商户可用余额重新发起OnRamp
6. **商户端一致性** → 商户只看到1笔退款或失败，内部多笔交易单为内部记账

> 各场景的详细异常处理说明已穿插在第5章各场景中（5.2~5.5），包含异常表格、时序图和流程图。

## 10. 预约单匹配规则

### 10.1 匹配总流程

渠道入账通知到达后，系统按以下流程进行预约单匹配：

```mermaid
flowchart TD
    A[渠道入账通知到达] --> B{幂等校验:channelrefno 是否已处理}
    B -->|是,重复通知| B1[忽略,返回已处理结果]
    B -->|否,新通知| C[解析通知:收款账户,金额,币种,汇款人]

    C --> D{Step 1:匹配收款账户}
    D -->|未找到对应账户| D1[异常:未知收款账户:告警 人工处理]
    D -->|找到账户| E{Step 2:匹配商户 MID}

    E -->|账户未绑定商户| E1[异常:孤立账户:告警]
    E -->|找到商户| F{Step 3:同名校验}

    F -->|汇款人名 不等于 商户名| F1[非同名转账:告警 人工确认]
    F -->|同名通过| G{Step 4:Purpose Code 校验}

    G -->|备注中无 Purpose Code| G1[缺少 Purpose Code:告警 人工]
    G -->|校验通过| H{Step 5:查找待匹配预约单}

    H -->|无待匹配预约单| H1[无预约单收款:挂账处理]
    H -->|找到候选预约单| I{Step 6:币种匹配}

    I -->|币种不一致| I1[币种不匹配:人工匹配队列]
    I -->|币种一致| J{Step 7:金额匹配}

    J -->|精确匹配 在0.01内| J1[精确匹配:自动触发承兑]
    J -->|偏差 在 5%之内| J2[容差匹配 :自动触发承兑:按实际金额]
    J -->|偏差 大于 5% 且 小于等于 20%| J3[大偏差 :人工确认后触发]
    J -->|偏差 大于 20%| J4[ 严重偏差:人工匹配队列]

    J1 --> K0{业务校验}
    J2 --> K0
    K0 -->|校验失败| K1[不创建商户单,资金留在法币账户]
    K0 -->|校验通过| K[创建商户单和交易单链: 执行承兑]
    J3 --> L[进入人工确认队列]
    L -->|人工确认匹配| K0
    L -->|人工拒绝| M[退款流程]
```

### 10.2 匹配步骤详细说明

| Step | 匹配维度               | 匹配规则                                                     | 失败处理                  |
| ---- | ---------------------- | ------------------------------------------------------------ | ------------------------- |
| 1    | **收款账户**     | 渠道通知中的收款账号 → 系统中的 VA/BB 代收付账户            | 未知账户 → 告警 + 人工   |
| 2    | **商户 MID**     | 收款账户 → 绑定的商户 MID                                   | 孤立账户 → 告警          |
| 3    | **同名校验**     | 汇款人名称 vs 商户注册名称（模糊匹配，忽略大小写/空格/标点） | 非同名 → 告警 + 人工确认 |
| 4    | **Purpose Code** | 汇款备注中提取 Purpose Code                                  | 缺失 → 告警 + 人工确认   |
| 5    | **预约单查找**   | 该商户 + 该收款账户 + 状态=AWAITING_FUNDS + 未过期           | 无预约单 → 挂账          |
| 6    | **币种**         | 入账币种 = 预约单币种                                        | 不匹配 → 人工队列        |
| 7    | **金额**         | 入账金额 vs 预约金额，按偏差比例分级处理                     | 见金额匹配规则            |

### 10.3 金额匹配分级规则

| 偏差范围          | 级别        | 处理方式                               | 说明                     |
| ----------------- | ----------- | -------------------------------------- | ------------------------ |
| ≤ ±0.01（精确） | 🟢 自动     | 直接匹配，自动触发承兑                 | 最理想情况               |
| ±0.01 ~ ±5%     | 🟢 自动     | 自动匹配，**按实际到账金额承兑** | 银行扣费导致的正常偏差   |
| ±5% ~ ±20%      | 🟡 人工确认 | 进入人工确认队列，运营审核后决定       | 可能是部分汇款或汇率损耗 |
| > ±20%           | 🔴 人工处理 | 进入人工匹配队列，需运营手动处理       | 可能是错误汇款或拆分汇款 |

```
金额偏差计算公式：
偏差率 = |实际到账金额 - 预约金额| / 预约金额 × 100%

示例：
· 预约 200 USD，到账 199.50 USD → 偏差 0.25% → 🟢 自动匹配
· 预约 200 USD，到账 190.00 USD → 偏差 5.0% → 🟢 自动匹配（边界）
· 预约 200 USD，到账 175.00 USD → 偏差 12.5% → 🟡 人工确认
· 预约 200 USD，到账 100.00 USD → 偏差 50% → 🔴 人工处理
```

## 11. 校验规则详解

> 本章明确区分两类校验的触发时机、校验内容和失败处理，解答"预约单下单时有没有校验"的问题。

### 11.1 核心结论

```
即时单（场景 A1/A2）：
  商户下单 → 【前置业务校验】→ 校验通过 → 创建商户单 → 执行承兑

预约单（场景 B1/B2）：
  商户下单 → 【预约单创建校验】→ 创建预约单 → AWAITING_FUNDS
                                                    ↓ 收款到账
                                              【后置业务校验（匹配校验）】
                                                    ↓ 校验通过
                                              创建商户单 → 执行承兑
```

**关键区别：**

- 即时单的业务校验是**前置**的（下单时立即执行，校验失败直接拒绝，不创建任何单据）
- 预约单有**两层校验**：下单时的创建校验（轻量）+ 收款到账后的业务校验（完整）

---

### 11.2 预约单创建校验（下单时，轻量校验）

> 商户选择"银行转账"提交预约单时触发。**校验失败 → 拒绝创建预约单，不展示收款账户信息。**

| # | 校验项                      | 校验内容                                     | 失败处理                   |
| - | --------------------------- | -------------------------------------------- | -------------------------- |
| 1 | **产品开通状态**      | OnRamp 产品是否已为该商户开通                | 拒绝，提示"产品未开通"     |
| 2 | **商户 KYC/合规状态** | 商户账户是否处于正常状态（未冻结/未注销）    | 拒绝，提示"账户状态异常"   |
| 3 | **收款账户有效性**    | 商户选择的 VA / 代收付账户是否仍有效、未过期 | 拒绝，提示"收款账户不可用" |
| 4 | **单笔金额范围**      | 预约金额是否在产品配置的 [最小, 最大] 范围内 | 拒绝，提示具体限额         |
| 5 | **货币对支持**        | 预约的法币币种是否在产品支持的货币对内       | 拒绝，提示"不支持该货币"   |
| 6 | **汇款人绑定**        | 是否已绑定至少一个汇款人（同名校验的前提）   | 拒绝，提示"请先添加汇款人" |

**注意：预约单创建校验不校验余额/额度**，因为资金尚未到账，无法也无需校验。

---

### 11.3 商户单业务校验（收款到账后，完整校验）

> 预约单匹配成功（MATCHED）后，在创建商户单之前触发。**校验失败 → 不创建商户单，预约单状态 = MATCH_FAILED，资金留在商户法币账户。**

| # | 校验项                       | 校验内容                                                                         | 失败处理                       |
| - | ---------------------------- | -------------------------------------------------------------------------------- | ------------------------------ |
| 1 | **产品开通状态**       | OnRamp 产品是否仍处于开通状态（可能在等待期间被关闭）                            | MATCH_FAILED，资金留在法币账户 |
| 2 | **商户合规状态**       | 商户账户是否仍正常（可能在等待期间被冻结）                                       | MATCH_FAILED，资金留在法币账户 |
| 3 | **目标数币钱包状态**   | 承兑目标钱包（BB USDT 钱包）是否正常、可入账                                     | MATCH_FAILED，资金留在法币账户 |
| 4 | **单笔金额范围**       | 实际到账金额是否在产品配置的 [最小, 最大] 范围内                                 | MATCH_FAILED，资金留在法币账户 |
| 5 | **单日/单月累计限额**  | 加上本笔后是否超出商户的单日/单月累计限额                                        | MATCH_FAILED，资金留在法币账户 |
| 6 | **货币对支持**         | 到账币种是否仍在支持的货币对内                                                   | MATCH_FAILED，资金留在法币账户 |
| 7 | **跨SP场景：IPL 余额** | BB+IPL 场景下，商户 IPL 法币账户余额是否足以发起提现（即到账金额是否已正确入账） | MATCH_FAILED，资金留在法币账户 |

**注意：商户单业务校验不再校验余额是否足够扣款**，因为预约单场景的资金来源是外部汇入（不是从账户余额扣），到账即为资金来源。

---

### 11.4 即时单业务校验（下单时，完整校验）

> 商户选择"法币钱包余额"提交即时单时触发。**校验失败 → 直接拒绝，不创建任何单据。**

| # | 校验项                      | 校验内容                                                   | 失败处理                 |
| - | --------------------------- | ---------------------------------------------------------- | ------------------------ |
| 1 | **产品开通状态**      | OnRamp 产品是否已为该商户开通                              | 拒绝，提示"产品未开通"   |
| 2 | **商户 KYC/合规状态** | 商户账户是否处于正常状态                                   | 拒绝，提示"账户状态异常" |
| 3 | **目标数币钱包状态**  | 承兑目标钱包是否正常、可入账                               | 拒绝，提示"钱包不可用"   |
| 4 | **单笔金额范围**      | 金额是否在产品配置的 [最小, 最大] 范围内                   | 拒绝，提示具体限额       |
| 5 | **单日/单月累计限额** | 加上本笔后是否超出累计限额                                 | 拒绝，提示"已超出限额"   |
| 6 | **货币对支持**        | 法币币种是否在支持的货币对内                               | 拒绝，提示"不支持该货币" |
| 7 | **法币账户余额**      | 商户法币账户余额是否 ≥ 本次承兑金额（含手续费，外扣模式） | 拒绝，提示"余额不足"     |

---

### 11.5 三类校验对比汇总

| 校验维度          |  预约单创建校验  | 即时单业务校验 |    预约单业务校验（收款后）    |
| ----------------- | :--------------: | :------------: | :----------------------------: |
| 触发时机          |    商户下单时    |   商户下单时   |         收款到账匹配后         |
| 产品开通状态      |        ✅        |       ✅       |               ✅               |
| 商户合规状态      |        ✅        |       ✅       |               ✅               |
| 收款账户有效性    |        ✅        |       —       |               —               |
| 汇款人已绑定      |        ✅        |       —       |               —               |
| 目标钱包状态      |        —        |       ✅       |               ✅               |
| 单笔金额范围      | ✅（按预约金额） |       ✅       |      ✅（按实际到账金额）      |
| 单日/单月累计限额 |    ❌ 不校验    |       ✅       |               ✅               |
| 法币账户余额      |    ❌ 不校验    |       ✅       |    ❌ 不校验（资金已到账）    |
| 货币对支持        |        ✅        |       ✅       |               ✅               |
| 校验失败结果      |  拒绝创建预约单  |    拒绝下单    | MATCH_FAILED，资金留在法币账户 |
| 是否创建单据      |        ❌        |       ❌       |       ❌（商户单不创建）       |

**为什么预约单下单时不校验累计限额？**

```
预约单下单 → 收款到账 之间可能间隔数小时到数天。
在此期间，商户可能有其他交易消耗了额度，也可能额度已重置。
因此，累计限额校验必须在收款到账时（业务校验阶段）以当时的实时额度为准，
而非下单时的预估额度。
```

**MATCH_FAILED 后资金如何处理？**

```
资金已入账到商户法币账户（IPL 或 BB），不会自动退款。
商户可选择：
  1. 修复问题后，由运营人工重新触发业务校验（如产品已重新开通）
  2. 直接使用法币账户余额发起即时单 OnRamp（走场景 A1/A2）
  3. 申请退款（退回原汇款人）
```

## 13. 场景汇总对比

### 13.1 OnRamp 场景一览

| 场景                  | 模式      | 支付方式     | 交易单数 | SP       | 渠道单 | 触发方式        |
| --------------------- | --------- | ------------ | -------- | -------- | ------ | --------------- |
| A1 BB法币→数币       | A(BB自己) | 法币钱包余额 | 1笔      | 单SP(BB) | ❌     | 商户发起        |
| A2 IPL法币→BB数币    | B(IPL-BB) | 法币钱包余额 | 2笔      | 双SP     | ❌     | 商户发起        |
| B1 BB VA收款→数币    | A(BB自己) | 银行转账     | 2笔      | 单SP(BB) | ✅     | 渠道触发+预约单 |
| B2 IPL VA收款→BB数币 | B(IPL-BB) | 银行转账     | 3笔      | 双SP     | ✅     | 渠道触发+预约单 |

### 13.2 模式对比

| 对比项                  | 模式A（BB自己）        | 模式B（IPL-BB打通）             |
| ----------------------- | ---------------------- | ------------------------------- |
| **SP数量**        | 单SP（BB）             | 双SP（BB+IPL）                  |
| **法币通道**      | BB代收付账户/XPAY      | IPL法币通道                     |
| **中间户**        | 不需要                 | 需要（BB和IPL各维护对方中间户） |
| **IPL侧交易性质** | 不涉及IPL              | 同名提现（IPL→BB）             |
| **风控**          | BB统一风控             | BB风控 + IPL合规，各自独立      |
| **清算**          | BB内部清算             | BB清算 + IPL清算，各自独立      |
| **适用场景**      | BB有自己的法币渠道能力 | 需要借助IPL的法币渠道能力       |

### 12.3 即时单 vs 预约单对比

| 对比项             | 即时单（场景A）   | 预约单（场景B）            |
| ------------------ | ----------------- | -------------------------- |
| **支付方式** | 法币钱包余额扣款  | 银行转账                   |
| **汇率**     | 不锁汇            | 不锁汇                     |
| **完成时间** | 实时              | 取决于银行转账到账时间     |
| **有效期**   | 无                | 72小时                     |
| **匹配逻辑** | 无需匹配          | 渠道入账通知→匹配预约单   |
| **风控**     | 余额校验+承兑风控 | 收款风控+同名校验+承兑风控 |
